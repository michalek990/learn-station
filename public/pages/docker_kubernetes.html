Docker — Fundament
Docker — architektura, obrazy, kontenery
Docker to platforma do budowania i uruchamiania kontenerów — izolowanych procesów współdzielących jądro systemu operacyjnego hosta. Lżejsze od VM, szybkie do uruchomienia, przenośne. Oparte na przestrzeniach nazw Linuksa i cgroups.
🏗️ Kluczowe koncepty
Image
Niemutowalny snapshot systemu plików + metadane (CMD, ENV, EXPOSE). Warstwowy (union filesystem). Bazuje na innych obrazach (FROM).
Container
Uruchomiona instancja image. Izolowany proces z własnym namespace (PID, NET, MNT, UTS, IPC, USER). Efemeryczny — dane giną po usunięciu.
Registry
Repozytorium obrazów — Docker Hub, GitHub Container Registry, AWS ECR, GCR, Harbor (self-hosted). docker pull/push
Layer
Każda instrukcja Dockerfile tworzy warstwę. Warstwy są cachowane i współdzielone między obrazami. Niezmienne po stworzeniu.
Volume
Persystentny storage zarządzany przez Docker. Przeżywa usunięcie kontenera. Typy: named volume, bind mount, tmpfs.
Network
Wirtualna sieć między kontenerami. Tryby: bridge (domyślny), host, none, overlay (Swarm/K8s). DNS między kontenerami w tej samej sieci.
Docker Daemon
Serwer zarządzający kontenerami (dockerd). Klient CLI (docker) komunikuje się przez socket. Alternatywy: containerd, Podman (rootless).

Dockerfile
Dockerfile — budowanie obrazów, multi-stage, optymalizacja
Dockerfile to przepis na obraz — sekwencja instrukcji tworzących warstwy. Każda instrukcja = warstwa (cache). Multi-stage builds drastycznie zmniejszają rozmiar obrazów produkcyjnych — builder ma narzędzia, final image tylko runtime.
Dockerfile — Spring Boot multi-stage (produkcyjny)
# ══ Stage 1: Build ════════════════════════════════════════════
# Duży obraz z Maven/JDK — zostaje TYLKO w build stage
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /workspace/app

# Kopiuj POM najpierw (cache dependencies — zmiana kodu nie rebuilda deps!)
COPY pom.xml .
COPY .mvn/ .mvn/
COPY mvnw .
RUN ./mvnw dependency:go-offline -q

# Teraz kopiuj kod źródłowy
COPY src src

# Buduj aplikację (bez testów = szybciej w CI gdy testy są osobno)
RUN ./mvnw package -DskipTests -q

# Rozpakuj Spring Boot JAR na warstwy (Spring Boot Layered JARs)
RUN java -Djarmode=layertools \
    -jar target/*.jar extract \
    --destination target/extracted

# ══ Stage 2: Runtime ══════════════════════════════════════════
# Mały obraz tylko z JRE — to trafia na produkcję!
FROM eclipse-temurin:21-jre-alpine AS runtime

# Security: nie uruchamiaj jako root!
RUN addgroup -S spring && adduser -S spring -G spring
USER spring:spring

WORKDIR /app

# Spring Boot Layered JARs — dependencies (rzadko się zmieniają) = oddzielna warstwa
# Dzięki temu push/pull zmienia tylko warstwy z kodem aplikacji!
COPY --from=builder /workspace/app/target/extracted/dependencies/ ./
COPY --from=builder /workspace/app/target/extracted/spring-boot-loader/ ./
COPY --from=builder /workspace/app/target/extracted/snapshot-dependencies/ ./
COPY --from=builder /workspace/app/target/extracted/application/ ./

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=60s --retries=3 \
  CMD wget -q --spider http://localhost:8080/actuator/health || exit 1

EXPOSE 8080

# Użyj exec form (nie shell form!) — prawidłowe przekazywanie sygnałów
ENTRYPOINT ["java",
  "-XX:+UseContainerSupport",
  "-XX:MaxRAMPercentage=75.0",
  "-Djava.security.egd=file:/dev/./urandom",
  "org.springframework.boot.loader.launch.JarLauncher"
]
📋 Instrukcje Dockerfile
FROM
Obraz bazowy. FROM scratch = pusty obraz (Go binaries). Zawsze pinuj wersję: openjdk:21-jre-slim nie openjdk:latest.
RUN
Wykonaj komendę w nowej warstwie. Łącz komendy przez && żeby nie tworzyć zbędnych warstw. rm -rf /var/cache/apk/* w tej samej warstwie.
COPY vs ADD
Zawsze COPY (explicit). ADD ma ukryte funkcje (rozpakowanie .tar, URL) — używaj tylko gdy potrzebujesz tych funkcji.
ENV
Zmienne środowiskowe dostępne w runtime. Nadpisywane przez docker run -e lub K8s env. Nie używaj do sekretów!
ARG
Zmienne tylko w czasie build (docker build --build-arg VERSION=1.0). Nie są dostępne w runtime. Używaj do wersji, tagów.
ENTRYPOINT vs CMD
ENTRYPOINT = komenda bazowa (nie nadpisywana łatwo). CMD = domyślne argumenty (łatwo nadpisywane). Razem: ENTRYPOINT=executable, CMD=default args.
USER
Uruchom jako non-root! Domyślnie Docker uruchamia jako root — ogromne ryzyko bezpieczeństwa. Zawsze USER appuser.
Optymalizacja — cache warstw Kolejność instrukcji ma ogromne znaczenie dla cache. Częste zmiany = na końcu. Wolno zmieniające się = na początku.
Złe: COPY . . → RUN mvn dependency:go-offline
Dobre: COPY pom.xml . → RUN mvn dependency:go-offline → COPY src src
Zmiana kodu nie reinstaluje dependencies!

🏔️ Alpine vs Slim vs Distroless

Alpine (~5MB base) — musl libc, minimalne narzędzia. Małe obrazy, ale czasem problemy z Java (JVM zakłada glibc).

Slim (~80MB) — Debian/Ubuntu stripped. Najbezpieczniejszy wybór dla JVM.

Distroless — tylko runtime (bez powłoki, pakietów). Najmniejsza powierzchnia ataku. Google's recommendation.

📦 Multi-stage korzyści

Typowy Spring Boot JAR: ~50MB

single-stage (JDK): ~400MB+

single-stage (JRE): ~250MB

multi-stage layered: ~120MB

Mniejszy obraz = szybszy pull, mniejsza powierzchnia ataku, tańszy registry storage.

⚙️ JVM w kontenerze

Problem: Stara JVM czytała RAM hosta, nie kontenera → OOM Killer.

-XX:+UseContainerSupport — JVM widzi limity cgroup (domyślnie od Java 10)

-XX:MaxRAMPercentage=75.0 — użyj 75% RAM limitu kontenera

Zawsze ustaw limit pamięci w K8s/Docker!

🔒 Security best practices

• Non-root user (USER)

• Read-only filesystem: --read-only

• No privileged: nigdy --privileged

• Skanuj obraz: trivy image myapp:1.0

• Pinuj dokładne wersje bazowe (nie latest)

• Secrets przez env lub mounted secrets, nigdy w Dockerfile

Docker Compose
Docker Compose — środowisko lokalne i integracyjne
Docker Compose to narzędzie do definiowania i uruchamiania multi-container aplikacji w jednym pliku YAML. Idealne dla środowisk lokalnych i testów integracyjnych. Compose v2 (plugin) zastąpił v1 (python).
docker-compose.yml — Spring Boot + PostgreSQL + Kafka + ELK
# docker-compose.yml
services:
  # ── Aplikacja Spring Boot ──────────────────────────────────
  order-service:
    build:
      context: .
      target: runtime          # multi-stage stage name
    image: order-service:"${VERSION:-latest}"
    ports:
      - "8080:8080"
    environment:
      SPRING_PROFILES_ACTIVE: docker
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/orders
      SPRING_DATASOURCE_USERNAME: orders_user
      # Sekrety lepiej przez .env lub secrets:
      SPRING_DATASOURCE_PASSWORD: "${DB_PASSWORD}"
      KAFKA_BOOTSTRAP_SERVERS: kafka:9092
    depends_on:
      postgres:
        condition: service_healthy    # czekaj na health check!
      kafka:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:8080/actuator/health"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 60s
    networks:
      - app-network

  # ── PostgreSQL ─────────────────────────────────────────────
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: orders
      POSTGRES_USER: orders_user
      POSTGRES_PASSWORD: "${DB_PASSWORD}"
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d  # SQL scripts przy starcie
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U orders_user"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network

  # ── Kafka (KRaft mode — bez ZooKeepera) ───────────────────
  kafka:
    image: confluentinc/cp-kafka:7.6.0
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_LISTENERS: PLAINTEXT://:9092,CONTROLLER://:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_CONTROLLER_QUORUM_VOTERS: "1@kafka:9093"
      CLUSTER_ID: MkU3OEVBNTcwNTJENDM2Qk
    healthcheck:
      test: kafka-topics.sh --bootstrap-server localhost:9092 --list
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 30s
    networks:
      - app-network

volumes:
  postgres-data:

networks:
  app-network:
    driver: bridge

depends_on — pułapka depends_on bez condition czeka tylko na start kontenera, nie na gotowość serwisu. PostgreSQL może być "running" ale jeszcze nie przyjmować połączeń.
Używaj condition: service_healthy z healthcheck na każdym zależnym serwisie. To eliminuje potrzebę wait-for-it.sh w starterach.
Docker Compose vs Kubernetes Docker Compose = lokalny development i testy integracyjne. Kubernetes = produkcja, staging. Nikt nie używa Compose na produkcji (może poza małymi projektami/VPS). Testcontainers pozwala używać Docker w testach integracyjnych bez "lokalnego" Compose.

Kubernetes — Fundament
Kubernetes — architektura klastra, komponenty Control Plane
Kubernetes to platforma orkiestracji kontenerów — automatyzuje deployment, skalowanie i zarządzanie aplikacjami. Klaster = Control Plane (zarządzanie) + Worker Nodes (uruchamianie workloadów). Deklaratywny model: opisujesz co chcesz, K8s realizuje.
🏗️ Komponenty klastra
CONTROL PLANE
API Server
Brama do klastra — każde kubectl i każdy komponent rozmawia przez REST API. Waliduje, autoryzuje i persystuje do etcd.
etcd
Rozproszony key-value store — jedyne źródło prawdy o stanie klastra. Replikowany (Raft). Krytyczny — backup etcd = backup klastra.
Scheduler
Przypisuje Pody do nodów na podstawie zasobów (requests/limits), selektorów, affinity, taints/tolerations. Nie uruchamia — tylko decyduje gdzie.
Controller Manager
Pętla sterowania — ciągle porównuje desired state z actual state i koryguje. Zawiera: ReplicaSet, Deployment, Node, Job, ServiceAccount controllers.
Cloud Controller
Integracja z cloud providerem (AWS/GCP/Azure) — Load Balancery, storage, node lifecycle. Oddziela logikę K8s od cloud-specific.
WORKER NODE
kubelet
Agent na każdym nodzie — komunikuje się z API Server, zarządza Podami przez CRI (Container Runtime Interface). Raportuje stan nodów.
kube-proxy
Implementuje Services (iptables/IPVS). Routing ruchu sieciowego do właściwych Podów. Jeden per node.
Container Runtime
containerd (domyślny od K8s 1.24) lub CRI-O. Uruchamia kontenery przez OCI. Docker Shim usunięty.

Kubernetes Workloads
Pod, Deployment, StatefulSet, DaemonSet, Job — obiekty workload
Workloady to obiekty K8s uruchamiające kontenery. Pod to atomowa jednostka — rzadko tworzysz bezpośrednio. Deployment zarządza bezstanowymi aplikacjami. StatefulSet dla baz danych. DaemonSet dla agentów per node.
Deployment — kompletny manifest Spring Boot
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
  namespace: myapp
  labels:
    app: order-service
    version: v1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: order-service

  # Strategia deploymentu
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # ile dodatkowych Podów podczas update
      maxUnavailable: 0  # ile może być niedostępnych (0 = zero downtime)

  template:
    metadata:
      labels:
        app: order-service
        version: v1
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/actuator/prometheus"

    spec:
      # Czas na graceful shutdown (SIGTERM → SIGKILL)
      terminationGracePeriodSeconds: 60

      # Init container — czekaj na PostgreSQL
      initContainers:
        - name: wait-for-db
          image: busybox:1.36
          command: ['sh', '-c',
            'until nc -z postgres 5432; do echo waiting; sleep 2; done']

      containers:
        - name: order-service
          image: registry.io/order-service:1.5.0
          imagePullPolicy: Always

          ports:
            - containerPort: 8080
              name: http

          # Zmienne środowiskowe
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: "kubernetes"
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: db-secret
                  key: password
            - name: POD_NAME
              valueFrom:
                fieldRef:
                  fieldPath: metadata.name

          envFrom:
            - configMapRef:
                name: order-service-config

          # Zasoby — OBOWIĄZEK na produkcji
          resources:
            requests:
              memory: "256Mi"
              cpu:    "100m"
            limits:
              memory: "512Mi"
              cpu:    "500m"

          # Health checks
          startupProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            failureThreshold: 30
            periodSeconds: 10        # max 5 min na startup

          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 0
            periodSeconds: 10
            failureThreshold: 3

          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 0
            periodSeconds: 5

          # Graceful shutdown
          lifecycle:
            preStop:
              exec:
                command: ["/bin/sh", "-c", "sleep 5"]

          volumeMounts:
            - name: app-config
              mountPath: /config
              readOnly: true

      volumes:
        - name: app-config
          configMap:
            name: order-service-config
📋 Typy workloadów
Pod
Jeden lub kilka kontenerów współdzielących network i storage. Atomowa jednostka K8s. Efemeryczny — nie twórz bezpośrednio w prod.
Deployment
Bezstanowe aplikacje (stateless). Zarządza ReplicaSet, rolling updates, rollback. To twój domyślny wybór dla Spring Boot serwisów.
StatefulSet
Stanowe aplikacje — bazy danych, Kafka, Elasticsearch. Stabilne nazwy Podów (pod-0, pod-1), persistent volumes per Pod, ordered start/stop.
DaemonSet
Jeden Pod na każdym nodzie. Używaj dla: Filebeat (logi), node-exporter (metryki), kube-proxy, CNI pluginy.
Job
Jednorazowe zadania — migracje DB, generowanie raportów. Uruchamia się do completion. parallelism i completions dla równoległości.
CronJob
Job po harmonogramie (cron syntax). Tworzy Job przy każdym uruchomieniu. Uwaga na concurrencyPolicy: Forbid.
💊 Probes — kiedy co używać
ProbeCelAkcja przy porażce
startupProbeAplikacja gotowa do health check. Blokuje liveness/readiness.Kontener restartowany po failureThreshold
livenessProbeCzy aplikacja żyje? Deadlock, OOM, nieskończona pętla.Kontener restartowany
readinessProbeCzy aplikacja gotowa przyjmować ruch? Warms up, circuit breaker.Pod usunięty z Service endpoints (brak ruchu)

Kubernetes Networking
Service, Ingress, DNS — sieć w Kubernetes
Każdy Pod ma unikalny IP. Service daje stabilny IP/DNS i load balancing do zestawu Podów. Ingress to HTTP/HTTPS router — mapuje ścieżki URL do serwisów. DNS w K8s: <service>.<namespace>.svc.cluster.local.
Service + Ingress — manifest
# ══ Service — ClusterIP (domyślny) ════════════════════════════
apiVersion: v1
kind: Service
metadata:
  name: order-service
  namespace: myapp
spec:
  selector:
    app: order-service     # musi pasować do Pod labels!
  ports:
    - name: http
      port: 80              # port Service
      targetPort: 8080    # port kontenera

🌐 Typy Service
ClusterIP
Domyślny. Wewnętrzny wirtualny IP (VIP) dostępny tylko w klastrze. Używaj dla komunikacji między serwisami. DNS: service.namespace.
NodePort
Otwiera port (30000-32767) na każdym nodzie → ruch idzie do Podów. Nieeleganckie, używaj dla debugowania lub małych projektów.
LoadBalancer
Cloud provider tworzy zewnętrzny Load Balancer (AWS ELB, GCP GLB). Jeden per Service = drogie! Używaj Ingress zamiast wielu LB.
ExternalName
DNS CNAME alias do zewnętrznej usługi. Pozwala używać K8s DNS dla zewnętrznych zasobów (np. managed DB).
Headless
clusterIP: None — DNS zwraca bezpośrednio IP Podów (nie VIP). Używany przez StatefulSet, service discovery (Consul, Kafka).

Zasoby & Autoscaling
Requests, Limits, HPA, VPA — zarządzanie zasobami
Requests i Limits to najważniejsza konfiguracja w K8s — wpływają na scheduling, QoS i stabilność klastra. HPA skaluje horyzontalnie (więcej Podów), VPA skaluje wertykalnie (więcej zasobów per Pod).
⚖️ Requests vs Limits
requests.cpu
Gwarancja CPU dla Scheduler. Node musi mieć tyle wolnego CPU żeby schedulować Pod. 100m = 0.1 vCPU (milli-CPU). Aplikacja może używać więcej jeśli dostępne.
limits.cpu
Maks CPU. Gdy aplikacja przekroczy → throttling (nie kill!). CPU jest "compressible". Uważaj: zbyt niskie limits.cpu = powolna aplikacja.
requests.memory
Gwarancja RAM. Używana do schedulowania. Aplikacja może używać więcej jeśli dostępne na nodzie.
limits.memory
Maks RAM. Gdy aplikacja przekroczy → OOMKilled (kontener zabity!). Dla JVM: ustaw limits.memory wyżej niż MaxRAMPercentage × JVM heap.
QOS CLASSES

Guaranteed — requests == limits. Pod ostatni do ewakuacji przy presji zasobów. ✅ Używaj na produkcji.

Burstable — requests < limits. Może być evicted jeśli node pod presją.

BestEffort — brak requests/limits. Pierwszy evicted. Nigdy na produkcji.

HPA — Horizontal Pod Autoscaler
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service

  minReplicas: 2      # nigdy poniżej 2 (HA)
  maxReplicas: 20     # max skalowanie

  metrics:
    # CPU — klasyczny trigger
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70  # % requests.cpu

    # Metryki zewnętrzne (z Prometheus via KEDA lub Adapter)
    - type: External
      external:
        metric:
          name: kafka_consumer_lag
          selector:
            matchLabels:
              topic: orders
        target:
          type: AverageValue
          averageValue: "1000"  # max 1000 wiadomości lag/pod

  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30     # szybki scale-up
      policies:
        - type: Pods
          value: 4
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300    # wolny scale-down (5 min)
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
Requests — zasada kciuka dla Spring Boot Startuj ostrożnie, mierz, dostosuj. Typowe wartości na start:
CPU requests: 100–250m (JVM idle ~10–50m, pod obciążeniem 200–500m)
Memory requests: 256–512Mi (JVM heap 128–256Mi + overhead)
Zawsze: requests == limits dla memory (Guaranteed QoS). CPU możesz zostawić burstable.

  name: order-service-sa

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: order-service-role
  namespace: myapp
rules:
  - apiGroups: [""]
    resources: [configmaps]
    verbs: [get, list, watch]   # tylko odczyt!  
  - apiGroups: [""]
    resources: [secrets]
    resourceNames: [db-secret]  # tylko konkretny Secret
    verbs: [get]
---

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: order-service-binding
  namespace: myapp
subjects:
  - kind: ServiceAccount
    name: order-service-sa
    namespace: myapp
roleRef:
  kind: Role
  name: order-service-role
  apiGroup: rbac.authorization.k8s.io
---
# ══ SecurityContext — hardening kontenera ════════════════════
spec:
  securityContext:               # poziom Pod
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 1000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: order-service
      securityContext:            # poziom Container
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: [ALL]              # usuń wszystkie Linux capabilities
🔐 RBAC — koncepty
Role
Zestaw uprawnień w konkretnym namespace. Verbs: get, list, watch, create, update, patch, delete.
ClusterRole
Jak Role ale cluster-wide (lub dla non-namespaced resources jak nodes, PV, CRD).
RoleBinding
Przypisuje Role do Subject (User, Group, ServiceAccount) w namespace.
ClusterRoleBinding
Przypisuje ClusterRole globalnie — ostrożnie! Daje dostęp do wszystkich namespaces.
ServiceAccount
Tożsamość Poda. Domyślny SA = default z minimalnym dostępem. Zawsze twórz dedykowane SA per aplikację. Tokeny montowane auto do /var/run/secrets/kubernetes.io/serviceaccount/.
Zasada minimalnych uprawnień Każdy Pod powinien mieć tylko tyle uprawnień ile potrzebuje. Domyślny ServiceAccount ma dostęp do K8s API — możliwy wektor ataku.
Praktyka: dedykowany SA per serwis + minimalne Role + automountServiceAccountToken: false jeśli aplikacja nie używa K8s API.

📦 Helm — koncepty
Chart
Pakiet K8s manifestów z szablonami. Jak aplikacja w apt/npm. Można wersjonować i dystrybuować.
Release
Instancja Chart zainstalowana w klastrze. Ten sam Chart możesz zainstalować wielokrotnie z różnymi nazwami.

Spring Boot na Kubernetes
Spring Boot + K8s — konfiguracja, graceful shutdown, health checks
Spring Boot ma świetną integrację z Kubernetes przez Spring Cloud Kubernetes, Actuator health groups i wbudowany graceful shutdown. Kilka kluczowych konfiguracji sprawia że aplikacja zachowuje się poprawnie w środowisku orkiestrowanym.
application-kubernetes.yml — produkcyjna konfiguracja
# application-kubernetes.yml
spring:
  application:
    name: order-service

  # Graceful shutdown — poczekaj na kończące się żądania
  lifecycle:
    timeout-per-shutdown-phase: 30s

  # ConfigMap / Secret auto-reload (Spring Cloud Kubernetes)
  cloud:
    kubernetes:
      config:
        enabled: true
        name: order-service-config
        namespace: myapp
      secrets:
        enabled: true
        name: db-secret
      reload:
        enabled: true
        mode: event            # auto-reload gdy ConfigMap zmieniony
        period: 15000

  # Datasource z connection pool (pod obciążeniem)
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 3000
      idle-timeout: 300000
      max-lifetime: 1200000

server:
  port: 8080
  # Graceful shutdown: K8s wysyła SIGTERM, Spring czeka na żądania
  shutdown: graceful

management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus,metrics
  endpoint:
    health:
      probes:
        enabled: true          # /actuator/health/liveness + /readiness
      show-details: always
      group:
        liveness:
          include: livenessState,diskSpace
        readiness:
          include: readinessState,db,kafka,redis  # sprawdź zewnętrzne zależności

  # Dodaj K8s info do /actuator/info
  info:
    kubernetes:
      enabled: true

/* ── Spring Boot Graceful Shutdown flow w K8s ──────────────
   1. K8s usuwa Pod z Service endpoints (readiness fails)
   2. K8s wysyła SIGTERM do kontenera
   3. preStop hook: sleep 5 (bufor dla load balancerów)
   4. Spring: server.shutdown=graceful — czeka na żądania
   5. terminationGracePeriodSeconds=60 — max czas
   6. SIGKILL jeśli czas przekroczony
── ────────────────────────────────────────────────────────*/

Produkcja & Dobre Praktyki
Docker i K8s na produkcji — checklist i antywzorce
Klaster K8s to złożony system. Najczęstsze problemy produkcyjne, antywzorce i konfiguracje których absolutnie nie wolno pominąć na produkcji.
🚫 Antywzorce Docker

🔴 latest tag — brak reproducibility. Zawsze pinuj: openjdk:21.0.2-jre-slim

🔴 Root user — USER root domyślnie. Zawsze USER appuser

🔴 Sekrety w Dockerfile/image — docker history to pokaże. Używaj ARG tylko dla build, sekrety przez env/mounted secrets

🔴 Jeden wielki RUN — brak granularności cache. Ale też nie za dużo warstw.

🔴 Brak .dockerignore — kopiujesz .git, node_modules, sekrety do obrazu

🔴 Dane w kontenerze — dane giną po usunięciu kontenera. Zawsze Volume.

🚫 Antywzorce Kubernetes

🔴 Brak requests/limits — BestEffort QoS, OOMKilled, niestabilny klaster

🔴 1 replika — brak HA. Minimum 2 repliki z PodDisruptionBudget

🔴 latest image tag — brak determinism. imagePullPolicy: Always + wersjonowanie

🔴 Sekrety w ConfigMap — hasła jako plain text w ConfigMap. Użyj Secret lub ESO

🔴 Brak readinessProbe — ruch do Poda zanim gotowy = błędy 503

🔴 Brak graceful shutdown — przerwane żądania przy rolling update

🔴 Brak NetworkPolicy — każdy Pod widzi każdy. Flat network = ryzyko lateral movement

✅ Produkcja — checklist

✅ Multi-stage Dockerfile + non-root user

✅ Pinowane wersje obrazów (nie latest)

✅ Skanowanie obrazów (trivy, Snyk)

✅ Requests == limits dla memory (Guaranteed)

✅ Min. 2 repliki + PodDisruptionBudget

✅ startupProbe + livenessProbe + readinessProbe

✅ Graceful shutdown (preStop + terminationGrace)

✅ ResourceQuota per namespace

✅ NetworkPolicy (deny-all + allowlist)

✅ Sekrety przez ESO/SealedSecrets, nie plain K8s Secret w Git

✅ RBAC: dedykowane ServiceAccount, zasada minimalnych uprawnień

✅ HPA dla stateless serwisów

✅ PodDisruptionBudget (minAvailable: 1)

✅ Pod Affinity/Anti-affinity — rozprosz repliki po nodach

✅ Observability: logi + metryki + traces (patrz: Observability guide)

PodDisruptionBudget + ResourceQuota + Affinity
# ══ PodDisruptionBudget — HA podczas maintenance ══════════════
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: order-service-pdb
spec:
  minAvailable: 1           # zawsze min 1 pod dostępny
  selector:
    matchLabels:
      app: order-service
---
# ══ ResourceQuota — limity namespace ══════════════════════════
apiVersion: v1
kind: ResourceQuota
metadata:
  name: myapp-quota
  namespace: myapp
spec:
  hard:
    requests.cpu:    "10"
    requests.memory: 20Gi
    limits.cpu:      "20"
    limits.memory:   40Gi
    pods:            "50"
    count/secrets:   "20"
---
# ══ Pod Anti-Affinity — rozprosz po nodach ════════════════════
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchLabels: { app: order-service }
          topologyKey: kubernetes.io/hostname
# Efekt: K8s preferuje umieszczanie replik na różnych nodach
# = brak single point of failure na poziomie noda
