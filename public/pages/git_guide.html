Model Danych
Jak Git naprawdę działa — obiekty, DAG, HEAD
Git to content-addressable filesystem — każdy obiekt jest adresowany przez SHA-1 hash swojej zawartości. Zrozumienie modelu danych wyjaśnia dlaczego komendy działają tak a nie inaczej, i usuwa magię z rebasing, cherry-picking i reflogów.
🗃️ Cztery typy obiektów Git
blob
Zawartość pliku (bez nazwy, bez ścieżki). Identyczny plik w dwóch miejscach = jeden blob. git cat-file -p <sha>
tree
Katalog — lista wpisów (mode, type, sha, name). Odpowiednik inodes w filesystemie. Zawiera blobsy i inne tree.
commit
Snapshot + metadane: author, committer, date, message, parent commit(s), wskaźnik na root tree. Commit to nie diff — to pełny snapshot!
tag
Annotated tag — obiekt z nazwą, taggerem, datą, wiadomością i wskaźnikiem na inny obiekt (zwykle commit). Lightweight tag to tylko ref.
STRUKTURA COMMITA
commit a3f1d7e
tree     9f2bc4a  ← root tree snapshot
parent  8d3ae1b  ← poprzedni commit
author  Jan Kowalski <jan@..> 1700000000
message feat: add payment processing
SHA = hash(tree + parent + author + message + date)
→ zmiana CZEGOKOLWIEK = nowy SHA = nowy commit
Eksploracja obiektów Git — narzędzia plumbing
# Refs — gdzie jesteś
cat .git/HEAD                   # ref: refs/heads/main
cat .git/refs/heads/main        # SHA aktualnego commita
cat .git/refs/remotes/origin/main

# Inspekcja obiektów (plumbing)
git cat-file -t a3f1d7e         # typ: commit/blob/tree/tag
git cat-file -p a3f1d7e         # zawartość obiektu
git cat-file -p HEAD            # zawartość aktualnego commita
git cat-file -p HEAD:src/Main.java  # zawartość pliku w commicie

# Inspekcja tree
git ls-tree HEAD                # tree root aktualnego commita
git ls-tree -r HEAD             # rekurencyjnie (wszystkie pliki)
git ls-tree HEAD src/           # tree konkretnego katalogu

# Index (staging area)
git ls-files --stage            # co jest w stage z SHA blobsów
git diff --cached               # diff staged vs last commit

# Graf commitów
git log --graph --oneline --all --decorate

# Hash pliku bez dodawania
git hash-object plik.txt        # SHA gdyby był dodany

# Znajdź wszystkie unreachable objects
git fsck --unreachable

# Wyczyść unreachable objects (GC)
git gc --prune=now
Trzy obszary robocze w Git Working Directory — pliki na dysku, które widzisz i edytujesz.
Staging Area (Index) — "poczekania" — co wejdzie do następnego commita. git add tu przenosi.
Repository (.git/) — baza danych obiektów. git commit tu zapisuje.

Podstawy
Komendy podstawowe — z opcjami których nie znasz
Komendy których używasz codziennie mają opcje które dramatycznie podnoszą produktywność. Większość devów używa 20% możliwości git add, commit, diff i log. Poniżej flagi które powinieneś znać.
git add, status, diff — zaawansowane opcje
# ══ git add ═══════════════════════════════════════════════════
git add .                       # wszystkie zmiany w bieżącym katalogu
git add -A                      # wszystkie: nowe, zmienione, usunięte
git add -u                      # tylko śledzone (bez nowych plików)
git add -p                      # interaktywny — wybierz hunki ⭐
git add -p src/Service.java     # interaktywny dla jednego pliku
# Opcje w trybie -p:
# y=stage, n=pomiń, s=podziel hunk, e=edytuj ręcznie, q=quit

git add -N plik.txt             # intent-to-add (widoczny w diff)

# ══ git status ════════════════════════════════════════════════
git status                      # verbose
git status -s                   # short format: XY filename
git status -sb                  # short + branch info
# Znaczenie flag: M=modified, A=added, D=deleted, ??=untracked
# Lewa kolumna=staging, prawa=working dir

# ══ git diff ══════════════════════════════════════════════════
git diff                        # working vs staging
git diff --cached               # staging vs HEAD (co wejdzie w commit)
git diff HEAD                   # working vs HEAD (wszystkie zmiany)
git diff main..feature          # między branchami
git diff main...feature         # zmiany od wspólnego przodka (merge-base)
git diff HEAD~3 HEAD            # ostatnie 3 commity
git diff --stat                 # podsumowanie (ile linii +/-)
git diff --word-diff            # diff słowo po słowie
git diff --name-only            # tylko nazwy zmienionych plików
git diff --name-status          # M/A/D + nazwy
git diff -w                     # ignoruj whitespace
git diff --ignore-blank-lines   # ignoruj puste linie
git commit — wszystkie opcje
# ══ git commit ════════════════════════════════════════════════
git commit -m "feat: add user registration"
git commit -am "fix: null check"    # add tracked + commit (nie nowe!)
git commit --amend                  # popraw ostatni commit ⭐
git commit --amend -m "nowa treść"  # zmień tylko message
git commit --amend --no-edit        # zmień zawartość, zachowaj message
git commit --allow-empty -m "trigger CI"  # pusty commit
git commit --fixup=HEAD~2           # commit przeznaczony do autosquash
git commit --squash=HEAD~2          # squash do autosquash

# Conventional Commits (standard)
git commit -m "feat(auth): add OAuth2 login"
git commit -m "fix(api): handle null userId in getUser"
git commit -m "docs: update README installation steps"
git commit -m "refactor(db): extract ConnectionPool class"
git commit -m "test: add integration tests for OrderService"
git commit -m "chore: bump Spring Boot to 3.2.0"
git commit -m "perf: cache product list in Redis"
git commit -m "ci: add SonarQube step to pipeline"
git commit -m "feat!: remove deprecated /v1/orders endpoint"
# feat! lub BREAKING CHANGE: w footer = major version bump

# ══ git rm / mv ═══════════════════════════════════════════════
git rm plik.txt                 # usuń z tracking i z dysku
git rm --cached plik.txt        # usuń z tracking, zachowaj na dysku
git rm --cached -r build/       # usuń katalog z tracking
git mv stara_nazwa.txt nowa.txt # rename (Git widzi jako rename ~50%+)

# ══ git restore (nowe, zastępuje checkout dla plików) ═════════
git restore plik.txt            # odrzuć zmiany w working dir
git restore --staged plik.txt   # unstage (z staging do working)
git restore --source=HEAD~2 plik.txt  # przywróć wersję sprzed 2 commitów
git restore .                   # odrzuć WSZYSTKIE zmiany w WD ⚠️
📝 Anatomia dobrego commit message
feat(payment): add idempotent payment processing <blank line> Payment service now accepts idempotency-key header. Duplicate requests with same key return original response without re-charging the customer. Resolves the double-charge issue reported in JIRA-1234. Tested with concurrent request simulation (100 RPS). <blank line> Closes: #1234 Reviewed-by: Anna Nowak Co-authored-by: Piotr Wiśniewski <[email protected]>

Zasady dobrego message:

• Subject line max 72 znaki — inaczej GitHub/GitLab tnie

• Tryb rozkazujący — "add" nie "added", "fix" nie "fixed"

• Subject = co, body = dlaczego i kontekst

• Blank line między subject a body — konieczne

• Conventional Commits — type(scope): description

• Typ: feat, fix, docs, refactor, test, chore, perf, ci, build, style

Złe message'y:

• "fix" — co naprawiłeś?

• "WIP" — nie commituj WIP na shared branches

• "changes" — to widać w diff

• "asdfasdf" — tak, to się zdarza

Branching
Branches — tworzenie, nawigacja, zarządzanie
Branch w Git to po prostu wskaźnik (ref) na commit — 41 bajtów w pliku. Tworzenie brancha jest natychmiastowe i tanie. Kluczowe: rozumieć HEAD, tracking branches i różnicę między lokalnym a remote branchem.
git branch, checkout, switch — kompletne opcje
# ══ git branch ════════════════════════════════════════════════
git branch                      # lista lokalnych branchy
git branch -a                   # lokalne + remote
git branch -r                   # tylko remote
git branch -v                   # z ostatnim commitem każdego
git branch -vv                  # + tracking info + ahead/behind ⭐
git branch feature/login        # utwórz branch (nie przełączaj)
git branch feature/login abc123 # utwórz od konkretnego commita
git branch -d feature/login     # usuń (bezpieczne — tylko merged)
git branch -D feature/login     # usuń force (nawet unmerged) ⚠️
git branch -m stara nowa        # rename
git branch -m nowa              # rename aktualnego brancha
git branch --merged main        # które są zmergowane do main
git branch --no-merged main     # które NIE są zmergowane
git branch --contains abc123    # które zawierają dany commit

# Masowe usuwanie zmergowanych branchy lokalnych
git branch --merged main | grep -v "^\* \|main\|develop" | xargs git branch -d

# ══ git switch (nowe, zastępuje checkout dla branchy) ═════════
git switch feature/login        # przełącz na branch
git switch -c feature/payment   # utwórz i przełącz ⭐
git switch -c hotfix origin/main  # nowy branch od remote
git switch -                    # wróć do poprzedniego brancha ⭐
git switch main                 # przełącz na main

# ══ git checkout (starszy, bardziej wielofunkcyjny) ══════════
git checkout -b feature/login   # utwórz i przełącz
git checkout abc123             # detached HEAD — konkretny commit
git checkout -                  # poprzedni branch/commit
git checkout -- plik.txt        # odrzuć zmiany (stare API, użyj restore)

# ══ Detached HEAD ══════════════════════════════════════════════
git checkout abc123             # HEAD wskazuje na commit, nie branch
# Możesz eksplorować, commitować, ALE commity będą "orphaned"
# po przełączeniu na branch. Żeby zachować:
git switch -c saved-work        # utwórz branch od detached HEAD

🏷️ Tagi — annotated vs lightweight
# Lightweight tag — tylko wskaźnik na commit
git tag v1.2.0                  # brak obiektu tag!
git tag v1.2.0 abc123           # tag na konkretny commit

# Annotated tag — pełny obiekt (zalecane dla release) ⭐
git tag -a v1.2.0 -m "Release 1.2.0 — payment module"
git tag -a v1.2.0 abc123 -m "..."  # na konkretny commit

git tag                         # lista tagów
git tag -l "v1.*"               # z filtrem
git show v1.2.0                 # szczegóły tagu + commit
git push origin v1.2.0          # push jednego tagu
git push origin --tags          # push wszystkich tagów
git tag -d v1.2.0               # usuń lokalnie
git push origin :refs/tags/v1.2.0  # usuń na remote
git push origin --delete v1.2.0 # nowszy sposób

Remote
Remote — fetch, pull, push i zarządzanie remotes
Remote to alias do URL zdalnego repozytorium. origin to konwencja nazwy dla głównego remote. Różnica między fetch (pobierz, nie merguj) a pull (fetch + merge/rebase) jest fundamentalna dla bezpiecznej pracy z zespołem.
Remote — komendy z opcjami
# ══ git remote ════════════════════════════════════════════════
git remote -v                   # lista remotes z URL (fetch + push)
git remote add origin https://github.com/user/repo.git
git remote add upstream https://github.com/original/repo.git
git remote rename origin old-origin
git remote remove upstream
git remote set-url origin [email protected]:user/repo.git  # zmień URL
git remote show origin          # szczegółowe info o remote
git remote prune origin         # usuń lokalnie remote tracking refs dla usuniętych branchy

# ══ git fetch — pobierz, NIE merguj ═══════════════════════════
git fetch                       # fetch z origin (default)
git fetch origin                # explicit
git fetch --all                 # fetch ze WSZYSTKICH remotes
git fetch origin main           # fetch jednego brancha
git fetch --prune               # + usuń deleted remote branches ⭐
git fetch --tags                # fetch wszystkich tagów
# Po fetch: origin/main jest zaktualizowany, lokalny main — nie!

# ══ git pull — fetch + merge/rebase ══════════════════════════
git pull                        # pull z tracked remote
git pull origin main            # explicit
git pull --rebase               # rebase zamiast merge commit ⭐
git pull --rebase=interactive   # interaktywny rebase przy pull
git pull --no-rebase            # force merge (override config)
git pull --ff-only              # tylko jeśli fast-forward (bezpieczne)
git pull --no-ff                # zawsze merge commit

# ══ git push ══════════════════════════════════════════════════
git push                        # push do tracked remote
git push origin main            # explicit
git push -u origin feature/x   # push + ustaw tracking (-u = --set-upstream) ⭐
git push --force-with-lease     # force bezpieczny (sprawdza czy inni nie pushali) ⭐⭐
git push --force                # force niebezpieczny ⚠️ może nadpisać cudze commity
git push origin --delete feature/old  # usuń remote branch
git push origin --all           # push wszystkich lokalnych branchy
git push origin --tags          # push tagów
git push origin HEAD            # push aktualnego brancha bez pamiętania nazwy

# ══ Tracking branches ═════════════════════════════════════════
git branch --set-upstream-to=origin/main main  # ustaw tracking ręcznie
git branch -u origin/feature feature           # alias
--force vs --force-with-lease git push --force nadpisuje remote bez sprawdzenia czy ktoś inny pushał. Jeśli Ania pushała commita który ty nie masz lokalnie — jej praca przepada.

git push --force-with-lease sprawdza czy twoja lokalna kopia remote brancha jest aktualna. Jeśli ktoś pushał od twojego ostatniego fetch — odmawia. Zawsze używaj --force-with-lease zamiast --force.

Jedyne legalne użycie force-push: na własnym feature branchu po rebase.

Rebase vs Merge
Rebase, Merge, Cherry-pick — kiedy co i jak bezpiecznie
To jeden z najważniejszych tematów w Git. Merge zachowuje historię "jak było" — zawsze z merge commitem (opcjonalnie). Rebase przepisuje historię — commity wyglądają jakby były napisane na szczycie targetu. Złota zasada: nie rebasuj publicznej historii.
git merge — typy i opcje
# ══ git merge ══════════════════════════════════════════════════
git merge feature/login         # merge do aktualnego brancha
git merge --no-ff feature/login # zawsze merge commit (nawet FF możliwy)
git merge --ff-only feature/login  # tylko jeśli fast-forward (czyste)
git merge --squash feature/login   # spłaszcz do jednego commita ⭐
git merge --squash && git commit   # squash wymaga ręcznego commita
git merge --abort               # przerwij merge (przy konflikcie)
git merge --continue            # kontynuuj po rozwiązaniu konfliktu
git merge -X theirs feature     # przy konflikcie preferuj ich zmiany
git merge -X ours feature       # przy konflikcie preferuj nasze
git merge --no-commit feature   # merge ale nie commituj

# Fast-Forward Merge (brak merge commita)

# main:    A --- B
# feature:         C --- D
# Po merge --ff:
# main:    A --- B --- C --- D

# 3-way Merge (merge commit)

# main:    A --- B --- M   (M = merge commit)
#                ╲   ╱
# feature:  C --- D

# ══ Rozwiązywanie konfliktów ══════════════════════════════════
# Konflikt wygląda tak w pliku:
<<<<<<< HEAD
        return "nasze";
=======
        return "ich";
>>>>>>> feature/login

# Narzędzia do konfliktów:
git mergetool                   # graficzne narzędzie (vimdiff, VS Code, IntelliJ)
git checkout --ours plik.txt    # weź naszą wersję całego pliku
git checkout --theirs plik.txt  # weź ich wersję całego pliku

# Konfiguracja merge strategy dla konkretnego pliku
# .gitattributes:
# CHANGELOG.md merge=union  ← łącz zamiast konfliktować
# package-lock.json merge=theirs  ← zawsze ich
git rebase — standard, interaktywny, exec
# ══ git rebase — podstawowy ═══════════════════════════════════
git rebase main                 # rebasuj aktualny branch na main
git rebase origin/main          # rebasuj na remote main
git rebase --onto main feature~5 feature  # przenieś ostatnie 5 commitów
git rebase --abort              # przerwij rebase
git rebase --continue           # kontynuuj po konflikcie
git rebase --skip               # pomiń problematyczny commit

# ══ git rebase --interactive (git rebase -i) ══════════════════
git rebase -i HEAD~5            # edytuj ostatnie 5 commitów ⭐⭐
git rebase -i main              # edytuj commity od rozwidlenia z main
git rebase -i --root            # edytuj od samego początku

# W edytorze — opcje per commit:
# pick   p  — zostaw jak jest (default)
# reword r  — zostaw, zmień tylko message
# edit   e  — zatrzymaj do edycji (można dodać zmiany)
# squash s  — połącz z poprzednim, edytuj message
# fixup  f  — połącz z poprzednim, zachowaj message poprzedniego ⭐
# drop   d  — usuń commit całkowicie
# exec   x  — wykonaj shell command po tym commicie

# Przykład — squash 3 commitów WIP w jeden:
# pick a1b2c3 feat: start user service
# fixup 9f3e2a1 wip
# fixup 8d4f1b3 fix typo
# → jeden commit z message "feat: start user service"

# Autosquash — fixup! i squash! commity
git commit --fixup=HEAD~2       # commit ze "fixup! [original message]"
git rebase -i --autosquash main # automatycznie sortuje fixup! commity

# ══ git cherry-pick ═══════════════════════════════════════════
git cherry-pick abc123          # przenieś jeden commit ⭐
git cherry-pick abc123 def456   # kilka commitów
git cherry-pick abc123..def456  # zakres (bez abc123)
git cherry-pick abc123^..def456 # zakres (z abc123)
git cherry-pick -n abc123       # bez commita (tylko zmiany)
git cherry-pick --abort
git cherry-pick --continue
Złota zasada Rebase — nigdy nie rebasuj publicznych branchy Rebase przepisuje historię — tworzy nowe commity z nowymi SHA. Jeśli ktoś ma lokalnie stare SHA i ty pushujesz po rebase — ich historia diverguje od remote. Chaos.

✅ Bezpieczne: rebase swojego feature brancha na main (przed merge)
✅ Bezpieczne: interactive rebase własnych WIP commitów przed push
❌ Niebezpieczne: rebase main, develop, release — branchy których używa zespół
❌ Niebezpieczne: rebase brancha po którym inni już pracują
⚖️ Merge vs Rebase vs Squash — kiedy co
StrategiaHistoriaKiedy używać

Merge --no-ffZachowuje branch topologyChcesz widzieć grupę zmian jako feature

Historia & Log
git log, blame, bisect — eksploracja historii jak detektyw
Historia Git to kopalnia wiedzy — kto, co, kiedy i dlaczego. git log ma dziesiątki opcji do filtrowania. git blame pokazuje kto napisał każdą linię. git bisect binarnie wyszukuje commit który wprowadził buga.
git log — formatowanie i filtrowanie
# ══ Format output ═════════════════════════════════════════════
git log                         # domyślny (verbose)
git log --oneline               # jedna linia per commit
git log --oneline --graph       # z grafem ASCII ⭐
git log --oneline --graph --all --decorate  # wszystkie branche ⭐⭐
git log --pretty=format:"%h %an %ar %s"    # custom format
git log --format="%C(yellow)%h%Creset %s %C(blue)(%an)%Creset"

# Format placeholders:
# %h=short sha, %H=full sha, %s=subject, %b=body
# %an=author name, %ae=author email, %ar=author date relative
# %cn=committer name, %cd=committer date
# %C(color)=kolor, %Creset=reset, %n=newline

git log --stat                  # z listą zmienionych plików
git log --patch                 # z pełnym diffem (=git log -p)
git log --patch -3              # ostatnie 3 z diffem
git log --shortstat             # krótkie statystyki

# ══ Filtrowanie ═══════════════════════════════════════════════
git log -5                      # ostatnie 5 commitów
git log --since="2024-01-01"    # od daty
git log --until="2024-12-31"    # do daty
git log --since="2 weeks ago"   # relatywnie
git log --author="Jan"          # filtr autora (regex)
git log --grep="JIRA-1234"      # szukaj w message
git log --grep="fix" -i         # case insensitive
git log -S "methodName"         # commity które dodały/usunęły string ⭐ (pickaxe)
git log -G "regex"              # commity pasujące do regex w diff ⭐⭐
git log --all-match --grep="feat" --author="Jan"  # AND logic

# ══ Filtrowanie po plikach/ścieżkach ═════════════════════════
git log -- src/Service.java     # commity dotyczące pliku ⭐
git log --follow -- src/New.java  # przez rename (follow renames)
git log main..feature           # commity w feature, nie w main
git log main...feature          # commity w jednym lub drugim (symmetric)
git log --merges                # tylko merge commity
git log --no-merges             # bez merge commitów
git log --first-parent main     # tylko "backbone" — bez feature commitów ⭐
git blame, show, bisect
# ══ git blame — kto napisał każdą linię ═════════════════════

Cofanie Błędów
Reset, Revert, Restore — cofanie zmian bezpiecznie
Git ma kilka mechanizmów cofania — różnią się tym co cofają i czy są bezpieczne dla publicznej historii. revert jest zawsze bezpieczny. reset modyfikuje historię. reflog to siatka bezpieczeństwa — można odzyskać prawie wszystko.
reset, revert, restore — kiedy co
# ══ git reset ═════════════════════════════════════════════════
# HEAD wskazuje na commit, zmienia gdzie gałąź "wskazuje"

git reset --soft HEAD~1         # cofnij commit, zachowaj w staged ⭐
git reset --mixed HEAD~1        # cofnij commit, zachowaj w working dir (default)
git reset --hard HEAD~1         # cofnij commit, USUŃ zmiany ⚠️ nieodwracalne!

git reset HEAD~3                # cofnij 3 commity (--mixed)
git reset abc123                # cofnij do konkretnego commita
git reset --hard origin/main    # synchronizuj z remote (usuwa lokalne!) ⚠️
git reset HEAD plik.java        # unstage konkretnego pliku (stary sposób)
git restore --staged plik.java  # nowy sposób unstage ⭐

# ══ git revert — BEZPIECZNE cofanie ══════════════════════════
# Tworzy NOWY commit który odwraca zmiany. Historia zachowana.
# Używaj na publicznych branchach!

git revert HEAD                 # revert ostatniego commita
git revert abc123               # revert konkretnego commita
git revert HEAD~3..HEAD         # revert ostatnich 3 (tworzy 3 revert commity)
git revert --no-commit HEAD~3..HEAD  # revert bez commitowania (zbiorczy)
git revert -m 1 abc123          # revert merge commita (1=mainline) ⭐
git revert --abort
git revert --continue

# ══ Porównanie ══════════════════════════════════════════════
#                  Working Dir  Staging  Repository  Historia
# reset --soft     ✅ zachowana  staged   cofnięty    zmieniona
# reset --mixed    ✅ zachowana  czyste   cofnięty    zmieniona
# reset --hard     ❌ usunięta   czyste   cofnięty    zmieniona
# revert           ✅ zachowana  new      new commit  ✅ zachowana
# restore          edytuje WD   opcja    bez zmian   ✅ zachowana
reflog — siatka bezpieczeństwa
# ══ git reflog — historia ruchów HEAD ════════════════════════
# Reflog rejestruje KAŻDY ruch HEAD (nawet reset --hard)
# Domyślnie trzymany 90 dni. Lokalne repozytorium.

git reflog                      # historia HEAD ⭐⭐
git reflog show main            # historia konkretnego brancha
git reflog --date=iso           # z timestamps
git reflog --all                # wszystkich refs

# Odtwarzanie stanu po reset --hard
git reflog
# HEAD@{1} abc123 commit: feat: add payment
# HEAD@{2} 9f3e2a reset --hard HEAD~3
git reset --hard HEAD@{1}       # cofnij do stanu przed resetem ⭐
git switch -c recovery HEAD@{1} # lub utwórz nowy branch

# Odtwarzanie usuniętego brancha
git reflog | grep "feat/deleted-branch"
git switch -c feat/deleted-branch abc123  # od ostatniego SHA brancha

# Odtwarzanie dropped stash
git fsck --unreachable | grep commit
git show <unreachable-sha>     # znajdź swój stash
git stash apply <unreachable-sha>

# ══ Przydatne scenariusze naprawy ════════════════════════════
# 1. Commit na złym branchu
git log --oneline -1            # zapamiętaj SHA
git reset --soft HEAD~1         # cofnij commit, zachowaj zmiany
git switch poprawny-branch
git commit -m "..."             # commituj na właściwym

# 2. Commity do main zamiast feature brancha
git switch -c feature/fix       # utwórz branch od aktualnego stanu
git switch main
git reset --hard origin/main    # cofnij main do remote

# 3. Accidentalnie zmergowany branch
git reset --hard ORIG_HEAD      # GIT zapisuje ORIG_HEAD przed merge!

# 4. Dodałeś duży plik do historii
git filter-branch --tree-filter 'rm -f duzy-plik.zip' HEAD  # stare
git filter-repo --path duzy-plik.zip --invert  # nowe (git-filter-repo)
Sekret seniora — prawie wszystko jest odwracalne w Git git reset --hard wygląda groźnie ale reflog ma twój backup przez 90 dni. Usunąłeś branch? git reflog → znajdź SHA → git switch -c odtworzony SHA. Zrobiłeś force push? Jeśli ktoś ma lokalną kopię — może ją push'nąć z powrotem. Jedyne co jest trudne do odzyskania: niezcommitowane zmiany po reset --hard lub restore. Dlatego commituj często — nawet WIP. Zawsze możesz rebase -i potem.

Stash & Worktree
git stash, worktree — praca z wieloma kontekstami
git stash to tymczasowy schowek — odkładasz niezcommitowane zmiany, przełączasz kontekst, wracasz. git worktree to nowsza, lepsza alternatywa — wiele working directories tego samego repo bez stash i clone.
git stash — wszystkie komendy
# ══ Podstawowe ════════════════════════════════════════════════

git stash push                  # jawny push (=git stash)
git stash push -m "WIP: payment refactor"  # z opisem ⭐
git stash push -u               # + untracked pliki (nowe)
git stash push --include-untracked  # alias -u
git stash push -a               # + ignored pliki (rzadko)
git stash push -p               # interaktywny (wybierz hunki) ⭐
git stash push -- src/Service.java  # tylko konkretny plik

# Lista i inspekcja
git stash list                  # lista stashów: stash@{0}, stash@{1}...
git stash show                  # co jest w najnowszym
git stash show -p               # z diffem
git stash show stash@{2}        # konkretny stash
git stash show -p stash@{2}

# Odtwarzanie
git stash pop                   # apply + drop ⭐
git stash apply                 # apply bez drop (zostaje w liście)
git stash apply stash@{2}       # konkretny stash
git stash pop stash@{2}

git stash branch feature/new stash@{0}  # utwórz branch od stasha ⭐

# Usuwanie
git stash drop                  # usuń najnowszy
git stash drop stash@{2}        # konkretny
git stash clear                 # usuń WSZYSTKIE ⚠️
git worktree — lepsza alternatywa dla stash
# ══ git worktree — wiele working dirs jednego repo ═══════════
# Zamiast stash + switch: osobny katalog dla każdego brancha
# Jeden .git/, wiele working directories. Szybkie, bezpieczne.

# Dodaj nowy worktree
git worktree add ../hotfix main  # nowy katalog od brancha main
git worktree add ../feature feature/payment  # istniejący branch

# /projekt/           ← twój główny worktree (main)
# /hotfix/            ← osobny katalog (branch: hotfix/v2)

git worktree prune              # usuń worktrees których katalogów nie ma

# ══ Ręczny hook — .git/hooks/pre-commit (chmod +x!) ═════════
#!/bin/bash
set -e

echo "▶ Running pre-commit checks..."

# 1. Sprawdź formatowanie (Spotless)
./mvnw spotless:check
if [ $? -ne 0 ]; then
  echo "❌ Code formatting issues. Run: ./mvnw spotless:apply"
  exit 1
fi

# 2. Znajdź sekrety (grep dla wzorców)
if git diff --cached | grep -iE "(password|secret|api_key|token)\s*=\s*['\"][^'\"]{8,}"; then
  echo "❌ Potential secrets found in staged files!"
  exit 1
fi

# 3. Sprawdź TODO w staged plikach (warning, nie blokuj)
TODO_COUNT=$(git diff --cached | grep "^\+" | grep -c "TODO" || true)
if [ "$TODO_COUNT" -gt 0 ]; then
  echo "⚠️  Found $TODO_COUNT TODO(s) — remember to create tickets!"
fi

echo "✅ Pre-commit checks passed!"

# ══ commit-msg hook — waliduj format ════════════════════════
#!/bin/bash
# $1 = plik z commit message
MSG=$(cat "$1")
PATTERN="^(feat|fix|docs|refactor|test|chore|perf|ci|build|style)(\(.+\))?!?: .{1,72}"

if ! echo "$MSG" | grep -qE "$PATTERN"; then
  echo "❌ Commit message doesn't follow Conventional Commits!"
  echo "   Expected: type(scope): description"
  echo "   Example:  feat(auth): add OAuth2 login"
  exit 1
fi

#!/bin/bash
# Auto-dodaj JIRA ticket z nazwy brancha

Branching Workflows
Git Flow, GitHub Flow, Trunk-Based — strategie dla zespołu
Nie ma jednego "najlepszego" workflow — to zależy od tempa wydań, wielkości zespołu i wymagań jakościowych. Kluczowe: wybrać jeden, trzymać się go konsekwentnie i zautomatyzować przez CI/CD.
🌿 GitHub Flow — prosty, szybki
d4e5f6a●feat: payment processingmain
│╲
c3b4e5f│ ●feat: user profile pagefeature/profile
a1b2c3d●fix: login redirect

Zasady:

• main zawsze deployable

• Feature branch → PR → code review → merge to main

• Po merge → natychmiastowy deploy

• Squash merge lub rebase + merge dla czystej historii

Kiedy: SaaS z continuous deployment, małe/średnie zespoły, brak potrzeby wsparcia wielu wersji jednocześnie

Unikaj gdy: potrzebujesz wsparcia wielu wersji produkcyjnych równocześnie

main     ●────────────────────●  v2.0

5 typów branchy: main (produkcja), develop (integracja), feature/*, release/*, hotfix/*

Kiedy: oprogramowanie z wersjonowaniem (mobilne, biblioteki), enterprise z scheduled releases

Unikaj gdy: continuous deployment — Git Flow spowalnia dostarczanie

Wymaga: feature flags, dobry CI/CD, dyscyplina teamowa, wysoka kultura testowania

Senior z Git: Nie boisz się reflogu — wiesz że prawie wszystko można odtworzyć. Używasz rebase -i do porządkowania historii przed merge. Zawsze --force-with-lease, nigdy --force. Aliasy dla 80% codziennych komend. Conventional Commits bo rozumiesz że historia to dokumentacja. Git bisect do szybkiego znajdowania regresji. Worktree zamiast stash przy context-switch. git log -S i -G do archeologicznych poszukiwań. Hooki do gate-keepingu jakości. Rozumiesz że branch to tylko wskaźnik, commit to snapshot, rebase to tworzenie nowych commitów — nie "przenoszenie". Git to supermoc — master it.
