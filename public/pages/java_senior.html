OOP — Głębokie Pułapki
Dziedziczenie, kompozycja i polimorfizm — co senior wie
OOP wygląda prosto — dopóki nie spotkasz się z problemem diamenta, konstruktorami w dziedziczeniu, kowariancją zwracanych typów czy przeciążaniem kontra przesłanianiem. To są pytania które padają na senior interview.
Overriding vs Overloading — różnice kluczowe
// ─── Overriding — polimorfizm w runtime ──────────────────
class Animal {
    public String sound() { return "..."; }
}
class Dog extends Animal {
    @Override
    public String sound() { return "Woof"; } // runtime dispatch
}
Animal a = new Dog();
a.sound(); // "Woof" — virtual dispatch

// ─── Overloading — rozwiązywane w compile-time! ───────────
void print(Object o)  { System.out.println("Object: " + o); }
void print(String s)  { System.out.println("String: " + s); }

Object obj = "hello";   // static type = Object
print(obj);             // ❌ "Object: hello" — NIE String!
// Overloading zależy od statycznego (compile-time) typu argumentu

// ─── Kowariantny typ zwracany (Java 5+) ──────────────────
class Builder {
    public Builder withName(String n) { ... return this; }
}
class SpecialBuilder extends Builder {
    @Override
    public SpecialBuilder withName(String n) { ... return this; }
    // ✅ Covariant return type — bardziej konkretny typ
}
Dziedziczenie — pułapki konstruktorów
// PUŁAPKA: wywołanie metody wirtualnej w konstruktorze!
class Base {
    public Base() {
        init();  // ❌ NIEBEZPIECZNE!
    }
    protected void init() { System.out.println("Base.init"); }
}
class Child extends Base {
    private final String value = "hello";

    @Override
    protected void init() {
        // value jest null tutaj! Pole nie zostało jeszcze zainicjalizowane
        System.out.println(value.toUpperCase()); // NPE!
    }
}
// Kolejność: Base() → Child.init() → Child fields init
// Rozwiązanie: nie wywołuj metod wirtualnych z konstruktora
// Użyj factory method pattern lub inicjalizuj przez super()

// ─── super() jest zawsze pierwszy! ───────────────────────
class Child extends Base {
    public Child(String name) {
        // super() wywołane niejawnie jeśli nie podasz explicite
        // ❌ nie możesz mieć kodu PRZED super()
        super(name);       // musi być pierwszą instrukcją
        this.extra = name; // dopiero potem
    }
}

🦆 duck typing & Structural Types

Java używa nominal typing — typy kompatybilne gdy explicite implementują interfejs. Brak duck typing jak w Python/Go.

Obejście: Generics + structural constraints, Reflection, lub Proxy dynamiczny.

Interfejsy funkcjonalne + lambdy dają praktyczny duck typing — dowolna metoda o pasującym sygnaturze może być użyta jako lambda.

⚠️ Fragile Base Class Problem Problem: dodanie lub zmiana metody w klasie bazowej może nieoczekiwanie zepsuć podklasę, która "przypadkowo" nadpisała tę metodę lub polegała na określonym zachowaniu. Efektywna Java (Bloch): "Design and document for inheritance or else prohibit it." Używaj final jeśli klasa nie jest projektowana do dziedziczenia. Dokumentuj które metody nadpisywać (i które wywołują inne wirtualne metody).

Kontrakt equals() i hashCode()
equals, hashCode, Comparable — pełny kontrakt
Kontrakt equals/hashCode to jeden z najważniejszych w Javie — złamanie go powoduje cichy błąd w kolekcjach. HashMap, HashSet, Hibernate cache — wszystkie zależą od poprawnej implementacji.
Kontrakt equals() — 5 właściwości
// Kontrakt equals() (java.lang.Object):
// 1. Reflexive:   x.equals(x) == true
// 2. Symmetric:   x.equals(y) == y.equals(x)
// 3. Transitive:  x.equals(y) && y.equals(z) → x.equals(z)
// 4. Consistent:  wielokrotne wywołanie = ten sam wynik
// 5. Non-null:    x.equals(null) == false (nigdy NPE)

// ✅ Poprawna implementacja (Java 7+ Objects.equals):
@Override
public boolean equals(Object o) {
    if (this == o) return true;            // 1. identity check
    if (!(o instanceof Money other)) return false; // 2. type + null
    return Objects.equals(amount, other.amount)
        && Objects.equals(currency, other.currency);  // 3. fields
}

// PUŁAPKA: equals z dziedziczeniem łamie symetrię!
class Point {
    int x, y;
    public boolean equals(Object o) {
        if (!(o instanceof Point p)) return false;
        return x == p.x && y == p.y;
    }
}
class ColorPoint extends Point {
    Color color;
    public boolean equals(Object o) {
        if (!(o instanceof ColorPoint cp)) return false;
        return super.equals(o) && color.equals(cp.color);
    }
}
Point p = new Point(1,2);
ColorPoint cp = new ColorPoint(1,2, RED);
p.equals(cp);   // true (Point.equals)
cp.equals(p);   // false (ColorPoint.equals)  ← ŁAMIE SYMETRIĘ!
// Rozwiązanie: kompozycja zamiast dziedziczenia + getClass() check
hashCode — kontrakt i implementacja
// Kontrakt hashCode:
// 1. Konsystencja: hashCode() zwraca ten sam int w życiu obiektu
// 2. equals → hashCode: jeśli a.equals(b) to a.hashCode() == b.hashCode()
// 3. !equals → hashCode: NIE musi być różny (ale powinien dla wydajności)

// ✅ Implementacja z Objects.hash():
@Override
public int hashCode() {
    return Objects.hash(amount, currency); // wygodne, nie najszybsze
}

// ✅ Ręczna implementacja (Bloch's recipe — wyższy hash quality):
public int hashCode() {
    int result = 31;                          // prime start
    result = 31 * result + amount.hashCode();  // prime multiplier
    result = 31 * result + currency.hashCode();
    return result;
}
// 31 = 2^5 - 1 = JVM może optimizować do shift+subtract

// PUŁAPKA: mutowalny klucz w HashMap!
Map<List<String>, String> map = new HashMap<>();
List<String> key = new ArrayList<>(List.of("a"));
map.put(key, "value");
key.add("b");              // modyfikacja! hashCode się zmieniło
map.get(key);              // null — klucz nie znajdzie się w buckecie!
Comparable vs Comparator
// Comparable — naturalne porządkowanie (wewnątrz klasy)
public class Money implements Comparable<Money> {
    @Override
    public int compareTo(Money other) {
        // Reguła: return negative gdy this < other
        // ❌ Nie używaj: return this.amount - other.amount (overflow!)
        // ✅ Użyj:
        return this.amount.compareTo(other.amount);
    }
}

// Comparator — zewnętrzne / wielokrotne porządkowania
Comparator<Employee> byDeptThenSalary = Comparator
    .comparing(Employee::getDept)
    .thenComparingDouble(Employee::getSalary).reversed()
    .thenComparing(Employee::getName, String.CASE_INSENSITIVE_ORDER);

// Comparator.naturalOrder() / reverseOrder()
// Comparator.nullsFirst() / nullsLast()
Comparator<String> safeComp = Comparator.nullsLast(
    String::compareToIgnoreCase);

// compareTo kontrakt:
// sgn(x.compareTo(y)) == -sgn(y.compareTo(x))
// transitivity: x > y && y > z → x > z
// consistency with equals (strongly recommended):
// x.compareTo(y) == 0 → x.equals(y) (TreeSet/TreeMap!)
🔑 Records — equals gratis

Records automatycznie generują equals, hashCode i toString na podstawie wszystkich komponentów (canonical fields).

public record Money(BigDecimal amount, String currency) {}

Money a = new Money(BigDecimal.TEN, "PLN");
Money b = new Money(BigDecimal.TEN, "PLN");
a.equals(b); // ✅ true — strukturalna równość

// Uwaga: equals na BigDecimal!
new BigDecimal("1.0").equals(new BigDecimal("1.00")) // false!
new BigDecimal("1.0").compareTo(new BigDecimal("1.00")) // 0 ✅

Collections — Internals
Jak naprawdę działają kolekcje Javy
HashMap, ArrayList, LinkedList — każda kolekcja ma unikalną implementację z innym profilem złożoności. Senior rozumie co dzieje się "pod maską" i wybiera właściwą strukturę dla kontekstu.
HashMap — jak działa wewnętrznie
// Struktura: tablica bucketów (Node[])
// index = (n-1) & hash(key)  gdzie n = capacity (potęga 2)

// Java 8+: bucket <= 8 elementów → LinkedList
//          bucket  > 8 elementów → TreeNode (red-black tree!)
// → gwarantuje O(log n) nawet przy słabym hashCode

// Parametry:
new HashMap<>(16, 0.75f);
//            ↑           ↑
//   initial capacity   load factor
// Resize gdy: size > capacity * loadFactor
// Resize = 2x capacity + rehash wszystkich kluczy = O(n)!

// ✅ Inicjalizuj z właściwą pojemnością:
int expectedSize = 1000;
new HashMap<>((int)(expectedSize / 0.75f) + 1); // bez resize

// hash() — perturbacja wyższych bitów:
static final int hash(Object key) {
    int h;
    return (key == null) ? 0
        : (h = key.hashCode()) ^ (h >>> 16);
    // XOR z górnymi bitami — redukuje kolizje gdy tablica mała
}

// Złożoności: get/put/remove = O(1) amortyzowane
// W najgorszym (wszystko w jednym buckecie): O(log n) w Java 8+
ArrayList vs LinkedList — kiedy co
// ArrayList — tablica dynamiczna
// Wewnętrznie: Object[] elementData
// Domyślny rozmiar: 10. Grow factor: 1.5x
// get(i)     → O(1) — random access
// add(e)     → O(1) amortyzowane (ostatni)
// add(i,e)   → O(n) — shift elementów w prawo
// remove(i)  → O(n) — shift w lewo
// contains   → O(n) — liniowe przeszukiwanie

// LinkedList — doubly-linked list
// Każdy element: Node { item, prev, next }
// get(i)     → O(n) — musi przejść od head/tail
// addFirst() → O(1)
// addLast()  → O(1)
// remove(i)  → O(n) — find, potem O(1) relink
// Overhead: każdy Node = 24 bytes (header + item + prev + next)

// W praktyce: ArrayList PRAWIE ZAWSZE lepsza niż LinkedList!
// LinkedList dobre tylko gdy: Deque semantics + dużo add/remove na końcach
// ArrayDeque > LinkedList jako Deque (lepsza lokalność cache)

// Inne kolekcje i ich złożoności:
// TreeMap/TreeSet   → O(log n) get/put/remove, sorted order
// LinkedHashMap     → O(1) jak HashMap + insertion order
// PriorityQueue     → O(log n) offer/poll, O(1) peek (min-heap)
// ArrayDeque        → O(1) addFirst/addLast/pollFirst/pollLast

Exception Design
Wyjątki — jak projektować, kiedy rzucać
Większość programistów używa wyjątków źle — łapią Exception, połykają stack trace, używają wyjątków do flow control. Senior wie kiedy checked vs unchecked, jak projektować hierarchię i kiedy wyjątek to zły wybór.
Checked vs Unchecked — kiedy co
// Checked (Exception) → wywołujący MUSI obsłużyć
// Używaj gdy: wywołujący może i powinien podjąć sensowne działanie
public class InsufficientFundsException extends Exception {
    private final BigDecimal deficit;
    public InsufficientFundsException(BigDecimal d) {
        super("Insufficient funds: deficit " + d);
        this.deficit = d;
    }
    public BigDecimal getDeficit() { return deficit; }
}
// Wywołujący może zaproponować doładowanie konta

// Unchecked (RuntimeException) → błąd programisty lub nie do naprawienia
public class OrderNotFoundException extends RuntimeException {
    public OrderNotFoundException(Long id) {
        super("Order not found: " + id);
    }
    // cause chain dla diagnostyki:
    public OrderNotFoundException(Long id, Throwable cause) {
        super("Order not found: " + id, cause);
    }
}

// ❌ Anty-pattern — łapanie Exception/Throwable:
try { ... } catch (Exception e) { log.error("error"); } // połykanie!

// ❌ Wyjątek do flow control (bardzo wolne — stack unwind):
try {
    return map.get(key);  // jeśli null → NullPointerException
} catch (NullPointerException e) {
    return defaultValue;   // ❌ użyj getOrDefault!
}
Exception chaining i multi-catch
// ✅ Zawsze zachowuj cause chain:
try {
    jdbcTemplate.query(sql, params);
} catch (DataAccessException e) {
    // ❌ nowy exception BEZ cause — trace utracony!
    throw new OrderNotFoundException(id);
    // ✅ z cause:
    throw new OrderNotFoundException(id, e); // zachowaj trace!
}

// Multi-catch (Java 7+):
try { ... }
catch (IOException | SQLException e) { // oba → jeden handler
    // e jest effectively final — nie możesz reassignować
    handleError(e);
}

// Catch more specific first:
try { ... }
catch (FileNotFoundException e) { ... } // bardziej szczegółowy first
catch (IOException e) { ... }           // ogólniejszy second

// Exception w lambda — problem!
list.forEach(item -> {
    try { process(item); }         // ❌ verbose
    catch (IOException e) { throw new RuntimeException(e); }
});
// ✅ Wrapper helper:
interface ThrowingConsumer<T> {
    void accept(T t) throws Exception;
    static <T> Consumer<T> wrap(ThrowingConsumer<T> tc) {
        return t -> { try { tc.accept(t); }
            catch(Exception e) { throw new RuntimeException(e); }};
    }
}
list.forEach(ThrowingConsumer.wrap(this::process));

□ Czytelna wiadomość z kontekstem (co, dlaczego, jakie wartości)

□ Pola z informacjami dla catch (np. entityId, attemptedValue)

Immutability Patterns
Niezmienność — jak budować i dlaczego
Niezmienne obiekty są thread-safe, bezpieczne jako klucze w HashMap i łatwiejsze do rozumowania. Ale "final field" nie wystarczy — musisz rozumieć defensive copy, shallow vs deep immutability i Builder pattern.
Budowanie immutable klasy
// Reguły immutable klasy (Effective Java):
// 1. Nie udostępniaj mutujących metod (setterów)
// 2. Klasa powinna być final (lub wszystkie konstruktory prywatne)
// 3. Wszystkie pola final
// 4. Wszystkie pola private
// 5. Defensive copy dla mutable komponentów

public final class Period {
    private final Date start;
    private final Date end;

    public Period(Date start, Date end) {
        // ✅ Defensive copy w konstruktorze PRZED walidacją
        // (TOCTOU attack: zmiana po sprawdzeniu)
        this.start = new Date(start.getTime());  // kopia!
        this.end   = new Date(end.getTime());    // kopia!
        if (this.start.after(this.end))
            throw new IllegalArgumentException("start after end");
    }

    public Date start() {
        return new Date(start.getTime()); // ✅ defensive copy w getterze
    }
}
// Dziś zamiast Date używaj java.time.LocalDate — immutable!

// Shallow vs Deep immutability:
public final class Order {
    private final List<Item> items;

    public Order(List<Item> items) {
        this.items = List.copyOf(items); // ✅ unmodifiable copy
    }
    public List<Item> items() { return items; } // już bezpieczne
    // Ale Item musi być też immutable dla deep immutability!
}
Builder Pattern — właściwa implementacja
public final class User {
    private final String firstName;
    private final String lastName;
    private final String email;
    private final int    age;

    private User(Builder b) {
        this.firstName = Objects.requireNonNull(b.firstName, "firstName");
        this.lastName  = Objects.requireNonNull(b.lastName,  "lastName");
        this.email     = b.email;
        this.age       = b.age;
    }

    public static class Builder {
        private String firstName, lastName, email;
        private int age;

        public Builder firstName(String v) { firstName = v; return this; }
        public Builder lastName(String v)  { lastName  = v; return this; }
        public Builder email(String v)     { email     = v; return this; }
        public Builder age(int v)          { age       = v; return this; }
        public User    build() { return new User(this); }
    }
}

📦 Unmodifiable vs Immutable

Unmodifiable view — Collections.unmodifiableList(list) — rzuca UnsupportedOperationException przy mutacji, ALE oryginalna lista wciąż może się zmienić!

Truly immutable — List.of(), List.copyOf() — nie można zmienić ani przez widok ani przez oryginalną referencję.

Używaj List.copyOf() gdy chcesz mieć pewność że kopia jest immutable — nawet jeśli oryginał jest mutowany.

⚡ Dlaczego String jest immutable

String Pool: wielokrotne referencje do tego samego obiektu — mutable String byłoby katastrofalne (jedna referencja zmienia, wszystkie widzą zmianę).

Thread safety: automatyczna bez synchronizacji.

Klucz HashMap: hashCode jest cachowany — bezpieczny jako klucz.

Bezpieczeństwo: ścieżki plików, adresy URL nie mogą być podmienione po walidacji.

Reflection API & Dynamic Proxy
Reflection, Dynamic Proxy i MethodHandles
Reflection to fundament Spring AOP, Hibernate, Jackson i wielu innych frameworków. Senior musi rozumieć jak działa, kiedy jest bezpieczna i kiedy lepiej użyć nowszego MethodHandles API.
Reflection — pełny zakres możliwości
// Inspekcja klasy:
Class<User> clazz = User.class;
clazz.getDeclaredFields();          // prywatne też!
clazz.getDeclaredMethods();
clazz.getDeclaredConstructors();
clazz.getAnnotations();
clazz.getSuperclass();
clazz.getGenericInterfaces();       // z parametrami typów

// Tworzenie instancji:
Constructor<User> ctor = clazz.getDeclaredConstructor(String.class);
ctor.setAccessible(true);           // omija private!
User user = ctor.newInstance("Jan");

// Dostęp do pól:
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);
field.set(user, "Nowa wartość");  // nawet private final!
Object value = field.get(user);

// Wywoływanie metod:
Method method = clazz.getDeclaredMethod("processOrder", Long.class);
method.setAccessible(true);
Object result = method.invoke(user, 42L);

// MethodHandles — szybsze niż Reflection, sprawdzane w compile-time:
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle mh = lookup.findVirtual(
    User.class, "getName",
    MethodType.methodType(String.class));
String name = (String) mh.invoke(user); // ~3x szybsze niż reflect
Dynamic Proxy — jak działa Spring AOP
// JDK Dynamic Proxy — tylko dla interfejsów
OrderService proxy = (OrderService) Proxy.newProxyInstance(
    OrderService.class.getClassLoader(),
    new Class[] { OrderService.class },
    new InvocationHandler() {
        private final OrderService delegate = new OrderServiceImpl();

        @Override
        public Object invoke(Object proxy, Method method, Object[] args)
                throws Throwable {
            // Around advice:
            log.info("Before: {}", method.getName());
            long start = System.nanoTime();
            try {
                return method.invoke(delegate, args); // wywołaj prawdziwe
            } finally {
                log.info("After: {}ms", (System.nanoTime()-start)/1e6);

Annotation Processor — generowanie kodu
// Annotation Processor działa w compile-time (javac)
// Jak Lombok, MapStruct, Dagger2 generują kod
@SupportedAnnotationTypes("com.app.Builder")
@SupportedSourceVersion(SourceVersion.RELEASE_21)
@AutoService(Processor.class)  // google-auto-service rejestruje
public class BuilderProcessor extends AbstractProcessor {

    @Override
    public boolean process(Set<? extends TypeElement> annotations,
                            RoundEnvironment roundEnv) {
        for (Element el : roundEnv.getElementsAnnotatedWith(Builder.class)) {
            if (el.getKind() != ElementKind.CLASS) continue;
            TypeElement typeEl = (TypeElement) el;

            // Generuj kod za pomocą JavaPoet lub ręcznie:
            String builderCode = generateBuilder(typeEl);
            try {
                JavaFileObject file = processingEnv.getFiler()
                    .createSourceFile(typeEl.getSimpleName() + "Builder");
                try (var writer = file.openWriter()) {
                    writer.write(builderCode);
                }
            } catch (IOException e) { throw new RuntimeException(e); }
        }
        return true; // claim the annotation
    }
}

// Either<L, R> — Left (błąd) lub Right (sukces)
// Pozwala na composable error handling

// Prosta implementacja (sealed):
sealed interface Either<L, R> permits Left, Right {
    static <L,R> Either<L,R> left(L v)  { return new Left<>(v); }
    static <L,R> Either<L,R> right(R v) { return new Right<>(v); }
    boolean isRight();

    default <T> Either<L,T> map(Function<R,T> fn) {
        return switch (this) {
            case Right<L,R>(R v) -> Either.right(fn.apply(v));
            case Left<L,R>(L l) -> Either.left(l);
        };
    }
}
record Left<L,R>(L value)  implements Either<L,R> { public boolean isRight(){return false;}}
record Right<L,R>(R value) implements Either<L,R> { public boolean isRight(){return true; }}

// ─── Memoizacja — cache wyników funkcji ──────────────────
public static <T, R> Function<T, R> memoize(Function<T, R> fn) {
    Map<T, R> cache = new ConcurrentHashMap<>();
    return key -> cache.computeIfAbsent(key, fn);
}

Function<Long, BigDecimal> cachedPrice = memoize(id ->
    priceService.calculateExpensive(id));  // wywołane raz per id

// ─── Currying — przekształcanie f(a,b) w f(a)(b) ─────────
static <A,B,C> Function<A, Function<B,C>> curry(BiFunction<A,B,C> f) {
    return a -> b -> f.apply(a, b);
}

Function<String, String> addPrefix =
    curry(String::concat).apply("PREFIX_");

public class EventBus<T> {
    private final List<Consumer<T>> listeners = new CopyOnWriteArrayList<>();

    public Runnable subscribe(Consumer<T> l) {
        listeners.add(l);

    public void publish(T event) { listeners.forEach(l -> l.accept(event)); }
