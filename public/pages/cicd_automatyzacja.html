01 — pojęcia
Czym jest CI/CD?

Trzy pojęcia które brzmią podobnie ale oznaczają różne etapy automatyzacji. Każde rozwiązuje inny problem.

⚙️ CI — Continuous Integration
Ciągła integracja
Developer pushe kod ↓ Automatycznie uruchamiają się: → testy jednostkowe → testy integracyjne → analiza kodu (linting) → sprawdzanie coverage ↓ ✅ przeszło lub ❌ błąd
Cel: wykryć błędy zanim trafią do gałęzi głównej.
📦 CD — Continuous Delivery
Ciągłe dostarczanie
CI przeszło pomyślnie ↓ Automatycznie tworzony jest: → artefakt (JAR, WAR, ZIP) → obraz Docker → gotowy do deploymentu ↓ Deploy na staging/UAT ↓ Oczekuje na zatwierdzenie
Cel: kod jest zawsze gotowy do wdrożenia na produkcję.
🚀 CD — Continuous Deployment
Ciągłe wdrożenie
Delivery przeszło ↓ Automatycznie wdrożone: → na produkcję bez ręcznego → zatwierdzenia człowieka! ↓ ✅ kod żyje na produkcji w ciągu minut od push'a
Cel: zero ręcznych kroków — push → produkcja automatycznie.
CI/CD = fabryka oprogramowania Bez CI/CD: developer ręcznie buduje, ręcznie testuje, ręcznie wgrywa na serwer. Z CI/CD: push do gałęzi głównej → 5 minut → kod na produkcji. Każda firma powyżej 2 programistów powinna to mieć.
02 — od kodu do produkcji
Pełny flow CI/CD

Każdy krok po kolei — od napisania kodu do działającej aplikacji na produkcji.

🔄 Etapy pipeline krok po kroku
1
git push / pull request
Developer pushuje zmiany. Pipeline startuje automatycznie dzięki webhook'owi z GitHub/GitLab.
2
Checkout + setup środowiska
Runner pobiera kod z repo, instaluje zależności (Maven/Gradle/npm), konfiguruje JDK/Node.
3
Kompilacja / Build
mvn package lub gradle build. Sprawdza czy kod w ogóle się kompiluje. Tworzy artefakt (JAR/WAR).
4
Testy automatyczne
Unit testy (JUnit), integracyjne, E2E. Jeśli cokolwiek padnie — pipeline zatrzymuje się i powiadamia.
5
Analiza kodu (Code Quality)
SonarQube, Checkstyle, SpotBugs. Sprawdza code coverage, security issues, code smells.
6
Docker build + push
Tworzy obraz Docker z aplikacją. Taguje wersją (commit SHA lub tag). Pushuje do Docker Registry (GHCR, Docker Hub, ECR).
7
Deploy na staging
Automatyczny deploy na środowisko testowe. Smoke testy, testy E2E. Opcjonalnie: manualne zatwierdzenie QA.
8
Deploy na produkcję 🚀
Automatyczny (Continuous Deployment) lub po ręcznym zatwierdzeniu (Continuous Delivery). Monitoring + alerty.
── UPROSZCZONY SCHEMAT ────────────────────── Developer GitHub Actions │ │ ├─ git push ─────────────────────► │ │ │ checkout repo │ setup Java/Node │ mvn test │ │ │ ┌─ FAIL → notify ─► email/Slack │ └─ PASS ─► │ docker build │ docker push → GHCR │ │ │ ▼ │ ssh deploy.sh │ docker pull + up │ │ ◄─── notify ───────────────── ✅ deployed: v1.4.2 (2m 14s)
Fail Fast — zatrzymaj się przy pierwszym błędzie Dobry pipeline zatrzymuje się i powiadamia od razu gdy cokolwiek padnie. Nie ma sensu budować obrazu Docker jeśli testy nie przeszły. Kolejność kroków ma znaczenie — najtańsze (najszybsze) sprawdzenia najpierw.
03 — narzędzie
GitHub Actions — konfiguracja

Pliki YAML w katalogu .github/workflows/. Każdy plik = jeden workflow. Triggerowany przez eventy (push, PR, schedule, ręcznie).

.github/workflows/ci.yml — kompletny CI
name: CI — Build and Test

# Kiedy uruchamiać
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      # 1. Pobierz kod
      - name: Checkout code
        uses: actions/checkout@v4

      # 2. Skonfiguruj Java
      - name: Setup Java 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      # 3. Cache Maven dependencies
      - name: Cache Maven
        uses: actions/cache@v4
        with:
          path: ~/.m2/repository
          key: maven-${{ hashFiles('pom.xml') }}

      # 4. Build + testy
      - name: Build and test
        run: mvn --batch-mode verify

      # 5. Raport testów jeśli fail
      - name: Test report
        uses: dorny/test-reporter@v1
        if: always()  # nawet przy fail!
        with:
          reporter: java-junit
          path: target/surefire-reports/*.xml
.github/workflows/cd.yml — deploy na VPS
name: CD — Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    # Uruchom tylko po CI
    needs: build-and-test

    steps:
      - uses: actions/checkout@v4

      # Zaloguj do GitHub Container Registry
      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # Zbuduj i push obraz Docker
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ghcr.io/${{ github.repo }}/app:latest
            ghcr.io/${{ github.repo }}/app:${{ github.sha }}

      # Wdróż na VPS przez SSH
      - name: Deploy to VPS
        uses: appleboy/ssh-action@v1
        with:
          host:     ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key:      ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/app
            docker compose pull
            docker compose up -d --no-build
            docker image prune -f

04 — konteneryzacja
Docker w pipeline

Docker gwarantuje że aplikacja działa tak samo na każdym środowisku. "Działa u mnie" znika — obraz Docker działa wszędzie identycznie.

Dockerfile — multi-stage build
# ── ETAP 1: Builder ──────────────────────
FROM maven:3.9-amazoncorretto-21 AS builder

WORKDIR /build

# Najpierw tylko pom.xml — cache zależności!
COPY pom.xml .
RUN mvn dependency:go-offline

# Teraz kod — jak zmienisz kod, zależności
# NIE będą pobierane od nowa (cache hit)
COPY src ./src
RUN mvn package -DskipTests

# ── ETAP 2: Runtime ──────────────────────
# Znacznie mniejszy obraz! Bez Maven, JDK
FROM amazoncorretto:21-alpine

# Nie uruchamiaj jako root!
RUN addgroup -S app && adduser -S app -G app
USER app

WORKDIR /app
COPY --from=builder /build/target/app.jar app.jar

EXPOSE 8080

# Health check wbudowany w obraz
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["java", "-jar", "app.jar"]
docker-compose.yml — na serwerze produkcyjnym
services:
  app:
    image: ghcr.io/user/myapp:latest
    restart: always
    ports:
      - "127.0.0.1:8080:8080" # tylko lokalnie!
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - DB_URL=${DB_URL}        # z .env
      - DB_PASSWORD=${DB_PASSWORD}
    healthcheck:
      test: ["CMD", "wget", "-qO-",
             "http://localhost:8080/health"]
      interval: 30s
      retries: 3
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    restart: always
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_PASSWORD=${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata: # dane przeżywają restart kontenera
Multi-stage build — po co? Etap 1 (builder): pełny JDK + Maven = ~600MB. Etap 2 (runtime): tylko JRE Alpine = ~90MB. Mniejszy obraz = szybszy pull, mniejsza powierzchnia ataku, tańsze przechowywanie. Kopiujesz tylko gotowy JAR, nie cały toolchain.
05 — środowiska
dev → staging → production

Każde środowisko ma inne dane, inne konfiguracje. Nigdy nie testujesz na produkcji. Staging to klon produkcji z prawdziwymi danymi.

🏗️ Środowiska — różnice
LOCAL / DEV
Komputer developera. H2 in-memory lub lokalny Docker. Hot-reload. Debug mode. Dummy dane. Każdy ma swoje — nie kolidują ze sobą.
STAGING / UAT
Identyczny z produkcją (ta sama konfiguracja, podobne dane). Tutaj QA testuje nowe funkcje. Deploy automatyczny po CI. Testowy Stripe/płatności.
PRODUCTION
Prawdziwi użytkownicy. Deploy tylko po zatwierdzeniu lub ręcznie. Logi, monitoring, alerty. Backupy. Nigdy nie debugujesz tu bezpośrednio.
Spring Boot — profile środowisk
# application.yml — bazowy
spring:
  application:
    name: myapp

---
# application-dev.yml

  datasource:

# Lub w GitHub Actions:

06 — bez downtime
Strategie deployment

Jak wdrożyć nową wersję bez przerwy w działaniu serwisu. Każda strategia ma inne ryzyko i koszt.

RYZYKO: HIGH
💥
Big Bang
// stop → deploy → start
Zatrzymaj stary, uruchom nowy. Prosta, ale zawsze jest downtime. OK dla małych projektów w nocy. Rollback trudny.
RYZYKO: MED
🔄
Rolling Deploy
// instancja po instancji
Aktualizujesz instancje jedna po drugiej. Load balancer przekierowuje ruch do już zaktualizowanych. Brak downtime. Przez chwilę działają dwie wersje jednocześnie.
RYZYKO: LOW
🔵🟢
Blue-Green
// dwa identyczne środowiska
Blue = stara produkcja, Green = nowa. Deploy na Green, testuj, przełącz ruch (DNS/LB). Rollback = zmień z powrotem na Blue. Droższe — podwójne zasoby.
RYZYKO: VERY LOW
🐦
Canary
// powolne przełączanie ruchu
Wyślij 5% ruchu na nową wersję, obserwuj metryki. Jeśli OK → 20% → 50% → 100%. Błąd u 5% userów zamiast 100%. Rollback = cofnij do 0%.
FEATURE FLAGS
🏳️
Feature Flags
// kod żyje, feature wyłączona
Nowa funkcja jest w kodzie ale wyłączona flagą. Włączasz dla wybranych userów (beta testers) bez ponownego deployu. Rollback = wyłącz flagę.
Blue-Green na VPS z Docker Compose
#!/bin/bash — deploy.sh

CURRENT=$(docker compose ps --services | head -1)
NEW_IMAGE="ghcr.io/user/app:$1"  # $1 = nowy tag

# Uruchom nową wersję na innym porcie
docker run -d --name app_new \
  -p 127.0.0.1:8081:8080 \
  -e SPRING_PROFILES_ACTIVE=prod \
  $NEW_IMAGE

# Poczekaj na health check
for i in $(seq 1 30); do
  if curl -sf http://localhost:8081/actuator/health; then
    echo "Health check OK!"; break
  fi
  sleep 2
done

# Przełącz nginx na nową instancję
sed -i 's/8080/8081/' /etc/nginx/sites-enabled/app
nginx -s reload

# Zatrzymaj starą instancję
docker stop app_old && docker rm app_old
docker rename app_new app_old
07 — bezpieczeństwo
Zarządzanie sekretami

Hasła, klucze API, certyfikaty — nigdy w kodzie, nigdy w repozytorium. Zawsze w zmiennych środowiskowych lub dedykowanych narzędziach.

GitHub Secrets — jak ustawić i używać
# Dodaj secret w GitHub:
# Settings → Secrets and variables → Actions
# → New repository secret
#
# Nazwa: VPS_HOST  Wartość: 123.45.67.89
# Nazwa: SSH_PRIVATE_KEY  Wartość: (klucz)
# Nazwa: DB_PASSWORD  Wartość: (hasło)

# W workflow — używasz przez ${{ secrets.XXX }}
steps:
  - name: Deploy
    env:
      HOST: ${{ secrets.VPS_HOST }}
      KEY:  ${{ secrets.SSH_PRIVATE_KEY }}
    run: |
      echo "$KEY" > /tmp/key && chmod 600 /tmp/key
      ssh -i /tmp/key user@$HOST "cd /app && ./deploy.sh"

# NIGDY nie rób tego!
password: "moje_haslo123"     # ❌ w kodzie!
DB_PASS=haslo123               # ❌ w .env w repo!

# .gitignore — obowiązkowo:
.env
.env.local
.env.prod
*.pem
*.key
🔑 Klucze SSH do deploymentu
1. Wygeneruj parę kluczy (lokalnie): ssh-keygen -t ed25519 -C "github-ci-deploy" → tworzy: deploy_key (prywatny) → deploy_key.pub (publiczny) 2. Klucz publiczny → na serwer VPS: cat deploy_key.pub >> ~/.ssh/authorized_keys 3. Klucz prywatny → do GitHub Secrets: cat deploy_key | pbcopy GitHub → Settings → Secrets → SSH_PRIVATE_KEY = (wklejasz) 4. W Actions używasz: ${{ secrets.SSH_PRIVATE_KEY }}
🚨 Wyciek sekretów — co robić? Jeśli przez pomyłkę wrzuciłeś secret do repo: 1) Natychmiast zmień/unieważnij klucz/hasło. 2) Wyczyść historię git (git filter-repo). 3) Force-push. 4) Poinformuj zespół. Samo usunięcie pliku z git nie wystarczy — historia git pamięta wszystko.

Powiadomienie na Slack
# W GitHub Actions — na końcu workflow
- name: Notify Slack on success
  if: success()
  uses: slackapi/slack-github-action@v1
  with:
    slack-bot-token: ${{ secrets.SLACK_TOKEN }}
    channel-id: deploys
    slack-message: |
      ✅ Deployed ${{ github.sha }}
      by ${{ github.actor }}
      https://myapp.com

- name: Notify Slack on failure
  if: failure()
  uses: slackapi/slack-github-action@v1
  with:
    slack-bot-token: ${{ secrets.SLACK_TOKEN }}
    channel-id: alerts
    slack-message: |
      ❌ DEPLOY FAILED!
      Branch: ${{ github.ref }}
      See: ${{ github.server_url }}/...
📊 Co monitorować po deploymencie
MetrykaCo obserwujesz
HTTP 5xx rate% błędów serwera — powinno być ~0%
Response timeP95/P99 latencja — czy deploy nie spowolnił
Error rateLogi błędów — nowe exception po deploymencie?
CPU / RAMCzy nowa wersja nie zużywa więcej zasobów
Health endpoint/actuator/health — czy aplikacja żyje
Złota zasada: deploy z możliwością rollbacku w <5 minut Trzymaj poprzedni tag obrazu Docker. W razie problemów: docker compose set image app=old_tag && docker compose up -d. Nie kasuj poprzednich obrazów od razu po deploymencie — poczekaj 24h.
