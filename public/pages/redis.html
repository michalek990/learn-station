Redis — Fundament
Architektura — jak działa Redis
Redis to in-memory data store — wszystkie dane trzyma w RAM. Single-threaded (jeden wątek dla komend), dlatego każda operacja jest atomowa. Latencja sub-milisekundowa. Używany jako cache, message broker, session store, rate limiter, leaderboard.
🏗️ Kluczowe cechy
In-memory
Wszystkie dane w RAM. Latencja <1ms. Rozmiar danych = rozmiar RAM. Persistence opcjonalna (RDB/AOF).
Single-threaded
Jeden wątek obsługuje komendy → automatyczna atomowość. Brak race conditions. I/O multiplexing (epoll) dla obsługi wielu klientów. Od Redis 6.0: I/O threads, ale logic nadal single-thread.
Rich data types
String, Hash, List, Set, Sorted Set, HyperLogLog, Bitmap, Stream, JSON (RedisJSON moduł). Operacje natywne — nie parsuj JSON żeby policzyć elementy.
TTL per key
Każdy klucz może mieć expiry — Redis sam usuwa po czasie. Fundamental dla cache patterns.
Persistence
Opcjonalna — RDB (snapshots) lub AOF (append-only log) lub obie. Możesz wyłączyć dla pure cache.
Replikacja
Master-Replica (async). Redis Sentinel — HA i auto-failover. Redis Cluster — sharding + HA.

Struktury danych
Typy danych — String, Hash, List, Set, Sorted Set i inne
Wybór właściwej struktury danych to fundament efektywnego użycia Redis. Każdy typ ma natywne operacje — nie trzymaj wszystkiego jako JSON String gdy Hash da ci częściowe odczyty i atomowe incrementy.
String, Hash, List — komendy
# ══ STRING — najprostszy typ ══════════════════════════════════
# Tekst, liczby, serializowany JSON, binarny (max 512MB)
SET user:1:name "Jan Kowalski"
SET user:1:name "Jan Kowalski" EX 3600  # z TTL 1h
SET user:1:name "Jan Kowalski" NX       # tylko jeśli NIE istnieje
SET user:1:name "Jan Kowalski" XX       # tylko jeśli istnieje
GET user:1:name
MSET k1 v1 k2 v2 k3 v3              # multi set (atomowy)
MGET k1 k2 k3
GETSET user:1:name "Nowe Imię"       # pobierz starą, ustaw nową
GETDEL user:1:name                   # pobierz i usuń

# Liczniki — atomowe, podstawa rate limiterów
INCR page:views                      # 0 → 1, non-existent → 1
INCRBY page:views 5
DECR stock:product:42
INCRBYFLOAT price:usd 1.50

# ══ HASH — obiekt z polami ════════════════════════════════════
# Idealny dla obiektów (user, session, product)
# Efektywny pamięciowo dla małych hashy (ziplist encoding)
HSET user:1 name "Jan" email "[email protected]" age 30
HGET user:1 name
HMGET user:1 name email              # pobierz kilka pól
HGETALL user:1                       # wszystkie pola + wartości
HKEYS user:1                         # tylko nazwy pól
HVALS user:1                         # tylko wartości
HINCRBY user:1 age 1                 # atomowy increment pola
HSETNX user:1 email "[email protected]"  # set tylko jeśli nie istnieje
HDEL user:1 age
HEXISTS user:1 email                 # 1 / 0
HLEN user:1                          # liczba pól

# Hash vs String z JSON:
# Hash: częściowy odczyt (HGET 1 pole), atomowy increment
# JSON string: read/deserialize/modify/serialize/write — nie atomowe

# ══ LIST — ordered collection, linked list ════════════════════
# Kolejka (LPUSH + RPOP), Stack (LPUSH + LPOP), historia
LPUSH notifications:user:1 "msg1"    # push na lewo (head)
RPUSH notifications:user:1 "msg2"    # push na prawo (tail)
LPOP notifications:user:1            # pop z lewej
RPOP notifications:user:1            # pop z prawej
LRANGE notifications:user:1 0 9    # pierwsze 10 elementów
LRANGE notifications:user:1 0 -1   # wszystkie
LLEN notifications:user:1
LINDEX notifications:user:1 0       # element po indeksie (O(n)!)
LTRIM notifications:user:1 0 99    # zachowaj pierwsze 100

# Blocking pop — consumer pattern (bez pollingu!)
BLPOP queue:jobs 0                  # czekaj w nieskończoność
BLPOP queue:jobs 30                 # czekaj max 30s
Set, Sorted Set, HyperLogLog, Bitmap
# ══ SET — unordered, unique elements ══════════════════════════
# Tagi, permissions, unique visitors, "kto obserwuje kogo"
SADD tags:article:42 "java" "spring" "redis"
SREM tags:article:42 "redis"
SMEMBERS tags:article:42
SISMEMBER tags:article:42 "java"    # O(1)!
SCARD tags:article:42               # liczba elementów

# Operacje na zbiorach
SUNION set1 set2                    # suma zbiorów
SINTER followers:1 followers:2     # część wspólna (wspólni followerzy)
SDIFF set1 set2                    # różnica
SUNIONSTORE dest set1 set2         # wynik zapisz w dest

# ══ SORTED SET (ZSET) — set z score ══════════════════════════
# Leaderboardy, rankingi, priority queues, zakresy czasowe
# Elementy unikalne, sortowane po score (float64)
ZADD leaderboard 1500 "alice"
ZADD leaderboard 2300 "bob"
ZADD leaderboard 1800 "charlie"
ZADD leaderboard NX 2000 "alice"  # tylko jeśli nie istnieje
ZINCRBY leaderboard 100 "alice"   # atomowy increment score
ZSCORE leaderboard "alice"
ZRANK leaderboard "alice"          # pozycja (0-indexed) ASC
ZREVRANK leaderboard "alice"       # pozycja DESC (1. miejsce = 0)
ZRANGE leaderboard 0 -1 WITHSCORES REV  # top N
ZRANGEBYSCORE leaderboard 1000 2000     # zakres score
ZRANGEBYLEX leaderboard "[alice" "[bob" # zakres leksyk.
ZCARD leaderboard
ZREM leaderboard "alice"
# TTL-based expiry pattern: score = timestamp
ZRANGEBYSCORE sessions 0 1700000000   # stare sesje

# ══ HYPERLOGLOG — probabilistyczna kardynalność ═══════════════
# ~1% błąd, stały rozmiar 12KB niezależnie od danych!
PFADD visitors:2024-01-15 "user:1" "user:2" "user:3"
PFCOUNT visitors:2024-01-15         # ile unikalnych użytkowników
PFMERGE visitors:week visitors:mon visitors:tue  # merge HLL

# ══ BITMAP — bit per position ══════════════════════════════════
# Daily active users, feature flags, compact boolean arrays
SETBIT active:2024-01-15 1001 1    # user 1001 był aktywny
GETBIT active:2024-01-15 1001
BITCOUNT active:2024-01-15         # ilu aktywnych userów
BITOP AND active:both active:mon active:tue  # aktywni w oba dni

# ══ Operacje na kluczach ══════════════════════════════════════
EXISTS user:1                       # 1 / 0
DEL user:1 user:2                   # usuń (sync)
UNLINK user:1                       # usuń async (non-blocking)
TYPE user:1                         # string/hash/list/set/zset
RENAME user:1 user:old:1
KEYS user:*                         # ⚠️ NIGDY na prod! Blokuje
SCAN 0 MATCH user:* COUNT 100     # ✅ iteracyjne skanowanie
String — kiedy

Liczniki (INCR), cache pojedynczych wartości, serializowany obiekt (JSON), session token, rate limit counter, feature flag

Max 512MB. Trzymaj mały.

Hash — kiedy

Reprezentacja obiektu (user, product, config) gdy potrzebujesz częściowych odczytów lub atomowych aktualizacji pól.

Efektywny dla <128 pól (ziplist encoding).

List — kiedy

Queue (LPUSH+RPOP), Stack (LPUSH+LPOP), historia (ostatnie N eventów — LTRIM), timeline, sidekiq-style job queue

LINDEX to O(n) — unikaj losowego dostępu.

Sorted Set — kiedy

Leaderboard, rankingi, priority queue, sliding window rate limiter (score = timestamp), scheduled tasks (score = execute_at)

O(log n) dla add/update. Najwszechstronniejszy typ.

Cache Patterns
Wzorce cache — Cache-Aside, Write-Through, Read-Through
Redis jako cache to najczęstszy use case. Wzorce różnią się tym kto zarządza spójnością między cache a bazą danych. Wybór wzorca wpływa na spójność, złożoność i wydajność.
Wzorce cache — implementacja Java
// ══ Cache-Aside (Lazy Loading) — najczęstszy wzorzec ══════════
// Aplikacja zarządza cache samodzielnie
// Read: cache miss → DB → zapisz do cache
// Write: uaktualnij DB → unieważnij/uaktualnij cache
public Product getProduct(long id) {
    String key = "product:" + id;

    // 1. Sprawdź cache
    String cached = redis.get(key);
    if (cached != null) {
        return objectMapper.readValue(cached, Product.class);
    }

    // 2. Cache miss → pobierz z DB
    Product product = productRepo.findById(id)
        .orElseThrow();

    // 3. Zapisz do cache z TTL
    redis.setex(key, 3600, objectMapper.writeValueAsString(product));

    return product;
}

public void updateProduct(Product product) {
    productRepo.save(product);
    redis.del("product:" + product.getId());  // invalidate!
}

// ══ Write-Through — cache zawsze aktualny ═════════════════════
// Write do cache I DB jednocześnie (synchronicznie)
// Zaleta: cache zawsze spójny. Wada: wyższy write latency
public void saveProduct(Product product) {
    productRepo.save(product);  // zapis do DB
    redis.setex("product:" + product.getId(),
                 3600,
                 objectMapper.writeValueAsString(product));
}

// ══ Write-Behind (Write-Back) — async DB write ════════════════
// Zapis tylko do cache, asynchronicznie do DB
// Zaleta: super szybki write. Wada: ryzyko utraty danych
public void updateCounter(long userId) {
    redis.hincrby("user:" + userId, "pageViews", 1);
    // Scheduled job co 5 min: flush do DB
}

// ══ Cache Stampede (dog-piling) — problem i rozwiązanie ══════
// Problem: wiele żądań trafia na cache miss jednocześnie → DB overwhelmed
// Rozwiązanie: distributed lock / probabilistic early recomputation
public Product getProductSafe(long id) {
    String key = "product:" + id;
    String lockKey = "lock:product:" + id;

    String cached = redis.get(key);
    if (cached != null) return deserialize(cached);

    // Tylko jeden wątek odświeża cache — reszta czeka lub zwraca stale
    Boolean lock = redis.setnx(lockKey, "1");
    if (lock != null && lock) {
        try {
            redis.expire(lockKey, 10);
            Product p = productRepo.findById(id).orElseThrow();
            redis.setex(key, 3600, serialize(p));
            return p;
        } finally {
            redis.del(lockKey);
        }
    }
    // Inne wątki: krótkie oczekiwanie lub zwróć stale value
    Thread.sleep(50);
    return getProductSafe(id); // retry
}

Write-ThroughWysoka (zawsze hit)Wyższa (2 zapisy)✅ Silna

Cache Invalidation — najtrudniejszy problem "There are only two hard things in CS: cache invalidation and naming things." — Phil Karlton

Strategie:
🔴 Delete on write — usuń klucz przy każdej zmianie. Prosto, bezpieczne. Następny read odświeży.
🟡 Update on write — zaktualizuj wartość. Ryzyko race condition przy concurrent writes.
🟢 TTL only — polegaj tylko na wygaśnięciu. Proste, akceptowalna nieaktualność (eventual consistency).
Klucze Redis — konwencja nazewnictwa Używaj dwukropka jako separatora hierarchii:
user:1001:profile · product:42:stock · session:abc123 · rate:ip:192.168.1.1
Unikaj spacji i specjalnych znaków. Trzymaj klucze krótkie — każdy bajt kosztuje w RAM. Prefiksy ułatwiają SCAN i organizację.

TTL & Eviction
Expiry, TTL i polityki eksmisji kluczy
Redis może automatycznie usuwać klucze po czasie (TTL) oraz eksmitować klucze gdy pamięć jest pełna (eviction policy). Dobra konfiguracja chroni przed OOM i zapewnia świeżość danych.
TTL — komendy
# Ustawianie TTL
SET session:abc "data" EX 3600      # sekundy
SET session:abc "data" PX 3600000   # milisekundy
SET session:abc "data" EXAT 1700000000  # unix timestamp
EXPIRE session:abc 3600             # ustaw/zmień TTL
PEXPIRE session:abc 3600000         # ms precision
EXPIREAT session:abc 1700000000     # absolute timestamp

# Sprawdzanie TTL
TTL session:abc     # sekundy pozostałe (-1=no expiry, -2=not exists)
PTTL session:abc    # ms precision

# Usuwanie TTL (uczyń permanent)
PERSIST session:abc

# Typowe TTL per use case:
# Session token:      1800s  (30 min inaktywności)
# Product cache:      3600s  (1h)
# Rate limit window:  60s    (per minute)
# Email verification: 86400s (24h)
# Hot data cache:     300s   (5 min)
# Config cache:       600s   (10 min)
🗑️ Eviction Policies

Lazy vs Active Expiry Redis używa dwóch mechanizmów usuwania wygasłych kluczy:

Lazy expiry — klucz usuwany dopiero gdy ktoś próbuje go odczytać. Szybkie, ale klucze mogą zajmować pamięć długo po wygaśnięciu.

Active expiry — Redis co 100ms losowo sprawdza 20 kluczy z TTL i usuwa wygasłe. Jeśli >25% jest wygasłe, powtarza. Nigdy nie sprawdza wszystkich naraz — nie blokuje.
Rate Limiter — dwa wzorce
// ══ Fixed Window — prosty ════════════════════════════════════
public boolean allowRequest(String userId) {
    String key = "rate:" + userId + ":"
               + (System.currentTimeMillis() / 60000); // minuta
    long count = redis.incr(key);
    if (count == 1) redis.expire(key, 60);  // ustaw TTL przy pierwszym
    return count <= 100;  // max 100 req/min
}

// ══ Sliding Window — Sorted Set (dokładniejszy) ═══════════════
public boolean allowRequestSlidingWindow(String userId) {
    String key = "rate:sliding:" + userId;
    long now = System.currentTimeMillis();
    long windowStart = now - 60_000; // ostatnia minuta

    // Pipeline dla atomowości
    redis.multi();
    redis.zremrangebyscore(key, 0, windowStart); // usuń stare
    redis.zadd(key, now, now + "-" + UUID.randomUUID()); // dodaj request
    redis.expire(key, 70);
    List<?> results = redis.exec();

    long count = redis.zcard(key);
    return count <= 100;
}

Pub/Sub & Streams
Pub/Sub i Redis Streams — messaging w Redis
Pub/Sub — fire-and-forget, brak persistence, subskrybenci muszą być online. Streams — persystentny log z consumer groups (jak uproszczona Kafka w Redis). Wybieraj Streams gdy potrzebujesz durability lub przetwarzania offline.
Pub/Sub i Streams — komendy
# ══ PUB/SUB — fire and forget ════════════════════════════════
# Publisher:
PUBLISH notifications:user:1001 '{"type":"order","id":42}'
PUBLISH channel:* "broadcast"  # pattern publish nie istnieje

# Subscriber (blokuje połączenie!):
SUBSCRIBE notifications:user:1001
SUBSCRIBE channel1 channel2          # wiele kanałów
PSUBSCRIBE notifications:user:*      # pattern glob
UNSUBSCRIBE notifications:user:1001
PUBSUB CHANNELS notifications:*      # aktywne kanały
PUBSUB NUMSUB channel1               # liczba subskrybentów

# ⚠️ Pub/Sub: wiadomość ginie jeśli nikt nie słucha!
# ⚠️ Wolny subscriber blokuje publishera (back-pressure)

# ══ STREAMS — persystentny log ═══════════════════════════════
# Dodaj wiadomość (auto ID = timestamp-sequence)
XADD orders * \
  orderId 42 \
  userId 1001 \
  amount "99.99"
# Zwraca ID: "1700000000123-0"

# Odczyt od początku
XRANGE orders - +              # wszystkie wiadomości
XRANGE orders - + COUNT 10   # pierwsze 10
XREVRANGE orders + -           # od końca
XLEN orders                    # liczba wiadomości

# ══ Consumer Groups — guaranteed delivery ════════════════════
XGROUP CREATE orders order-processors $ MKSTREAM
# $ = tylko nowe wiadomości, 0 = od początku

# Consumer czyta (nie-acknowledged = pending)
XREADGROUP GROUP order-processors consumer-1 \
  COUNT 10 BLOCK 0 \
  STREAMS orders >
# > = tylko nieprzypisane do grupy wiadomości

# Potwierdź przetworzenie
XACK orders order-processors 1700000000123-0

# Sprawdź pending (niepotwierdzonych)
XPENDING orders order-processors - + 10

# Claim wiadomości od umarłego consumera (po 30s)
XAUTOCLAIM orders order-processors consumer-2 \
  30000 0-0 COUNT 10

# Trim stream (max 10000 wiadomości)
XTRIM orders MAXLEN ~ 10000  # ~ = approximate (szybsze)

Kiedy Streams zamiast Pub/Sub Używaj Streams gdy: potrzebujesz gwarancji dostarczenia, consumer może być offline, chcesz consumer groups (load balancing), potrzebujesz reprocessing.

Używaj Pub/Sub gdy: "co najmniej raz" jest OK, wszyscy subskrybenci są online, prosty broadcast (powiadomienia real-time, cache invalidation w klastrze).

Transakcje & Lua
MULTI/EXEC, WATCH, Pipeline i skrypty Lua
Redis transakcje (MULTI/EXEC) są atomowe ale nie rollbackują przy błędach logicznych. WATCH implementuje optimistic locking. Lua scripts to najpotężniejszy mechanizm — atomowe złożone operacje wykonywane server-side.
MULTI/EXEC, WATCH, Pipeline, Lua
# ══ MULTI/EXEC — transakcja ═══════════════════════════════════
MULTI
SET balance:user:1 900
SET balance:user:2 1100
INCR transfers:count
EXEC
# Wszystkie komendy wykonują się atomowo (nie ma interleaving)
# Uwaga: błąd w jednej komendzie NIE rollbackuje pozostałych!
# DISCARD - anuluj transakcję

# ══ WATCH — optimistic locking ════════════════════════════════
# WATCH monitoruje klucz — jeśli zmieniony przed EXEC → EXEC zwróci nil
WATCH balance:user:1
val = GET balance:user:1    # odczytaj aktualną wartość
MULTI
SET balance:user:1 (val - 100)
EXEC   # null jeśli balance:user:1 zmienił się między WATCH a EXEC
# Wzorzec: retry pętla dopóki EXEC się powiedzie

# ══ PIPELINE — batching (NIE transakcja) ══════════════════════
# Wysyła wiele komend naraz, zmniejsza RTT
# Komendy NIE są atomowe (może być interleaving z innymi klientami)
# W Jedis/Lettuce: pipeline { ... } → flush → receive responses

# ══ Lua Script — atomowe złożone operacje ════════════════════
# Cały skrypt wykonuje się atomowo (single-threaded Redis)
# KEYS[1], KEYS[2] — klucze | ARGV[1], ARGV[2] — argumenty

# Przykład: Transfer sald (atomowy check-and-transfer)
EVAL "
local from = KEYS[1]
local to   = KEYS[2]
local amount = tonumber(ARGV[1])

local balance = tonumber(redis.call('GET', from))
if balance == nil or balance < amount then
  return redis.error_reply('INSUFFICIENT_FUNDS')
end
redis.call('DECRBY', from, amount)
redis.call('INCRBY', to,   amount)
return redis.call('GET', from)
" 2 balance:user:1 balance:user:2 100

# Rejestracja skryptu (sha1 hash → szybsze wywołanie)
SCRIPT LOAD "return redis.call('GET', KEYS[1])"
# Zwraca SHA1 hash skryptu
EVALSHA abc123sha1hash... 1 mykey

# Distributed Lock (Redlock pattern — prosty wariant)
# SET lock:resource unique-value NX PX 30000
# Lua do bezpiecznego release (tylko własny lock!):
EVAL "
if redis.call('GET', KEYS[1]) == ARGV[1] then
  return redis.call('DEL', KEYS[1])
else
  return 0
end
" 1 lock:resource my-unique-uuid
🔒 Distributed Lock — Redlock

Prosty lock (1 Redis):

SET lock:res uuid NX PX 30000 — ustaw tylko jeśli nie istnieje, TTL = auto-release

Release: Lua script (sprawdź uuid przed DEL)

Redlock (N=5 Redis instances):

Próbuj ustawić lock na każdej z N instancji. Lock acquired = >N/2 sukcesów w czasie < lock_ttl / 10. Bezpieczniejszy przy failover.

⚠️ Gotowe biblioteki: Redisson (Java) ma RLock z Redlock implementacją.

⚡ Pipeline vs MULTI vs Lua
AtomowośćRTTWarunek

Lua✅ silna1 RTTDowolny

Lua = najlepsza opcja dla złożonej logiki. Pipeline = bulk reads/writes bez atomowości.

Persistence
RDB, AOF i hybrydowe — trwałość danych
Redis może persystować dane na dysk na dwa sposoby: RDB (snapshots w określonych interwałach) i AOF (append-only log każdej operacji zapisu). Dla cache możesz wyłączyć persistence całkowicie.
💾 RDB vs AOF
CechaRDB (snapshot)AOF (log)
MechanizmBinarny snapshot całego datasetuLog każdej operacji zapisu
Utrata danychDo ostatniego snapshot (minuty)appendfsync always = 0
everysec = max 1s
Rozmiar plikuMały (kompaktowy)Duży (rośnie ciągle)
Restart speedSzybkiWolniejszy (replay log)
Wpływ na wydajnośćfork() co N minutfsync + rewrite background
RekomendacjaBackup, cache z tolerancją utratyProdukcja z durability
redis.conf — persistence
# ══ RDB — snapshots ══════════════════════════════════════════
# save <seconds> <changes>
save 900 1     # po 900s jeśli >= 1 zmiana
save 300 10    # po 300s jeśli >= 10 zmian
save 60 10000  # po 60s jeśli >= 10000 zmian
# save "" — wyłącz RDB

dbfilename dump.rdb
dir /var/lib/redis

# Ręczny snapshot:
# BGSAVE  — async (fork child)
# SAVE    — sync (blokuje!) — nigdy na prod
# LASTSAVE — timestamp ostatniego snapshota

# ══ AOF — append only file ════════════════════════════════════
appendonly yes
appendfilename "appendonly.aof"

# appendfsync always  — fsync po każdej operacji (durabilność max, wolno)
# appendfsync everysec — fsync co sekundę (rekomendowane ✅)
# appendfsync no       — OS decyduje (szybko, ryzyko utraty)
appendfsync everysec

# AOF rewrite — kompresja (usuń nadmiarowe operacje)
auto-aof-rewrite-percentage 100   # rewrite gdy AOF 2x większy niż baseline
auto-aof-rewrite-min-size 64mb    # minimum rozmiar do rewrite
# BGREWRITEAOF — ręczny rewrite

# ══ Hybrid mode (Redis 4+) — najlepsze z obu ════════════════
# AOF zawiera RDB snapshot + delta AOF od tamtego momentu
# Szybki restart + pełna durability
aof-use-rdb-preamble yes          # rekomendowane na produkcji ✅

# ══ Dla pure cache — wyłącz wszystko ═════════════════════════
# save ""
# appendonly no

Replikacja & HA
Master-Replica, Sentinel i Redis Cluster
Redis oferuje trzy poziomy HA: replikacja (master + repliki read-only), Sentinel (auto-failover bez shardingu), Cluster (sharding + HA, 16384 slotów). Wybór zależy od wymagań na skalę i złożoność.
🏗️ Tryby wdrożenia
Standalone
Jeden serwer. Dev/testy. Brak HA. Prosty setup.
Master-Replica
Async replikacja. Repliki = read-only. Skalowanie odczytów. Brak auto-failover — manualny promote repliki przy awarii mastera.
Sentinel
HA dla jednej instancji. Monitoruje master, automatyczny failover (promote repliki). Min 3 Sentinele (quorum). Brak shardingu — wszystko na jednym nodzie. Rekomendowany dla większości aplikacji.
Redis Cluster
Sharding + HA. 16384 hash slotów rozdzielonych między master nody. Każdy master ma repliki. Min 3 masterów (6 nodów). Automatyczny failover. Niektóre operacje (multi-key) wymagają hash tags.
Managed Cloud
AWS ElastiCache, GCP Memorystore, Azure Cache for Redis — Sentinel lub Cluster bez ręcznego zarządzania. Rekomendowane na chmurze.
Replikacja + Cluster — konfiguracja
# ══ Replika — redis.conf ══════════════════════════════════════
replicaof master-host 6379
masterauth yourpassword
replica-read-only yes              # repliki są read-only (domyślnie)
replica-lazy-flush yes             # async flush przy resync

# Sprawdź replikację:
INFO replication
# master_replid, master_repl_offset, connected_slaves
# slave0: ip=...,port=...,state=online,offset=...,lag=0

# ══ Redis Cluster — hash slots ════════════════════════════════
# Klucz → CRC16(key) % 16384 → slot → node
# Hash tags: {user}.orders i {user}.profile → ten sam slot!
MSET {user:1}.name "Jan" {user:1}.age "30"  # OK — ten sam slot
MSET user:1:name "Jan" user:2:age "25"     # ❌ różne sloty!

# Sprawdź slot dla klucza:
CLUSTER KEYSLOT mykey

# redis.conf dla Cluster node:
cluster-enabled yes
cluster-config-file nodes.conf
cluster-node-timeout 5000
cluster-require-full-coverage no   # działaj nawet gdy niektóre sloty offline

# Tworzenie klastra (redis-cli):
redis-cli --cluster create \
  host1:7001 host2:7002 host3:7003 \
  host4:7004 host5:7005 host6:7006 \
  --cluster-replicas 1

CLUSTER INFO
CLUSTER NODES
Sentinel vs Cluster — kiedy co Sentinel — dane mieszczą się w RAM jednego serwera, potrzebujesz tylko HA bez shardingu. Prostszy, mniej problemów z multi-key operacjami.

Cluster — dane nie mieszczą się w jednym nodzie lub potrzebujesz bardzo wysokiego throughput rozłożonego na wiele nodów. Uwaga na ograniczenia: brak cross-slot transakcji, Lua musi używać hash tags.

Spring Data Redis
Spring Boot + Redis — cache, session, RedisTemplate
Spring Boot auto-konfiguruje Redis przez spring-boot-starter-data-redis. RedisTemplate — niskopoziomowe API. @Cacheable — deklaratywny cache. Spring Session — sesje HTTP w Redis (skalowalne, sticky sessions niepotrzebne).
Spring Boot — konfiguracja i użycie
/* ── application.yml ──────────────────────────────────────── */
spring:
  data:
    redis:
      host: localhost
      port: 6379
      password: "${REDIS_PASSWORD}"
      timeout: 2000ms
      lettuce:
        pool:
          max-active: 20      # max connections
          max-idle: 10
          min-idle: 5
          max-wait: 1000ms
        # Dla Sentinel:
        # sentinel:
        #   master: mymaster
        #   nodes: sentinel1:26379,sentinel2:26379
        # Dla Cluster:
        # cluster:
        #   nodes: host1:7001,host2:7002,host3:7003
  cache:
    type: redis
    redis:
      time-to-live: 3600000   # 1h w ms (domyślny TTL)
      key-prefix: "myapp:"
      use-key-prefix: true
      cache-null-values: false

/* ── RedisConfig.java ─────────────────────────────────────── */
@Configuration
@EnableCaching
public class RedisConfig {

  @Bean
  public RedisTemplate<String, Object> redisTemplate(
      RedisConnectionFactory factory) {
    RedisTemplate<String, Object> tpl = new RedisTemplate<>();
    tpl.setConnectionFactory(factory);
    tpl.setKeySerializer(new StringRedisSerializer());
    tpl.setValueSerializer(new GenericJackson2JsonRedisSerializer());
    tpl.setHashKeySerializer(new StringRedisSerializer());
    tpl.setHashValueSerializer(new GenericJackson2JsonRedisSerializer());
    return tpl;
  }

  // Per-cache TTL konfiguracja
  @Bean
  public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration defaults = RedisCacheConfiguration
        .defaultCacheConfig()
        .entryTtl(Duration.ofMinutes(60))
        .disableCachingNullValues()
        .serializeValuesWith(RedisSerializationContext.SerializationPair
            .fromSerializer(new GenericJackson2JsonRedisSerializer()));

    return RedisCacheManager.builder(factory)
        .cacheDefaults(defaults)
        .withCacheConfiguration("products",
            defaults.entryTtl(Duration.ofMinutes(10)))
        .withCacheConfiguration("sessions",
            defaults.entryTtl(Duration.ofMinutes(30)))
        .build();
  }
}
@Cacheable + RedisTemplate + Spring Session
// ══ @Cacheable — deklaratywny cache ══════════════════════════
@Service
public class ProductService {

  // Cachuje wynik pod kluczem "products::42"
  @Cacheable(value = "products", key = "#id")
  public Product getProduct(long id) {
    return repo.findById(id).orElseThrow();
  }

  // Zawsze wywołaj metodę i zaktualizuj cache
  @CachePut(value = "products", key = "#result.id")
  public Product updateProduct(Product p) {
    return repo.save(p);
  }

  // Usuń z cache
  @CacheEvict(value = "products", key = "#id")
  public void deleteProduct(long id) {
    repo.deleteById(id);
  }

  // Warunek cache — tylko gdy status == ACTIVE
  @Cacheable(value = "products", key = "#id",
             condition = "#result.status == 'ACTIVE'")
  public Product getActiveProduct(long id) { ... }
}

// ══ RedisTemplate — niskopoziomowe operacje ═══════════════════
@Autowired
private RedisTemplate<String, Object> redis;

// String ops
redis.opsForValue().set("key", "value", Duration.ofHours(1));
Object val = redis.opsForValue().get("key");

// Hash ops
redis.opsForHash().put("user:1", "name", "Jan");
redis.opsForHash().entries("user:1"); // Map

// ZSet ops (Leaderboard)
redis.opsForZSet().add("leaderboard", "alice", 1500.0);
redis.opsForZSet().reverseRangeWithScores("leaderboard", 0, 9);

// Pipeline — batch ops
redis.executePipelined((RedisCallback<?>) connection -> {
    for (int i = 0; i < 100; i++) {
        connection.stringCommands().set(
            ("key:"+i).getBytes(), "val".getBytes());
    }
    return null;
});

// Adnotacja na klasie config:
@EnableRedisIndexedHttpSession(maxInactiveIntervalInSeconds = 1800)
// Automatycznie: HttpSession → Redis. Klucz: spring:session:sessions:*
// Efekt: stateless serwery, horizontal scaling bez sticky sessions

Senior z Redis: Wiesz że Redis to nie tylko cache — to szwajcarski scyzoryk: rate limiter, distributed lock, leaderboard, session store, job queue, pub/sub broker. Zawsze ustawiasz TTL i maxmemory-policy — bez tego Redis rozsadzi serwer. Single-threaded = atomowość komend, ale KEYS * na produkcji blokuje cały serwer (używasz SCAN). Lua script zamiast WATCH+retry dla złożonej logiki. Dla HA: Sentinel na jedną instancję, Cluster gdy dane nie mieszczą się w jednym serwerze. I pamiętasz: Redis trzyma dane w RAM — jeśli serwer padnie bez persistence, tracisz wszystko.
