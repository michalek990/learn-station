Fundamenty Systemów Rozproszonych
CAP Theorem, BASE, Fallacies of Distributed Computing
Zanim zbudujesz rozproszony system — zrozum fundamenty. CAP Theorem definiuje ograniczenia każdego systemu sieciowego. 8 Fallacies to najczęstsze błędne założenia które prowadzą do katastrof produkcyjnych.
🔺 CAP Theorem

W systemie rozproszonym możesz zagwarantować maksymalnie 2 z 3:

C — Consistency

Każdy odczyt widzi najnowszy zapis lub błąd. Wszyscy klienci widzą te same dane w tym samym czasie.

A — Availability

Każde żądanie otrzymuje odpowiedź (nie błąd), ale niekoniecznie najnowsze dane.

P — Partition Tolerance

System działa mimo utraty/opóźnienia komunikacji między węzłami. W sieci jest ZAWSZE możliwa.

Praktycznie: P jest obowiązkowe → wybierasz CP lub AP.

⚖️ CP vs AP — przykłady
CP — bazy
PostgreSQL, MySQL, MongoDB (domyślnie), ZooKeeper, etcd, Consul — spójność ponad dostępność
AP — bazy
Cassandra, CouchDB, DynamoDB, DNS — dostępność ponad spójność. Eventual consistency.
ACID
Atomicity, Consistency, Isolation, Durability — gwarancje transakcji RDBMS. Trudne w dist. systemach.
BASE
Basically Available, Soft state, Eventually consistent — alternatywa dla ACID w NoSQL.
PACELC
Rozszerzenie CAP: przy braku partycji → Latency vs Consistency. Ważne w praktyce!
💀 8 Fallacies — pułapki

Błędne założenia o sieciach (Deutsch, Gosling, 1994-1997):

2.

3.
Przepustowość jest nieskończona
4.
Sieć jest bezpieczna
5.
Topologia sieci się nie zmienia
6.
Jest jeden administrator
7.
Koszt transportu wynosi 0
8.
Sieć jest jednorodna
Idempotency — kluczowe pojęcie W systemach rozproszonych żądanie może być dostarczone więcej niż raz (sieć nie jest niezawodna). Operacja idempotentna — wielokrotne wykonanie daje ten sam wynik co jednokrotne. HTTP: GET, PUT, DELETE są idempotentne. POST nie jest. Zawsze projektuj API z myślą o idempotencji — użyj Idempotency-Key w nagłówku dla operacji finansowych.

Komunikacja między serwisami
REST vs gRPC vs Message Queue — kiedy co wybrać
Trzy główne paradygmaty komunikacji w mikroserwisach. Synchroniczna (REST, gRPC) vs Asynchroniczna (MQ, Kafka). Wybór zależy od wymagań: latencja, coupling, throughput, failure isolation.
🌐 REST / HTTP
Protokół
HTTP/1.1, HTTP/2. JSON lub XML. Human-readable.
Zalety
Powszechny, łatwy debug, fire-and-forget, stateless, cache-able (GET).
Wady
Loose contract (OpenAPI pomaga), narzut JSON parsowania, HTTP/1.1 no multiplexing.
Kiedy
Public API, browser clients, niska częstotliwość, proste integracje, CRUD.
HTTP Metody
GET (read), POST (create), PUT/PATCH (update), DELETE. Status codes: 2xx/4xx/5xx.
⚡ gRPC
Protokół
HTTP/2. Protobuf (binary) — 3-10x szybszy niż JSON.
Zalety
Silny kontrakt (.proto file), streaming (bidirectional!), multiplexing, code generation dla wielu języków.
Wady
Trudny debug (binary), słaba browserowa obsługa, potrzeba Protobuf toolchain.
Kiedy
Internal service-to-service, high throughput, multi-language (polyglot), real-time streaming.
Typy RPC
Unary, Server streaming, Client streaming, Bidirectional streaming.
📨 Message Queue / Async
Technologie
Apache Kafka, RabbitMQ, AWS SQS/SNS, ActiveMQ, Azure Service Bus.
Zalety
Temporal decoupling, buffer na obciążenie, retry + dead letter, fan-out, event sourcing.
Wady
Eventual consistency, trudniejszy debugging, ordering gwarancje zależne od implementacji.
Kiedy
Long-running operations, email/notifications, event-driven arch., inter-BC communication, load leveling.

// ── orders.proto — kontrakt ───────────────────────────────────
// syntax = "proto3";
// package orders;
//
// service OrderService {
//   rpc PlaceOrder (PlaceOrderRequest) returns (OrderResponse);
//   rpc StreamOrders (CustomerId) returns (stream OrderEvent);
// }
// message PlaceOrderRequest {
//   string customer_id = 1;
//   repeated OrderItem items = 2;
// }

// ── Java implementacja serwera ────────────────────────────────
@GrpcService
public class OrderGrpcService extends OrderServiceGrpc.OrderServiceImplBase {

  @Override
  public void placeOrder(PlaceOrderRequest request,
      StreamObserver<OrderResponse> responseObserver) {
    // protobuf → domain
    Order order = mapToDomain(request);
    OrderId id = orderService.placeOrder(order);
    // domain → protobuf
    OrderResponse response = OrderResponse.newBuilder()
        .setOrderId(id.getValue())
        .setStatus("PLACED")
        .build();
    responseObserver.onNext(response);
    responseObserver.onCompleted();
  }
}

Apache Kafka
Kafka — distributed log, core concepts i Spring integration
Apache Kafka to rozproszony, persystentny log — nie kolejka! Wiadomości są trzymane i mogą być odczytane wielokrotnie przez różnych konsumentów. Zaprojektowany dla milionów wiadomości/sekundę.
🗂️ Kafka — fundamentalne pojęcia
Topic
Kanał komunikacji — "orders", "payments". Producer pisze do topicu, Consumer czyta.
Partition
Topic = N partycji. Ordering gwarantowany TYLKO w obrębie jednej partycji. Klucz → partycja.
Offset
Numer wiadomości w partycji — Consumer commituje offset po przetworzeniu.
Consumer Group
Partycje rozdzielane między instancje w grupie — load balancing. Każda partycja → jeden consumer w grupie.
Retention
Wiadomości nie znikają po odczytaniu! Domyślnie 7 dni lub max rozmiar. Event replay możliwy.
Replication Factor
Kopia partycji na N brokerach. RF=3 → wytrzymuje 2 awarie brokerów.
ISR
In-Sync Replicas — repliki zgodne z liderem. min.insync.replicas=2 → durability.
acks
0=fire&forget, 1=leader ack, all=wszystkie ISR. acks=all + RF=3 = bezpieczne.
Spring Kafka — Producer i Consumer
// ── application.yml ───────────────────────────────────────────
// spring:
//   kafka:
//     bootstrap-servers: kafka:9092
//     producer:
//       key-serializer: org.apache.kafka.common.serialization.StringSerializer
//       value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
//       acks: all
//       retries: 3
//     consumer:
//       group-id: order-service
//       auto-offset-reset: earliest
//       enable-auto-commit: false  ← ważne! manual commit

// ── Producer ──────────────────────────────────────────────────
@Service
public class OrderEventProducer {
  private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

  public void publishOrderPlaced(Order order) {
    OrderPlacedEvent event = new OrderPlacedEvent(
        order.getId().getValue(),
        order.getCustomerId().getValue(),
        order.getTotal()
    );
    // key = orderId → ta sama partycja dla jednego zamówienia
    kafkaTemplate.send("orders.placed", event.orderId(), event)
        .whenComplete((result, ex) -> {
          if (ex != null) log.error("Kafka send failed", ex);
          else log.info("Sent to partition {}",
               result.getRecordMetadata().partition());
        });
  }
}

// ── Consumer ──────────────────────────────────────────────────
@Component
public class PaymentEventConsumer {

  @KafkaListener(topics = "orders.placed",
                  groupId = "payment-service")
  public void handleOrderPlaced(
      @Payload OrderPlacedEvent event,
      @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
      Acknowledgment ack) {

    try {
      paymentService.processPayment(event);
      ack.acknowledge(); // commit offset po sukcesie
    } catch (RetryableException e) {
      // nie commit → retry
      throw e;
    } catch (Exception e) {
      ack.acknowledge(); // commit + wyślij do DLQ
      deadLetterProducer.send(event, e);
    }
  }
}

// ── Dead Letter Queue setup ────────────────────────────────────
@Bean
public DefaultErrorHandler errorHandler(DeadLetterPublishingRecoverer recoverer) {
  return new DefaultErrorHandler(recoverer,
      new FixedBackOff(1000L, 3)); // 3 retry co 1s
}
Partycje i skalowanie partitions = max_consumers_in_group. Więcej partycji = więcej równoległości, ale większy overhead dla brokerów. Nie możesz zmniejszyć liczby partycji — planuj z zapasem.
Exactly-once Semantics Kafka 0.11+: enable.idempotence=true + transactional.id. Używaj przy operacjach finansowych. Duży overhead — rozważ at-least-once + idempotent consumer.
Consumer Lag Opóźnienie konsumenta względem producenta. Monitoruj przez kafka-consumer-groups.sh lub metryki. Duży lag = problem z wydajnością lub awaria konsumenta.
Kafka vs RabbitMQ Kafka = log, retention, replay, high throughput, ordering. RabbitMQ = routing (exchanges), ttl per message, priorytet, lepszy do task queues. Kafka dla event streaming, Rabbit dla task distribution.

Spójność i Transakcje Rozproszone
Distributed Transactions — 2PC, Outbox Pattern, Eventual Consistency
Transakcja obejmująca kilka serwisów to jeden z najtrudniejszych problemów w distributed systems. Klasyczne 2PC jest zbyt wolne i podatne na błędy. Nowoczesne podejście: Outbox Pattern + Saga.
🔄 Modele spójności
Strong Consistency
Każdy odczyt widzi ostatni zapis. Wymaga koordynacji (kosztowne). RDBMS w single-node.
Eventual Consistency
Wszystkie węzły dojdą do tego samego stanu w końcu. Cassandra, DynamoDB, DNS.
Causal Consistency
Operacje powiązane przyczynowo są widoczne w kolejności. Kompromis między strong a eventual.
Monotonic Read
Nigdy nie cofasz się do starszego stanu. Raz widziałeś rekord v2 — nie zobaczysz v1.
2PC (Two Phase Commit)
Prepare + Commit. Koordynator zbiera gotowość wszystkich uczestników, potem commituje lub rollback. Problem: koordynator może failować w środku. Używaj tylko gdy musisz (legacy).
Outbox Pattern
Rozwiązanie! Zapisz event do tabeli outbox w tej samej transakcji co dane. Osobny process publikuje do Kafki.
Transactional Outbox Pattern
// Problem: jak atomowo zapisać Order + opublikować event?
// Zapis do DB i wysyłka do Kafki nie są w jednej transakcji!
// Rozwiązanie: Outbox — event idzie do tabeli w tej samej tx

// ── Tabela outbox (SQL) ───────────────────────────────────────
// CREATE TABLE outbox_events (
//   id           UUID PRIMARY KEY,
//   aggregate_id VARCHAR(255) NOT NULL,
//   event_type   VARCHAR(255) NOT NULL,
//   payload      JSONB NOT NULL,
//   created_at   TIMESTAMPTZ DEFAULT NOW(),
//   published    BOOLEAN DEFAULT FALSE
// );

// ── Zapis w jednej transakcji ────────────────────────────────
@Service
public class PlaceOrderService {
  private final OrderRepository orderRepo;
  private final OutboxRepository outboxRepo;

  @Transactional
  public OrderId placeOrder(PlaceOrderCommand cmd) {
    Order order = Order.create(cmd);
    orderRepo.save(order); // zapisz dane

    // zapisz event w tej samej transakcji!
    OutboxEvent event = OutboxEvent.builder()
        .id(UUID.randomUUID())
        .aggregateId(order.getId().getValue())
        .eventType("OrderPlaced")
        .payload(objectMapper.writeValueAsString(order))
        .build();
    outboxRepo.save(event);

    return order.getId();
    // commit → oba zapisy atomowe!
  }
}

// ── Outbox Relay (osobny serwis/wątek) ───────────────────────
@Scheduled(fixedDelay = 1000)
public void publishPendingEvents() {
  outboxRepo.findUnpublished(Pageable.ofSize(100))
      .forEach(event -> {
        kafkaTemplate.send(event.getEventType(), event.getPayload());
        outboxRepo.markPublished(event.getId());
      });
}

Transakcje Rozproszone
Saga Pattern — Choreography vs Orchestration
Saga to sekwencja lokalnych transakcji z kompensującymi akcjami w razie błędu. Dwie implementacje: Choreography (eventy, luźne powiązanie) i Orchestration (centralny koordynator, łatwiejszy monitoring).
🎭 Saga Choreography vs Orchestration
Choreography — tańczą razem bez dyrygenta

Każdy serwis reaguje na event i publikuje następny. Luźne powiązanie, łatwe dodawanie uczestników. Wada: trudny debuging (śledzenie przez wiele eventów), ryzyko cykli.

OrderPlaced → PaymentService (PaymentProcessed) → InventoryService (ItemsReserved) → ShippingService (ShipmentScheduled)

Orchestration — dyrygent zarządza

Centralny Saga Orchestrator wydaje polecenia serwis po serwisie. Łatwiejszy monitoring i rollback. Wada: single point of logic, coupling do orchestratora.

OrderSaga → placeOrder → processPayment → reserveItems → scheduleShipping → DONE

ChoreographyOrchestration
CouplingLuźneŚcisłe (do saga)
DebugowanieTrudneŁatwe
TestowanieTrudniejszeProstsze
Dodawanie krokówŁatweZmiana saga
Saga Orchestration — implementacja
// Saga Orchestrator zarządza krokami i kompensacjami

@Component
public class PlaceOrderSaga {
  private final PaymentService paymentService;
  private final InventoryService inventoryService;
  private final ShippingService shippingService;

  @Transactional
  public void execute(PlaceOrderCommand cmd) {
    SagaState state = new SagaState(cmd);

    try {
      // Krok 1 — płatność
      PaymentId paymentId = paymentService.charge(cmd.getPayment());
      state.setPaymentId(paymentId);

      // Krok 2 — rezerwacja towaru
      try {
        ReservationId resId = inventoryService.reserve(cmd.getItems());
        state.setReservationId(resId);
      } catch (OutOfStockException e) {
        // Kompensacja — oddaj płatność
        paymentService.refund(state.getPaymentId());
        throw new SagaFailedException("Brak towaru, płatność zwrócona", e);
      }

      // Krok 3 — wysyłka
      try {
        shippingService.schedule(state.getReservationId(), cmd.getAddress());
      } catch (Exception e) {
        // Kompensacje w odwrotnej kolejności!
        inventoryService.cancelReservation(state.getReservationId());
        paymentService.refund(state.getPaymentId());
        throw new SagaFailedException("Błąd wysyłki, rollback", e);
      }

    } catch (SagaFailedException e) {
      // log, alert, DLQ
      sagaRepo.markFailed(state.getSagaId(), e.getMessage());
      throw e;
    }
  }
}

Resilience Patterns
Circuit Breaker, Retry, Bulkhead, Timeout, Rate Limiting
W systemach rozproszonych serwisy czasowo zawodzą. Wzorce odporności zapobiegają kaskadowym awariom (cascade failure) — jeden serwis nie wysypuje całego systemu. Biblioteka: Resilience4j.
Resilience4j — Circuit Breaker, Retry, Bulkhead
// ── application.yml ───────────────────────────────────────────
// resilience4j:
//   circuitbreaker:
//     instances:
//       paymentService:
//         slidingWindowSize: 10
//         failureRateThreshold: 50    # 50% failures → OPEN
//         waitDurationInOpenState: 30s
//         permittedCallsInHalfOpenState: 5
//   retry:
//     instances:
//       paymentService:
//         maxAttempts: 3
//         waitDuration: 500ms
//         exponentialBackoffMultiplier: 2  # 500ms, 1s, 2s
//         retryExceptions: java.net.ConnectException
//   bulkhead:
//     instances:
//       paymentService:
//         maxConcurrentCalls: 20  # max równoległych wywołań

// ── Adnotacje Resilience4j ────────────────────────────────────
@Service
public class PaymentClient {

  @CircuitBreaker(name = "paymentService", fallbackMethod = "fallback")
  @Retry(name = "paymentService")
  @Bulkhead(name = "paymentService")
  @TimeLimiter(name = "paymentService") // timeout
  public CompletableFuture<PaymentResult> charge(PaymentRequest req) {
    return httpClient.post("/payments", req);
  }

  // Fallback — co gdy CB jest OPEN lub retry się skończą
  private CompletableFuture<PaymentResult> fallback(
      PaymentRequest req, Exception ex) {
    log.warn("Payment service unavailable, queuing: {}", ex.getMessage());
    // Wyślij do kolejki do późniejszego przetwarzania
    paymentQueue.enqueue(req);
    return CompletableFuture.completedFuture(
        PaymentResult.pending("Przetwarzanie opóźnione"));
  }
}

// ── Stany Circuit Breaker ────────────────────────────────────
// CLOSED → normalne działanie, liczy błędy
// OPEN   → failfast, nie woluje downstream
// HALF-OPEN → testuje kilka wywołań, decyduje
🛡️ Wzorce odporności — przegląd
Circuit Breaker
Zapobiega kaskadowym awariom. Failfast gdy downstream jest niedostępny — zamiast czekać na timeout. 3 stany: CLOSED/OPEN/HALF-OPEN.
Retry + Backoff
Ponawia nieudane wywołania. Exponential backoff + jitter = unikanie thundering herd. Tylko dla idempotentnych operacji!
Bulkhead
Izolacja zasobów — jeden slow/broken service nie wyczerpuje thread pool dla innych. Threadpool lub semaphore bulkhead.
Timeout
Zawsze definiuj timeout! Bez timeoutu jeden zawieszony serwis blokuje wątki na wieki. Hierarchia timeoutów: DB < service < gateway.
Rate Limiting
Ogranicz liczbę żądań per klient/IP. Algorytmy: Token Bucket, Leaky Bucket, Sliding Window. Implementacja: Redis + Resilience4j lub API Gateway.
Fallback
Degraded mode — gdy serwis niedostępny zwróć cache, wartość domyślną lub zakolejkuj do późniejszego przetworzenia.
Kaskadowa awaria — jak zapobiegać Service A wywołuje B, B wywołuje C. C zawodzi → B czeka na timeout → wątek B zablokowany → A czeka → cały system zawieszony. Rozwiązanie: Circuit Breaker + Bulkhead + Timeout na każdym poziomie. Nigdy nie ufaj zewnętrznemu serwisowi — zawsze zakładaj że może failować.

Service Discovery i Load Balancing
Jak serwisy się znajdują — Client vs Server-side Discovery
W dynamicznych środowiskach (Kubernetes, cloud) adresy IP serwisów się zmieniają. Service Discovery pozwala znaleźć aktualny adres serwisu przez jego logiczną nazwę. Load Balancing rozdziela ruch.
🔍 Service Discovery — typy
Client-side Discovery

Klient odpytuje Service Registry (Eureka, Consul), dostaje listę instancji, sam wybiera (load balancing). Spring Cloud Netflix Eureka + Ribbon (legacy) lub Spring Cloud LoadBalancer.

Server-side Discovery (preferowane)

Klient wysyła żądanie do Load Balancer (AWS ALB, NGINX, Kubernetes Service). LB pyta Registry i przekierowuje. Klient nie wie nic o Service Discovery.

Kubernetes
Wbudowany Service Discovery przez DNS. http://order-service:8080 → kube-proxy → Pod. Domyślny wybór dla nowych projektów.
Consul
Service Registry + Config Store + Health Checking. Dobry poza Kubernetes lub w multi-cloud.
Eureka
Netflix OSS, Spring Cloud. AP — dostępny nawet przy partycji. Legacy — dziś Kubernetes wygrywa.
Spring Cloud — Load Balancing i Discovery
// ── Spring Cloud LoadBalancer (nowoczesne) ────────────────────
@Configuration
public class WebClientConfig {

  @Bean
  @LoadBalanced // ← magia — ResolvesServiceId przez registry
  public WebClient.Builder webClientBuilder() {
    return WebClient.builder();
  }
}

@Service
public class PaymentClient {
  private final WebClient.Builder webClientBuilder;

  public Mono<PaymentResult> charge(PaymentRequest req) {
    // "payment-service" = nazwa serwisu w registrze
    // @LoadBalanced WebClient resolwuje do realnego IP
    return webClientBuilder.build()
        .post()
        .uri("http://payment-service/api/payments")
        .bodyValue(req)
        .retrieve()
        .bodyToMono(PaymentResult.class);
  }
}

// → Tworzy Service object z stałym DNS
// → order-service.default.svc.cluster.local
// → Aplikacja używa: http://order-service:8080

Observability
Logs, Metrics, Traces — Three Pillars of Observability
W systemach rozproszonych bez obserwowalności jesteś ślepy. Trzy filary: Logs (co się stało), Metrics (jak wiele / jak szybko), Traces (przez jakie serwisy przeszło żądanie). Niezbędne na produkcji.
📝 Structured Logging

Structured logs (JSON) — nie plain text. Każde pole to klucz-wartość. Łatwe filtrowanie i indeksowanie w ELK/Loki.

{
  "timestamp": "2025-01-15T10:30:00Z",
  "level": "INFO",
  "traceId": "abc123",
  "spanId": "def456",
  "service": "order-service",
  "orderId": "ord-789",
  "message": "Order placed"
}

Stack: Logback/SLF4J + JSON encoder → Filebeat → Elasticsearch → Kibana. Lub Grafana Loki (lżejszy).

Zawsze loguj: traceId, requestId, userId, czas operacji. Nigdy: hasła, PII bez maskowania.

📊 Metryki — Micrometer + Prometheus

Micrometer = fasada jak SLF4J, ale dla metryk. Spring Boot Actuator integruje go automatycznie.

Counter
Monotonicznie rosnące. Liczba żądań, błędów. Counter.increment()
Gauge
Aktualna wartość. Liczba wątków, rozmiar kolejki.
Timer
Czas trwania + liczba wywołań. Percentyle (p95, p99).
Histogram
Rozkład wartości. Request size, response time.

Golden Signals (SRE): Latency, Traffic, Errors, Saturation. Monitoruj te 4 dla każdego serwisu.

Stack: Prometheus (scrape) → Grafana (dashboard + alerts)

🔍 Distributed Tracing

@Service
public class OrderService {
  private final Counter ordersPlaced;
  private final Counter ordersFailed;
  private final Timer orderTimer;

  public OrderService(MeterRegistry registry) {
    ordersPlaced = Counter.builder("orders.placed")
        .tag("region", "eu-west")
        .description("Total orders placed")
        .register(registry);
    ordersFailed = registry.counter("orders.failed");
    orderTimer = registry.timer("orders.place.duration");
  }

  public OrderId placeOrder(PlaceOrderCommand cmd) {
    return orderTimer.record(() -> {
      try {
        OrderId id = doPlaceOrder(cmd);
        ordersPlaced.increment();
        return id;
      } catch (Exception e) {
        ordersFailed.increment();
        throw e;
      }
    });
  }
}

Service Mesh i API Gateway
Istio, Linkerd — infrastruktura jako kod komunikacji
Service Mesh przenosi cross-cutting concerns (mTLS, circuit breaking, tracing, retry) z kodu aplikacji do infrastruktury. Sidecar proxy (Envoy) przy każdym Podzie obsługuje całą komunikację sieciową.
🕸️ Service Mesh — architektura

Sidecar Pattern: Każdy Pod = kontener aplikacji + kontener Envoy proxy. Aplikacja nie wie o sidecarze — cały ruch przez proxy.

Data Plane
Envoy sidecar proxies — obsługują faktyczny ruch. Zero zmian w aplikacji.
Control Plane
Istio/Linkerd — konfiguruje i zarządza sidecarami przez K8s CRDs.
mTLS
Wzajemna auth TLS między serwisami — zero trust networking. Automatyczne certyfikaty.
Traffic management
A/B testing, canary deployments, traffic mirroring — bez zmiany kodu.
Istio vs Linkerd
Istio = więcej features, większy overhead. Linkerd = prostszy, lżejszy, Rust-based proxy.
Kiedy NIE potrzebujesz Service Mesh Mały zespół, mało serwisów, brak ekspertyzy Kubernetes. Service Mesh to dodatkowa złożoność — najpierw rozwiąż problemy w kodzie, potem w infrastrukturze.

@Component
public class AuthenticationFilter implements GlobalFilter, Ordered {

  @Override
  public Mono<Void> filter(ServerWebExchange exchange,
      GatewayFilterChain chain) {
    String token = exchange.getRequest().getHeaders()
        .getFirst("Authorization");

    if (token == null || !isValid(token)) {
      exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
      return exchange.getResponse().setComplete();
    }

    // Dodaj user info jako nagłówek dla downstream serwisów
    String userId = extractUserId(token);
    ServerHttpRequest mutated = exchange.getRequest().mutate()
        .header("X-User-Id", userId)
        .build();
    return chain.filter(exchange.mutate().request(mutated).build());
  }

  @Override
  public int getOrder() { return -1; } // najwyższy priorytet
}

CQRS oddziela ścieżkę zapisu (Commands) od ścieżki odczytu (Queries) — różne modele, różne bazy. Event Sourcing przechowuje historię zdarzeń zamiast aktualnego stanu. Używaj świadomie — duży overhead.

// ── Event Store (tabela) ──────────────────────────────────────
// CREATE TABLE event_store (
//   id            UUID PRIMARY KEY,
//   aggregate_id  VARCHAR(255) NOT NULL,
//   aggregate_type VARCHAR(255) NOT NULL,
//   version       BIGINT NOT NULL,  ← optimistic locking!
//   event_type    VARCHAR(255) NOT NULL,
//   payload       JSONB NOT NULL,
//   created_at    TIMESTAMPTZ DEFAULT NOW()
// );

// ── Aggregate z event sourcing ────────────────────────────────
public class Order {
  private OrderId id;
  private OrderStatus status;
  private List<OrderItem> items;
  private long version = 0;

  // Odtworzenie stanu z historii eventów
  public static Order reconstitute(List<DomainEvent> history) {
    Order order = new Order();
    history.forEach(order::apply);
    return order;
  }

  private void apply(DomainEvent event) {
    version++;
    switch (event) {
      case OrderPlacedEvent e -> {
        this.id = new OrderId(e.orderId());
        this.status = OrderStatus.PLACED;
        this.items = e.items();
      }
      case OrderConfirmedEvent e ->
        this.status = OrderStatus.CONFIRMED;
      case OrderCancelledEvent e ->
        this.status = OrderStatus.CANCELLED;
      default -> {}
    }
  }
}

CQRS — kiedy stosować Gdy read/write mają różne wymagania (np. złożone raporty vs proste zapisy). Nie stosuj wszędzie — dla CRUD to over-engineering. Dobrze łączy się z DDD i Event Sourcing.
Event Sourcing — zalety Pełny audit log "w cenie". Temporal queries (stan w dowolnym punkcie historii). Replay eventów do nowych projekcji. Naturalne dla systemów finansowych i audit-heavy.
Event Sourcing — wady Eventual consistency read models. Trudny schema evolution eventów. "Eventual" queries utrudniają proste UI. Myśl dwa razy zanim użyjesz — frameworki: Axon, EventStoreDB.
