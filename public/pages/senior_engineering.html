01 / 07
Code Review
Code Review — jak robić go dobrze jako senior
Code review to nie tylko "szukanie bugów" — to narzędzie transferu wiedzy, utrzymania spójności architektury i podnoszenia jakości całego zespołu. Senior prowadzi review które uczy, nie demotywuje, i skupia się na tym co ważne.
🎯 Co sprawdzać — priorytety
BLOCKER
Bezpieczeństwo — SQL injection, XSS, hardcoded secrets, brak autoryzacji, wrażliwe dane w logach, nieprawidłowe uprawnienia. Poprawność — race conditions, NPE, off-by-one, utrata danych, złe warunki brzegowe. Data integrity — brak transakcji gdzie potrzeba, błędna logika biznesowa.
MAJOR
Architektura — naruszenie granic modułów, tight coupling, brak abstrakcji tam gdzie potrzeba (lub niepotrzebna abstrakcja). Performance — N+1 queries, brak indeksów, zbędne alokacje w hot path, blokujące I/O. Testowalność — brak testów dla krytycznej logiki, testy które testują implementację nie zachowanie.
MINOR
Czytelność — niejasne nazwy, zbyt długie metody (>20 linii), deep nesting (>3 poziomy), brak komentarzy do nieoczywistej logiki. Spójność — inne konwencje niż reszta projektu, własne rozwiązania gdy istnieje firmowy standard.
NIT
Styl, formatowanie (jeśli linter nie pilnuje), drobne preferencje. Wyraźnie oznaczaj jako "nit:" — autor wie że może zignorować. Nie blokuj PR z powodu nitów.
Złota zasada — "why" nie "what" Zły komentarz: "Zmień to na HashMap"
Dobry komentarz: "Rozważ HashMap zamiast ArrayList — wyszukiwanie po userId będzie O(1) zamiast O(n). Przy 10k userów to może być odczuwalne."

Zawsze tłumacz dlaczego coś jest problemem i co zyskujemy zmianą. Review to mentoring przy skali.

👁️ Czego NIE sprawdzać w review

Formatowanie kodu — od tego jest linter (Checkstyle, Spotless, Prettier). Nie trać czasu seniora na wcięcia.

Styl osobisty — jeśli działa i jest czytelne, nie narzucaj własnych preferencji jako standardu.

Zbyt szczegółowe implementacyjne decyzje — jeśli cel jest osiągnięty i kod jest czytelny, autor mógł mieć powód.

Przerabianie całego PRa — scope review = scope PRa. Nie żądaj refaktorów spoza zakresu zmiany.

🔧 Automatyzacja — co delegate do narzędzi

SonarQube / SonarCloud — code smells, pokrycie testami, duplikacja, security hotspots. Blokuj merge jeśli quality gate nie przechodzi.

SpotBugs / PMD — statyczna analiza dla Java: null pointer, resource leaks, concurrency bugs.

Checkstyle / Spotless — formatowanie i konwencje kodu. Zero dyskusji o stylu.

OWASP Dependency-Check / Dependabot — znane CVE w zależnościach. Automatyczne alerty i PRy.

🧠 Psychologia review

Krytykuj kod, nie autora — "ten kod nie obsługuje X" zamiast "nie obsłużyłeś X".

Dawaj komplementy — szczególnie juniorom. Pozytywne wzmocnienie działa. "Dobre podejście z X".

Traktuj własne sugestie jako opcje — "rozważ", "co myślisz o", "możliwe jest też" zamiast "zmień na".

Bądź otwarty na odpowiedź — autor może znać kontekst który Ty nie znasz. Dyskusja > dyktowanie.

02 / 07
Tech Debt
Dług techniczny — identyfikacja, kategoryzacja, zarządzanie
Technical debt to nieunikniony koszt decyzji podjętych pod presją czasu lub niewiedzą. Problemem nie jest istnienie długu — problem to niezarządzany dług który rośnie compound interest. Senior potrafi go mierzyć, priorytetyzować i "sprzedać" go biznesowi.
📊 Kwadrant długu technicznego (Fowler)
→Reckless (nierozważny)Prudent (roztropny)
Deliberate (celowy) "Nie mamy czasu na design" — najgorszy typ. Świadoma rezygnacja z jakości bez planu spłaty. "Shipmujemy teraz, refaktorujemy potem" — akceptowalny gdy jest realny plan i termin.
Inadvertent (nieświadomy) "Co to są warstwy?" — brak wiedzy i refleksji. Edukacja i review pomagają. "Teraz wiemy lepiej jak to zrobić" — naturalny efekt uczenia się. Akceptowalny.

Cel: minimalizuj Reckless/Deliberate. Akceptuj Prudent/Deliberate krótkoterminowo z jawnym TODO + ticket. Akceptuj Prudent/Inadvertent — to naturalna ewolucja.

🔍 Jak identyfikować i mierzyć
SonarQube Metrics
Technical Debt Ratio (%), Code Smell count, Coverage %, Duplication %. Skonfiguruj Quality Gate — automatyczne ostrzeżenia przy regresji.
Change Failure Rate
Jak często deploy prowadzi do incydentów lub hotfixów? Wysoki rate = dług w testach, brak observability lub krucha architektura.
Cognitive Complexity
Metryka SonarQube — ile wysiłku żeby zrozumieć metodę. >15 = problem. Koreluje z liczbą bugów lepiej niż cyclomatic complexity.
DORA Metrics
Lead time for changes, Deployment frequency, MTTR, Change failure rate — pośrednio mierzą jakość i dług.
Hotspot analysis
git log --format='%H %s' -- path/to/file | wc -l — pliki z największą liczbą commitów = hotspoty = wysoki koszt długu.
💰 Jak "sprzedać" dług biznesowi

Biznes rozumie pieniądze i czas, nie "code smells". Tłumacz dług na język biznesu:

❌ Źle (technicznie):

"Mamy silnie sprzężony monolit z wysoką cyclomatic complexity i brak dependency injection w serwisach domenowych."

✅ Dobrze (biznesowo):

"Każda nowa funkcja w module płatności zajmuje 3 sprinty zamiast 1, bo kod jest trudny do zmiany bez ryzyka regresji. Szacujemy że inwestycja 2 tygodni refactoringu zmniejszy czas dostarczenia feature'ów o 40% i liczbę bugów produkcyjnych o połowę."

Techniki argumentacji:

• Velocity chart — pokaż jak velocity spada z kwartału na kwartał przez dług

• Incident cost — policz koszt ostatnich 3 incydentów spowodowanych długiem

• Feature lead time trend — podobne feature'y zajmują coraz więcej czasu

• Onboarding time — jak długo trwa wdrożenie nowego dewelopera

🗓️ Strategie spłaty długu
Boy Scout Rule
"Zostaw obóz czystszym niż go zastałeś." Przy każdym dotyku pliku — małe poprawki: rename, extract method, usuń dead code. Dług spada organicznie bez dedykowanego sprintu.
20% rule
Rezerwuj 20% capacity sprintu na dług techniczny. Stała pozycja w definicji sprintu. Biznes planuje z 80% feature capacity.
Debt Sprint
Co N sprintów — jeden dedykowany tech debt sprint. Transparentny dla biznesu. Ryzyko: traktowany jako "opcja" do cięcia przy deadlines.
Strangler Fig
Nowe funkcje piszesz we właściwy sposób, stopniowo zastępując stary kod. Nie przepisujesz wszystkiego naraz — za duże ryzyko. Wzorzec Martina Fowlera.
Opportunistic
Spłacaj dług przy okazji related feature'ów — wychodzi naturalnie, minimalny friction z PM.
Priorytetyzacja — Impact vs Effort matrix Nie cały dług jest równy. Priorytetyzuj po: Frequency (jak często dotykamy tego kodu?) × Impact (jak bardzo spowalnia? ile bugów powoduje?) × Risk (czy jest niebezpieczny?). Dług w rzadko używanym module legacy który nikt nie dotyka = niski priorytet. Dług w module płatności który jest zmieniany co sprint = krytyczny priorytet. Użyj Tech Debt backlog — osobna lista z priorytetami, reviewed co kwartał z leadem technicznym.
03 / 07
Refactoring
Strategie refaktoryzacji — bezpiecznie, stopniowo, bez regresji
Refactoring = zmiana struktury kodu bez zmiany jego zachowania. Kluczowe słowo: bez zmiany zachowania. Bezpieczny refactoring wymaga testów przed zmianą, małych kroków i weryfikacji na każdym etapie. Nie "przepisz wszystko" — to prawie zawsze błąd.
Kluczowe techniki refaktoryzacji — Java
// ══ 1. EXTRACT METHOD — najczęstszy refactor ══════════════════
// Przed: długa metoda robiąca zbyt wiele
public void processOrder(Order order) {
    // validate
    if (order.getItems().isEmpty()) throw new IllegalArgumentException();
    if (order.getCustomer() == null) throw new IllegalArgumentException();
    // calculate total
    double total = 0;
    for (OrderItem i : order.getItems()) total += i.getPrice() * i.getQty();
    // apply discount
    if (order.getCustomer().isPremium()) total *= 0.9;
    // save ... 30 more lines
}

// Po: czytelne, testowalne w izolacji
public void processOrder(Order order) {
    validateOrder(order);
    double total = calculateTotal(order);
    saveOrder(order, total);
}
private void validateOrder(Order order) { ... }
private double calculateTotal(Order order) { ... }

// ══ 2. STRANGLER FIG — stopniowe zastępowanie ════════════════
// Nie przepisuj — owijaj i przekierowuj
public class PaymentServiceFacade {
    private final LegacyPaymentService legacy;
    private final NewPaymentService newService;
    private final FeatureFlags flags;

    public PaymentResult processPayment(Payment payment) {
        if (flags.isEnabled("new-payment-service")) {
            return newService.process(payment);  // nowy
        }
        return legacy.process(payment);           // stary
    }
}
// Feature flag → stopniowe włączanie (1% → 10% → 100%)
// Rollback w sekundy bez deployu

// ══ 3. EXTRACT SERVICE — rozdzielenie odpowiedzialności ══════
// Przed: serwis który robi za dużo (1000+ linii)
public class UserService {
    // logika rejestracji, autentykacji, profilu,
    // wysyłania emaili, zarządzania rolami... ← BŁĄD SRP
}
// Po: rozdziel po domenach (Small Incremental Changes!)
class UserRegistrationService { ... }
class UserProfileService      { ... }
class UserAuthService         { ... }
class UserRoleService         { ... }

// ══ 4. REPLACE CONDITIONAL WITH POLYMORPHISM ═════════════════
// Przed: rozrastający się switch po typie
double calculate(Employee e) {
    return switch (e.getType()) {
        case ENGINEER  -> e.salary * 1.1;
        case MANAGER   -> e.salary * 1.2 + bonus;
        case DIRECTOR  -> e.salary * 1.5 + bonus + stock;
    };
}
// Po: każdy typ odpowiada za siebie
interface Employee { double calculatePay(); }
class Engineer  implements Employee { ... }
class Manager   implements Employee { ... }
// Dodanie nowego typu = nowa klasa, bez modyfikacji istniejących (OCP)
🪜 Bezpieczny proces refaktoringu
REFACTORING CHECKLIST
01
Napisz testy PRZED zmianą — jeśli nie ma testów, napisz characterization tests (opisują obecne zachowanie, nie pożądane). To siatka bezpieczeństwa dla refactoringu.
02
Małe, atomowe kroki — każdy krok = jeden commit. "Extract method" to jeden commit. Każdy krok musi przejść testy. Nie łącz refactoringu z bugfixami — osobny PR.
03
Jeden typ refactoru na raz — nie rób rename + extract + move jednocześnie. Przy błędzie nie wiesz co go spowodowało.
04
Feature flag dla dużych zmian — Strangler Fig pattern. Nowy kod obok starego, przełącznik feature flag. Bezpieczny rollback.
05
Refactoring ≠ nowe feature'y — absolutnie separuj. PR refactoringowy zawiera ZERO nowej funkcjonalności. Ułatwia review i bisectowanie błędów.
06
Zdefiniuj "done" — jaki jest cel? Lepsza testowalność? Szybsze zmiany? Bez celu refactoring nigdy się nie kończy i rośnie w scope.
Największy błąd — "Big Bang Rewrite" "Przepiszemy wszystko od nowa w 3 miesiące" — to prawie zawsze katastrofa. Nowy kod ma własne bugi. Stary kod ma utajone reguły biznesowe których nikt nie pamięta. Przez 3 miesiące produkt stoi. Po 3 miesiącach jest 6 miesięcy opóźnienia. Joel Spolsky: "Things You Should Never Do". Zamiast tego: Strangler Fig, stopniowe zastępowanie, migration scripts. Wyjątek: naprawdę martwy kod legacy który nie jest aktywnie rozwijany i masz pełne zrozumienie wymagań.

Mentoring
Mentoring — jak efektywnie rozwijać innych developerów
Mentoring to jeden z największych multiplicatorów wartości seniora. Zamiast robić rzeczy sam — uczysz innych jak je robić. Senior który efektywnie mentoruje = 3x developer. Kluczowe: dawaj zadania z "productive struggle", nie dawaj gotowych odpowiedzi zbyt szybko.

💬 Techniki mentoringu w praktyce
Pair Programming
Driver + Navigator. Zmieniajcie role. Watch me do it → Do it together → You do it alone. Najszybszy transfer wiedzy, ale kosztowny czasowo — używaj strategicznie dla nowych konceptów.
Pytania Sokratejskie
"Co się stanie gdy..." / "Jak byś to przetestował?" / "Jakie są alternatywy?" Zamiast dawać odpowiedź — prowadź do niej pytaniami. Podopieczny uczy się myśleć, nie zapamiętywać.
Deleguj z kontekstem
Nie: "Zrób feature X." Tak: "Mamy problem Y — użytkownicy robią Z. Jednym rozwiązaniem jest feature X, ale mogą być inne. Zbadaj i zaproponuj." Rozwijasz myślenie systemowe.
Deliberate Feedback
SBI: Situation (konkretna sytuacja) → Behavior (co zrobiłeś) → Impact (jaki efekt). Feedback bliski zdarzeniu, konkretny, prywatny jeśli negatywny.
1:1 Meetings
Regular 30-45 min. Nie status update — to mentee prowadzi agendę. Pytaj: "Co cię blokuje?", "Z czego jesteś dumny z ostatniego tygodnia?", "Czego chcesz się nauczyć?"
ANTI-PATTERN — Seagull Manager Problem: wlatać, natychmiast dawać odpowiedź, wylatywać. Podopieczny nie uczy się myśleć — tylko czeka na ciebie.

Fix: daj 24h na samodzielne badanie przed udzieleniem odpowiedzi. "Spróbuj to zbadać, wróć do mnie jutro z tym co znalazłeś."
ANTI-PATTERN — Perfectionist Blocker Problem: robisz za podopiecznego "bo to szybciej" lub blokujesz ich PRy z zbyt wieloma komentarzami.

Fix: akceptuj 80% rozwiązania jeśli działa. Dług komentarzami nie pomaga — wybierz 1-2 najważniejsze lekcje z danego PR.
PRODUCTIVE STRUGGLE — klucz Pozwól podopiecznemu zmagać się z problemem przez rozsądny czas zanim pomożesz. Frustracja i próby są częścią uczenia się — interweniuj gdy są stuck >2h bez postępu, nie po 20 minutach.
Individual Development Plan (IDP) Co kwartał wspólnie z podopiecznym twórz plan rozwoju: cel (co chce osiągnąć za 6-12 miesięcy), luki (co musi rozwinąć żeby tam dojść), akcje (konkretne zadania, kursy, projekty), miernik (jak wiemy że osiągnęliśmy?). Bez IDP mentoring jest reaktywny — reaguje na problemy. Z IDP jest proaktywny — buduje konkretną ścieżkę.
05 / 07
Decyzje Architektoniczne
Jak podejmować decyzje architektoniczne — framework
Decyzje architektoniczne są kosztowne do odwrócenia — dlatego wymagają struktury, danych i komunikacji. Senior nie podejmuje decyzji solo bez kontekstu — angażuje stakeholderów, analizuje trade-offy i dokumentuje dlaczego tak, a nie inaczej.
🧭 Framework decyzji architektonicznych
PROCESS — od problemu do decyzji
01
Zdefiniuj problem, nie rozwiązanie — "potrzebujemy Kafki" to nie problem. "Mamy 50k eventów/s i obecna synchroniczna komunikacja powoduje latencję 3s i cascade failures" — to problem.
02
Zbierz constraints — co jest nienaruszalne? Budżet, czas dostarczenia, kompetencje zespołu, compliance, SLA. Constraints redukują przestrzeń rozwiązań.
03
Generuj opcje (min 3) — zawsze minimum 3 opcje: status quo (koszt braku działania!), opcja minimalna, opcja docelowa. Monitoruj bias konfirmacji — celowo szukaj argumentów przeciw faworytowi.
04
Analiza trade-offów — dla każdej opcji: pros, cons, risks, cost, reversibility. Nie szukasz "najlepszej" — szukasz "najlepszej w tych constraints". Nie ma silver bullet.
05
Angażuj stakeholderów — RFC (Request for Comments) do zespołu, tech lead, DevOps jeśli dotyczy infrastruktury. Decyzje podjęte kolektywnie mają lepszy buy-in i mniej blind spots.
06
PoC dla dużych ryzyk — jeśli nie wiesz czy A jest lepsze od B w waszym kontekście — timebox'd PoC (3-5 dni). Dane > opinie.
07
Udokumentuj jako ADR — dlaczego podjęto tę decyzję, jakie były opcje, co zdecydowało. Przyszły ty będzie ci wdzięczny. Patrz rozdział ADR.

🚩 Bias i pułapki decyzyjne
Hype Driven
"Wszyscy używają Kafki / Kubernetes / GraphQL" — nie jest argumentem. Twoje wymagania są inne niż Netflix.
Resume Driven
Wybierasz technologię żeby ją mieć w CV, nie bo rozwiązuje problem lepiej. Uczciwie pytaj siebie: "Czy wybrałbym to samo anonimowo?"
Sunk Cost Bias
"Już 3 miesiące pracujemy na X, nie możemy teraz zmienić." Przeszły koszt nie jest argumentem za złą decyzją. Oceń przyszłość, nie przeszłość.
Authority Bias
"CTO powiedział że..." — decyzja techniczna powinna być oparta na danych, nie autorytecie. Szanuj hierarchy, ale wychylaj się z danych jeśli direction jest błędna.

Architecture Decision Records
ADR — jak dokumentować decyzje architektoniczne

TITLE

ADR-0012: Użycie Azure Service Bus do asynchronicznej komunikacji między mikroserwisami

DATE & AUTHORS

2024-11-15 | Jan Kowalski (tech lead), Anna Nowak (arch)

STATUS

Accepted — zastępuje ADR-0007 (direct HTTP calls) | Supersedes: ADR-0007

CONTEXT

Order Service wywołuje synchronicznie Inventory Service i Payment Service przez REST. Przy awarii jednego serwisu cascade failure blokuje cały flow zamówień. Obserwujemy 15% błędów zamówień przy incydentach Inventory Service. Throughput wynosi ~500 zamówień/min w peak hours.

DECISION DRIVERS

• Eliminacja cascade failures między serwisami
• Utrzymanie zamówień przy chwilowej niedostępności downstream serwisów
• Team ma zero doświadczenia z self-hosted Kafka (constraint)
• Jesteśmy już na Azure (ecosystem constraint)

CONSIDERED OPTIONS

Option A: Azure Service Bus (Topics + Subscriptions) — managed, wbudowany dead-letter queue, retry policies, sessions ordering. Koszt: ~$10/mln wiadomości.

Option B: Apache Kafka (managed — Azure Event Hubs) — wyższy throughput, replay, partycje. Overhead operacyjny, team nie zna. Przerost dla naszego throughput.

Option C: Status quo — synchroniczny REST — zero kosztu zmiany. Nie rozwiązuje cascade failures. Rosnący tech debt.

DECISION

Wybieramy Azure Service Bus (Option A). Throughput 500 zam/min nie uzasadnia Kafki. Managed service eliminuje overhead operacyjny. Dead-letter queue daje naturalną obsługę błędów. Ecosystem fit z Azure.

CONSEQUENCES

Positive: brak cascade failures, at-least-once delivery, retry out-of-box, ordering per session.
Negative: eventual consistency (order confirmed przed payment confirmed), złożoność testowania asynchronicznego, potrzeba implementacji idempotency w consumers.
Risks: Service Bus SLA 99.9% — potrzeba dead-letter queue monitoring.

LINKS

Azure Service Bus docs · RFC-0031 (dyskusja) · Ticket ARCH-142

Statusy ADR:

PROPOSED → ACCEPTED → DEPRECATED

DEPRECATED nie znaczy błędna decyzja — znaczy że kontekst się zmienił. Stare ADRy zostawiasz (historia), dodajesz link do nowego.

Kiedy pisać ADR Pisz ADR gdy: decyzja jest trudna do odwrócenia, wielu ludzi będzie dotkniętych zmianą, była nieoczywista kontrowersja przy podejmowaniu decyzji, decyzja wpływa na strukturę systemu lub sposób pracy zespołu.

Nie pisz ADR dla: oczywistych decyzji implementacyjnych, rutynowych wyborów bibliotek bez alternatyw, rzeczy które można łatwo zmienić bez kosztów.

Dokumentacja Systemów
Dokumentowanie systemów — co, jak, gdzie i ile
Dokumentacja ma jedną zasadę: minimalna wystarczająca — wystarczająca żeby nowy developer mógł pracować efektywnie, minimalna żeby nie było drugiej pracy w utrzymaniu (kod się zmienia, dokumentacja gnije). Dobra dokumentacja opisuje dlaczego, nie co — co jest w kodzie.
📖 Co dokumentować — hierarchia
README.md
Obowiązkowe. Jak uruchomić lokalnie (prerequisites, steps), krótki opis co robi serwis, linki do dalszej dokumentacji, główne konwencje. Max 1-2 strony. Nowy dev powinien uruchomić w <30 min.
ADR
Obowiązkowe. Dlaczego kluczowe decyzje architektoniczne są takie a nie inne. Patrz rozdział ADR. Chronione przed gniciem bo decyzje są historyczne.
OpenAPI / AsyncAPI
Obowiązkowe dla API. Kontrakty między serwisami. Generuj z kodu (SpringDoc) — zawsze aktualne. AsyncAPI dla event-driven (Kafka, Service Bus topics).
Architecture Overview
Zalecane. C4 Model poziom 1-2 (System Context + Container diagram). Opisuje jak serwis się wpisuje w większy ekosystem. Aktualizuj przy dużych zmianach, nie przy każdym commicie.
Runbook / Playbook
Zalecane dla krytycznych serwisów. Co robić przy incydencie: jak sprawdzić health, typowe błędy i ich przyczyny, jak restartować, gdzie są logi, kogo kontaktować. Ratuje życie o 3 w nocy.
Komentarze w kodzie
Selektywnie. Tylko dla nieoczywistej logiki biznesowej, obejść znanych bugów (z linkiem do issue), skomplikowanych algorytmów. Nie komentuj "// increment i" przed i++.
🏗️ C4 Model — 4 poziomy abstrakcji
L1 System Context
Twój system w kontekście użytkowników i zewnętrznych systemów. Dla non-technical stakeholders. Jeden diagram — gdzie jesteśmy w ekosystemie.
L2 Container
Główne "kontenery" (nie Docker!) — serwisy, bazy danych, frontendy. Jak komunikują się ze sobą. Dla tech leads i devops. Najważniejszy poziom.
L3 Component
Wewnętrzna struktura jednego kontenera — główne moduły/komponenty. Dla developerów pracujących w tym serwisie. Rzadziej potrzebny.
L4 Code
UML klas i sekwencji. Prawie zawsze zbyt droge w utrzymaniu. Generuj automatycznie (PlantUML z kodu) lub nie rób wcale.

Senior Engineer — pełen obraz: Code review to nie polowanie na błędy — to mentoring przy skali. Dług techniczny tłumaczysz biznesowi przez velocity i incydent cost, nie "code smells". Refaktorujesz małymi krokami z testami jako siatką bezpieczeństwa — nigdy Big Bang Rewrite. Mentorujesz przez productive struggle i pytania sokratejskie, nie przez dawanie odpowiedzi. Decyzje architektoniczne podejmujesz z frameworkiem — problem → constraints → opcje → trade-offs → RFC → PoC → ADR. Dokumentujesz dlaczego, nie co — ADR + README + OpenAPI + C4.
