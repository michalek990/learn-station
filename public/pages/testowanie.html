01 — fundamenty
Dlaczego w ogóle testować?

Testy to pewność że twój kod działa — teraz i za rok. Refactoring bez testów to gra w rosyjską ruletkę. Jedno źle zmienione miejsce i cała aplikacja pada.

❌ Bez testów
  • ✗ Zmiana kodu = lęk przed crashem produkcji
  • ✗ Ręczne testowanie po każdej zmianie = godziny
  • ✗ Nowy developer nie ma pojęcia czy coś zepsuł
  • ✗ Refactoring = loteria
  • ✗ Regresja (stary bug wraca) — nie wiesz kiedy
  • ✗ Deployment = stres
✓ Z testami
  • ✓ Pewność że kod działa (proof)
  • ✓ Refactoring bez strachu — testy pokażą błędy
  • ✓ Dokumentacja "jak używać" — testy pokazują API
  • ✓ Catching bugs wcześnie = tanie naprawy
  • ✓ CI/CD — deploy automatyczny bo testy pilnują
  • ✓ Onboarding nowych devów — widzą jak działa
Złota zasada: kod bez testów to legacy code Michael Feathers: "Legacy code to kod bez testów". Możesz mieć piękny, czysty kod napisany wczoraj — jeśli nie ma testów, to legacy. Nikt nie będzie go chciał zmieniać bo lęk przed zepsuciem.
02 — strategia
Piramida Testów

Nie wszystkie testy są równe. Piramida mówi ile jakiego typu testów powinieneś mieć. Większość to testy jednostkowe — szybkie i tanie. Na szczycie E2E — wolne ale kompleksowe.

E2E / UI Tests
Wolne · Drogie · Kruche · Mało (5-10%)
Integration Tests
Średnie · Testują moduły razem (20-30%)
Unit Tests
Szybkie · Tanie · Stabilne · Dużo (60-70%)
✓ Dobre pokrycie 70% unit, 20% integration, 10% E2E. Szybki feedback loop — większość błędów łapana na unit testach (sekundy). Integration weryfikuje współpracę modułów. E2E sprawdza kluczowe scenariusze user flow.
✗ Ice cream cone (anti-pattern) Odwrócona piramida: dużo E2E, mało unit. Pipeline trwa 30 minut, testy często failują bez powodu (flaky), debugowanie koszmarne. Drogie, wolne, kruchse.
03 — rodzaje
Typy Testów

Każdy typ testuje coś innego i ma inne zastosowanie. Wybieraj mądrze co testujesz i na którym poziomie.

🧪
Unit Tests
Testują pojedynczą funkcję/metodę w izolacji. Mockujesz wszystkie zależności. Najszybsze, najwięcej.
JUnit, Jest, pytest
🔗
Integration Tests
Testują współpracę kilku modułów razem. Np. service + repository + database. Wolniejsze niż unit.
@SpringBootTest, Testcontainers
🌐
E2E Tests
Symulują prawdziwego użytkownika w przeglądarce. Klikają, wypełniają formularze. Wolne ale kompleksowe.
Selenium, Cypress, Playwright
💥
Smoke Tests
Szybkie testy "czy aplikacja w ogóle działa". Uruchom po deploymencie — czy główne funkcje żyją.
Health endpoint check
📊
Performance Tests
Czy aplikacja wytrzymuje obciążenie. Load testing, stress testing, spike testing.
JMeter, k6, Gatling
🔐
Security Tests
Skanowanie podatności: SQL injection, XSS, broken auth. OWASP ZAP, penetration testing.
OWASP ZAP, Burp Suite
04 — fundament piramidy
Unit Tests — Testy Jednostkowe

Testują najmniejszą możliwą jednostkę kodu — pojedynczą funkcję czy metodę. Wszystkie zależności są zamockowane. Szybkie, deterministyczne, łatwe do pisania.

Klasa do testowania — Calculator.java
public class Calculator {
  
  public int add(int a, int b) {
    return a + b;
  }
  
  public int divide(int a, int b) {
    if (b == 0) {
      throw new IllegalArgumentException(
        "Cannot divide by zero"
      );
    }
    return a / b;
  }
}
Test — CalculatorTest.java (JUnit 5)
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {
  
  private Calculator calculator;
  
  @BeforeEach
  void setUp() {
    calculator = new Calculator();
  }
  
  @Test
  void add_shouldReturnSum() {
    // given
    int a = 5, b = 3;
    
    // when
    int result = calculator.add(a, b);
    
    // then
    assertEquals(8, result);
  }
  
  @Test
  void divide_shouldThrowWhenDivideByZero() {
    // given
    int a = 10, b = 0;
    
    // when & then
    assertThrows(
      IllegalArgumentException.class,
      () -> calculator.divide(a, b)
    );
  }
}
📋 AAA Pattern — Given/When/Then
KrokCo robiszPrzykład
Arrange (Given) Przygotuj dane testowe, zmockuj zależności int a = 5, b = 3;
Act (When) Wywołaj testowaną metodę int result = calc.add(a, b);
Assert (Then) Sprawdź czy wynik jest poprawny assertEquals(8, result);
05 — moduły razem
Integration Tests

Testują jak kilka komponentów współpracuje. Np. Controller + Service + Repository + prawdziwa baza danych. Wolniejsze niż unit ale łapią błędy integracyjne.

UserServiceIntegrationTest.java — Spring Boot + Testcontainers
@SpringBootTest
@Testcontainers
class UserServiceIntegrationTest {
  
  // Testcontainers uruchamia prawdziwą bazę w Dockerze!
  @Container
  static PostgreSQLContainer<?> postgres = 
    new PostgreSQLContainer<>("postgres:16-alpine");
  
  @Autowired
  private UserService userService;
  
  @Autowired
  private UserRepository userRepository;
  
  @Test
  void createUser_shouldSaveToDatabase() {
    // given
    var request = new CreateUserRequest(
      "[email protected]", "John Doe"
    );
    
    // when
    var user = userService.createUser(request);
    
    // then — sprawdź w bazie!
    var saved = userRepository.findById(user.getId());
    assertTrue(saved.isPresent());
    assertEquals("[email protected]", saved.get().getEmail());
  }
  
  @AfterEach
  void cleanUp() {
    userRepository.deleteAll(); // wyczyść po teście
  }
}
✓ Testcontainers — prawdziwa baza w Dockerze Nie mockuj bazy danych w integration testach! Testcontainers uruchamia prawdziwy Postgres/MySQL/MongoDB w kontenerze Docker. Test działa na prawdziwej bazie → łapiesz błędy SQL, problemy z indeksami, constraint violations.
06 — cały flow użytkownika
E2E Tests — End-to-End

Symulują prawdziwego użytkownika. Otwierają przeglądarkę, klikają przyciski, wypełniają formularze. Najwolniejsze ale testują cały system razem — frontend + backend + baza.

login.spec.js — Playwright (JavaScript)
import { test, expect } from '@playwright/test';

test('user can log in successfully', async ({ page }) => {
  // given — otwórz stronę logowania
  await page.goto('http://localhost:3000/login');
  
  // when — wypełnij formularz
  await page.fill('input[name="email"]', '[email protected]');
  await page.fill('input[name="password"]', 'password123');
  await page.click('button[type="submit"]');
  
  // then — sprawdź czy przekierowano na dashboard
  await expect(page).toHaveURL('http://localhost:3000/dashboard');
  
  // i czy pojawił się user menu
  const userName = page.locator('.user-menu .name');
  await expect(userName).toHaveText('Test User');
});
⚠️ E2E są kruche (flaky) Testy E2E często failują bez powodu: timing issues, wolny load strony, animacje, race conditions. Musisz używać explicit waits, retry logic, stabilnych selektorów. Nie testuj wszystkiego przez E2E!
🎯 Co testować przez E2E
  • ✓ Krytyczne user flows (login, checkout, payment)
  • ✓ Happy path głównych funkcji
  • ✓ Smoke tests po deploymencie
  • ✗ Edge cases (za wolne)
  • ✗ Wszystkie kombinacje (za drogie)
  • ✗ Walidacja formularzy (unit testy)
07 — metodologia
TDD — Test-Driven Development

Najpierw piszesz test (który failuje), potem kod który go naprawia. Red → Green → Refactor. Wymusza myślenie o API przed implementacją.

🔴 RED → 🟢 GREEN → 🔵 REFACTOR
1. 🔴 RED — Napisz test który failuje
Zanim napiszesz kod, napisz test który sprawdza czy funkcja działa. Test będzie czerwony (fail) bo funkcja jeszcze nie istnieje.
@Test
void getUserById_shouldReturnUser() {
  // test failuje — metoda nie istnieje!
  User user = userService.getUserById(1L);
  assertNotNull(user);
}
2. 🟢 GREEN — Napisz minimalny kod żeby test przeszedł
Napisz najprostszy możliwy kod który sprawi że test przejdzie. Nie optymalizuj, nie dodawaj funkcji — tylko żeby było zielone.
public User getUserById(Long id) {
  // minimalny kod — test przechodzi!
  return userRepository.findById(id).orElse(null);
}
3. 🔵 REFACTOR — Popraw kod, testy pilnują
Teraz możesz refactorować — poprawić strukturę, wyciągnąć metody, usunąć duplikację. Testy zagwarantują że nic nie zepsułeś.
public User getUserById(Long id) {
  // refactor — rzuć wyjątek zamiast null
  return userRepository.findById(id)
    .orElseThrow(() -> new NotFoundException("User not found"));
}
✓ Korzyści z TDD Wymusza prosty design (trudno testować = źle zaprojektowane). Masz testy od razu (nie "napiszę później"). API przemyślane przed implementacją. Pewność że każda linia kodu działa.
⚠️ TDD nie zawsze TDD świetne dla logiki biznesowej, algorytmów, utils. Gorsze dla UI, eksperymentów, prototypów. Nie dogmatyzuj — używaj gdy pomaga, pomiń gdy przeszkadza.
08 — best practices
Dobre Praktyki Testowania

Pisanie testów to skill. Złe testy są gorsze niż brak testów — dają fałszywe poczucie bezpieczeństwa i spowalniają development.

✓ DO
Test behaviors, not implementation
Testuj "co robi" nie "jak robi". Zmiana implementacji nie powinna psuć testów.
Jeden assert per test (lub kilka related)
Test powinien testować jedną rzecz. Łatwiej debugować failure.
Testy muszą być deterministyczne
Zawsze ten sam wynik. Nie zależą od czasu, random, sieci, filesystem.
Szybkie unit testy
Cały suite unit testów < 10 sekund. Wolne = nie uruchamiasz = bezużyteczne.
Nazwa testu = dokumentacja
shouldReturnUserWhenValidId() mówi co test robi.
Arrange/Act/Assert pattern
Given/When/Then. Struktura czytelna dla każdego.
✗ DON'T
Testować private methods
Testuj public API. Private to szczegół implementacji.
Testować gettery/settery
Testy getterów/setterów to marnowanie czasu. Zero value.
Jeden wielki test na wszystko
Test 200 linii testujący 10 rzeczy = nie wiesz co padło.
Testy zależne od kolejności
Test A musi być przed testem B = kruche. Każdy test izolowany.
Ignorować flaky tests
Test który czasem failuje = zaufanie do testów spada. Fix it ASAP.
100% coverage za wszelką cenę
Coverage to metryka, nie cel. 80% dobrego pokrycia > 100% bezsensownych testów.
📊 Test Coverage — ile wystarczy?
Coverage pokazuje ile % kodu jest wykonywane przez testy. NIE mówi czy testy są dobre — tylko czy kod jest uruchamiany.
Logika biznesowa (UserService, OrderCalculator)
90%+ Coverage
Controllers (REST endpoints)
80%+ Coverage
Utils, helpers
85%+ Coverage
DTOs, Entities (getters/setters)
30%+ OK (nie ma co testować)
Cel: 80% overall coverage. Krytyczna logika biznesowa 90%+. Nie gonić 100% — diminishing returns.
