Architektura Platformy Java
JDK, JRE, JVM — co jest czym
Fundamentalna hierarchia platformy Java. Wiele osób myli te pojęcia — każda ma inne zadanie i inny zakres. Od Java 11 granica JDK/JRE zaciera się (JRE nie jest dostarczane osobno).

JVM — Architektura wewnętrzna
Co robi JVM od .java do wykonania
JVM jest specyfikacją — HotSpot to jej najpopularniejsza implementacja. Kompilacja dwuetapowa (javac → bytecode, JIT → native) daje Javie unikalną kombinację przenośności i wydajności.
Droga od kodu do wykonania
Source Code (.java)
        ↓  javac (kompilacja)
Bytecode (.class)          // platform-independent
        ↓  ClassLoader
Loaded Class (Method Area)
        ↓  Bytecode Verifier   // bezpieczeństwo
        ↓  Interpreter          // pierwsze wykonania
        ↓  profiling — "hot spots"?
        ↓  JIT Compiler (C1/C2)  // kompilacja do native
Native Machine Code         // cached w Code Cache

// Tiered Compilation (HotSpot):
Tier 0: Interpreter
Tier 1: C1 lekka optymalizacja (szybka kompilacja)
Tier 2: C1 z profilowaniem
Tier 3: C1 z pełnym profilowaniem
Tier 4: C2 agresywna optymalizacja (wolniejsza kompilacja)

🔧 Runtime Data Areas
Method Area
Metadane klas, bytecode metod, stałe. Implementacja: Metaspace (od Java 8, off-heap)
Heap
Obiekty i tablice. Dzielony między wątki. Zarządzany przez GC. Young + Old Generation.
Stack (per thread)
Ramki wywołań (frames): zmienne lokalne, operand stack, reference do current method. StackOverflow = za głęboka rekurencja.
PC Register
Program Counter — adres aktualnie wykonywanej instrukcji bytecode. Per thread.
Native Method Stack
Stack dla metod natywnych (C/C++ przez JNI). Oddzielny od Java Stack.
Code Cache
Cache skompilowanego kodu JIT. Gdy pełny — JIT przestaje kompilować (degradacja!). Monitoruj.
⚡ JIT Optymalizacje Inlining — wklejanie ciał małych metod. Loop unrolling. Escape analysis — alokacja na stosie zamiast heap. Dead code elimination. Devirtualization — wywołania wirtualne → statyczne. JIT może cofnąć optymalizację (deoptimization) gdy profil się zmieni.

Zarządzanie Pamięcią
Heap, GC algorytmy i tuning
Garbage Collection to jeden z najważniejszych aspektów JVM dla wydajności. Zrozumienie regionów pamięci, algorytmów i jak czytać GC logi to niezbędna umiejętność dla każdego senior Java developera.

📊 Cykl GC — G1

Minor GC (Young) — najczęstszy. STW. Kopiuje ocalałe z Eden/Survivor do S1 lub Old. Trwa 1–100ms.

Concurrent Marking — G1 równolegle z aplikacją oznacza żywe obiekty w Old Gen.

Mixed GC — czyści Young + część Old (regiony o największym garbage). Cel: <200ms.

Full GC — ostateczność gdy G1 nie nadąża. Długa pauza STW. Każdy Full GC to problem.

🔍 Finalizatory i References

finalize() — nie używaj! Nieprzewidywalne, może powodować wycieki, deprecated Java 9, usunięte Java 18.

SoftReference — GC usuwa gdy potrzebuje pamięci. Użyj jako cache.

WeakReference — GC usuwa przy następnym cyklu. WeakHashMap — cache z auto-eviction.

PhantomReference — po GC, przed zwolnieniem. Używaj do cleanup (zamiast finalize). Cleaner API (Java 9+).

🚨 Memory Leaks w Java

Static collections — rosną bez ograniczeń. Nieusuwane elementy.

ThreadLocal bez remove() — wycieki w thread poolach (dane z poprzednich requestów).

Listeners/Observers — niezderejstrowane subskrypcje trzymają obiekty żywymi.

Inner classes — nonstatic inner class trzyma ref do outer class.

Heap dump → jmap -dump:live,format=b,file=heap.hprof <pid> → analiza Eclipse MAT.

public class HotReloadClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytes = loadBytesFromDisk(name);  // czytaj .class
        return defineClass(name, bytes, 0, bytes.length);

⚠️ ClassNotFoundException vs NoClassDefFoundError

ClassNotFoundException (checked) — klasa nie znaleziona przy Class.forName() lub ClassLoader.loadClass(). Brak na classpath.

NoClassDefFoundError (error) — klasa była dostępna przy kompilacji, ale nie w runtime. Błąd w static inizjalizatorze może też ją wywołać.

LinkageError — klasa załadowana przez dwa różne ClassLoadery — problem w OSGi, Tomcat, Spring Boot fat-jar.

System Typów & Generics
Typy, Generics i Type Erasure
System typów Javy to fundament — prymitywy vs obiekty, autoboxing, generyki i kluczowy mechanizm type erasure. Bez rozumienia tych mechanizmów trudno debugować ClassCastException i pisać generyczne biblioteki.
Generics — zaawansowane użycie
// Bounded wildcards — PECS (Producer Extends, Consumer Super)

// ? extends T — Producer (odczyt): możesz CZYTAĆ T
public void sumList(List<? extends Number> list) {
    Number n = list.get(0);  // ✅ bezpieczne
    list.add(1);              // ❌ kompilator zabrani!
}

// ? super T — Consumer (zapis): możesz PISAĆ T
public void addNumbers(List<? super Integer> list) {
    list.add(42);             // ✅ bezpieczne
    Integer i = list.get(0); // ❌ Object — nie Integer
}

// Multiple bounds
public <T extends Comparable<T> & Serializable> T max(T a, T b) {
    return a.compareTo(b) >= 0 ? a : b;
}

// Type Token — omijanie type erasure
public class TypeToken<T> {
    private final Type type;
    public TypeToken() {
        // getGenericSuperclass() zachowuje parametry w runtime!
        type = ((ParameterizedType) getClass().getGenericSuperclass())
                   .getActualTypeArguments()[0];
    }
}
// Użycie: new TypeToken<List<User>>(){}.getType()
// Jackson ParameterizedTypeReference robi to samo
Type Erasure — jak działają generyki
// Generyki są tylko w czasie kompilacji!
// JVM widzi List, nie List<String>

// Kod źródłowy:
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// Po erasure (co widzi JVM):
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0); // checkcast

// Konsekwencje erasure:
new T();                            // ❌ Cannot instantiate T
new T[];                            // ❌ Generic array
list instanceof List<String>       // ❌ Illegal

// ✅ Heap pollution — kompilator ostrzega:
@SuppressWarnings("unchecked")
List<String> strings = (List<String>) new ArrayList<Integer>();

// Reifiable types (zachowane w runtime):
// int[], String[], List (raw), List<?>, int
// Non-reifiable: List<String>, Map<K,V>, T

Współbieżność
Wątki, synchronizacja i java.util.concurrent
Współbieżność to jeden z najtrudniejszych aspektów Javy. Od synchronized przez Lock, executor services, CompletableFuture, po nowe Virtual Threads — każdy mechanizm ma inne trade-offs i przypadki użycia.
Synchronizacja — mechanizmy
// ─── synchronized ────────────────────────────────────────
public synchronized void increment() { count++; }
// = synchronized(this) — blokuje instancję

synchronized (MyClass.class) { // blokuje klasę }

// ─── ReentrantLock — więcej kontroli ─────────────────────
private final Lock lock = new ReentrantLock(true); // fair!

boolean acquired = lock.tryLock(100, TimeUnit.MILLISECONDS);
if (acquired) {
    try { doWork(); }
    finally { lock.unlock(); }   // ZAWSZE w finally!
}

// ─── ReadWriteLock — wiele czytających, jeden piszący ─────
private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

// Read — wiele wątków równocześnie
rwLock.readLock().lock();
try { return cache.get(key); }
finally { rwLock.readLock().unlock(); }

// Write — wyłączny dostęp
rwLock.writeLock().lock();
try { cache.put(key, value); }
finally { rwLock.writeLock().unlock(); }
Atomic i concurrent collections
// ─── Atomic — lock-free operacje ──────────────────────────
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();         // atomowy ++
counter.compareAndSet(5, 10);    // CAS — serce lock-free

AtomicReference<State> state = new AtomicReference<>(initial);
state.updateAndGet(s -> s.transition(Event.CONFIRM)); // atomowy update

// ─── java.util.concurrent collections ────────────────────
ConcurrentHashMap<K,V> map;   // segment locking, nie blokuje całej mapy
map.computeIfAbsent(key, k -> expensive()); // atomowa operacja

CopyOnWriteArrayList<T> list; // kopia przy każdym write — read bez locka

CompletableFuture<User> userFuture = CompletableFuture

🔒 Deadlock — diagnoza

Warunki deadlocka (Coffman): Mutual exclusion, Hold & wait, No preemption, Circular wait — wszystkie cztery muszą być spełnione.

Diagnostyka: jstack <pid> → szukaj "deadlock" w output. JDK Mission Control → thread dumps. VisualVM.

Zapobieganie: acquire locki zawsze w tej samej kolejności. Używaj tryLock(timeout). Preferuj lock-free struktury.

// Semaphore — limituje concurrent access

Java Memory Model (JMM)
Visibility, Happens-Before i volatile
JMM definiuje reguły widoczności zmian w pamięci między wątkami. Bez zrozumienia happens-before nie możesz pisać poprawnego kodu wielowątkowego — problemy są nieodtwarzalne i specyficzne dla architektury CPU.
volatile — gwarancje widoczności
// Problem: CPU cache — wątek może nie widzieć zmian innych!
private boolean running = true; // ❌ może być cached w CPU

private volatile boolean running = true; // ✅
// volatile gwarantuje:
// 1. Każdy odczyt z main memory (nie cache CPU)
// 2. Każdy zapis od razu do main memory
// 3. Happens-before: zapis volatile HB odczyt tego samego pola

// ❌ volatile NIE gwarantuje atomowości złożonych operacji!
volatile int count = 0;
count++;  // ❌ nie atomic: read, increment, write = 3 operacje
// ✅ Użyj AtomicInteger dla atomowych operacji na liczbach

// Gdzie volatile wystarcza:
// - jeden wątek pisze, wiele czyta
// - boolean flagi (stop flag, initialized)
// - referencje do niezmiennych obiektów

class Singleton {
    private volatile static Singleton instance;
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null)
                    instance = new Singleton(); // volatile chroni!
            }
        }
        return instance;
    }
}
Happens-Before — reguły JMM
// Happens-Before (HB): jeśli A HB B, to B widzi efekty A

// Reguły HB w Javie:
// 1. Program Order Rule
//    Każda akcja w wątku HB kolejnej akcji w tym wątku

// 2. Monitor Lock Rule
//    Unlock monitora HB kolejny lock tego samego monitora

// 3. Volatile Variable Rule
//    Zapis volatile HB kolejny odczyt tego pola

// 4. Thread Start Rule
//    thread.start() HB jakiekolwiek działanie w nowym wątku

// 5. Thread Termination Rule
//    Wszelkie akcje w wątku HB thread.join() w innym wątku

// 6. Transitivity
//    Jeśli A HB B i B HB C, to A HB C

// Praktyczne konsekwencje:
int x = 0;
Thread t = new Thread(() -> {
    x = 42;  // Thread Start Rule → t.start() HB to
});
t.start();
t.join();
// Thread Termination Rule → join() HB ten kod
System.out.println(x);  // ✅ widzi 42 — gwarantowane
🚨 Data Race — najgroźniejszy bug współbieżności Data race = dwa wątki odczytują/zapisują to samo pole bez synchronizacji, gdy przynajmniej jeden pisze. Wynik jest nieokreślony (undefined behavior). Może się przejawiać sporadycznie, tylko na wielordzeniowych maszynach, lub tylko pod obciążeniem. Narzędzia: ThreadSanitizer (w GraalVM native), Helgrind (Valgrind), Java Thread Sanitizer. Zasada: jeśli pole jest dostępne z wielu wątków i choć jeden pisze → potrzebujesz synchronizacji.

Pattern Matching — switch (Java 21)
// Stary sposób:
if (shape instanceof Circle) {
    Circle c = (Circle) shape;
    return Math.PI * c.radius() * c.radius();
}

// Java 16+ — pattern matching instanceof:
if (shape instanceof Circle c) {  // c automatycznie Circle
    return Math.PI * c.radius() * c.radius();
}

// Java 21 — pattern matching switch (final):
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.width() * r.height();
    case Triangle t  -> t.base() * t.height() / 2.0;
    // Sealed class → compiler wie że wyczerpałeś przypadki!
};

// Guarded patterns:
case Circle c when c.radius() > 0 -> "valid circle";
case Circle c                       -> "degenerate circle";

// Record patterns (Java 21):
if (obj instanceof Point(int x, int y)) { // destrukturyzacja!
    System.out.println(x + ", " + y);
}

Streams, Lambdas & Functional
Stream API — zaawansowane wzorce
Stream API to potężne narzędzie, ale ma pułapki. Lazy evaluation, short-circuiting, parallel streams — każdy aspekt ma implikacje dla wydajności. Functional interfaces i method references to fundament nowoczesnej Javy.
Stream — zaawansowane operacje
// ─── Lazy evaluation — ważne! ────────────────────────────
// Intermediate ops (filter, map) NIE są wykonywane bez terminal op
Stream<String> s = list.stream()
    .filter(x -> { log("filter"); return x.length() > 3; })
    .map(x -> { log("map"); return x.toUpperCase(); });
// Nic nie zostało wykonane jeszcze!
s.findFirst(); // teraz: filter→map→findFirst. SHORT-CIRCUIT!

// ─── Collectors zaawansowane ──────────────────────────────
Map<String, Long> countByDept = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDept,
        Collectors.counting()));

Map<Boolean, List<Order>> partition = orders.stream()
    .collect(Collectors.partitioningBy(o -> o.getTotal() > 1000));

// Downstream collector:
Map<String, Optional<Employee>> topEarner = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDept,
        Collectors.maxBy(Comparator.comparing(Employee::getSalary))));

// Joining:
String csv = names.stream()
    .collect(Collectors.joining(", ", "[", "]")); // [a, b, c]

// Custom Collector:
Collector<Integer, ?, Statistics> stats = Collector.of(
    Statistics::new,        // supplier
    Statistics::accumulate,  // accumulator
    Statistics::combine,     // combiner (parallel)
    Statistics::finish       // finisher
);

⚡ Parallel Streams — kiedy pomagają, kiedy szkodzą Używaj gdy: duże kolekcje (>10k elementów), operacje CPU-bound, niezależne elementy. Unikaj gdy: operacje I/O-bound (zajmują ForkJoinPool.commonPool), mało danych (overhead splittingu > zysk), operacje z side effects, ordered streams z stateful operations. 🗜️ Serializacja — i dlaczego unikać

Java Serialization — problemy: security vulnerabilities (deserializacja = RCE!), kruche (zmiana klasy łamie format), wolne, duże pliki.

Performance & Benchmarking
Optymalizacja i mierzenie wydajności Java
Optymalizacja bez mierzenia to zgadywanie. JMH (Java Microbenchmark Harness) to jedyne narzędzie do rzetelnych benchmarków w JVM — eliminuje pułapki jak JIT warm-up, dead code elimination i czas GC.
JMH — właściwe benchmarki
// Dependency: org.openjdk.jmh:jmh-core
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
@State(Scope.Thread)
@Warmup(iterations = 5, time = 1)      // JIT musi się "rozgrzać"!
@Measurement(iterations = 10, time = 1)
@Fork(2)                              // osobny JVM fork
public class StringConcatBenchmark {

    @Param({"10", "100", "1000"})    // parametryzacja
    private int size;

    @Benchmark
    public String plusOperator() {
        String result = "";
        for (int i = 0; i < size; i++) result += i;
        return result;
    }

    @Benchmark
    public String stringBuilder() {
        var sb = new StringBuilder();
        for (int i = 0; i < size; i++) sb.append(i);
        return sb.toString();
    }
}
// Uruchom: java -jar benchmarks.jar -rf json -rff results.json

🔢 String Pool

Literały String są automatycznie internowane w String Pool (PermGen → Heap od Java 8).

String a = "hello";     // pool
String b = "hello";     // ten sam obiekt
a == b;                   // true!
String c = new String("hello"); // heap
a == c;                   // false
a.equals(c);              // true ✅
📏 Rozmiary obiektów

Każdy obiekt Java ma object header: 12–16 bajtów (mark word + klass pointer).

Pola wyrównywane do 8 bajtów (padding). Odniesienie: 4 bajty (compressed oops, heap <32GB) lub 8 bajtów.

jol-core (Java Object Layout) — narzędzie do sprawdzania rozmiaru obiektów w pamięci.

⚡ CPU Cache Efficiency

CPU cache line = 64 bajty. Dostęp do danych poza cache (cache miss) kosztuje ~100x więcej.

Array of Structs vs Struct of Arrays: iteracja po tablicy prymitywów jest cache-friendly.

False sharing: dwa wątki zapisują do różnych pól tego samego cache line → konflikt. @Contended (JDK 8+) dodaje padding.

Projekty OpenJDK — Przyszłość Javy
Loom, Valhalla, Panama, Amber
Cztery wielkie projekty OpenJDK definiują przyszłość Javy. Część już trafiła do produkcji (Loom → Virtual Threads), reszta jest w zaawansowanym stadium. Senior powinien rozumieć kierunek ewolucji platformy.
🧵 Project Loom — Virtual Threads

Status: DONE — Java 21 (final)

Virtual Threads, Structured Concurrency, Scoped Values. Miliony lekkich wątków bez kosztów platform threads. Blokowanie nie marnuje OS thread — carrier thread wraca do puli.

Structured Concurrency (preview Java 21) — zarządzanie grupami powiązanych wątków jako jednostką. Automatyczny cancel przy błędzie lub wyjściu ze scope.

Scoped Values (preview Java 21) — bezpieczna alternatywa dla ThreadLocal przy milionach VT. Immutable, dziedziczone przez child threads.

💎 Project Valhalla — Value Types

Status: W toku — JEP 401 (preview)

Eliminuje granicę prymityw/obiekt. Value Classes — obiekty bez identity (jak int), alokowane na stosie lub inline w tablicach. Znikają object headers dla small immutable objects.

// Przyszłość — value class
value class Point {
    double x, y;  // inline w tablicy!
}
Point[] arr = new Point[1_000_000];
// Dziś: 1M pointer + 1M object (header + fields)
// Valhalla: 1M * (x,y) — jak double[][]!
🌉 Project Panama — Native Interop

Status: DONE — Java 22 (final) / Java 21 preview

Foreign Function & Memory API — zastępuje JNI. Wywołuj natywne biblioteki C bez pisania C kodu. Bezpieczniejsze zarządzanie off-heap memory.

// Wywołanie C strlen() bez JNI:
try (Arena arena = Arena.ofConfined()) {

        .downcallHandle("strlen", ...);

}

Vector API (incubator) — SIMD instrukcje dla przetwarzania wektorowego (ML, sygnały, multimedia).

🍂 Project Amber — Language Features
