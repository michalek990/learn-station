GoF — Creational Patterns
Wzorce kreacyjne — tworzenie obiektów
Oddzielają logikę tworzenia obiektów od reszty kodu. Klasy nie tworzą obiektów bezpośrednio — zamiast tego delegują to do wyspecjalizowanych mechanizmów.
Singleton — Thread-safe (Java)
// ── Singleton — jeden z najbardziej znanych i krytykowanych ──
// Problem: jedna instancja w całej aplikacji
// Użycie: konfiguracja, logowanie, connection pool

public class DatabasePool {

  // volatile — zapobiega reorder optimization CPU
  private static volatile DatabasePool instance;

  private DatabasePool() {
    // prywatny konstruktor — nikt nie tworzy ręcznie
  }

  // Double-Checked Locking — thread-safe lazy init
  public static DatabasePool getInstance() {
    if (instance == null) {
      synchronized (DatabasePool.class) {
        if (instance == null) {
          instance = new DatabasePool();
        }
      }
    }
    return instance;
  }
}

// ── Lepiej: Enum Singleton (Joshua Bloch) ────────────────────
public enum AppConfig {
  INSTANCE;
  private final String dbUrl = "jdbc:postgresql://...";
  public String getDbUrl() { return dbUrl; }
}
// Serializacja, reflection-safe! Spring @Bean domyślnie Singleton.
Builder — złożone obiekty krok po kroku
// Problem: konstruktor z 8 parametrami = koszmar
// Rozwiązanie: płynne API budowania obiektów

public class HttpRequest {
  private final String url;
  private final String method;
  private final Map<String, String> headers;
  private final String body;
  private final int timeout;

  private HttpRequest(Builder builder) {
    this.url = builder.url;
    this.method = builder.method;
    this.headers = builder.headers;
    this.body = builder.body;
    this.timeout = builder.timeout;
  }

  public static class Builder {
    private String url;
    private String method = "GET";
    private Map<String, String> headers = new HashMap<>();
    private String body;
    private int timeout = 30;

    public Builder url(String url) { this.url=url; return this; }
    public Builder method(String m) { this.method=m; return this; }
    public Builder header(String k, String v) {
      headers.put(k,v); return this;
    }
    public Builder body(String b) { this.body=b; return this; }
    public Builder timeout(int t) { this.timeout=t; return this; }
    public HttpRequest build() { return new HttpRequest(this); }
  }
}

// Użycie — czytelne i niemutowalny wynik
HttpRequest req = new HttpRequest.Builder()
    .url("https://api.example.com/users")
    .method("POST")
    .header("Authorization", "Bearer token")
    .body("{\"name\":\"Jan\"}")
    .timeout(10)
    .build();
// @Builder z Lombok generuje to automatycznie!
Factory Method vs Abstract Factory
// ── Factory Method — podklasa decyduje co tworzyć ────────────
public abstract class NotificationSender {
  // Factory Method — abstrakcyjny, do nadpisania
  protected abstract Notification createNotification(String msg);

  public void send(String message) {
    Notification n = createNotification(message); // fabryka
    n.prepare();
    n.deliver();
  }
}

public class EmailSender extends NotificationSender {
  @Override
  protected Notification createNotification(String msg) {
    return new EmailNotification(msg);
  }
}

GoF — Structural Patterns
Wzorce strukturalne — kompozycja klas
Opisują jak składać klasy i obiekty w większe struktury. Kluczowe dla architektury warstw, integracji zewnętrznych API i rozszerzania funkcjonalności bez modyfikacji istniejącego kodu.
Adapter, Decorator, Facade
// ── ADAPTER — konwersja interfejsu ───────────────────────────
// Mamy stary PaymentGateway, potrzebujemy nowego interfejsu
public interface PaymentProcessor {
  void processPayment(BigDecimal amount, String currency);
}

public class LegacyBankAdapter implements PaymentProcessor {
  private final LegacyBankSystem legacy;

  public LegacyBankAdapter(LegacyBankSystem legacy) {
    this.legacy = legacy;
  }
  @Override
  public void processPayment(BigDecimal amount, String currency) {
    // konwertuj do starego API
    double converted = amount.doubleValue();
    legacy.executeTransfer(converted, currency.toLowerCase());
  }
}

// ── DECORATOR — dynamiczne rozszerzanie ──────────────────────
public interface DataReader {
  String read();
}
public class EncryptionDecorator implements DataReader {
  private final DataReader wrapped;
  public EncryptionDecorator(DataReader r) { wrapped = r; }
  @Override
  public String read() { return decrypt(wrapped.read()); }
}
public class CompressionDecorator implements DataReader {
  private final DataReader wrapped;
  public CompressionDecorator(DataReader r) { wrapped = r; }
  @Override
  public String read() { return decompress(wrapped.read()); }
}
// Składanie dekoratorów jak warstwy cebuli:
DataReader reader = new CompressionDecorator(
                     new EncryptionDecorator(
                     new FileDataReader("data.bin")));
Proxy, Composite, Facade
// ── PROXY — zastępnik / pośrednik ────────────────────────────
// Typy: Virtual (lazy init), Protection (auth), Remote, Cache
public class CachingServiceProxy implements UserService {
  private final UserService real;
  private final Map<Long, User> cache = new HashMap<>();

  @Override
  public User findById(Long id) {
    return cache.computeIfAbsent(id, real::findById);
  }
}
// Spring AOP = Dynamic Proxy (JDK lub CGLIB)
// @Transactional, @Cacheable — to wzorzec Proxy!

// ── COMPOSITE — drzewa, hierarchie ───────────────────────────
public interface FileSystemItem {
  long getSize();
  void print(String indent);
}
public class File implements FileSystemItem { /* leaf */ }
public class Directory implements FileSystemItem {
  private List<FileSystemItem> children = new ArrayList<>();
  public long getSize() {
    return children.stream().mapToLong(FileSystemItem::getSize).sum();
  }
}

public class OrderFacade {

GoF — Behavioral Patterns
Wzorce behawioralne — interakcje i odpowiedzialność
Najliczniejsza grupa — opisują jak obiekty współpracują i dzielą odpowiedzialność. Najczęściej używane w codziennym kodzie: Strategy, Observer, Command, Template Method, Chain of Responsibility.
Strategy — wymienna logika algorytmu
// Problem: if/else dla różnych algorytmów sortowania, płatności itp.
// Rozwiązanie: interfejs + implementacje + wstrzykiwanie

@FunctionalInterface
public interface DiscountStrategy {
  BigDecimal calculate(BigDecimal price, Customer customer);
}

// Różne implementacje
public class VipDiscountStrategy implements DiscountStrategy {
  public BigDecimal calculate(BigDecimal price, Customer c) {
    return price.multiply(new BigDecimal("0.8")); // 20% off
  }
}
public class SeasonalDiscountStrategy implements DiscountStrategy {
  public BigDecimal calculate(BigDecimal price, Customer c) {
    return price.multiply(new BigDecimal("0.9"));
  }
}

// Context używa strategii
public class OrderService {
  private final DiscountStrategy discountStrategy;

  // Wstrzyknięta przez DI — łatwa zamiana
  public OrderService(DiscountStrategy strategy) {
    this.discountStrategy = strategy;
  }

  public BigDecimal calculateTotal(Order order, Customer c) {
    return discountStrategy.calculate(order.getSubtotal(), c);
  }
}

// Java 8+ — Strategy jako lambda (bo @FunctionalInterface)
DiscountStrategy blackFriday = (price, c) -> price.multiply(
    new BigDecimal("0.5"));
OrderService svc = new OrderService(blackFriday);
Observer, Command, Template Method
// ── OBSERVER — subskrypcja na zdarzenia ──────────────────────
public interface OrderEventListener {
  void onOrderPlaced(OrderPlacedEvent event);
}
public class OrderEventPublisher {
  private List<OrderEventListener> listeners = new ArrayList<>();
  public void subscribe(OrderEventListener l) { listeners.add(l); }
  public void publish(OrderPlacedEvent event) {
    listeners.forEach(l -> l.onOrderPlaced(event));
  }
}
// Spring: ApplicationEventPublisher + @EventListener — to Observer!

// ── COMMAND — akcja jako obiekt ───────────────────────────────
public interface Command {
  void execute();
  void undo();
}
public class TransferMoneyCommand implements Command {
  private final Account from, to;
  private final BigDecimal amount;
  public void execute() { from.debit(amount); to.credit(amount); }
  public void undo()    { to.debit(amount); from.credit(amount); }
}
// Dobre dla: undo/redo, kolejkowanie, transakcje, makra

// ── TEMPLATE METHOD — szkielet algorytmu ─────────────────────
public abstract class DataMigrator {
  // Template Method — definiuje kolejność kroków
  public final void migrate() {
    extractData();
    transformData();
    loadData();
    validateResult();
  }
  protected abstract void extractData();
  protected abstract void transformData();
  protected abstract void loadData();
  protected void validateResult() { /* domyślna impl */ }
}
// Spring: AbstractController, JdbcTemplate — to Template Method!
Chain of Responsibility — łańcuch handlerów
// Problem: seria walidacji / filtrów — if/else hell
// Rozwiązanie: każdy handler przetwarza LUB przekazuje dalej

public abstract class RequestHandler {
  private RequestHandler next;

  public RequestHandler setNext(RequestHandler next) {
    this.next = next;
    return next; // fluent chain building
  }
  public abstract void handle(HttpRequest request);

  protected void passToNext(HttpRequest request) {
    if (next != null) next.handle(request);
  }
}

public class AuthHandler extends RequestHandler {
  @Override
  public void handle(HttpRequest req) {
    if (!req.hasValidToken()) throw new UnauthorizedException();
    passToNext(req);
  }
}
public class RateLimitHandler extends RequestHandler { /* ... */ }
public class LoggingHandler extends RequestHandler { /* ... */ }

// Budowanie łańcucha
RequestHandler chain = new LoggingHandler();
chain.setNext(new AuthHandler())
     .setNext(new RateLimitHandler())
     .setNext(new BusinessHandler());
// Spring Security FilterChain = Chain of Responsibility!

Architektura Systemowa
Monolit vs Mikroserwisy — kiedy co stosować
Nie ma jednej właściwej odpowiedzi — oba podejścia mają swoje miejsce. Kluczowe: złożoność operacyjna vs złożoność domenowa. Mikroserwisy nie są srebrną kulą — nieodpowiednie dla małych teamów i prostych domen.
🧱 Monolit — zalety i typy
Modular Monolith
Złoty środek — moduły z wyraźnymi granicami, wspólny deployment. Używaj do momentu gdy masz problem z skalowaniem.
Big Ball of Mud
Chaos — brak modularyzacji, wszystko powiązane ze wszystkim. Należy unikać.
Zalety monolitu
Prosty deployment, łatwe debugowanie, transakcje lokalne (ACID), brak latencji sieciowej, łatwy refactoring.
Wady monolitu
Jeden deployment dla całości, trudne skalowanie fragmentów, ryzyko coupling, jeden język/stack.
Kiedy wybrać
Startup, mały team (<10 dev), niezdefiniowane granice domenowe, ograniczone zasoby operacyjne.
🔬 Mikroserwisy — zalety i wyzwania
Zalety
Niezależny deployment, skalowanie per-serwis, różne stacki tech, izolacja błędów, małe zespoły = szybkie delivery.
Wyzwania
Distributed transactions (saga/2PC), latencja sieciowa, service discovery, monitoring rozproszony, eventual consistency.
Kiedy wybrać
Duży team (Conway's Law!), dobrze zdefiniowane Bounded Contexts, potrzeba niezależnego skalowania, wysoka dojrzałość operacyjna.
Wzorce mikro
API Gateway, Service Mesh, Sidecar, Circuit Breaker, Saga, Strangler Fig (migracja z monolitu).
Decomposition
By Business Capability (rekomendowane) lub by Subdomain (DDD). Nie by technicality!
Conway's Law — kluczowa zasada "Organizations design systems that mirror their own communication structure." — Jeśli masz 3 zespoły, dostaniesz 3-modułowy system. Mikroserwisy warto wdrażać gdy struktura organizacji już to wspiera — 2-pizza team per service.
Strangler Fig Pattern Migracja monolitu do mikroserwisów stopniowo — nowe feature w nowym serwisie, stare zawijamy proxy. Nigdy nie pisz od zera ("Big Bang rewrite")!
API Gateway Jeden punkt wejścia dla klientów — routing, auth, rate limiting, SSL termination, agregacja odpowiedzi. Przykłady: Kong, AWS API GW, Spring Cloud Gateway.
Circuit Breaker Gdy downstream serwis zawodzi — otwieramy circuit (failfast). Po czasie testujemy (half-open). Implementacje: Resilience4j, Spring Cloud Circuit Breaker.

Architektura Hexagonalna
Ports & Adapters — izolacja domeny
Architektura Hexagonalna (Alistair Cockburn, 2005) — rdzeń aplikacji (domena + logika) jest izolowany od świata zewnętrznego przez Ports (interfejsy) i Adapters (implementacje). Cel: domena nie zna Springa, JPA, HTTP, ani żadnego frameworka.
Ports & Adapters — struktura pakietów i kod
// ── Struktura projektu ──────────────────────────────────────
// src/main/java/com/example/
// ├── domain/           ← czysta Java, ZERO frameworków!
// │   ├── model/        ← Aggregates, Entities, Value Objects
// │   ├── port/
// │   │   ├── in/       ← Use Case interfaces (Driving Ports)
// │   │   └── out/      ← Repository interfaces (Driven Ports)
// │   └── service/      ← Domain Services
// ├── application/      ← Use Case implementations
// └── adapter/
//     ├── in/
//     │   ├── web/      ← REST Controllers
//     │   └── cli/      ← CLI Commands
//     └── out/
//         ├── persistence/ ← JPA Repositories
//         └── messaging/   ← Kafka Producers

// ── Driving Port (in) — Use Case interface ───────────────────
public interface PlaceOrderUseCase {
  OrderId placeOrder(PlaceOrderCommand command);
}

// ── Driven Port (out) — Repository interface ─────────────────
public interface OrderRepository {
  void save(Order order);
  Optional<Order> findById(OrderId id);
}

// ── Application Service — implementuje Port in ───────────────
@Service
public class PlaceOrderService implements PlaceOrderUseCase {
  private final OrderRepository orderRepo; // Port out
  private final PaymentPort payment;       // Port out

  @Override
  public OrderId placeOrder(PlaceOrderCommand cmd) {
    Order order = Order.create(cmd.getItems(), cmd.getCustomer());
    payment.charge(order.getTotal());
    orderRepo.save(order);
    return order.getId();
  }
}
Adapters — implementacje portów
// ── Adapter in (Driving) — REST Controller ───────────────────
@RestController
@RequestMapping("/api/orders")
public class OrderController {
  // Zależy od USE CASE INTERFACE — nie od Service bezpośrednio
  private final PlaceOrderUseCase placeOrderUseCase;

  @PostMapping
  public ResponseEntity<OrderResponse> placeOrder(
      @RequestBody PlaceOrderRequest request) {
    // mapowanie HTTP → Command (domena nie zna HTTP)
    PlaceOrderCommand command = toCommand(request);
    OrderId orderId = placeOrderUseCase.placeOrder(command);
    return ResponseEntity.ok(new OrderResponse(orderId.getValue()));
  }
}

// ── Adapter out (Driven) — JPA Repository ────────────────────
@Repository
public class JpaOrderRepository implements OrderRepository {
  private final SpringDataOrderRepo jpaRepo;
  private final OrderMapper mapper;

  @Override
  public void save(Order order) {
    // mapowanie Domain → JPA Entity (domena nie zna JPA!)
    OrderJpaEntity entity = mapper.toEntity(order);
    jpaRepo.save(entity);
  }

  @Override
  public Optional<Order> findById(OrderId id) {
    return jpaRepo.findById(id.getValue())
        .map(mapper::toDomain); // JPA Entity → Domain
  }
}

// Korzyść: możemy zamienić JPA na MongoDB lub InMemory w testach!
// OrderRepository to interfejs — testujemy use case bez bazy danych
Kluczowa zasada Hexagonal Zależności zawsze wskazują DO WEWNĄTRZ (do domeny). Domena nie zależy od niczego zewnętrznego. Adaptery zależą od Portów (interfejsów domenowych). Dzięki temu testowanie logiki biznesowej = czysty Java test, bez Springa, bez bazy.

Clean Architecture
Czysta architektura — Robert C. Martin (Uncle Bob)
Clean Architecture (2017) — 4 koncentryczne warstwy z regułą zależności: kod zewnętrzny zależy od wewnętrznego, nigdy odwrotnie. Podobna do Hexagonal, ale z wyraźniejszym podziałem na Entities → Use Cases → Interface Adapters → Frameworks.
🎯 4 warstwy Clean Architecture
1. Entities (Enterprise Business Rules)

Obiekty biznesowe — reguły które byłyby prawdziwe nawet bez komputerów. Aggregates, Value Objects, Domain Events. Zero zależności zewnętrznych.

2. Use Cases (Application Business Rules)

Logika aplikacji — co system robi. Orkiestracja Entities. Zależy tylko od Entities. Przykład: PlaceOrderUseCase, RegisterUserUseCase.

3. Interface Adapters

Controllers, Presenters, Gateways, Repositories. Konwertują dane między formatem Use Cases a formatem zewnętrznym (HTTP, DB, UI).

4. Frameworks & Drivers

Najzewnętrzniejsza warstwa — Spring, Hibernate, Angular, DB. Szczegóły implementacji. Powinna być łatwa do wymiany.

The Dependency Rule — złota zasada
// ════════════════════════════════════════════════════════════
//  DEPENDENCY RULE: zależności tylko do wewnątrz!
// ════════════════════════════════════════════════════════════
//
//  [ Frameworks ]  →  [ Interface Adapters ]
//                           ↓
//                    [ Use Cases ]
//                           ↓
//                      [ Entities ]
//
// Entities NIE ZNAJĄ Use Cases
// Use Cases NIE ZNAJĄ Controllerów
// Controllers NIE ZNAJĄ Spring Security (bezpośrednio)
// ════════════════════════════════════════════════════════════

// ── Entities — czysta logika biznesowa ───────────────────────
public class Order { // Aggregate Root
  private OrderId id;
  private CustomerId customerId;
  private List<OrderItem> items;
  private OrderStatus status;

  // Logika biznesowa TUTAJ — nie w serwisie!
  public void confirm() {
    if (status != OrderStatus.PENDING)
      throw new IllegalStateException("Tylko PENDING można potwierdzić");
    this.status = OrderStatus.CONFIRMED;
    registerEvent(new OrderConfirmedEvent(this.id));
  }

  public BigDecimal getTotal() {
    return items.stream()
        .map(OrderItem::getSubtotal)
        .reduce(BigDecimal.ZERO, BigDecimal::add);
  }
}

// ── Use Case — orchestracja ───────────────────────────────────
public class ConfirmOrderUseCase {
  private final OrderRepository orderRepo;
  private final EventPublisher events;

  public void execute(OrderId orderId) {
    Order order = orderRepo.findById(orderId)
        .orElseThrow(OrderNotFoundException::new);
    order.confirm(); // logika w Entity!
    orderRepo.save(order);
    events.publishAll(order.getDomainEvents());
  }
}
Hexagonal vs Clean Bardzo podobne! Hexagonal = Ports & Adapters. Clean = warstwy + Dependency Rule. Oba osiągają to samo: izolacja domeny. Clean Architecture jest bardziej prescriptive co do struktury warstw.
Screaming Architecture Uncle Bob: architektura powinna "krzyczeć" o tym co robi system, nie o tym jakich frameworków używa. Pakiety: order/, payment/, customer/ — nie controllers/, services/, repositories/.
Kiedy nie stosować CRUD aplikacje z prostą logiką, prototypy, narzędzia wewnętrzne. Overhead warstw dla SELECT * FROM users to over-engineering. Stosuj proporcjonalnie do złożoności domeny.

Domain-Driven Design — Strategic
DDD Strategiczne — Bounded Contexts i mapy
DDD (Eric Evans, "Blue Book" 2003) — podejście do projektowania złożonych systemów skupione na modelu domeny. Strategiczne DDD: jak dzielić duży system na mniejsze, spójne konteksty z jasną granicą odpowiedzialności i językiem.
🗣️ Ubiquitous Language

Jeden wspólny język — developerzy, eksperci domenowi, Product Owner mówią tymi samymi słowami. Język odzwierciedla się w kodzie:

❌ processItem(), handleEntity()

✅ shipOrder(), chargeCustomer()

Kiedy ekspert mówi "rezerwacja" — klasa to Reservation, nie BookingEntry. Niekonsekwencja językowa = bug w myśleniu.

🗺️ Bounded Context

Granica kontekstu — w obrębie której model jest spójny i Ubiquitous Language ma jednoznaczne znaczenie.

Przykład: "User" w Identity Context (login, auth) ≠ "Customer" w Sales Context (koszyk, zamówienia) ≠ "Passenger" w Shipping Context. Ten sam człowiek — różne modele!

Bounded Context ≈ mikroserwis (ale niekoniecznie — może być modułem w monolicie).

// ACL chroni nasz model przed "zatruciem" modelem CRM

// ── Zewnętrzny model CRM (nie nasz!) ─────────────────────────
public class CrmAccountDto {
  public String account_id;          // snake_case, brzydkie
  public String first_name_field;
  public String billing_address_line1;
  public String billing_address_city;
  public boolean is_vip_flag;
}

// ── Nasz model domenowy (czysty) ─────────────────────────────
public record Customer(CustomerId id, CustomerName name,
                         Address billingAddress, boolean vip) {}

// ── Anti-Corruption Layer — translator ───────────────────────
@Component
public class CrmToOrdersTranslator {

  public Customer translate(CrmAccountDto crmAccount) {
    return new Customer(
        new CustomerId(crmAccount.account_id),
        new CustomerName(crmAccount.first_name_field),
        new Address(crmAccount.billing_address_line1,
                    crmAccount.billing_address_city),
        crmAccount.is_vip_flag
    );
  }
}

// ── Port do zewnętrznego systemu ─────────────────────────────
public interface CustomerLookupPort {
  Optional<Customer> findCustomer(CustomerId id);
}

@Component
public class CrmCustomerAdapter implements CustomerLookupPort {
  private final CrmApiClient crm;
  private final CrmToOrdersTranslator translator;

  public Optional<Customer> findCustomer(CustomerId id) {
    return crm.getAccount(id.getValue())
               .map(translator::translate); // ACL tu działa
  }
}

Domain-Driven Design — Tactical
DDD Taktyczne — building blocks
Taktyczne DDD to wzorce budowania modelu domeny: Aggregates, Entities, Value Objects, Domain Events, Repositories, Domain Services. Każdy ma konkretną rolę i zasady.
Aggregate, Entity, Value Object
// ── VALUE OBJECT — niemutowalny, bez tożsamości ──────────────
// Porównywane przez wartość, nie referencję. Zawsze niemutowalne.
public record Money(BigDecimal amount, String currency) {
  public Money {  // compact constructor — walidacja
    Objects.requireNonNull(amount, "Amount required");
    if (amount.compareTo(BigDecimal.ZERO) < 0)
      throw new IllegalArgumentException("Kwota nie może być ujemna");
  }
  public Money add(Money other) {
    if (!currency.equals(other.currency))
      throw new IllegalArgumentException("Różne waluty!");
    return new Money(amount.add(other.amount), currency);
  }
}
// Money(100, "PLN").equals(Money(100, "PLN")) → true (record!)

// ── ENTITY — identyfikowana przez ID ─────────────────────────
public class OrderItem { // Entity (wewnątrz Aggregate)
  private final OrderItemId id; // tożsamość!
  private ProductId productId;
  private int quantity;
  private Money unitPrice;

  public Money getSubtotal() {
    return new Money(unitPrice.amount().multiply(
        new BigDecimal(quantity)), unitPrice.currency());
  }
}

// ── AGGREGATE ROOT — granica transakcyjna ────────────────────
public class Order { // Aggregate Root
  private final OrderId id;
  private CustomerId customerId;
  private final List<OrderItem> items = new ArrayList<>();
  private OrderStatus status;
  private final List<DomainEvent> events = new ArrayList<>();

  // ZAWSZE modyfikuj items przez AR — nie bezpośrednio!
  public void addItem(ProductId pid, int qty, Money price) {
    if (status != OrderStatus.DRAFT)
      throw new DomainException("Nie można dodać do potwierdzonego");
    items.add(new OrderItem(...));
  }

  public void submit() {
    if (items.isEmpty()) throw new DomainException("Brak pozycji");
    status = OrderStatus.SUBMITTED;
    events.add(new OrderSubmittedEvent(id, customerId, getTotal()));
  }

  public List<DomainEvent> pullEvents() {
    List<DomainEvent> result = new ArrayList<>(events);
    events.clear(); return result;
  }
}

// Reaction: EmailService, InventoryService, AnalyticsService

// ── REPOSITORY — abstrakcja kolekcji agregatów ───────────────
public interface OrderRepository {
  // CRUD przez Aggregate Root — nigdy przez child entities!
  void save(Order order);
  Optional<Order> findById(OrderId id);
  List<Order> findByCustomer(CustomerId customerId);
  // Brak: findByItemProductId() — to narusza granicę AR!
}

// ── DOMAIN SERVICE — logika między agregatami ────────────────
// Kiedy: logika nie pasuje do żadnego Aggregate
public class TransferMoneyService { // Domain Service
  public void transfer(Account from, Account to, Money amount) {
    from.debit(amount);
    to.credit(amount);
    // Nie pasuje do Account — operuje na 2 agregatach
  }
}
// UWAGA: nie mylić z Application Service (orchestracja)
// Domain Service: czysta logika domenowa, bez IO
// Application Service: orchestracja + IO (repo, events)

// ── SPECIFICATION — enkapsulacja reguły biznesowej ───────────
public class EligibleForDiscount implements Specification<Customer> {
  @Override
  public boolean isSatisfiedBy(Customer customer) {
    return customer.isVip() &&
           customer.getTotalSpend().compareTo(
               new Money(new BigDecimal("1000"), "PLN")) > 0;
  }
}
// Kompozycja: eligible.and(new ActiveCustomer()).isSatisfiedBy(c)
🏗️ Aggregate Rules

1. Granica transakcyjna — 1 AR = 1 transakcja

2. Referencja do innych AR tylko przez ID

3. Modyfikacje tylko przez AR, nie dzieci

4. Małe Aggregates — łatwiejsze concurrency

5. Eventual consistency między AR jest OK

💎 Value Object

Niemutowalny — brak setterów

Porównanie przez wartość — equals/hashCode

Self-validating — walidacja w konstruktorze

Przykłady: Money, Email, Address, PhoneNumber, Percentage, DateRange

Java 16+ record = idealny dla VO

📨 Domain Events

Fact — coś się stało, niezmienne, past tense

Decoupling — reakcje bez bezpośrednich wywołań

Transactional Outbox — publi-kuj po commicie

Naming: OrderSubmitted, PaymentProcessed, UserRegistered

🔧 Domain vs Application Service

Domain Service: czysta logika biznesowa, operuje na kilku AR, zero IO

Application Service: orkiestracja — wczytaj z repo, wywołaj domenę, zapisz, opublikuj event

Błąd: "Anemic Domain Model" — logika w serwisach, modele = czyste data klasy
