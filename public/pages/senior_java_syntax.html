Typy Primitywne & Operatory
Integer overflow, rzutowanie, kolejność operatorów
Najbardziej podstawowe pułapki — wyniki które są "oczywiste" ale Java zwraca coś innego. Overflow, integer division, pre/post increment, bitwise ops.
Overflow, casting, integer arithmetic
// ══ OVERFLOW ══════════════════════════════════════
int max = Integer.MAX_VALUE;   // 2147483647
System.out.println(max + 1);    // ➜ -2147483648 (wraparound!)
System.out.println(max * 2);    // ➜ -2 (overflow)

long result = 1_000_000 * 1_000_000;  // ➜ -727379968 !!!!
long ok     = 1_000_000L * 1_000_000; // ➜ 1000000000000 (L suffix!)

// ══ INTEGER DIVISION ══════════════════════════════
System.out.println(5 / 2);     // ➜ 2 (nie 2.5!)
System.out.println(5 / 2.0);   // ➜ 2.5
System.out.println(5.0 / 2);   // ➜ 2.5
System.out.println(-7 / 2);    // ➜ -3 (truncation toward zero)
System.out.println(-7 % 2);    // ➜ -1 (sign = sign of dividend)

// ══ PRE vs POST INCREMENT ═════════════════════════
int a = 5;
int b = a++;   // b=5, a=6
int c = ++a;   // c=7, a=7
int d = a++ + ++a;  // d = 7 + 9 = 16, a=9

// ══ CASTING PUŁAPKI ═══════════════════════════════
byte x = (byte) 200;   // ➜ -56 (200 - 256)
byte y = (byte) 300;   // ➜ 44  (300 - 256)
int  z = (int) 9.9;    // ➜ 9   (truncation, nie rounding)
char ch = (char) 65;   // ➜ 'A'
int  n = (int) 'A';    // ➜ 65

// ══ BITWISE ═══════════════════════════════════════
System.out.println(5 & 3);    // ➜ 1  (0101 & 0011 = 0001)
System.out.println(5 | 3);    // ➜ 7  (0101 | 0011 = 0111)
System.out.println(5 ^ 3);    // ➜ 6  (0101 ^ 0011 = 0110)
System.out.println(5 << 1);   // ➜ 10 (multiply by 2)
System.out.println(-1 >> 1);  // ➜ -1 (arithmetic shift, sign preserved)
System.out.println(-1 >>> 1); // ➜ 2147483647 (logical shift, zero fill)

// ══ CONDITIONAL OPERATOR TRAP ══════════════════════
Object o = true ? new Integer(1) : 2.0;
// typ: double! ternary unboxuje Integer do int, promuje do double
// potem autoboxuje do Double → o instanceof Double == true !
char w wyrażeniach — klasyczna pułapka
// char jest numeryczny — promuje się do int w wyrażeniach
char c1 = 'a';
char c2 = 'b';
// System.out.println(c1 + c2);  ➜ 195 (97 + 98) NIE "ab"!
System.out.println("" + c1 + c2);  // ➜ "ab" (String concat)
System.out.println(c1 + "" + c2);  // ➜ "ab"
System.out.println((char)(c1 + 1)); // ➜ 'b'

// String concatenation evaluation order
int i = 1, j = 2;
System.out.println("sum: " + i + j);   // ➜ "sum: 12" (left-to-right)
System.out.println("sum: " + (i + j)); // ➜ "sum: 3"
System.out.println(i + j + " sum");   // ➜ "3 sum" (i+j=3 first!)
Operator precedence — najczęstsze błędy
// && ma wyższy precedence niż ||
boolean r = true || false && false;
// = true || (false && false) = true || false = true

// Short-circuit evaluation
int x = 0;
boolean b = (x != 0) && (10/x > 1); // ➜ false (nie ArithmeticException)
boolean c = (true)  || (1/0 > 0);   // ➜ true  (nie ArithmeticException)

// Bitwise & vs &&  — BRAK short-circuit!
boolean d = (false) & (++x > 0);  // x = 1 (side effect wykonany!)
boolean e = (false) && (++x > 0); // x = 1 (nie zmienił się!)

// Compound assignment z rzutowaniem
byte bb = 10;
bb += 5;    // OK — += zawiera implicit cast: bb = (byte)(bb + 5)
// bb = bb + 5;  // BŁĄD kompilacji! bb + 5 = int, trzeba cast
Klasyczne pytanie: co wydrukuje? System.out.println(1 + 2 + "3" + 4 + 5) → "3345". Ewaluacja lewa→prawa: 1+2=3, 3+"3"="33", "33"+4="334", "334"+5="3345".

Autoboxing & Integer Cache
Integer cache -128..127, NullPointerException, == vs equals
Autoboxing to mechanizm automatycznej konwersji między prymitywami a ich wrapperami. Ukrywa pułapki: Integer cache powoduje że == działa tylko dla małych liczb, unboxing null = NPE.
Integer Cache — pułapka na rozmowach
// JVM cache'uje Integer od -128 do 127
Integer a = 127;
Integer b = 127;
System.out.println(a == b);        // ➜ true  (ten sam obiekt z cache)

Integer c = 128;
Integer d = 128;
System.out.println(c == d);        // ➜ false (nowe obiekty!)
System.out.println(c.equals(d));   // ➜ true  (zawsze używaj equals)

// Long, Short, Byte, Character też mają cache (-128..127)
Long l1 = 100L, l2 = 100L;
System.out.println(l1 == l2);      // ➜ true

// Mieszanie typów — unboxing
Integer x = 10;
int     y = 10;
System.out.println(x == y);        // ➜ true (x unboxed to int)

Integer p = 200;
Integer q = 200;
System.out.println(p == q);        // ➜ false
System.out.println(p == (q + 0));  // ➜ true! q+0 unboxuje q do int → unboxuje p

// NPE z unboxing
Integer nullInt = null;
int     val = nullInt;  // NullPointerException przy unboxing!

// Subtelna pułapka w wyrażeniach warunkowych
Integer n = null;
int result = n != null ? n : 0;  // OK
int trap   = true ? n : 0;        // NPE! 0 wymusza int → unboxing n=null
Autoboxing w kolekcjach i switch
// Autoboxing w operacjach arytmetycznych na kolekcjach
List<Integer> list = new ArrayList<>();
list.add(1); list.add(2); list.add(3);

// remove(int index) vs remove(Object o) — overload!
list.remove(1);            // ➜ usuwa INDEX 1, pozostaje [1,3]
list.remove(Integer.valueOf(1)); // ➜ usuwa WARTOŚĆ 1

// Switch z Integer — unboxing NPE risk
Integer val = null;
switch (val) {  // ➜ NullPointerException! (unboxing null)
  case 1: break;
}

// Performance pułapka — autoboxing w pętli
Long sum = 0L;
for (long i = 0; i < 1_000_000; i++) {
  sum += i;  // unbox + add + box przy każdej iteracji!
}             // milion zbędnych obiektów Long → GC pressure

// Prawidłowo:
long sumOk = 0L;
for (long i = 0; i < 1_000_000; i++) {
  sumOk += i;
}
Zasada: nigdy == dla wrapperów Jedyny bezpieczny wyjątek: porównanie z null (x == null). Wszędzie indziej: equals() dla wartości, Objects.equals(a, b) gdy któryś może być null. Pamiętaj że Boolean.TRUE == Boolean.TRUE działa (cache), ale nie polegaj na tym.

String Pool & Immutability
String pool, intern(), immutability, StringBuilder vs StringBuffer
String w Javie jest immutable i interned. Literały idą do string pool (PermGen → Metaspace od Java 8). new String() tworzy nowy obiekt poza poolem. To generuje klasyczne pytania na rozmowach.
String pool — == vs equals, intern()
// Literały → String Pool (współdzielone)
String s1 = "hello";
String s2 = "hello";
System.out.println(s1 == s2);       // ➜ true  (ten sam obiekt z pool)
System.out.println(s1.equals(s2));  // ➜ true

// new String() → poza poolem
String s3 = new String("hello");
System.out.println(s1 == s3);       // ➜ false (różne obiekty)
System.out.println(s1.equals(s3));  // ➜ true

// intern() — wróć do pool
String s4 = s3.intern();
System.out.println(s1 == s4);       // ➜ true!

// Concatenacja compile-time (stałe) → pool
String s5 = "hel" + "lo";           // compile-time → "hello" w pool
System.out.println(s1 == s5);       // ➜ true!

// Concatenacja runtime → nowy obiekt (poza pool)
String part = "hel";
String s6 = part + "lo";            // runtime → nowy obiekt
System.out.println(s1 == s6);       // ➜ false!

// final String → compile-time constant!
final String fp = "hel";
String s7 = fp + "lo";              // compile-time (final!)
System.out.println(s1 == s7);       // ➜ true!
Immutability, StringBuilder, pułapki metod
// String jest immutable — modyfikacje tworzą nowy obiekt
String s = "hello";
s.toUpperCase();          // nic nie robi — wynik jest ignorowany!
System.out.println(s);    // ➜ "hello" (nie zmienił się)

s = s.toUpperCase();      // ➜ "HELLO" (przypisz wynik!)

// substring() — pułapka memory leak (Java 6)
// Java 7+ substring tworzy nową kopię — brak leak
String big = "..." ; // milion znaków
String sub = big.substring(0, 5); // Java 7+: nowa kopia (bezpieczne)

// equals vs equalsIgnoreCase vs compareTo
System.out.println("abc".compareTo("abd"));     // ➜ -1 (leksykograficznie)
System.out.println("abc".compareTo("ABC"));     // ➜ 32 (różnica ASCII)
System.out.println(null.equals("x"));          // ➜ NPE!
System.out.println("x".equals(null));          // ➜ false (bezpieczne)

// StringBuilder NIE jest thread-safe, StringBuffer TAK
// StringBuilder szybszy (brak synchronizacji)
StringBuilder sb = new StringBuilder("hello");
sb.reverse();             // ➜ "olleh" (in-place!)
sb.insert(0, "!");        // ➜ "!olleh"
sb.delete(0, 1);          // ➜ "olleh"
sb.replace(0, 2, "XX"); // ➜ "XXleh"

// String.format vs + w pętli
// W pętli: StringBuilder.append() >> "str" + val (n² copies)
String switch — jak działa switch(str) używa hashCode() + equals() — nie ==. Bezpieczne dla wartości z pool i z new String(). Ale uwaga: switch(null) = NullPointerException zawsze!

OOP & Polymorphism
Overloading vs overriding, covariant return, this/super, inicjalizacja
Polimorfizm i dziedziczenie mają mnóstwo subtelnych zasad. Overloading rozwiązywany w compile-time, overriding w runtime. Kolejność inicjalizacji statycznych pól vs instancyjnych — klasyczny materiał na pytania.
Overloading — compile-time, zaskakujące wybory
// Overloading: metoda wybierana na podstawie DEKLAROWANEGO typu
class Printer {
  void print(Object o)  { System.out.println("Object"); }
  void print(String s)  { System.out.println("String"); }
  void print(Integer i) { System.out.println("Integer");}
}

Printer p = new Printer();
Object  o = "hello";   // declared type: Object, actual: String
p.print(o);            // ➜ "Object"! (compile-time typ = Object)
p.print("hello");      // ➜ "String"  (compile-time typ = String)
p.print((String) o);   // ➜ "String"  (cast = compile-time info)

// Overloading z null — niejednoznaczność
// p.print(null); // BŁĄD KOMPILACJI - niejednoznaczne (String? Integer?)
p.print((String) null); // OK — wybrany print(String)

// Varargs vs konkretne typy
static void f(int a, int b) { System.out.println("two ints"); }
static void f(int... a)     { System.out.println("varargs"); }
f(1, 2);   // ➜ "two ints" (konkretna wersja ma priorytet!)
f(1,2,3); // ➜ "varargs"
Kolejność inicjalizacji — klasyczne pytanie
class Base {
  static { System.out.println("1: Base static"); }
  { System.out.println("3: Base instance block"); }
  Base() { System.out.println("4: Base constructor"); }
}

class Child extends Base {
  static { System.out.println("2: Child static"); }
  { System.out.println("5: Child instance block"); }
  Child() { System.out.println("6: Child constructor"); }
}

new Child();
// Wydrukuje kolejno 1,2,3,4,5,6
// Kolejność: static parent → static child
//            → instance parent → ctor parent
//            → instance child → ctor child

// PUŁAPKA: wywołanie overridden metody w konstruktorze!
class A {
  int x = 10;
  A() { print(); }  // wywołuje overridden metodę!
  void print() { System.out.println("A.x=" + x); }
}
class B extends A {
  int y = 20;
  B() { super(); }
  @Override
  void print() { System.out.println("B.y=" + y); }
}
new B(); // ➜ "B.y=0" !!! y jeszcze nie zainicjalizowane (=0 default)
// NIGDY nie wywołuj metod wirtualnych w konstruktorze!
Hiding vs Overriding — static methods Static metody nie są overridowane — są hidden. Parent p = new Child(), p.staticMethod() wywoła metodę z Parent (compile-time typ), nie Child. Static metody nie uczestniczą w dynamicznym polimorfizmie.
Covariant return, final, interface defaults — zaawansowane
// Covariant return type — override może zwrócić podtyp
class Animal { Animal create() { return new Animal(); } }
class Dog extends Animal {
  @Override
  Dog create() { return new Dog(); }  // OK — Dog extends Animal
}

// Interface default method conflict — class wins
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B {
  @Override
  public void hello() { A.super.hello(); }  // WYMAGANE — niejednoznaczność
}

// final field vs effectively final — lambda/anonymous class
int i = 10;  // effectively final — nie deklarujemy final ale nie zmieniamy
Runnable r = () -> System.out.println(i);  // OK
// i++; — po tej linii: BŁĄD KOMPILACJI w lambdzie!

Generics & Type Erasure
Type erasure, wildcards, bounded types, heap pollution
Generics są compile-time only — w runtime wszystkie parametry typów są erasowane do Object lub upper bound. To wyjaśnia dlaczego nie możesz new T() ani instanceof List<String>.
Type Erasure — konsekwencje
// Type erasure — w bytecode List<String> == List<Integer> == List
List<String>  strings = new ArrayList<>();
List<Integer> ints    = new ArrayList<>();
System.out.println(strings.getClass() == ints.getClass()); // ➜ true!

// Nie możesz: instanceof z parametrem typów
// if (list instanceof List<String>) {}  // BŁĄD KOMPILACJI
if (strings instanceof List<?>) {}      // OK — wildcard unbounded

// Nie możesz: tworzyć tablic sparametryzowanych typów
// List<String>[] arr = new List<String>[10]; // BŁĄD (unchecked)
List<?>[] arr = new List[10];              // OK z warnings

// Wildcards: <? extends T> vs <? super T> — PECS rule
// Producer Extends, Consumer Super

// PRODUCER (czytasz z kolekcji) → ? extends T
static double sum(List<? extends Number> list) {
  return list.stream().mapToDouble(Number::doubleValue).sum();
}
// CONSUMER (piszesz do kolekcji) → ? super T
static void addNumbers(List<? super Integer> list) {
  list.add(1); list.add(2);  // OK
}

// Wywołania:
sum(new ArrayList<Integer>());   // OK
sum(new ArrayList<Double>());    // OK
addNumbers(new ArrayList<Number>()); // OK (Number super Integer)
Heap pollution, @SafeVarargs, type tokens
// Heap Pollution — mieszanie raw types i generics
List rawList = new ArrayList<String>();
rawList.add(42);              // unchecked warning — kompiluje się!
List<String> strings = rawList; // unchecked warning
String s = strings.get(0);    // ➜ ClassCastException w runtime!

// @SafeVarargs — suppresses heap pollution warnings
@SafeVarargs
static <T> List<T> listOf(T... elements) {
  return Arrays.asList(elements);
}

// Nie możesz: new T() → użyj Supplier lub Class<T> (type token)
static <T> T create(Class<T> clazz) throws Exception {
  return clazz.getDeclaredConstructor().newInstance();
}
// Użycie: create(String.class)

// Generic method — type inference
static <T extends Comparable<T>> T max(T a, T b) {
  return a.compareTo(b) >= 0 ? a : b;
}
Integer maxInt = max(3, 7);    // type inferred: T=Integer
String  maxStr = max("a", "b");// type inferred: T=String

// Multiple bounds
static <T extends Comparable<T> & Serializable> void store(T t) {}

Exceptions
try-catch-finally, multi-catch, try-with-resources, pułapki zwracania
Wyjątki mają dużo nieintuicyjnych zachowań — finally który nadpisuje return, exception który "gubi" poprzedni, multi-catch który zabrania dziedziczenia.
finally i return — co wygrywa
// finally ZAWSZE się wykonuje (poza System.exit() i kill)
static int tricky() {
  try {
    return 1;   // zapisuje "1" do return value slot
  } finally {
    return 2;   // nadpisuje! finally wins
  }
}
// tricky() ➜ 2

// Exception w finally "połyka" exception z try
static void danger() {
  try {
    throw new RuntimeException("from try");
  } finally {
    throw new RuntimeException("from finally"); // "from try" znika!
  }
}

// Zmienna lokalna w finally — NIE nadpisuje return value
static int notTricky() {
  int x = 1;
  try {
    return x;   // zwraca kopię x = 1
  } finally {
    x = 2;      // zmienia x ale nie zwracaną wartość!
  }
}
// notTricky() ➜ 1 (nie 2!)
Multi-catch, try-with-resources, suppressed
// Multi-catch — typy nie mogą być w relacji dziedziczenia
try { ... }
catch (IOException | SQLException e) {  // OK — niezwiązane
  // e jest effectively final — nie można przypisać nowej wartości!
}
// catch (Exception | IOException e) {}  // BŁĄD KOMPILACJI!
// IOException extends Exception — redundant

// Try-with-resources — close() wywoływane automatycznie
try (InputStream is = new FileInputStream("f");
     OutputStream os = new FileOutputStream("g")) {
  // os.close() wywoływane PRZED is.close() (LIFO)
  // Exception z close() → Suppressed exception
}

// Suppressed exceptions — Java 7+
try {
  throw new RuntimeException("primary");
} catch (RuntimeException e) {
  e.addSuppressed(new Exception("suppressed"));
  throw e;
}
// e.getSuppressed() → tablica suppressed exceptions

// Checked vs Unchecked
// Checked: extends Exception (nie RuntimeException)
// Unchecked: extends RuntimeException lub Error
// Error: OutOfMemoryError, StackOverflowError — nie catch!

// StackOverflowError — rekurencja bez base case
static void infinite() { infinite(); }  // StackOverflowError
Rethrow z Java 7 catch (Exception e) { throw e; } — Java 7+ jest wystarczająco mądra żeby wiedzieć że jeśli w try może lecieć tylko IOException, to rethrow jest bezpieczny jako checked. Kompilator wie dokładnie co leci — nie musi być throws Exception.

Lambdy & Streams
Functional interfaces, method references, lazy evaluation, pułapki streamów
Lambdy i Stream API mają wiele subtelnych zachowań. Streams są lazy — terminal operation go uruchamia. Stream nie może być użyty dwa razy. Pułapki z modyfikacją kolekcji podczas streamowania.
Stream — lazy, single-use, pułapki
// Stream jest LAZY — intermediate ops nie wykonują się bez terminal
Stream<Integer> stream = Stream.of(1, 2, 3)
  .filter(n -> { System.out.println("filter: "+n); return n > 1; })
  .map(n -> { System.out.println("map: "+n); return n * 2; });
// Nic nie wydrukowano jeszcze!
stream.forEach(System.out::println);
// Teraz: filter:1, filter:2, map:2, 4, filter:3, map:3, 6
// Vertical processing — każdy element przez cały pipeline

// Stream single-use — drugi terminal operation = exception
Stream<String> s = Stream.of("a", "b");
s.forEach(System.out::println);
// s.count(); // ➜ IllegalStateException: stream already operated upon

// findFirst vs findAny — parallel streams
Optional<String> first = Stream.of("a","b","c")
  .parallel().filter(x -> x.compareTo("b") >= 0)
  .findFirst();  // zawsze "b" — deterministyczny

Optional<String> any = Stream.of("a","b","c")
  .parallel().filter(x -> x.compareTo("b") >= 0)
  .findAny();    // może być "b" lub "c" — niedeterministyczny

// reduce — akumulator musi być asocjatywny dla parallel
int sum = IntStream.rangeClosed(1, 10)
  .reduce(0, Integer::sum);  // ➜ 55

// collect vs toList (Java 16+)
List<String> mutable   = stream.collect(Collectors.toList());   // mutable
List<String> immutable = stream.toList();                        // immutable (Java 16+)
Method references, Optional, functional interfaces
// 4 typy method references
// 1. Static method
Function<String, Integer> f1 = Integer::parseInt;

// 2. Instance method konkretnego obiektu
String str = "hello";
Supplier<String> f2 = str::toUpperCase;

// 3. Instance method arbitrary obiektu (pierwszy arg = this)
Function<String, String> f3 = String::toUpperCase;
// equivalent: s -> s.toUpperCase()

// 4. Constructor reference
Supplier<ArrayList> f4 = ArrayList::new;

// Optional — pułapki
Optional<String> opt = Optional.of("hello");
opt.get();              // OK, ale rzuca NoSuchElementException gdy empty
opt.orElse("default"); // bezpieczne

// orElse vs orElseGet — różnica gdy tworzysz obiekt!
Optional<String> present = Optional.of("val");
present.orElse(expensiveOp());      // expensiveOp() ZAWSZE wywoływana
present.orElseGet(() -> expensive()); // expensiveOp() tylko gdy empty!

// flatMap dla Optional — unikaj nested Optional
Optional<String> nested =
  Optional.of("hello")
    .flatMap(s -> Optional.of(s.toUpperCase()));

// NIE rób Optional.get() bez isPresent() — użyj ifPresent/map
opt.ifPresent(System.out::println);
String upper = opt.map(String::toUpperCase).orElse("");

Concurrency
volatile, synchronized, happens-before, AtomicXxx, pułapki wątków
Współbieżność w Javie to temat-rzeka na rozmowach. volatile nie zastępuje synchronizacji, synchronized ma semantykę happens-before, i++ nie jest atomowy nawet dla int.
volatile, happens-before, pułapki
// volatile — widoczność, NIE atomowość złożonych operacji
volatile boolean running = true;   // OK dla flag boolean
volatile int counter = 0;
// counter++ NIE jest atomowe! = read + increment + write
// Dwa wątki: obydwa czytają 0, obydwa zapisują 1 = lost update

// Poprawnie: AtomicInteger
AtomicInteger atomic = new AtomicInteger(0);
atomic.incrementAndGet();    // atomowy
atomic.compareAndSet(5, 6); // CAS — set 6 tylko jeśli jest 5

// synchronized — happens-before guarantee
class Counter {
  private int count = 0;
  synchronized void increment() { count++; }  // lock na this
  static synchronized void staticOp() { }     // lock na Class
}

// Double-checked locking — klasyczny pattern (Java 5+)
class Singleton {
  private volatile static Singleton instance; // MUST be volatile
  static Singleton getInstance() {
    if (instance == null) {
      synchronized (Singleton.class) {
        if (instance == null) {       // drugi check
          instance = new Singleton(); // BEZ volatile: reorder risk!
        }
      }
    }
    return instance;
  }
}
// Lepiej: Initialization-on-demand holder idiom:
class Better {
  private static class Holder { static final Better INSTANCE = new Better(); }
  static Better get() { return Holder.INSTANCE; }
}
wait/notify, ThreadLocal, CompletableFuture
// wait() MUSI być w synchronized — inaczej IllegalMonitorStateException
synchronized (obj) {
  while (!condition) {   // WHILE nie IF — spurious wakeups!
    obj.wait();
  }
  // praca z condition
  obj.notifyAll();       // notify vs notifyAll — bezpieczniej notifyAll
}

// ThreadLocal — per-thread state
ThreadLocal<SimpleDateFormat> sdf = 
  ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
// PAMIĘTAJ: remove() po użyciu w thread pool (thread reuse!)
try {
  sdf.get().format(new Date());
} finally {
  sdf.remove();  // bez tego: memory leak + wrong state w pooled threads
}

// CompletableFuture — async chain
CompletableFuture.supplyAsync(() -> fetchData())
  .thenApply(String::toUpperCase)  // same thread (often)
  .thenApplyAsync(String::trim)    // ForkJoinPool.commonPool()
  .exceptionally(e -> "error")
  .thenAccept(System.out::println);

// join() vs get() — join rzuca unchecked (CompletionException)
// get() rzuca checked InterruptedException + ExecutionException
Deadlock — 4 warunki Coffmana Mutex (zasoby wyłączne) + Hold & Wait (trzymasz i czekasz) + No Preemption (nie można odebrać) + Circular Wait (cykl zależności). Zapobieganie: zawsze lockuj w tej samej kolejności. Diagnoza: jstack <pid> → szukaj "BLOCKED" i "waiting to lock".

Collections & equals/hashCode
equals/hashCode kontrakt, ConcurrentModificationException, iterator
equals() i hashCode() mają kontrakt który musisz znać. Złamanie go powoduje że obiekty znikają w HashSet lub duplikują się. ConcurrentModificationException przy modyfikacji podczas iteracji to klasyczna pułapka.
equals/hashCode kontrakt — skutki złamania
// Kontrakt equals(): reflexive, symmetric, transitive, consistent
// Kontrakt hashCode(): jeśli a.equals(b) to a.hashCode() == b.hashCode()
// Odwrotnie NIE musi być! (hash collision = OK)

// Klasa BEZ override hashCode — złamanie kontraktu
class Bad {
  int id;
  Bad(int id) { this.id = id; }
  @Override
  public boolean equals(Object o) {
    return o instanceof Bad b && b.id == this.id;
  }
  // BRAK hashCode() → używa Object.hashCode() (address-based)
}

Bad b1 = new Bad(1), b2 = new Bad(1);
b1.equals(b2);                        // ➜ true
new HashSet<>().add(b1); add(b2);    // ➜ 2 elementy! (różne hash buckets)
new HashMap<>().put(b1, "x").get(b2); // ➜ null!!! (b2 w złym bucket)

// Poprawny hashCode dla powyższej klasy:
@Override
public int hashCode() { return Integer.hashCode(id); }

// Mutable key w HashMap — katastrofa!
List<Integer> key = new ArrayList<>(Arrays.asList(1, 2));
Map<List<Integer>, String> map = new HashMap<>();
map.put(key, "value");
key.add(3);          // hashCode zmienił się!
map.get(key);         // ➜ null! (szuka w złym buckecie)
ConcurrentModificationException, iterator, fail-fast
// ConcurrentModificationException — modyfikacja podczas iteracji
List<Integer> list = new ArrayList<>(Arrays.asList(1,2,3,4));

// ❌ Pętla for-each + remove → ConcurrentModificationException
for (Integer i : list) {
  if (i % 2 == 0) list.remove(i); // ➜ ConcurrentModificationException
}

// ✅ Iterator.remove() — bezpieczne
Iterator<Integer> it = list.iterator();
while (it.hasNext()) {
  if (it.next() % 2 == 0) it.remove(); // OK
}

// ✅ removeIf (Java 8) — najlepsze
list.removeIf(i -> i % 2 == 0);

// ✅ Stream filter — nowa lista
List<Integer> filtered = list.stream()
  .filter(i -> i % 2 != 0)
  .collect(Collectors.toList());

// Fail-fast vs fail-safe iteratory
// ArrayList, HashMap → fail-fast (rzuca CME)
// CopyOnWriteArrayList, ConcurrentHashMap → fail-safe (kopia)

Records, Sealed classes, Pattern Matching, Switch Expressions, Text Blocks
Nowoczesna Java ma eleganckie features które pojawiają się na rozmowach. Records (niezmienne data classes), sealed classes (zamknięta hierarchia), pattern matching w instanceof i switch.
Records, Sealed Classes, Pattern Matching
// ══ RECORDS (Java 16) ════════════════════════════
// Auto-generuje: constructor, getters, equals, hashCode, toString
record Point(int x, int y) {
  // Compact constructor — walidacja
  Point {
    if (x < 0 || y < 0) throw new IllegalArgumentException();
  }
  // Można dodać metody i static fields (nie instance!)
  double distance() { return Math.sqrt(x*x + y*y); }
}
// Getter to p.x() nie p.getX()!
Point p = new Point(3, 4);
p.x();   // ➜ 3
// p.x = 5;  // BŁĄD — fields są final!

// ══ SEALED CLASSES (Java 17) ══════════════════
sealed interface Shape permits Circle, Rectangle, Triangle {}
record Circle(double r) implements Shape {}
record Rectangle(double w, double h) implements Shape {}
final class Triangle implements Shape {}
// Klasy nie w 'permits' → BŁĄD KOMPILACJI

// ══ PATTERN MATCHING w SWITCH (Java 21) ══════
double area(Shape shape) {
  return switch (shape) {
    case Circle c    -> Math.PI * c.r() * c.r();
    case Rectangle r -> r.w() * r.h();
    case Triangle t  -> 0.0; // placeholder
    // Sealed: nie trzeba default! Kompilator wie że to wszystkie typy
  };
}

// Guarded patterns (Java 21)
return switch (shape) {
  case Circle c when c.r() > 10 -> "big circle";
  case Circle c                  -> "small circle";
  default                        -> "other";
};
