01 — mapowanie obiektów
JPA Entities — @Entity

Entity to POJO mapowany na tabelę w bazie. @Entity + @Table + @Id i masz object-relational mapping. Hibernate generuje DDL i zapytania SQL automatycznie.

User.java — podstawowa entity
import jakarta.persistence.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "users")
public class User {
  
  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;
  
  @Column(nullable = false, unique = true, length = 100)
  private String email;
  
  @Column(name = "full_name", length = 255)
  private String fullName;
  
  @Column(nullable = false)
  private String password;
  
  @Enumerated(EnumType.STRING)
  private UserRole role;
  
  @Column(name = "created_at", updatable = false)
  private LocalDateTime createdAt;
  
  @Column(name = "updated_at")
  private LocalDateTime updatedAt;
  
  // Automatyczne timestampy
  @PrePersist
  protected void onCreate() {
    createdAt = LocalDateTime.now();
    updatedAt = LocalDateTime.now();
  }
  
  @PreUpdate
  protected void onUpdate() {
    updatedAt = LocalDateTime.now();
  }
  
  // getters, setters, constructors...
}
📋 Najważniejsze adnotacje Entity
AdnotacjaZastosowanie
@EntityOznacza klasę jako JPA entity (mapowana na tabelę)
@Table(name="...")Nazwa tabeli w bazie (jeśli inna niż nazwa klasy)
@IdPrimary key
@GeneratedValueAuto-generowanie ID: IDENTITY (auto_increment), SEQUENCE, UUID
@ColumnKonfiguracja kolumny: name, nullable, unique, length
@EnumeratedMapuj enum: STRING (nazwa) lub ORDINAL (liczba, BAD)
@TemporalMapuj Date/Calendar: DATE, TIME, TIMESTAMP (deprecated w Javie 8+)
@LobLarge Object — TEXT/BLOB w bazie
@TransientPole NIE mapowane do bazy (tylko w pamięci)
@PrePersist/@PreUpdateCallback przed save/update (np. timestamp)
02 — data access layer
Spring Data Repositories

Zero implementacji DAO! Definiujesz interface extends JpaRepository — Spring generuje implementację w runtime. CRUD + paginacja + sortowanie za darmo.

UserRepository.java
import org.springframework.data.jpa.repository.*;
import java.util.*;

public interface UserRepository 
    extends JpaRepository<User, Long> {
  
  // Derived query methods — Spring generuje SQL!
  Optional<User> findByEmail(String email);
  
  List<User> findByRole(UserRole role);
  
  List<User> findByFullNameContainingIgnoreCase(
    String name
  );
  
  boolean existsByEmail(String email);
  
  long countByRole(UserRole role);
  
  void deleteByEmail(String email);
}

Użycie repository w Service
@Service
public class UserService {
  
  private final UserRepository userRepository;
  
  public UserService(UserRepository repo) {
    this.userRepository = repo;
  }
  
  // CRUD za darmo!
  public User save(User user) {
    return userRepository.save(user);
  }
  
  public Optional<User> findById(Long id) {
    return userRepository.findById(id);
  }
  
  public List<User> findAll() {
    return userRepository.findAll();
  }
  
  public void deleteById(Long id) {
    userRepository.deleteById(id);
  }
  
  // Paginacja + sortowanie
  public Page<User> findAll(int page, int size) {
    Pageable pageable = PageRequest.of(
      page, 
      size, 
      Sort.by("createdAt").descending()
    );
    return userRepository.findAll(pageable);
  }
}
✓ JpaRepository daje za darmo: save(), saveAll(), findById(), findAll(), findAllById(), count(), deleteById(), delete(), deleteAll(), existsById(), flush(). Plus wsparcie dla Page<T>, Sort, Pageable. Zero SQL!
03 — relacje między encjami
JPA Relationships

@OneToMany, @ManyToOne, @ManyToMany, @OneToOne — mapowanie relacji bazodanowych. Cascade, fetch type (LAZY/EAGER), orphan removal — kontroluj zachowanie.

@OneToMany / @ManyToOne — User ma wiele Orders
// User.java
@Entity
public class User {
  @Id @GeneratedValue
  private Long id;
  
  // Jeden user ma wiele zamówień
  @OneToMany(
    mappedBy = "user",           // pole w Order.java
    cascade = CascadeType.ALL,
    orphanRemoval = true,
    fetch = FetchType.LAZY      // DEFAULT, dobre!
  )
  private List<Order> orders = new ArrayList<>();
}

// Order.java
@Entity
@Table(name = "orders")
public class Order {
  @Id @GeneratedValue
  private Long id;
  
  // Wiele zamówień należy do jednego usera
  @ManyToOne(fetch = FetchType.LAZY)
  @JoinColumn(name = "user_id")  // FK w tabeli orders
  private User user;
}
@ManyToMany — Student ↔ Course (join table)
// Student.java
@Entity
public class Student {
  @Id @GeneratedValue
  private Long id;
  
  @ManyToMany
  @JoinTable(
    name = "student_course",              // join table
    joinColumns = @JoinColumn(name = "student_id"),
    inverseJoinColumns = @JoinColumn(name = "course_id")
  )
  private Set<Course> courses = new HashSet<>();
}

// Course.java
@Entity
public class Course {
  @Id @GeneratedValue
  private Long id;
  
  @ManyToMany(mappedBy = "courses")
  private Set<Student> students = new HashSet<>();
}
⚙️ Cascade Types & Fetch Strategies
OpcjaCo robi
CascadeType.PERSISTsave() parent → save() children automatycznie
CascadeType.MERGEmerge() parent → merge() children
CascadeType.REMOVEdelete() parent → delete() children
CascadeType.ALLWszystkie cascade operations
orphanRemoval=trueUsuń child jeśli usunięty z kolekcji parent
FetchType.LAZYZaładuj tylko gdy dostęp (domyślne dla @OneToMany/@ManyToMany)
FetchType.EAGERZaładuj od razu JOIN (domyślne dla @ManyToOne/@OneToOne)
⚠️ N+1 Problem! LAZY loading może powodować N+1 queries: pobierz 100 orders → dla każdego order osobne query po user (101 zapytań!). Rozwiązanie: @EntityGraph, JOIN FETCH w JPQL, lub projection DTO. Zawsze sprawdzaj logi SQL w development!
04 — własne zapytania
Custom Queries — @Query

Derived queries mają limit złożoności. Dla skomplikowanych zapytań używaj @Query z JPQL (Java Persistence Query Language) lub native SQL.

JPQL — object-oriented SQL
public interface UserRepository extends JpaRepository<User, Long> {
  
  // JPQL — query po encjach (nie tabelach!)
  @Query("SELECT u FROM User u WHERE u.email = :email")
  Optional<User> findByEmailJpql(@Param("email") String email);
  
  // JOIN FETCH — rozwiązuje N+1 problem
  @Query("SELECT u FROM User u LEFT JOIN FETCH u.orders WHERE u.id = :id")
  Optional<User> findByIdWithOrders(@Param("id") Long id);
  
  // Projekcja — zwróć tylko wybrane pola (DTO)
  @Query("""
    SELECT new com.example.dto.UserSummary(u.id, u.email, u.fullName)
    FROM User u
    WHERE u.role = :role
    """)
  List<UserSummary> findSummaryByRole(@Param("role") UserRole role);
  
  // Modifying query — UPDATE/DELETE
  @Modifying
  @Query("UPDATE User u SET u.role = :newRole WHERE u.email = :email")
  int updateRoleByEmail(
    @Param("email") String email,
    @Param("newRole") UserRole newRole
  );
}
Native SQL — surowy SQL gdy JPQL nie wystarcza
@Query(
  value = "SELECT * FROM users WHERE email LIKE :pattern",
  nativeQuery = true  // native SQL!
)
List<User> findByEmailPattern(@Param("pattern") String pattern);

// Native query z pagination
@Query(
  value = "SELECT u.* FROM users u JOIN orders o ON u.id = o.user_id GROUP BY u.id HAVING COUNT(o.id) > :minOrders",
  countQuery = "SELECT COUNT(DISTINCT u.id) FROM users u JOIN orders o ON u.id = o.user_id HAVING COUNT(o.id) > :minOrders",
  nativeQuery = true
)
Page<User> findUsersWithMinOrders(
  @Param("minOrders") int minOrders,
  Pageable pageable
);
💡 JPQL vs Native SQL JPQL: Database-agnostic (działa na PostgreSQL, MySQL, H2), type-safe, refactoring-friendly (zmiana nazwy entity → IDE podpowie). Native SQL: Database-specific features (window functions, JSONB operators w Postgres), performance tuning, legacy databases. Preferuj JPQL, używaj native tylko gdy musisz.
05 — dynamiczne zapytania
JPA Specifications — Criteria API

Dynamiczne budowanie zapytań w runtime. Filtry przekazane z UI → Specification → SQL. Type-safe alternative dla string concatenation w WHERE.

UserSpecifications.java — criteria builders
import org.springframework.data.jpa.domain.Specification;

public class UserSpecifications {
  
  public static Specification<User> hasEmail(String email) {
    return (root, query, cb) -> 
      email == null ? null : cb.equal(root.get("email"), email);
  }
  
  public static Specification<User> hasRole(UserRole role) {
    return (root, query, cb) ->
      role == null ? null : cb.equal(root.get("role"), role);
  }
  
  public static Specification<User> fullNameContains(String name) {
    return (root, query, cb) ->
      name == null ? null : 
        cb.like(cb.lower(root.get("fullName")), "%" + name.toLowerCase() + "%");
  }
  
  public static Specification<User> createdAfter(LocalDateTime date) {
    return (root, query, cb) ->
      date == null ? null : cb.greaterThan(root.get("createdAt"), date);
  }
}
Repository z Specification support
public interface UserRepository 
    extends JpaRepository<User, Long>,
            JpaSpecificationExecutor<User> {
  // JpaSpecificationExecutor dodaje:
  // findAll(Specification<User> spec)
  // findOne(Specification<User> spec)
  // count(Specification<User> spec)
}
Użycie — łączenie Specifications
@Service
public class UserService {
  
  public List<User> search(
      String email,
      UserRole role,
      String name
  ) {
    Specification<User> spec = Specification
      .where(UserSpecifications.hasEmail(email))
      .and(UserSpecifications.hasRole(role))
      .and(UserSpecifications.fullNameContains(name));
    
    return userRepository.findAll(spec);
  }
}
✓ Kiedy używać Specifications? Dynamiczne filtry z UI (użytkownik wybiera które pola filtrować), search endpoints z wieloma opcjonalnymi parametrami, reusable query fragments (np. "active users" używane w wielu miejscach). Specifications są type-safe i można je łączyć (and/or/not).
06 — ACID properties
@Transactional

Transakcje zapewniają ACID: Atomicity (wszystko albo nic), Consistency, Isolation, Durability. @Transactional na service layer — metoda w transakcji, rollback przy exception.

Transactional service methods
@Service
public class OrderService {
  
  private final OrderRepository orderRepo;
  private final InventoryService inventoryService;
  private final PaymentService paymentService;
  
  // @Transactional — rollback przy exception
  @Transactional
  public Order createOrder(CreateOrderRequest request) {
    // 1. Utwórz zamówienie
    Order order = new Order(request);
    order = orderRepo.save(order);
    
    // 2. Zmniejsz inventory
    inventoryService.decreaseStock(
      request.getProductId(), 
      request.getQuantity()
    );
    
    // 3. Przetwórz payment
    paymentService.charge(
      request.getPaymentMethod(),
      order.getTotalPrice()
    );
    
    // Jeśli cokolwiek rzuci exception → ROLLBACK!
    return order;
  }
  
  // readOnly=true — optymalizacja (bez dirty checking)
  @Transactional(readOnly = true)
  public List<Order> findAll() {
    return orderRepo.findAll();
  }
  
  // Rollback tylko przy konkretnym exception
  @Transactional(rollbackFor = PaymentException.class)
  public void processPayment(Long orderId) {
    // ...
  }
  
  // Nie rollback przy biznesowym exception
  @Transactional(noRollbackFor = OutOfStockException.class)
  public void tryFulfillOrder(Long orderId) {
    // ...
  }
}
⚙️ @Transactional opcje
OpcjaCo robi
readOnly=trueOptymalizacja dla read-only operations (brak dirty checking)
timeout=30Timeout transakcji w sekundach
propagation=...REQUIRED (default), REQUIRES_NEW, NESTED, SUPPORTS...
isolation=...READ_COMMITTED (default), REPEATABLE_READ, SERIALIZABLE
rollbackFor=...Rollback przy określonych exceptions
noRollbackFor=...Nie rollback przy określonych exceptions
⚠️ @Transactional działa tylko na public methods i tylko przez proxy! Jeśli wywołasz @Transactional method z tej samej klasy (this.method()) — transakcja NIE zadziała (self-invocation problem). Rozwiązanie: wyciągnij metodę do osobnego @Service lub użyj AspectJ proxy mode.
07 — optymalizacja
JPA Performance Tips

Hibernate jest potężny ale łatwo popełnić błędy performance. N+1 queries, brak indeksów, EAGER loading — każdy z tych problemów może położyć produkcję.

🐌 Problem: N+1 Queries Pobierasz 100 orders → Hibernate robi 1 query SELECT orders + 100 queries SELECT user dla każdego order = 101 queries!

Fix: JOIN FETCH w JPQL lub @EntityGraph
JOIN FETCH solution
@Query("""
  SELECT o FROM Order o
  LEFT JOIN FETCH o.user
  LEFT JOIN FETCH o.items
  """)
List<Order> findAllWithUserAndItems();

// Alternatywnie: @EntityGraph
@EntityGraph(
  attributePaths = {"user", "items"}
)
List<Order> findAll();
✓ LAZY loading (default) @OneToMany/@ManyToMany domyślnie LAZY. Ładuj relacje tylko gdy potrzeba → mniejszy memory footprint, szybsze queries.
⚠️ Unikaj EAGER EAGER loading ładuje wszystko od razu (JOIN). Dla głębokich grafów obiektów → ogromne queries, performance hell.
💡 Projekcje DTO Zamiast pełnej entity → zwracaj tylko potrzebne pola w DTO. Mniejszy transfer, szybsza serializacja JSON.
Performance best practices checklist
// ✅ 1. Indeksy na frequently queried columns
@Table(indexes = {
  @Index(name = "idx_user_email", columnList = "email"),
  @Index(name = "idx_order_user", columnList = "user_id")
})

// ✅ 2. Batch operations — save() 1000 razy = 1000 INSERTs
// Lepiej: saveAll() z batch size
spring.jpa.properties.hibernate.jdbc.batch_size=50

// ✅ 3. Pagination dla dużych list
Page<User> page = userRepo.findAll(PageRequest.of(0, 20));

// ✅ 4. ReadOnly transactions dla queries
@Transactional(readOnly = true)

// ✅ 5. Loguj SQL w development
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true

// ✅ 6. @BatchSize dla kolekcji — ładuj po N elementów
@BatchSize(size = 10)
@OneToMany(mappedBy = "user")
private List<Order> orders;
