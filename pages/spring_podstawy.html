01 — wprowadzenie
Czym jest Spring Framework?

Spring to framework do budowy aplikacji enterprise w Javie. Spring Boot to nadbudowa która automatyzuje większość konfiguracji. Zero XML, konwencja nad konfiguracją, embedded Tomcat — działająca aplikacja w 5 minut.

📦
Dependency Injection
Framework zarządza obiektami (beans) i ich zależnościami. Nie musisz ręcznie tworzyć instancji.
⚙️
Auto-Configuration
Spring Boot automatycznie konfiguruje komponenty na podstawie classpath. H2 w classpath → auto-konfiguruje DataSource.
🚀
Embedded Server
Wbudowany Tomcat/Jetty/Undertow. Nie musisz deployować WAR na zewnętrzny serwer. Uruchamiasz JAR.
🔌
Spring Boot Starters
Gotowe paczki zależności. spring-boot-starter-web daje MVC + Tomcat + JSON. Jedna linijka w pom.xml.
📊
Actuator
Production-ready features: health checks, metrics, monitoring endpoints. /actuator/health out of the box.
🛠️
DevTools
Hot reload — zmiana kodu → automatyczny restart aplikacji. Przyspiesza development.
🍃 Dlaczego Spring Boot jest wszędzie? 95% aplikacji enterprise Java używa Spring. Dlaczego? Ogromny ekosystem (Security, Data, Cloud), dojrzałość (20+ lat), świetna dokumentacja, komercyjne wsparcie (VMware), ogromna społeczność. Jeśli uczysz się Java backend — musisz znać Spring.
02 — fundament
IoC & Dependency Injection

Inversion of Control — odwrócenie kontroli. Nie ty tworzysz obiekty, Spring tworzy i wstrzykuje je tam gdzie potrzeba. Dependency Injection to mechanizm dostarczania zależności.

📄Bez DI — ręczne tworzenie
public class OrderService {
  
  // Musisz sam utworzyć zależności!
  private OrderRepository repository;
  private EmailService emailService;
  
  public OrderService() {
    // Tight coupling — trudno testować
    this.repository = new OrderRepository();
    this.emailService = new EmailService();
  }
  
  public void createOrder(Order order) {
    repository.save(order);
    emailService.send(order);
  }
}
🍃Z DI — Spring wstrzykuje
@Service // Spring zarządza tym obiektem
public class OrderService {
  
  // Spring wstrzyknie zależności!
  private final OrderRepository repository;
  private final EmailService emailService;
  
  // Constructor injection (BEST PRACTICE)
  public OrderService(
      OrderRepository repository,
      EmailService emailService) {
    this.repository = repository;
    this.emailService = emailService;
  }
  
  public void createOrder(Order order) {
    repository.save(order);
    emailService.send(order);
  }
}
🔍 3 typy Dependency Injection
TypJak wyglądaKiedy używać
Constructor public Service(Repo repo) {...} ✅ BEST PRACTICE — wymagane zależności, immutable, łatwe testowanie
Setter @Autowired
public void setRepo(Repo r) {...}		 ⚠️ Opcjonalne zależności. Rzadko używane.
Field @Autowired
private Repo repo;		 ❌ AVOID — nie da się testować bez Springa, ukrywa zależności
⚠️ Constructor Injection — zawsze! Field injection (@Autowired na polu) wygląda prościej ale ma problemy: nie da się testować bez Springa, ukrywa liczbę zależności (jeśli klasa ma 10 @Autowired to znak że robi za dużo), nie działa z final fields. Constructor injection wymusza immutability i pokazuje prawdziwą liczbę zależności.
03 — zarządzanie obiektami
Spring Beans

Bean to obiekt zarządzany przez Spring IoC Container. Spring tworzy bean, zarządza jego lifecycle, wstrzykuje zależności. Beans są singletonami domyślnie — jedna instancja na cały application context.

☕Rejestracja beans — @Component i pochodne
// @Component — generyczny bean
@Component
public class MyComponent {
  // Spring utworzy instancję i zarządza
}

// @Service — logika biznesowa (warstwa service)
@Service
public class UserService { }

// @Repository — dostęp do danych (warstwa DAO)
@Repository
public class UserRepository { }

// @Controller / @RestController — warstwa HTTP
@RestController
public class UserController { }

// @Configuration — klasa konfiguracyjna z @Bean methods
@Configuration
public class AppConfig {
  
  // @Bean — ręczna rejestracja bean
  @Bean
  public ObjectMapper objectMapper() {
    return new ObjectMapper()
      .registerModule(new JavaTimeModule());
  }
}
📦 Bean Scopes
singleton (default)
Jedna instancja na cały ApplicationContext. Wszystkie injections dostają tę samą instancję.
prototype
Nowa instancja przy każdym wstrzyknięciu. Spring nie zarządza lifecycle.
request (web)
Jedna instancja per HTTP request. Umiera po zakończeniu requestu.
session (web)
Jedna instancja per HTTP session. Żyje przez całą sesję usera.
Użycie scope
// Domyślnie singleton
@Service
public class UserService { }

// Prototype — nowa instancja zawsze
@Component
@Scope("prototype")
public class ShoppingCart { }

// Request scope (web)
@Component
@Scope(
  value = WebApplicationContext.SCOPE_REQUEST,
  proxyMode = ScopedProxyMode.TARGET_CLASS
)
public class RequestContext { }
💡 Kiedy używać @Bean vs @Component? @Component (i @Service/@Repository) — gdy masz dostęp do kodu klasy i możesz dodać adnotację. @Bean w @Configuration — gdy konfigurujesz klasę z zewnętrznej biblioteki (np. ObjectMapper, RestTemplate) lub potrzebujesz złożonej logiki tworzenia obiektu.
04 — adnotacje
Najważniejsze Adnotacje Spring

Spring Boot używa adnotacji do wszystkiego — rejestracja beans, mapowanie HTTP, validation, transakcje. Poznaj te najważniejsze.

@SpringBootApplication
Meta-adnotacja = @Configuration + @EnableAutoConfiguration + @ComponentScan. Punkt startowy aplikacji. Skanuje pakiet i podpakiety.
@RestController
= @Controller + @ResponseBody. Każda metoda zwraca JSON (nie view). Używaj do REST API.
@RequestMapping
Mapuje HTTP request na metodę. @GetMapping/@PostMapping/@PutMapping/@DeleteMapping to skróty.
@Autowired
Wstrzykuje zależność. Od Spring 4.3 opcjonalne w constructor injection (jeśli tylko 1 konstruktor).
@Value
Wstrzykuje wartość z application.properties: @Value("${app.name}") String name;
@ConfigurationProperties
Binduje grupę properties do POJO. Type-safe, walidacja, auto-complete w IDE.
@Transactional
Opakowuje metodę w transakcję bazodanową. Rollback przy exception. Używaj na service layer.
@Async
Wykonuje metodę asynchronicznie w osobnym wątku. Wymaga @EnableAsync w konfiguracji.
@Scheduled
Zaplanowane wykonanie metody (cron, fixedDelay, fixedRate). Wymaga @EnableScheduling.
@Profile
Aktywuj bean tylko w określonym profilu. @Profile("dev") — tylko gdy spring.profiles.active=dev
@Conditional
Warunkowa rejestracja bean. @ConditionalOnProperty, @ConditionalOnClass, @ConditionalOnMissingBean.
05 — szybki start
Spring Boot Starter Project

Najprostszy sposób na start — pom.xml + jedna klasa @SpringBootApplication. Spring Boot zrobi resztę: skonfiguruje Tomcat, Jackson, logging, wszystko.

📦pom.xml — minimalna konfiguracja
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.2.0</version>
</parent>

<dependencies>
  <!-- Web starter: MVC + Tomcat + JSON -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
  </dependency>
  
  <!-- JPA + Hibernate -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
  </dependency>
  
  <!-- PostgreSQL driver -->
  <dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
  </dependency>
</dependencies>
☕Application.java — main class
package com.example.demo;

import org.springframework.boot.*;
import org.springframework.boot.autoconfigure.*;

@SpringBootApplication // 🍃 Magic happens here

  public static void main(String[] args) {
    SpringApplication.run(
      Application.class, 
      args
    );
  }
}

06 — REST API
Spring MVC — REST Controllers

Spring MVC to framework do budowy aplikacji web. W kontekście Spring Boot używamy go głównie do REST API. @RestController + @GetMapping/@PostMapping i masz endpoint w 5 linijek.

🎮UserController.java — kompletny CRUD REST
@RestController
@RequestMapping("/api/users")
public class UserController {
  
  private final UserService userService;
  
  public UserController(UserService userService) {
    this.userService = userService;
  }
  
  // GET /api/users — lista wszystkich
  @GetMapping
  public List<User> getAllUsers() {
    return userService.findAll();
  }
  
  // GET /api/users/5 — jeden user po ID
  @GetMapping("/{id}")
  public User getUserById(@PathVariable Long id) {
    return userService.findById(id)
      .orElseThrow(() -> new NotFoundException("User not found"));
  }
  
  // POST /api/users — utwórz nowego
  @PostMapping
  @ResponseStatus(HttpStatus.CREATED)
  public User createUser(@RequestBody CreateUserRequest request) {
    return userService.create(request);
  }
  
  // PUT /api/users/5 — update całego usera
  @PutMapping("/{id}")
  public User updateUser(
      @PathVariable Long id,
      @RequestBody UpdateUserRequest request) {
    return userService.update(id, request);
  }
  
  // DELETE /api/users/5 — usuń usera
  @DeleteMapping("/{id}")
  @ResponseStatus(HttpStatus.NO_CONTENT)
  public void deleteUser(@PathVariable Long id) {
    userService.delete(id);
  }
}
🔍 Adnotacje parametrów
AdnotacjaSkąd bierze danePrzykład
@PathVariable Z URL path: /users/{id} @GetMapping("/{id}")
User get(@PathVariable Long id)
@RequestParam Z query string: ?page=1&size=20 @GetMapping
List get(@RequestParam int page)
@RequestBody Z body HTTP (JSON → POJO) @PostMapping
User create(@RequestBody UserDTO dto)
@RequestHeader Z HTTP header @GetMapping
void get(@RequestHeader("Auth") String token)
💡 @RequestBody → JSON automatycznie! Spring Boot ma wbudowany Jackson (JSON library). Jak widzisz @RequestBody — automatycznie deserializuje JSON z body do POJO. Zwracasz obiekt z metody → Jackson serializuje do JSON. Zero konfiguracji!
⚠️ Validation z Jakarta Bean Validation Dodaj spring-boot-starter-validation → możesz używać @Valid + adnotacje: @NotNull, @NotBlank, @Email, @Size, @Min/@Max. Spring automatycznie waliduje i rzuca MethodArgumentNotValidException jeśli błąd.
07 — konfiguracja
application.properties / application.yml

Cała konfiguracja aplikacji w jednym pliku. Connection string do bazy, port serwera, profile, logging — wszystko w application.properties lub application.yml (YAML format).

⚙️application.properties
# Server config
server.port=8080
server.servlet.context-path=/api

# Database (PostgreSQL)
spring.datasource.url=jdbc:postgresql://localhost:5432/mydb
spring.datasource.username=postgres
spring.datasource.password=secret

# JPA / Hibernate
spring.jpa.hibernate.ddl-auto=validate
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true

# Logging
logging.level.root=INFO
logging.level.com.example=DEBUG

# Custom properties
app.name=My Spring App
app.version=1.0.0

☕Użycie properties w kodzie — @Value i @ConfigurationProperties
// METODA 1: @Value — pojedyncze wartości
@Component
public class AppInfo {
  
  @Value("${app.name}")
  private String appName;
  
  @Value("${app.version}")
  private String version;
}

// METODA 2: @ConfigurationProperties — type-safe binding
@ConfigurationProperties(prefix = "app")
@Component
public class AppProperties {
  
  private String name;     // binduje app.name
  private String version;  // binduje app.version
  
  // getters & setters
  public String getName() { return name; }
  public void setName(String name) { this.name = name; }
  // ...
}
🍃 Profile-specific properties application-dev.yml (development), application-prod.yml (production). Aktywuj przez spring.profiles.active=dev lub environment variable SPRING_PROFILES_ACTIVE=prod. Spring Boot automatycznie ładuje odpowiedni plik!
08 — best practices
Struktura Projektu Spring Boot

Dobrze zorganizowany projekt to połowa sukcesu. Podział na warstwy (controller/service/repository), pakiety po feature nie po typie, jasne nazewnictwo.

📁Struktura katalogów — wariant 1: Package by Layer
src/main/java/com/example/myapp/
├── Application.java         // @SpringBootApplication
├── config/                  // @Configuration classes
│   ├── SecurityConfig.java
│   └── DatabaseConfig.java
├── controller/              // @RestController
│   ├── UserController.java
│   └── OrderController.java
├── service/                 // @Service (business logic)
│   ├── UserService.java
│   └── OrderService.java
├── repository/              // @Repository / JPA interfaces
│   ├── UserRepository.java
│   └── OrderRepository.java
├── model/                   // @Entity classes
│   ├── User.java
│   └── Order.java
├── dto/                     // Data Transfer Objects
│   ├── CreateUserRequest.java
│   └── UserResponse.java
└── exception/               // Custom exceptions
    ├── NotFoundException.java
    └── GlobalExceptionHandler.java
📁Struktura — wariant 2: Package by Feature (LEPSZY)
src/main/java/com/example/myapp/
├── Application.java
├── config/
│   └── SecurityConfig.java
├── user/                    // Wszystko o userach w jednym miejscu
│   ├── User.java
│   ├── UserController.java
│   ├── UserService.java
│   ├── UserRepository.java
│   ├── CreateUserRequest.java
│   └── UserResponse.java
├── order/                   // Wszystko o zamówieniach
│   ├── Order.java
│   ├── OrderController.java
│   ├── OrderService.java
│   ├── OrderRepository.java
│   └── OrderDTO.java
└── common/                  // Shared code
    ├── exception/
    │   └── GlobalExceptionHandler.java
    └── util/
        └── DateUtils.java
💡 Package by Feature > Package by Layer Package by feature jest lepszy dla większych projektów — wszystko związane z userem jest w pakiecie user/. Łatwiej znaleźć kod, łatwiej dodać nową feature (nowy pakiet), lepsze enkapsulowanie. Package by layer wymusza przeskakiwanie między pakietami.
🍃 Naming conventions Entity: User, Order (bez sufixu)
Repository: UserRepository
Service: UserService
Controller: UserController
DTO: CreateUserRequest, UserResponse (cel w nazwie)
