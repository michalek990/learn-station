Architektura Sprzętu
Z czego zbudowany jest komputer
Komputer to hierarchia warstw abstrakcji — od tranzystorów przez bramki logiczne, mikrokod, ISA (Instruction Set Architecture) aż po system operacyjny. Każda warstwa ukrywa szczegóły warstwy poniżej.
🔌 Główne komponenty PC
CPU
Central Processing Unit — "mózg". Wykonuje instrukcje. Zawiera: rdzenie (cores), cache L1/L2/L3, kontroler pamięci, PCIe lanes, iGPU (często). Intel / AMD / ARM.
RAM
Random Access Memory — pamięć operacyjna. Ulotna (dane giną po wyłączeniu). DDR4/DDR5. Szeroka magistrala (64-bit per kanał). Bezpośrednio adresowalna przez CPU.
Motherboard
Płyta główna — łączy wszystko. Chipset zarządza komunikacją. Socket CPU, sloty RAM, PCIe, SATA, M.2, USB, audio, LAN. BIOS/UEFI chip.
GPU
Graphics Processing Unit — tysiące małych rdzeni do obliczeń równoległych. VRAM (GDDR6). Połączony przez PCIe x16. Dedykowany (Nvidia/AMD) lub zintegrowany (Intel/AMD iGPU).
Storage
Dysk — HDD (talerze magnetyczne), SSD SATA, SSD NVMe (PCIe). NVMe przez M.2 slot = 7 GB/s+. Nieulotna (dane zostają po wyłączeniu).
PSU
Power Supply Unit — zamienia 230V AC na DC (3.3V, 5V, 12V). ATX standard. 80 PLUS certyfikaty = sprawność. Zły PSU = niestabilność/uszkodzenia.

🧮 Chipset — North/South Bridge (historycznie)

Dawniej: Northbridge (CPU ↔ RAM, GPU) + Southbridge (USB, SATA, audio, LAN).

Dziś: Kontroler pamięci i PCIe są wewnątrz CPU (on-die). Chipset (PCH — Platform Controller Hub) obsługuje tylko wolniejsze peryferia (USB, SATA, audio). CPU i PCH komunikują przez DMI.

AMD: kontrolery wewnątrz CPU (Infinity Fabric łączy CCD i IOD). Intel: podobnie.

🔬 Od tranzystora do instrukcji Tranzystor (MOSFET, 3nm–7nm) → bramki logiczne (AND, OR, NOT, XOR) → przerzutniki/rejestry → ALU (Arithmetic Logic Unit) → rdzeń CPU → instrukcja. Nowoczesny CPU ma ~50 miliardów tranzystorów. Intel i AMD używają TSMC i własnych fabryk. Tranzystory tak małe że zawierają kilka atomów krzemu.
02 / 12
CPU — Procesor w szczegółach
Jak CPU wykonuje instrukcje
Cykl fetch-decode-execute, pipeline, branch prediction, out-of-order execution, hyper-threading — to mechanizmy które sprawiają że nowoczesny CPU jest 1000x szybszy niż prosty model sugeruje. Każdy developer powinien to rozumieć.
Cykl CPU — Fetch → Decode → Execute
Każda instrukcja przechodzi przez etapy:

1. FETCH       // Pobierz instrukcję z pamięci (cache L1i)
                // PC (Program Counter) wskazuje adres
                // PC++ po pobraniu

2. DECODE      // Zdekoduj opcode → mikrooperacje (µops)
                // x86 → wewnętrznie RISC-like µops

3. DISPATCH    // Wyślij do odpowiedniej jednostki wykonawczej
                // ALU, FPU, Load/Store unit, Branch unit

4. EXECUTE     // Wykonaj operację
                // Out-of-order: kolejność może być zmieniona!
                // Wynik zapisany tymczasowo (ROB)

5. WRITE-BACK  // Zapisz wynik do rejestru / cache
                // In-order commit — zachowanie wyglądania sequential

─── Pipeline — wiele instrukcji jednocześnie ──────────────
Instrukcja 1:  F  D  E  W
Instrukcja 2:     F  D  E  W
Instrukcja 3:        F  D  E  W
Instrukcja 4:           F  D  E  W
→ 4 instrukcje "jednocześnie" w różnych etapach

─── Branch Misprediction — kosztowne! ─────────────────────
if (x > 0) { ... }  // CPU spekuluje która gałąź
Błędna spekulacja → flush pipeline → 15-20 cykli stracone
Nowoczesne predyktory: 95%+ accuracy

🧵 Hyper-Threading / SMT

Simultaneous Multi-Threading (SMT) — jeden fizyczny rdzeń wygląda jak dwa logiczne. Osobny zestaw rejestrów dla dwóch wątków, ale współdzielone jednostki wykonawcze i cache.

Zysk: ~20–30% wyższa przepustowość gdy jeden wątek czeka (cache miss, I/O, branch stall). Drugi wątek wykorzystuje te bezczynne cykle.

Intel: Hyper-Threading (HT). AMD: SMT. Czasem wyłącza się dla bezpieczeństwa (Spectre/Meltdown).

⚡ SIMD — równoległe dane

Single Instruction, Multiple Data — jedna instrukcja operuje na wielu danych jednocześnie. SSE, AVX, AVX-512.

AVX-512: 512-bit rejestr = 8 × double lub 16 × float operacji jednocześnie. Kluczowe dla: ML, kompresja, kryptografia, przetwarzanie multimediów.

Java: Kompilator JIT automatycznie wektoryzuje pętle (auto-vectorization). Ręcznie: Vector API (Panama).

🔒 Spectre & Meltdown

Speculative execution vulnerability — CPU spekulacyjnie wykonuje kod i zostawia ślady w cache. Atakujący może odczytać pamięć innych procesów przez timing attack.

Mitygacje: KPTI (Kernel Page Table Isolation) — oddzielne tablice stron dla kernel/user. Retpoline — bezpieczne indirect jumps. Overhead: 5–30% dla I/O intensive workloads.

Procesory Intel mocniej dotknięte niż AMD. ARM też podatny.

RAM & Pamięć Wirtualna
Jak działa pamięć — RAM, stronicowanie, swap
RAM to nie tylko "miejsce na programy". System operacyjny tworzy wirtualną przestrzeń adresową dla każdego procesu, zarządza stronicowaniem (paging) i używa dysku jako rozszerzenia RAM (swap/page file).
Pamięć wirtualna — jak działa
Każdy proces widzi swoją WŁASNĄ przestrzeń adresową
Proces nie ma bezpośredniego dostępu do fizycznego RAM

Virtual Address Space (64-bit, 48 bity używane):
0x0000 0000 0000 0000
│  Text segment       (kod programu, read-only)
│  Data segment       (zmienne globalne, statyczne)
│  BSS segment        (niezainicjalizowane globale)
│  Heap               (malloc/new → rośnie w górę ↑)
│  (free space)
│  Stack              (lokaln. zmienne, rośnie w dół ↓)
│  Kernel space       (tylko OS ma dostęp)
0xFFFF FFFF FFFF FFFF

─── Page Table — tłumaczenie wirtualny→fizyczny ───────────
Virtual Page Number    │ Physical Frame Number
0x7fff_dead_beef00     │→ 0x0000_0042_1234 (w RAM)
0x7fff_dead_c000       │→ 0x0000_0000 (NOT PRESENT)
                       │  → Page Fault! OS ładuje z dysku

─── MMU (Memory Management Unit) ──────────────────────────
Hardware w CPU które robi translację
TLB (Translation Lookaside Buffer) = cache tablic stron
TLB miss → page walk → 100+ cykli kary
Strona = 4KB (domyślnie). Huge pages = 2MB lub 1GB
RAM, SWAP, Page Fault
─── DRAM — jak działa fizycznie ───────────────────────────
Kondensatory (naładowany = 1, rozładowany = 0)
Rozładowują się → REFRESH co ~64ms (aktywne!)
Stąd DRAM = Dynamic RAM (vs SRAM w cache = statyczny)
SRAM = szybszy, droższy (6 tranzystorów vs 1)

─── DDR5 vs DDR4 ──────────────────────────────────────────
DDR4: 2133–3200 MT/s, 1.2V, 64-bit bus
DDR5: 4800–7200 MT/s, 1.1V, 2×32-bit (on-die ECC)
ECC RAM: dodatkowy bit do korekcji błędów → serwery

─── SWAP / Page File ──────────────────────────────────────
Gdy RAM pełny → OS przenosi nieużywane strony na dysk
Windows: pagefile.sys (domyślnie 1.5x RAM)
Linux:   swap partition lub swapfile
Page Fault = proces chce strony która jest na dysku
  → OS ładuje stronę do RAM (może wyrzucić inną)
  → proces wznawia → transparentnie dla procesu

─── Page Fault types ──────────────────────────────────────
Minor fault: strona w RAM, brak w tablicy → mapuj. Szybki.
Major fault: strona na dysku → I/O. Wolny (ms!).
Invalid: dostęp do niezaalokowanej strony → SIGSEGV
Windows: Access Violation (0xC0000005)
04 / 12
Dyski — HDD, SSD, NVMe
Jak działają dyski i systemy plików
HDD to wirujące talerze. SSD to komórki NAND flash. NVMe eliminuje warstwy pośrednie. Każdy typ ma inne właściwości i ograniczenia. RAID, partycje, MBR vs GPT — fundament każdego systemu.
HDD vs SSD vs NVMe — internals
─── HDD (Hard Disk Drive) ─────────────────────────────────
Talerze (platters) pokryte materiałem magnetycznym
Głowica odczytuje/zapisuje przez zmianę polaryzacji
Czas dostępu: seek time (głowica) + rotational latency
Typowo: 5400–7200 RPM, seek ~3ms, latency ~4ms
Sequential read: 150–200 MB/s. Random: BARDZO wolny
Podatny na wstrząsy! Mechaniczny = awaryjny

─── SSD SATA (2.5") ───────────────────────────────────────
NAND Flash: komórki przechowują ładunek elektryczny
SLC (1 bit) → MLC (2 bit) → TLC (3 bit) → QLC (4 bit)
Więcej bitów = taniej, wolniej, mniej trwały
Ograniczenia P/E cycles (program/erase): SLC 100k, TLC 3k
Sequential: 550 MB/s (ograniczenie SATA)

─── NVMe (Non-Volatile Memory Express) ────────────────────
Protokół zaprojektowany dla flash (nie recycled SCSI/ATA)
Przez PCIe 4.0 x4: 7000 MB/s sequential, 1M IOPS
Niskie latencje: ~100µs vs ~100ms HDD
Wear leveling: OS rozkłada zapisy równomiernie
TRIM: OS informuje SSD które bloki są wolne

─── Struktura dysku ───────────────────────────────────────
Sektor = 512B (legacy) lub 4096B (4K Advanced Format)
Cluster = N sektorów (unit alokacji systemu plików)
Partition Table: MBR (max 2TB, 4 partycje) lub GPT (8ZB)
💾 MBR vs GPT
MBR (Master Boot Record)

Pierwszy sektor dysku (512B). Zawiera bootloader + partition table.

Max 4 partycje podstawowe (extended hack → logical)

Max 2TB dysk (32-bit LBA)

Jeden punkt awarii — uszkodzony sektor = koniec

Legacy BIOS. Windows XP era.

GPT (GUID Partition Table)

UEFI standard. Nagłówek + kopia zapasowa na końcu dysku.

128 partycji (Windows domyślnie)

Max 8 ZB (zettabytes)

CRC32 checksums — wykrywa uszkodzenia

Wymagany dla UEFI boot. Każda partycja ma unikalny GUID.

RAID RAID 0: striping — szybkość, bez redundancji. RAID 1: mirror — kopia lustrzana. RAID 5: striping + parity (odtworzenie z jednej awarii, min 3 dyski). RAID 6: dwie parity (2 awarie). RAID 10: 1+0, mirror+stripe. Software RAID (mdadm w Linux, Storage Spaces w Windows) vs hardware RAID (controller).
05 / 12
Boot Process — krok po kroku
Od naciśnięcia przycisku do zalogowanego użytkownika
Uruchomienie komputera to sekwencja dziesiątek precyzyjnych kroków. UEFI/BIOS, POST, bootloader, kernel, init/systemd, login — każdy etap musi zakończyć się sukcesem. Znajomość tego procesu jest kluczowa dla diagnostyki problemów ze startem.
🚀 Pełny Boot Process — wszystkie kroki
Krok 1
Power On — napięcie stabilizuje się
Naciśnięcie przycisku zamyka obwód. PSU podaje napięcia (12V, 5V, 3.3V). Kondensatory stabilizują napięcia (~200ms). PSU wysyła sygnał Power Good do płyty głównej. Bez Power Good — CPU nie startuje.
Krok 2
CPU Reset Vector — pierwszy kod
CPU "budzi się" w trybie Real Mode (16-bit, 1MB pamięci). Rejestr CS:IP wskazuje na Reset Vector: 0xFFFF0 (lub 0xFFFFFFF0 w UEFI). To jest ostatnie 16 bajtów 1MB przestrzeni — zawiera JMP do kodu BIOS/UEFI w ROM chipie.
Krok 3
POST — Power On Self Test
BIOS/UEFI testuje sprzęt: CPU rejestry → Cache → RAM (memory test, sprawdza każdy bit) → Video card → Storage kontrolery → PCI/PCIe urządzenia. Błędy sygnalizowane przez kody dźwiękowe (POST beep codes) lub kody na małym wyświetlaczu płyty (POST debug LED). RAM nie działa → 3 beepy. GPU problem → inne beepy.
Krok 4
UEFI Initialization — DXE & BDS
UEFI (nowoczesny BIOS) przechodzi przez fazy: SEC (Security, init cache jako RAM) → PEI (Pre-EFI Init, init RAM) → DXE (Driver eXecution Environment, ładuje sterowniki dla urządzeń) → BDS (Boot Device Selection, wyświetla menu, wybiera bootloader). UEFI ma własny bootloader w partycji ESP (EFI System Partition, FAT32, zwykle 100–500 MB).
Krok 5
Secure Boot (opcjonalnie)
UEFI sprawdza podpis cyfrowy bootloadera używając certyfikatów przechowywanych w NVRAM (PK, KEK, db). Jeśli podpis nieważny → odmowa uruchomienia. Chroni przed bootkitami. Windows: wymaga Secure Boot od Windows 11. Linux: shim (podpisany przez Microsoft) → GRUB. Można wyłączyć w UEFI Setup.
Krok 6
Bootloader — Windows: bootmgr / Linux: GRUB2
Windows: bootmgr (w ESP lub MBR) → czyta BCD (Boot Configuration Data) → uruchamia winload.efi → ładuje kernel ntoskrnl.exe.

Linux: grub2 ładuje plik grub.cfg → wyświetla menu wyboru systemu → ładuje kernel (vmlinuz) i initial ramdisk (initrd/initramfs) do RAM → przekazuje sterowanie kernelowi.
Krok 7
Kernel Init — przejście do Protected Mode
Kernel startuje w Real Mode (16-bit), przełącza się na Protected Mode (32-bit) lub bezpośrednio Long Mode (64-bit). Inicjalizuje: GDT (Global Descriptor Table), IDT (Interrupt Descriptor Table), page tables (włącza paging), CPU features (wykrywa procesor, włącza NX bit, SMEP). Konfiguruje scheduler, memory allocator (buddy system). Montuje initramfs jako tymczasowy root filesystem.
Krok 8
initramfs / initrd — tymczasowy root
Mały filesystem w RAM zawierający sterowniki potrzebne do montowania prawdziwego roota (RAID, LVM, szyfrowanie LUKS, sterowniki NVMe). Kernel rozpakowuje initramfs → uruchamia /init (mały skrypt/program). Init ładuje moduły, konfiguruje storage, montuje prawdziwy root filesystem (/dev/sda1 lub UUID). Następnie pivot_root lub switch_root — przełączenie na prawdziwy /.
Krok 9
PID 1 — systemd (Linux) / smss.exe (Windows)
Linux — systemd: Kernel uruchamia /sbin/init (lub systemd) jako PID 1. systemd czyta unit files (/etc/systemd/system/), tworzy dependency graph, uruchamia serwisy równolegle. Targets: basic.target → multi-user.target → graphical.target.

Windows — smss.exe: Session Manager Subsystem. Uruchamia csrss.exe (Win32 subsystem), wininit.exe (init services), winlogon.exe (logowanie). Dalej: services.exe (SCM — Service Control Manager) startuje wszystkie serwisy Windows.
Krok 10
Sterowniki i usługi systemowe
Linux: udev wykrywa urządzenia, ładuje moduły jądra, tworzy pliki w /dev/. NetworkManager startuje sieć. D-Bus uruchamia bus komunikacji. PAM konfiguruje logowanie.

Windows: services.exe uruchamia serwisy: Plug&Play, Event Log, WMI, RPC, DCOM, Windows Update, Defender, NetworkManager. Sterowniki ładowane z C:\Windows\System32\drivers\.
Krok 11
Display Manager / Winlogon — ekran logowania
Linux: GDM (GNOME), SDDM (KDE), LightDM — Display Manager uruchamia X.org lub Wayland, wyświetla ekran logowania, po zalogowaniu uruchamia środowisko (GNOME, KDE, XFCE).

Windows: winlogon.exe → LogonUI.exe wyświetla ekran logowania. Po zalogowaniu: userinit.exe → explorer.exe (shell — pasek zadań, pulpit).
Krok 12
✅ Zalogowany użytkownik — desktop gotowy
User profile załadowany (~/.bashrc, ~/.profile lub HKCU registry). Autostart programy uruchomione. Czas od Power On do Desktop: 5–30 sekund (SSD, nowoczesny CPU, dobrze skonfigurowany OS) lub 2–5 minut (HDD, wiele serwisów autostart, stary sprzęt).
06 / 12
Kernel & Tryby CPU
Jądro systemu — kernel space vs user space
Kernel to rdzeń systemu operacyjnego. Zarządza pamięcią, procesami, urządzeniami i bezpieczeństwem. Separacja kernel space / user space to fundament bezpieczeństwa nowoczesnych systemów.
Ring 0–3 — tryby procesora
x86 definiuje 4 poziomy uprawnień (Rings):

Ring 0 — Kernel Mode
  Pełny dostęp do sprzętu i pamięci
  Wszystkie instrukcje dozwolone (in/out, cli, sti...)
  Błąd tu = kernel panic (Linux) / BSOD (Windows)
  Kod: jądro OS, sterowniki urządzeń

Ring 1, 2 — rzadko używane
  Historycznie: OS drivers. Dziś praktycznie nieużywane.
  Wirtualizacja: Ring 1 dla gościa OS (stary model)

Ring 3 — User Mode
  Ograniczony dostęp. Nie może bezpośrednio dotknąć sprzętu.
  Kod: Twoja aplikacja, przeglądarka, gra
  Naruszenie → SIGSEGV / Access Violation

─── System Call — przeprawa z Ring 3 → Ring 0 ──────────────
Aplikacja chce otworzyć plik:
  1. User space: open("file.txt") — biblioteka libc
  2. libc → syscall instrukcja (nr syscall w RAX)
  3. CPU przełącza się Ring 3 → Ring 0
  4. Kernel: sprawdza uprawnienia, otwiera plik
  5. Zwraca file descriptor do user space
  6. CPU wraca Ring 0 → Ring 3
  Koszt: ~1000 cykli CPU na jeden syscall
Monolithic vs Microkernel vs Hybrid
─── Monolithic Kernel (Linux) ─────────────────────────────
Cały kernel w Ring 0: filesystem, drivers, network, MM
Zalety: wydajność (brak context switch między komponentami)
Wady: błąd w sterowniku = kernel crash
Moduły (loadable kernel modules) dają elastyczność
lsmod, modprobe, insmod — zarządzanie modułami

─── Microkernel (L4, QNX, Mach) ──────────────────────────
Kernel: tylko scheduling, IPC, basic MM
Filesystem, drivers, network → osobne procesy (Ring 3)
Zalety: izolacja, bezpieczeństwo, stabilność
Wady: IPC overhead, wolniejszy

─── Hybrid (Windows NT, macOS XNU) ───────────────────────
Mieszanka: część serwisów w kernel, część jako procesy
Windows: NT kernel + Win32 subsystem jako serwis
macOS: Mach microkernel + BSD UNIX layer

─── Interrupt — ważne zdarzenia ──────────────────────────
Hardware: klawiatura, sieć, dysk → IRQ → IDT handler
Software: syscall, page fault, divide-by-zero
CPU przerywa aktualny wątek → kernel handler → powrót
APIC (Advanced PIC) zarządza IRQ w wielordzeniowym CPU
07 / 12
Procesy & Wątki
Jak OS zarządza procesami
Każda uruchomiona aplikacja to jeden lub więcej procesów. Kernel zarządza harmonogramowaniem, priorytetami, komunikacją między procesami (IPC) i sygnałami. fork/exec to fundament Uniksa.
Stany procesu i cykl życia
─── Stany procesu ─────────────────────────────────────────
NEW      → Tworzony (fork/CreateProcess)
READY    → Czeka na CPU (w kolejce schedulera)
RUNNING  → Wykonywany na CPU
WAITING  → Czeka na I/O, sygnał, mutex
ZOMBIE   → Zakończony, czeka aż rodzic odczyta exit code
DEAD     → Zasoby zwolnione

─── fork/exec — Unix model tworzenia procesów ─────────────
fork()   — tworzy kopię (clone) procesu rodzica
           Child dostaje własną przestrzeń adresową
           Copy-on-Write: fizyczna kopia tylko przy zapisie!
exec()   — zastępuje kod procesu nowym programem
           Ładuje nowy ELF, czyści stack/heap, startuje

Shell uruchamia polecenie:
  1. fork() → child process
  2. child: exec("/bin/ls", args) → staje się ls
  3. parent: wait() → czeka na zakończenie child
  4. child kończy → zombi → parent zbiera exit code

─── Windows model ─────────────────────────────────────────
CreateProcess() — tworzy nowy proces + nowy wątek
Brak fork() — zawsze nowy proces (bez dziedziczenia)
Handle dziedziczenie przez SECURITY_ATTRIBUTES
Scheduler, priorytety, IPC
─── Scheduler — kto dostaje CPU ──────────────────────────
Linux CFS (Completely Fair Scheduler):
  vruntime — wirtualny czas pracy procesu
  Red-black tree posortowany po vruntime
  Zawsze uruchamia process z najmniejszym vruntime
  nice (-20 do 19): -20 = wysoki priorytet, 19 = najniższy

Windows Scheduler:
  32 poziomy priorytetów (0–31)
  Real-time (16–31) i Normal (0–15)
  Priority boost: foreground app dostaje wyższy priorytet
  preemptive: może przerwać proces w każdej chwili

─── IPC — komunikacja między procesami ───────────────────
Pipe:          │ ls | grep txt      unidirectional
Named Pipe:    │ /tmp/mypipe        bidirectional, przez nazwę
Shared Memory: │ mmap(MAP_SHARED)   najszybsze, ryzyko race

Jak FS przechowuje pliki — inode (Linux)
─── inode — metadane pliku (ext4) ────────────────────────
Plik = nazwa (w katalogu) + inode (metadane) + bloki danych

inode zawiera:
  typ pliku (regular, directory, symlink, device...)
  uprawnienia (owner, group, mode: rwxrwxrwx)
  właściciel (UID, GID)
  rozmiar, atime, mtime, ctime
  liczba hard linków
  wskaźniki do bloków danych (extents w ext4)

─── Hard link vs Symlink ──────────────────────────────────
Hard link: inna nazwa wskazuje na TEN SAMY inode
  ln /etc/hosts /tmp/myhosts
  Plik istnieje dopóki jest choć jeden hard link
  Nie może przekraczać granic filesystemu

Symlink (soft link): osobny inode → zawiera ścieżkę
  ln -s /etc/hosts /tmp/myhosts
  Może wskazywać na inne filesystem, na katalogi
  Dangling symlink jeśli cel usunięty

─── Journaling — odporność na awarie ────────────────────
Przed zapisem na dysk → zapisz do Journal (logu)
Po awarii → odtwórz niezakończone operacje z Journal
Modes: writeback (metadata only), ordered (domyślny), journal (all)
COW (Copy-on-Write): Btrfs, ZFS — nowe dane na nowe bloki
  Nigdy nie nadpisuje starych danych → atomowe snapshoty

Sieć & Network Stack
Jak OS obsługuje sieć — od karty do aplikacji
Stos sieciowy to jeden z najważniejszych podsystemów OS. Od przerwania w karcie sieciowej przez DMA, protokoły TCP/IP, sockety aż do aplikacji — każda warstwa dodaje abstrakcję i funkcjonalność.
Pakiet sieciowy — droga przez system
Pakiet TCP przychodzi na kartę sieciową (NIC):

1. NIC  → DMA: kopia pakietu do RAM (ring buffer)
         → IRQ: przerwanie do CPU

2. Kernel driver  → odbiera z ring buffer
                    → sk_buff (socket buffer) — struktura pakietu

3. Network stack (softirq)
   → Ethernet: sprawdź MAC, odrzuć obcy
   → IP: sprawdź IP, routing, TTL--
   → TCP: sprawdź porty, sequence numbers, checksum
   → Reassembly fragmentów (jeśli potrzeba)

4. Socket receive buffer
   → Dane trafiają do bufora socketa (SO_RCVBUF)
   → Jeśli bufor pełny → pakiet odrzucony (drop!)

5. Aplikacja
   → recv()/read() → syscall → dane z bufora → user space

Diagnostyka & Troubleshooting
Diagnozowanie problemów — od BSOD do OOM
Senior developer powinien umieć diagnozować problemy systemowe, czytać logi, analizować dump pamięci i rozumieć komunikaty o błędach. To umiejętność odróżniająca kogoś kto "instaluje" od kogoś kto "rozumie".
💀 Windows BSOD — co mówi Blue Screen
IRQL_NOT_LESS_OR_EQUAL
Sterownik próbuje dostępu do niedostępnej pamięci przy wysokim poziomie IRQ. Uszkodzony/niezgodny sterownik.
PAGE_FAULT_IN_NONPAGED_AREA
Dostęp do strony pamięci która nie może być wyrzucona. Zepsuta RAM lub sterownik.
MEMORY_MANAGEMENT
Błąd zarządzania pamięcią. Zepsuta pamięć RAM → memtest86+
CRITICAL_PROCESS_DIED
Krytyczny proces systemowy (lsass, csrss) zakończył się nieoczekiwanie.
DRIVER_IRQL_NOT_LESS_EQUAL
Plik *.sys podany w komunikacie to winowajca. Zaktualizuj lub usuń ten sterownik.
Analiza dumpów
WinDbg (debugger Microsoft) → !analyze -v → pełna analiza. Minidumpy w C:\Windows\Minidump\. Online: WinDbg Preview w Microsoft Store.
🐧 Linux Kernel Panic & OOM
Kernel Panic
Odpowiednik BSOD. Sprawdź: dmesg, /var/log/kern.log. Przyczyny: uszkodzony moduł, null pointer, stack overflow w sterowniku.
OOM Killer
Out Of Memory Killer — kernel zabija procesy gdy brakuje RAM. Sprawdź: dmesg | grep -i "out of memory". Pojawi się który PID i proces został zabity.
Segfault
dmesg | grep segfault — pokaże program i adres naruszenia. Wskaże crash jądra lub aplikacji.
Dysk pełny
df -h + du -sh /var/log/*. Typowi winowajcy: logi aplikacji bez rotacji, coredumps, Docker layers. journalctl --vacuum-size=1G
Zombie procesy
ps aux | grep Z. Zombie = zakończony ale rodzic nie wywołał wait(). Nie zajmuje CPU/RAM. Zniknie gdy rodzic zostanie zabity.
🔍 Boot nie startuje

Windows: Startup Repair → Automatic Repair → bootrec /fixmbr, /fixboot, /rebuildbcd

Linux: Live USB → mount dysk → chroot → grub-install → update-grub

Sprawdź kolejność boot w UEFI. Sprawdź czy dysk wykrywany.
