01 — warstwa aplikacji
HTTP — HyperText Transfer Protocol

Protokół tekstowy request-response. Klient pyta, serwer odpowiada. Każde żądanie jest niezależne (bezstanowe) — serwer nie pamięta poprzednich requestów.

budowa żądania HTTP
── Request ──────────────────────────────
GET /api/users/5 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGc...
Accept: application/json
Content-Type: application/json
(pusta linia)
{ "name": "Jan" }   ← ciało (body)

── Response ─────────────────────────────
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 45
Cache-Control: max-age=3600
(pusta linia)
{ "id": 5, "name": "Jan" }

02 — ewolucja protokołu
HTTP/1.1 vs HTTP/2 vs HTTP/3

Każda wersja rozwiązuje problemy wydajnościowe poprzedniej. HTTP/2 dodał multiplexing, HTTP/3 zamienił TCP na QUIC.

HTTP/1.1
Jedno żądanie na raz Request 1 → Response 1 Request 2 → Response 2 Request 3 → Response 3 Przeglądarka otwiera 6 połączeń do jednego hosta żeby działać równolegle. Tekstowy. Nagłówki bez kompresji (te same nagłówki powtarzane w kółko).
HTTP/2
Multiplexing! Req 1 ┐ Req 2 ├──► jedno połączenie TCP Req 3 ┘ Resp 2 ┐ Resp 1 ├──► w dowolnej kolejności Resp 3 ┘ Kompresja nagłówków (HPACK) Binarny (nie tekstowy) Server Push (server wysyła zasoby zanim poproszą)
HTTP/3
Zamiast TCP używa QUIC (zbudowany na UDP!) QUIC = UDP + niezawodność + TLS 1.3 wbudowane Rozwiązuje head-of-line blocking na poziomie transportu. 0-RTT lub 1-RTT handshake (szybsze połączenie) Działa lepiej na wolnych sieciach i mobile.
Head-of-line blocking: W HTTP/1.1 jeden wolny request blokuje kolejne. W HTTP/2 naprawiono to dla HTTP, ale TCP nadal blokuje jak zgubi pakiet. HTTP/3 (QUIC) naprawia problem na poziomie transportu — utrata pakietu jednego strumienia nie blokuje pozostałych.
03 — szyfrowanie
HTTPS i TLS — szyfrowanie połączenia

TLS (Transport Layer Security) szyfruje komunikację HTTP. Zapewnia: szyfrowanie (nikt nie czyta), integralność (nikt nie zmienia), uwierzytelnienie serwera (wiesz z kim rozmawiasz).

🔗 Łańcuch zaufania certyfikatów

🛡️ HSTS — wymuszenie HTTPS
Nagłówek odpowiedzi:
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Mówi przeglądarce: "Zawsze używaj HTTPS dla tej domeny, nawet jeśli użytkownik wpisze http://."

preload = wpisz domenę do listy w przeglądarce — HTTPS zawsze, nawet przy pierwszej wizycie.
04 — warstwa transportowa
TCP — nawiązywanie połączenia

Przed wysłaniem jakichkolwiek danych TCP wykonuje 3-etapowy handshake. Zapewnia niezawodność: numerowanie bajtów, potwierdzenia, retransmisje, kontrola przepływu.

🤝 Three-way handshake (SYN → SYN-ACK → ACK)
KLIENT SERWER ──── SYN (seq=100) ──────────────────► "Chcę się połączyć, mój numer to 100" ◄─── SYN-ACK (seq=200, ack=101) ───── "Ok! Mój numer to 200, potwierdzam twoje 100+1" ──── ACK (ack=201) ──────────────────► "Potwierdzam twoje 200+1" ══ Połączenie nawiązane! ══ ──── dane: GET /index.html ──────────► ◄─── dane: HTTP/1.1 200 OK ──────────
👋 Zamknięcie — four-way handshake
Klient: FIN ──────────────────────────► ◄───────────────────────── ACK :Serwer ◄───────────────────────── FIN :Serwer Klient: ACK ──────────────────────────► Stan TIME_WAIT — klient czeka 2×MSL (~2min) żeby upewnić się że ostatni ACK dotarł. Dlatego nie możesz od razu zbindować tego samego portu → użyj SO_REUSEADDR!
🚩 Flagi TCP
FlagaZnaczenie
SYNSynchronize — nawiązanie połączenia
ACKAcknowledge — potwierdzenie odbioru
FINFinish — zakończenie połączenia (graceful)
RSTReset — brutalne zerwanie (błąd)
PSHPush — wyślij dane od razu, bez buforowania
URGUrgent — pilne dane (rzadko używane)
📊 Kontrola przepływu — Sliding Window
Odbiorca mówi ile może przyjąć: Window Size = 65535 bajtów Nadawca nie może wysłać więcej niż rozmiar okna bez ACK. Slow Start: zacznij od małego okna, podwajaj po każdym ACK. Utrata pakietu: zmniejsz okno o połowę. Algorytmy: Reno, CUBIC, BBR BBR (Google) = najnowszy, najlepszy dla dużych opóźnień.
05 — transport
TCP vs UDP

Dwa protokoły transportowe. TCP zapewnia niezawodność kosztem latencji. UDP jest szybki ale nie gwarantuje dostarczenia.

TCP — niezawodny
CechaWartość
PołączenieTak (handshake przed danymi)
KolejnośćGwarantowana (numery sekwencji)
DostarczenieGwarantowane (ACK + retransmisja)
Kontrola przepływuTak (sliding window)
Kontrola przeciążeniaTak (slow start, CUBIC)
NarzutWiększy (nagłówek 20+ bajtów)
Kiedy używaćHTTP, e-mail, FTP, SSH, bazy danych — wszędzie gdzie liczy się kompletność
UDP — szybki
CechaWartość
PołączenieBrak (bezpołączeniowy)
KolejnośćNie gwarantowana
DostarczenieNie gwarantowane (best effort)
Kontrola przepływuBrak
Kontrola przeciążeniaBrak
NarzutMały (nagłówek 8 bajtów)
Kiedy używaćDNS, VoIP, gry online, streaming, QUIC/HTTP3 — gdzie ważna latencja
QUIC = UDP + własna niezawodność + TLS 1.3 Zamiast polegać na TCP, Google zbudował QUIC na UDP i zaimplementował własną kontrolę przepływu i szyfrowanie. Zysk: szybszy handshake (0-RTT), brak head-of-line blocking na poziomie transportu. To jest fundament HTTP/3.
06 — tunelowanie
VPN — Virtual Private Network

VPN tworzy szyfrowany tunel między urządzeniami. Twój ruch internetowy wychodzi przez serwer VPN — zmienia się widoczny IP i szyfruje dane w drodze.

🔐 Jak działa tunel VPN — enkapsulacja
BEZ VPN: Ty ──────────── widoczny ruch ──────────► Google 192.168.1.5 każdy router widzi treść 142.250.x.x Z VPN — paczka w paczce: ┌────────────────────────────────┐ │ IP dst: 5.5.5.5 (serwer VPN) │ ← zewnętrzna │ 🔐 SZYFROWANE │ │ ┌──────────────────────────┐ │ │ │ IP dst: 142.250.x.x │ │ ← wewnętrzna │ │ GET /search?q=... │ │ │ └──────────────────────────┘ │ └────────────────────────────────┘ Serwer VPN odpakuje, wyśle do Google ze swojego IP. Google widzi IP VPN, nie Twój.
⚡ WireGuard — nowoczesny VPN
Dlaczego WireGuard > OpenVPN? ~4000 linii kodu vs ~70000 (OpenVPN) Mniejszy = łatwiejszy do audytu bezpieczeństwa ChaCha20 szyfrowanie + Curve25519 DH UDP (nie TCP) = mniej narzutu Brak negocjacji algorytmów — jeden zestaw kryptograficzny, bez słabszych. Działa nawet jak zmieniasz sieć (WiFi → 4G) bez zrywania tunelu.

07 — przegląd
Inne protokoły warte znajomości
WebSocket
Pełnodupleksowy kanał na istniejącym połączeniu TCP. Zaczyna od HTTP Upgrade:
Upgrade: websocket
Serwer odpowiada 101 Switching Protocols.

Używaj do: czatów, powiadomień real-time, gier, live dashboardów. Alternatywa: SSE (Server-Sent Events) — jednostronne serwer→klient, prostsze.
SSH
Szyfrowany zdalny dostęp (port 22). Klucze asymetryczne: generujesz parę (ssh-keygen), klucz publiczny wrzucasz na serwer do ~/.ssh/authorized_keys.

SSH tunneling:
ssh -L 3306:localhost:3306 user@serwer
Lokalny port 3306 → przez SSH → port 3306 na serwerze.
gRPC
Nowoczesny RPC od Google. Binarny (Protocol Buffers), HTTP/2, ze schemą kontraktu (.proto pliki).

Zamiast JSON wysyłasz skompresowane binarne dane — 5-10x mniejsze i szybsze od REST/JSON. Popularny w mikroserwisach. Wymaga code generation z .proto.
SMTP / IMAP / POP3
SMTP (25/587) — wysyłanie poczty.
IMAP (143/993) — odbieranie z synchronizacją (mail zostaje na serwerze).
POP3 (110/995) — pobieranie i kasowanie z serwera.

SPF, DKIM, DMARC — rekordy DNS które chronią przed podszywaniem się pod Twoją domenę w e-mailach.
DHCP
Automatyczne przydzielanie IP. Twój komputer broadcastuje "Hej, potrzebuję IP!" → Router odpowiada "Masz 192.168.1.47, brama to .1, DNS to 8.8.8.8".

DORA: Discover → Offer → Request → Ack. Lease time — IP jest "pożyczone" na określony czas.

diagnostyka

# Każdy hop po drodze
traceroute google.com
