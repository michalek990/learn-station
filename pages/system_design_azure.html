Load Balancing
Strategie load balancingu — algorytmy i warstwy
Load balancer dystrybuuje ruch między instancjami serwisu. Działa na warstwie L4 (TCP/UDP) lub L7 (HTTP). Algorytm wyboru serwera ma ogromne znaczenie przy różnych profilach ruchu i typach workloadów.
⚖️ Algorytmy load balancingu
Round Robin
Każdy request do kolejnego serwera po kolei. Prosto, równomiernie — ale ignoruje rzeczywiste obciążenie. Kiedy: jednородne requesty, podobny czas odpowiedzi.
Weighted Round Robin
Jak Round Robin, ale serwery dostają proporcjonalnie więcej requestów wg wagi. Kiedy: heterogeniczne serwery (różna moc obliczeniowa).
Least Connections
Request trafia do serwera z najmniejszą liczbą aktywnych połączeń. Kiedy: long-lived connections (WebSocket, SSE), różny czas przetwarzania requestów.
Least Response Time
Połączenie aktywnych połączeń + najniższy avg response time. Najbardziej dynamiczny. Kiedy: zróżnicowane latencje serwerów (różne lokalizacje, sprzęt).
IP Hash
hash(client_IP) % N — ten sam klient zawsze do tego samego serwera (sticky sessions bez cookie). Problem: nierówny rozkład jeśli dużo ruchu z jednego IP (NAT, proxy).
Random
Losowy wybór. Statystycznie równomierny przy dużej liczbie requestów. Prosty do implementacji, brak state.
Power of Two Choices
Losuj 2 serwery, wybierz mniej obciążony. Lepszy niż czysty random, prostszy niż pełny Least Connections. Używany przez Nginx i NGINX Plus.

Rate Limiting
Algorytmy rate limitingu — Token Bucket, Sliding Window i inne
Rate limiting chroni system przed przeciążeniem i nadużyciami. Różne algorytmy mają różne trade-offy — burst handling, pamięć, dokładność. Na Azure: API Management, Application Gateway WAF, custom implementacja z Redis.
Algorytmy rate limiting — porównanie i implementacja
// ══ 1. TOKEN BUCKET — najpopularniejszy ══════════════════════
// Wiaderko N tokenów, refill M tokenów/sekundę.
// Request = pobierz token. Brak tokenów = odrzuć (429).
// Zaleta: burst do pojemności bucket. Wada: wymaga persystencji stanu.

// Implementacja Redis (Lua — atomowe!)
local key       = KEYS[1]
local capacity  = tonumber(ARGV[1])  -- max tokens
local refill    = tonumber(ARGV[2])  -- tokens/second
local now       = tonumber(ARGV[3])  -- current timestamp ms
local requested = tonumber(ARGV[4])  -- tokens requested

local state = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens     = tonumber(state[1]) or capacity
local last_refill = tonumber(state[2]) or now

-- Przelicz tokens od ostatniego refill
local elapsed = (now - last_refill) / 1000
tokens = math.min(capacity, tokens + elapsed * refill)

if tokens >= requested then
  tokens = tokens - requested
  redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
  redis.call('EXPIRE', key, 60)
  return 1  -- allowed
end
return 0    -- rejected

// ══ 2. LEAKY BUCKET — stały output rate ══════════════════════
// Jak wiaderko z dziurą — niezależnie od input, output stały.
// Żadnego bursta — wygładza ruch. Używany w sieci (QoS).
// Zaleta: przewidywalny output. Wada: odrzuca burst nawet jeśli idle.

// ══ 3. FIXED WINDOW COUNTER — prosty ══════════════════════════
// Licznik per okno czasowe (np. minuta).
// Problem: boundary burst — 100 req w :59, 100 req w :01 = 200 w 2 sek
key = "rate:" + userId + ":" + (timestamp / 60000)
count = redis.INCR(key)
if count == 1: redis.EXPIRE(key, 60)
return count <= limit

// ══ 4. SLIDING WINDOW LOG — dokładny, drogi ═══════════════════
// Pamiętaj timestamp każdego requestu. Odrzuć starsze niż window.
// Zaleta: brak boundary burst. Wada: O(n) pamięci per user.
redis.ZREMRANGEBYSCORE(key, 0, now - 60000)
redis.ZADD(key, now, requestId)
return redis.ZCARD(key) <= limit

// ══ 5. SLIDING WINDOW COUNTER — najlepszy balans ══════════════
// Aproksymacja sliding window używając tylko 2 liczników.
// current_window_count + (prev_window_count × overlap%)
long prevWindowMs = (nowMs / windowMs - 1) * windowMs;
double overlap = 1.0 - (nowMs - windowMs) / (double) windowMs;
double approxCount = prevCount * overlap + currentCount;
return approxCount < limit;

Sharding & Partitioning
Horizontal partitioning, sharding strategies i Azure Cosmos DB
Partitioning = dzielenie danych wewnątrz jednej bazy. Sharding = rozdzielenie danych na wiele niezależnych baz/nodów (horizontal scaling). Kluczowa decyzja: jak wybierać shard — hash, range, directory, geographic.
🗂️ Strategie partycjonowania
Hash Partitioning
shard = hash(key) % N. Równomierny rozkład danych. Problem: dodanie nowego shardu = re-hash wszystkich danych (rozwiązanie: consistent hashing). Kiedy: równomierny dostęp, brak range queries po kluczu shardującym.
Range Partitioning
Zakresy wartości → shard (A-M → shard1, N-Z → shard2). Zaleta: range queries na jednym shardzie. Problem: hotspot jeśli dane nie są równomiernie rozłożone (np. "A" dużo popularniejsze niż "Z"). Używany w Cassandra (partition key), HBase.
Directory Partitioning
Lookup service (mapa klucz → shard). Elastyczne re-sharding bez downtime. Problem: lookup service = single point of failure, dodatkowy hop. Używany w Flickr, Pinterest.
Geographic
Dane użytkowników europejskich → EU shard, US → US shard. Niskie latencje, compliance (GDPR). Problem: nierówny rozkład jeśli regiony różnej wielkości.
Vertical Partitioning
Kolumny zamiast wierszy — różne tabele/bazy dla różnych domen. Np. tabela User: kolumny podstawowe w PostgreSQL, zdjęcia profilowe w Blob Storage. Normalizacja vs denormalizacja.

Consistent Hashing
Consistent Hashing — ring, virtual nodes, re-balancing
Problem klasycznego hash(key) % N: dodanie/usunięcie jednego serwera zmienia przypisanie prawie wszystkich kluczy. Consistent hashing rozwiązuje to — zmiana N remapuje tylko K/N kluczy (K=liczba kluczy). Używany w Cassandra, DynamoDB, Redis Cluster, CDN.
🔄 Jak działa Consistent Hashing

Ring (pierścień) — wyobraź sobie hash space [0, 2³²) jako okrąg. Serwery są umieszczone w punktach pierścienia (przez hash ich adresu/nazwy). Klucz hashowany → punkt na pierścieniu → przejdź clockwise → pierwszy serwer = serwer dla tego klucza.

RING — przykład
[0 ─── S1:1a2b ─── S2:3c4d ─── S3:8e9f ─── 2³²]
key="user:42" → hash → 2f1e → pierwsza S? = S3
key="order:1" → hash → 1b3c → pierwsza S? = S2

Dodanie serwera S4 — tylko klucze między S3 a S4 są przenoszone. Reszta bez zmian.

Usunięcie serwera S2 — tylko klucze S2 przechodzą do S3. Reszta bez zmian.

Virtual Nodes (vnodes) — każdy fizyczny serwer ma N pozycji na pierścieniu (np. 150 vnodes). Efekt: bardziej równomierny rozkład, naturalny load balancing gdy serwery mają różną pojemność (silniejszy serwer = więcej vnodes).

Consistent Hashing — implementacja Java
public class ConsistentHashRing<T> {
    private final TreeMap<Integer, T> ring = new TreeMap<>();
    private final int virtualNodes;

    public ConsistentHashRing(int virtualNodes) {
        this.virtualNodes = virtualNodes;
    }

    public void addNode(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            int hash = hash(node.toString() + "#" + i);
            ring.put(hash, node);
        }
    }

    public void removeNode(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            ring.remove(hash(node.toString() + "#" + i));
        }
    }

    public T getNode(String key) {
        if (ring.isEmpty()) return null;
        int hash = hash(key);
        // Pierwsza pozycja na pierścieniu >= hash
        Map.Entry<Integer, T> entry = ring.ceilingEntry(hash);
        // Jeśli nie ma (koniec pierścienia) → wróć do początku
        return entry != null ? entry.getValue()
                              : ring.firstEntry().getValue();
    }

    private int hash(String key) {
        // MurmurHash lub FNV — lepszy rozkład niż Java hashCode()
        return MurmurHash3.murmurhash3_x86_32(
            key.getBytes(), 0, key.length(), 42);
    }
}

// Użycie:
ConsistentHashRing<String> ring =
    new ConsistentHashRing<>(150); // 150 virtual nodes
ring.addNode("redis-node-1:6379");
ring.addNode("redis-node-2:6379");
ring.addNode("redis-node-3:6379");

String node = ring.getNode("user:12345");
// → "redis-node-2:6379" (deterministic)

Caching Strategies
Cache patterns w głąb — CDN, multi-tier, cache coherence
Caching redukuje latencję i obciążenie bazy. Różne warstwy (CDN → Application Cache → DB cache) rozwiązują różne problemy. Kluczowe wyzwania: spójność, thundering herd, cache poisoning.
🏛️ Warstwy cache i strategie
Cache-Aside (Lazy)
Aplikacja sprawdza cache → miss → pobiera z DB → zapisuje do cache. Zalety: proste, cache tylko to co czytane. Wada: pierwsze żądanie zawsze cache miss, ryzyko stampede. Domyślny wybór.
Read-Through
Cache layer (np. Ehcache, NCache) automatycznie ładuje z DB przy miss — aplikacja zawsze czyta tylko z cache. Prostsze API, ale cache musi znać strukturę danych.
Write-Through
Zapis do cache I DB synchronicznie. Cache zawsze aktualny. Wada: wyższa write latency, cache zapełnia się danymi które mogą nie być czytane (write-heavy workloads). Używaj gdy read/write ratio jest zrównoważony.
Write-Behind (Write-Back)
Zapis do cache, asynchronicznie do DB. Super szybkie zapisy. Ryzyko: utrata danych przy awarii cache przed flush do DB. Używaj dla non-critical data (view counters, sessions).
Refresh-Ahead
Cache aktywnie odświeża klucze przed ich wygaśnięciem (gdy TTL zbliża się do końca). Eliminuje cache miss dla popularnych kluczy. Implementacja: background job który refetchuje hot keys.
CDN (Edge Cache)
Statyczne i semi-statyczne zasoby (CSS, JS, images, API responses) na edge nodach globalnie. Azure Front Door / Azure CDN. Cache-Control headers sterują TTL. Origin shield = CDN chroni origin przed stampede.

🌩️ Thundering Herd + Cache Stampede

Problem: popularny klucz wygasa → setki requestów jednocześnie → cache miss → wszystkie uderzają w DB → DB overwhelmed → latencja rośnie → więcej timeoutów → kaskada.

Rozwiązania:

• Mutex/Locking — tylko jeden request odświeża, reszta czeka (lub zwraca stale)

• Probabilistic early expiration — losowo odśwież przed TTL (XFetch algorithm)

• Stale-while-revalidate — serwuj starą wartość podczas odświeżania w tle

• TTL jitter — TTL = base ± random(0, 60s) — rozrzuć wygasanie w czasie

CAP Theorem
CAP Theorem głębiej — PACELC, Consistency Levels, real-world
CAP mówi: rozproszony system może gwarantować najwyżej 2 z 3: Consistency, Availability, Partition Tolerance. Ale partition tolerance jest niezbędna w każdej sieci → realny wybór: CP vs AP. PACELC idzie dalej — co gdy partycji NIE ma?
🔺 CAP — co naprawdę oznacza
Consistency (C)
Każdy odczyt zwraca najnowszy zapis lub błąd. Nie "stare dane". W praktyce: linearizability — każda operacja wygląda jakby była atomowa i natychmiastowa.
Availability (A)
Każde żądanie zawsze dostaje odpowiedź (nie timeout/error) — choć może być nieaktualna. 100% dostępność nodów które działają.
Partition Tolerance (P)
System działa mimo utraty sieci między nodami (network partition). W praktyce: musisz wybrać P, bo partycje sieciowe są nieuniknione. Wybierasz CP lub AP.
PACELC — rozszerzenie CAP

When Partition: C lub A (jak CAP)

Else (normalnie): Latency vs Consistency

Przykład: DynamoDB = PA/EL (gdy partycja: dostępny; normalnie: niska latencja kosztem spójności)

PostgreSQL = PC/EC (gdy partycja: spójny; normalnie: spójny kosztem latencji)

CAP w praktyce — real-world nuances CAP to model akademicki — rzeczywistość jest bardziej skomplikowana. Consistency nie jest binarna — jest spektrum. Availability nie jest binarna — SLA 99.99% to nie 100%. Network partitions nie zdarzają się codziennie — ważniejszy jest PACELC (latency vs consistency w normalnym działaniu). Przy projektowaniu systemu pytaj: "Jakie dane mogę serwować stale? Gdzie wymagam linearizability?" — np. saldo bankowe wymaga strong consistency, feed social media może być eventual.

Backpressure Patterns
Backpressure — kontrola przepływu w systemach rozproszonych
Backpressure to mechanizm sygnalizacji do producenta by zwolnił gdy konsument nie nadąża. Bez backpressure: producent wysyła szybciej niż konsument przetwarza → przepełnienie buforów → OOM → crash. Zarządzanie przepływem na każdej warstwie stosu.
🔧 Wzorce backpressure
Load Shedding
Celowe odrzucanie requestów gdy system jest przeciążony. Zwróć 503 z nagłówkiem Retry-After. Lepsze niż degradacja całego systemu. Priorytetyzuj — odrzucaj mniej ważne requesty (batch, low-priority), zachowaj capacity dla krytycznych.
Queue-based Leveling
Bufor między producentem a konsumentem — kolejka (Azure Service Bus, Kafka). Producent: szybko pisze do kolejki. Konsument: przetwarza we własnym tempie. Kolejka = elastyczny bufor na spiki. Monitoruj głębokość kolejki!
Circuit Breaker
Wzorzec ochronny — gdy dependency zaczyna failować powyżej threshold, circuit breaker "otwiera się" i zwraca fast-fail (nie czeka na timeout). Po czasie próbuje ponownie (half-open). Resilience4j, Polly (.NET). Zapobiega kaskadowym awariom.
Bulkhead
Izolacja zasobów per serwis — każda zależność ma własną pulę threadów/połączeń. Awaria jednego nie zajmuje zasobów całego systemu (jak grodzie na statku). Resilience4j: @Bulkhead.
Reactive Streams
Protokół backpressure między publisherem a subscriberem. Publisher produkuje tyle ile subscriber żąda (request(n)). Project Reactor (Spring WebFlux), RxJava. Operator onBackpressureDrop, onBackpressureBuffer, onBackpressureLatest.
Throttling
Ogranicz liczbę jednoczesnych żądań do downstream serwisu. Semaphore(maxConcurrent) lub rate limiter. Różnica od rate limiting: rate limiting = prędkość w czasie, throttling = równoległość.
Backpressure — Resilience4j + Spring WebFlux
// ══ Circuit Breaker — Resilience4j ═══════════════════════════
@CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
@TimeLimiter(name = "paymentService")
@Retry(name = "paymentService")
public Mono<PaymentResult> processPayment(PaymentRequest req) {
    return paymentClient.process(req);
}

private Mono<PaymentResult> paymentFallback(
        PaymentRequest req, Throwable ex) {
    // Circuit open lub timeout → kolejkuj do retry później
    queueForRetry(req);
    return Mono.just(PaymentResult.queued(req.getId()));
}

Flux<Order> orderStream = kafka.receive()
    .limitRate(100)              // request(100) od razu, potem po 75%

Case Study — Azure
URL Shortener na Azure — jak projektować krok po kroku
Klasyczny system design interview case. Projektujemy od wymagań przez skalę do konkretnej architektury Azure. Metodologia: wymagania → estymacja skali → API → dane → high-level design → głębokie dive → bottlenecks.
📋 Wymagania i estymacja skali

Funkcjonalne:

• Utwórz krótki URL (shortcode 7 znaków) dla długiego URL

• Redirect: krótki URL → oryginalny URL

• TTL dla URLi (opcjonalnie)

• Statystyki kliknięć (opcjonalnie)

Niefunkcjonalne:

• Wysoka dostępność (99.99%) — redirect musi działać zawsze

• Niska latencja redirect <10ms

• Redirect 10x częstszy niż create

Estymacja:

• 100M create/dzień = ~1160 req/s

• 1B redirect/dzień = ~11600 req/s

• Storage: 100M × 500 bytes = ~50 GB/dzień (tylko URLe)

• Read-heavy (10:1) → cache jest kluczowy

Generowanie shortcode: Base62 z auto-increment ID (Azure SQL Sequence lub Cosmos DB auto-id). Alternatywa: random 7 znaków + sprawdzenie unikalności.

Redirect flow: shortcode → Redis (hit: 301/302 redirect <1ms) → miss → Cosmos DB → Redis (write-through) → redirect

301 vs 302: 301 (Permanent) — przeglądarka cache'uje, mniej serwerów hits ale brak analytics. 302 (Temporary) — każde kliknięcie przez serwer → lepsza analityka. Wybierz 302 jeśli chcesz stats.

Analytics (async): Redirect Service → Azure Event Hubs → Azure Stream Analytics → Azure SQL / Cosmos DB dla stats

🔑 Kluczowe decyzje projektowe

Product Catalog (read-heavy):

Cosmos DB z Redis cache. Cache-aside z 1h TTL. Azure CDN dla obrazów produktów. CQRS: osobne modele dla read/write — denormalizowany widok dla listy produktów (bez JOIN-ów).

Inventory (strong consistency):

PostgreSQL z SELECT ... FOR UPDATE lub optimistic locking (version column). Nigdy nie aktualizuj stock w cache — zawsze przez DB z rowlock. Redis tylko do "last seen stock" display, nie do rezerwacji.

Order Saga (distributed transaction):

Choreography lub Orchestration saga przez Service Bus. Kompensujące transakcje: OrderCreated → ReserveInventory (fail → ReleaseInventory) → ProcessPayment (fail → RefundPayment) → ShipOrder. Outbox pattern dla atomowości DB + Service Bus.

Payment (idempotency):

Każde żądanie ma idempotency key. PostgreSQL: ON CONFLICT (idempotency_key) DO NOTHING. Webhook od procesora płatności: verify signature, process once (Redis check), return 200 always.

Flash Sale (spik 10000x):

Rate limiting per user (APIM), pre-warm Redis cache produktów, inventory reservation w Redis DECR (atomic) + eventual sync do DB, queue-based leveling przez Service Bus dla zamówień.

💬 Kluczowe decyzje projektowe

Azure SignalR Service zamiast własnych WebSocketów:

Managed WebSocket infrastructure — obsługuje sticky sessions automatycznie. Chat Service jest stateless. Skaluje do milionów połączeń. Integruje z AKS przez backplane (Redis lub built-in).

Message Storage — Cosmos DB z partition po roomId:

Każda wiadomość: {id, roomId, senderId, content, timestamp}. Partition key = roomId — wszystkie wiadomości pokoju na tym samym shardzie → fast pagination. TTL na stare wiadomości (>1 rok do archiwum Blob Storage).

Presence (online/offline) — Redis:

SET presence:userId 1 EX 30 — heartbeat co 15s. KEYS presence:*? Nie — użyj Bloom Filter lub Sorted Set ze score=timestamp. Subskrybenci Pub/Sub dostają notyfikację o zmianie statusu.

Fanout do dużych pokojów (10k+ użytkowników):

Nie push do każdego WebSocket synchronicznie. Użyj SignalR Groups — Azure SignalR Service obsługuje broadcast do grup. Dla very large rooms (100k+): hierarchiczny fanout z pośrednimi węzłami lub pull-based (klient polluje nowe wiadomości).

Wiadomości offline / push notifications:

Sprawdź status presence w Redis. Jeśli offline → Azure Notification Hubs (Firebase/APNs push) lub email. Watermark per user per room — "last read message id" → badge count = nowsze wiadomości od watermark.

System Design Interview — framework 1. Wymagania (5 min) — funkcjonalne i niefunkcjonalne. Pytaj o scale, availability, consistency.
2. Estymacja (3 min) — req/s, storage/dzień, bandwidth. Identyfikuj czy read-heavy czy write-heavy.
3. High-level design (10 min) — główne komponenty, API, model danych.
4. Deep dive (15 min) — bottlenecks, sharding, caching, edge cases.
5. Bottlenecks (5 min) — single point of failure, hotspots, jak skalować dalej.
Zawsze pytaj prowadzącego o priorytety — nie możesz zaprojektować wszystkiego w 45 min.
