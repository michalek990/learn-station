Apache Kafka — Fundament
Architektura — czym jest Kafka i jak działa
Kafka to rozproszony, persystentny log zdarzeń — nie kolejka. Wiadomości są trzymane na dysku i mogą być odczytane wielokrotnie przez wielu konsumentów niezależnie. Zaprojektowana dla milionów wiadomości/sekundę przy niskiej latencji.
🏗️ Kluczowe komponenty
Broker
Serwer Kafka — przechowuje dane, obsługuje producentów i konsumentów. Klaster = N brokerów.
Topic
Logiczny kanał komunikacji. Producenci piszą, konsumenci czytają. Podzielony na partycje.
Partition
Fizyczna jednostka topicu na dysku brokera. Wiadomości append-only, niezmienne. Ordering gwarantowany TYLKO wewnątrz partycji.
Offset
Numer kolejny wiadomości w partycji (0, 1, 2...). Consumer commituje offset — zapamiętuje postęp.
Replica
Kopia partycji na innym brokerze. Leader obsługuje I/O, follower replikuje.
ZooKeeper / KRaft
ZooKeeper: metadata i koordynacja (legacy). KRaft (Kafka 3.3+): Kafka zarządza sobą samodzielnie — rekomendowane.
Producer
Klient wysyłający wiadomości do topicu.
Consumer
Klient czytający wiadomości z topicu. Grupowany w Consumer Groups.
Consumer Group
Grupa konsumentów przetwarzających topic równolegle. Każda partycja → jeden consumer w grupie.

Tematy i Partycje
Topics, Partitions, Offsets, Replication — fundament danych
Topic to logiczny kanał. Partition to fizyczny log na dysku. Razem tworzą fundament skalowalności i durability Kafki. Liczba partycji determinuje maksymalną równoległość konsumpcji.
Zarządzanie topicami — CLI i AdminClient
# ── CLI — tworzenie i zarządzanie topicami ────────────────────
# Utwórz topic
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic orders \
  --partitions 6 \
  --replication-factor 3 \
  --config retention.ms=604800000 \
  --config min.insync.replicas=2

# Listuj topici
kafka-topics.sh --bootstrap-server localhost:9092 --list

# Szczegóły topicu — partycje, liderzy, repliki
kafka-topics.sh --bootstrap-server localhost:9092 \
  --describe --topic orders

# Zmiana konfiguracji
kafka-configs.sh --bootstrap-server localhost:9092 \
  --entity-type topics --entity-name orders \
  --alter --add-config retention.ms=86400000

# Zwiększenie liczby partycji (zmniejszenie niemożliwe!)
kafka-topics.sh --bootstrap-server localhost:9092 \
  --alter --topic orders --partitions 12
AdminClient — Java API
// Programowe tworzenie topicu (np. przy starcie aplikacji)
@Bean
public NewTopic ordersTopic() {
  return TopicBuilder.name("orders")
      .partitions(6)
      .replicas(3)
      .config(TopicConfig.RETENTION_MS_CONFIG, "604800000")
      .config(TopicConfig.MIN_IN_SYNC_REPLICAS_CONFIG, "2")
      .compact() // lub .delete() — domyślna polityka usuwania
      .build();
}

// Konfiguracje retention:
// retention.ms       = jak długo (ms), -1 = forever
// retention.bytes    = max rozmiar partycji, -1 = unlimited
// cleanup.policy     = delete (domyślna) | compact

// Log Compaction — zachowuje tylko ostatnią wartość per klucz
// Idealne dla: snapshots stanu, konfiguracje, user profiles
// Wiadomości z null value (tombstone) = usunięcie klucza

// Strategia partycjonowania:
// Key = null → Round Robin (domyślnie równomierne)
// Key = "orderId" → hash(key) % partitions
//   → gwarancja: ten sam klucz → ta sama partycja
//   → ordering per klucz!
// Custom Partitioner → własna logika rozdziału
📏 Ile partycji?

Zasada: liczba partycji = max planowana liczba konsumentów w grupie

Zbyt mało → ograniczona równoległość

Zbyt wiele → overhead metadanych, więcej file handles, wolniejszy leader election

Praktyka: zacznij od 6-12, skaluj w górę

⚠️ Nie możesz zmniejszyć liczby partycji!

🔁 Replication Factor

RF = N → wytrzymuje N-1 awarii brokerów

RF=1 → dev, testy (brak durability)

RF=2 → wytrzymuje 1 awarię (minimum prod)

RF=3 → standard produkcyjny ✅

min.insync.replicas=2 + acks=all = bezpieczne

📦 Rozmiar wiadomości

Domyślny max: 1 MB

message.max.bytes — broker

max.request.size — producer

fetch.max.bytes — consumer

Praktyka: trzymaj wiadomości małe (<1MB). Duże payloady → external storage + referencja w Kafce (Claim Check Pattern).

⏱️ Retention

retention.ms — czas życia (domyślnie 7 dni)

retention.bytes — max rozmiar log per partycja

Log Compaction — zachowuje tylko ostatnią wartość dla danego klucza. Używaj dla state stores, cache invalidation.

Tombstone — wiadomość z value=null → usuwa klucz z compacted topic

Producers
Producer API — wysyłanie wiadomości, konfiguracja, batching
Producer odpowiada za wysyłanie wiadomości do Kafki. Kluczowe parametry: acks (gwarancja zapisu), batching (wydajność), kompresja, retries i idempotencja. Błędna konfiguracja może skutkować utratą danych lub duplikatami.
Producer — konfiguracja i wysyłka
// ── Konfiguracja Producenta ───────────────────────────────────
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG,
         "kafka1:9092,kafka2:9092,kafka3:9092");

// Serializery
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
         StringSerializer.class);
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
         JsonSerializer.class);

// ── Gwarancje dostarczania ────────────────────────────────────
// acks=0 → fire and forget — najszybszy, ryzyko utraty
// acks=1 → leader potwierdza — domyślny, możliwa utrata przy failover
// acks=all → wszystkie ISR potwierdzają — najbezpieczniejszy
props.put(ProducerConfig.ACKS_CONFIG, "all");

// ── Idempotencja — exactly-once na poziomie producenta ────────
// Zapobiega duplikatom przy retry (sieć może dostarczyć 2x)
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
// Wymaga: acks=all, retries>0, max.in.flight.requests.per.connection≤5

// ── Batching — wydajność ──────────────────────────────────────
// linger.ms: ile czekać na wypełnienie batcha (domyślnie 0!)
props.put(ProducerConfig.LINGER_MS_CONFIG, 5);
// batch.size: max rozmiar batcha w bajtach
props.put(ProducerConfig.BATCH_SIZE_CONFIG, 32768); // 32KB
// buffer.memory: łączna pamięć dla buforów producenta
props.put(ProducerConfig.BUFFER_MEMORY_CONFIG, 33554432);

// ── Kompresja ─────────────────────────────────────────────────
// snappy: dobry balans CPU/kompresja (rekomendowany)
// lz4: najszybszy, mniejsza kompresja
// gzip: najlepsza kompresja, drogi CPU
// zstd: najlepszy (Kafka 2.1+), rekomendowany dla nowych projektów
props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "snappy");

// ── Retry ─────────────────────────────────────────────────────
props.put(ProducerConfig.RETRIES_CONFIG, Integer.MAX_VALUE);
props.put(ProducerConfig.RETRY_BACKOFF_MS_CONFIG, 100);
props.put(ProducerConfig.DELIVERY_TIMEOUT_MS_CONFIG, 120000); // 2min
Producer — wysyłka sync/async/transakcje
KafkaProducer<String, OrderEvent> producer =
    new KafkaProducer<>(props);

// ── Async (zalecana dla throughput) ──────────────────────────
ProducerRecord<String, OrderEvent> record =
    new ProducerRecord<>(
        "orders",          // topic
        "order-123",        // key → determinuje partycję
        new OrderEvent(...)  // value
    );

producer.send(record, (metadata, exception) -> {
  if (exception != null) {
    log.error("Send failed: {}", exception.getMessage());
    // alert, DLQ, retry logic
  } else {
    log.debug("Sent to partition={} offset={}",
        metadata.partition(), metadata.offset());
  }
});

// ── Sync (gdy potrzebujesz potwierdzenia przed kontynuacją) ──
RecordMetadata meta = producer.send(record).get(); // blokuje!

// ── Transakcyjny Producer — exactly-once semantics ────────────
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "order-producer-1");
KafkaProducer<String, OrderEvent> txProducer = new KafkaProducer<>(props);
txProducer.initTransactions();

try {
  txProducer.beginTransaction();
  txProducer.send(new ProducerRecord<>("orders", event1));
  txProducer.send(new ProducerRecord<>("payments", event2));
  txProducer.commitTransaction(); // atomowo oba topici
} catch (Exception e) {
  txProducer.abortTransaction(); // rollback
  throw e;
} finally {
  txProducer.close();
}

// Custom Partitioner
public class PriorityPartitioner implements Partitioner {
  public int partition(String topic, Object key, byte[] keyBytes,
      Object value, byte[] valueBytes, Cluster cluster) {
    // VIP orders → partition 0, reszta → hash-based
    if (value instanceof OrderEvent e && e.isVip())
      return 0;
    return Utils.abs(Utils.murmur2(keyBytes)) %
           cluster.partitionCountForTopic(topic);
  }
}
Producer — złota konfiguracja produkcyjna acks=all + enable.idempotence=true + retries=Integer.MAX_VALUE + max.in.flight.requests.per.connection=5 + linger.ms=5 + compression.type=snappy. To daje: durability, brak duplikatów, dobry throughput.

Consumers i Consumer Groups
Consumer API, offset management, rebalancing, lag
Consumer czyta z Kafki. Consumer Group umożliwia równoległe przetwarzanie — partycje są rozdzielane między instancje. Kluczowe: offset commit strategia, obsługa rebalance, unikanie consumer lag.
Consumer — konfiguracja i pętla przetwarzania
Properties props = new Properties();
props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
props.put(ConsumerConfig.GROUP_ID_CONFIG, "order-processor");
props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
         StringDeserializer.class);
props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
         JsonDeserializer.class);

// ── Offset management ─────────────────────────────────────────
// auto.offset.reset: co zrobić gdy brak commita dla tej grupy
// earliest → od początku logu (replay!)
// latest   → tylko nowe wiadomości (domyślnie)
// none     → throw exception jeśli brak offset
props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest");

// ZAWSZE wyłącz auto-commit! Manualna kontrola = lepsza
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, false);

// ── Tuning wydajności ─────────────────────────────────────────
// max.poll.records: max rekordów per poll() — default 500
props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, 100);
// max.poll.interval.ms: max czas między poll() — default 5min
// Jeśli przetwarzanie trwa dłużej → rebalance! Ustaw realistycznie
props.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, 300000);
// session.timeout.ms: heartbeat timeout — default 45s
props.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG, 45000);

KafkaConsumer<String, OrderEvent> consumer = new KafkaConsumer<>(props);
consumer.subscribe(List.of("orders"));

while (true) {
  ConsumerRecords<String, OrderEvent> records =
      consumer.poll(Duration.ofMillis(1000));

  for (ConsumerRecord<String, OrderEvent> record : records) {
    try {
      processOrder(record.value());
      // commit per-partition z precyzyjnym offsetem
      consumer.commitSync(Map.of(
          new TopicPartition(record.topic(), record.partition()),
          new OffsetAndMetadata(record.offset() + 1)
      ));
    } catch (RetryableException e) {
      // nie commituj → zostanie przetworzone ponownie
      break;
    }
  }
}
Rebalance Listener i Seek
// ── ConsumerRebalanceListener — obsługa rebalance ─────────────
// Rebalance = partycje są na nowo przydzielane między konsumentów
// Powody: nowy konsument dołącza, odpada, zmiana partycji topicu
// Podczas rebalance: przetwarzanie ZATRZYMANE dla całej grupy!

consumer.subscribe(List.of("orders"), new ConsumerRebalanceListener() {
  @Override
  public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
    // Wywołane PRZED rebalance — zapisz postęp!
    log.info("Partitions revoked: {}", partitions);
    consumer.commitSync(currentOffsets); // commit przed oddaniem
  }
  @Override
  public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
    // Wywołane PO rebalance — można inicjalizować stan
    log.info("Partitions assigned: {}", partitions);
  }
});

// ── Seek — czytaj od konkretnego offsetu ──────────────────────
consumer.subscribe(List.of("orders"));
consumer.poll(Duration.ofMillis(0)); // trigger assignment

// Cofnij do początku (replay!)
consumer.seekToBeginning(consumer.assignment());

// Skocz na koniec (pomiń historię)
consumer.seekToEnd(consumer.assignment());

// Precyzyjny seek do offsetu
consumer.seek(new TopicPartition("orders", 0), 1000L);

// Seek do czasu (np. ostatnie 24h)
Map<TopicPartition, Long> timestamps = consumer.assignment().stream()
    .collect(Collectors.toMap(tp -> tp,
        tp -> System.currentTimeMillis() - 86400000L));
consumer.offsetsForTimes(timestamps).forEach((tp, oam) -> {
  if (oam != null) consumer.seek(tp, oam.offset());
});

// ── Sprawdzenie consumer lag z kodu ───────────────────────────
Map<TopicPartition, Long> endOffsets =
    consumer.endOffsets(consumer.assignment());
consumer.assignment().forEach(tp -> {
  long lag = endOffsets.get(tp) -
             consumer.position(tp);
  log.info("Partition {} lag: {}", tp, lag);
});
Consumer Group — jak działa skalowanie 1 partition → max 1 consumer w grupie. 6 partycji + 6 konsumentów = pełna równoległość. 6 partycji + 8 konsumentów = 2 konsumentów bezczynnych. Zawsze partitions ≥ max_consumers.
Stop-the-world Rebalance Klasyczny rebalance zatrzymuje całą grupę. Unikaj częstych rebalance: stabilny group.instance.id (Static Membership), session.timeout.ms ≥ max GC pause, max.poll.interval.ms > max czas przetwarzania.
Commit Strategy — at-least-once Commituj po przetworzeniu (manual commit). Commituj przed → at-most-once (możliwa utrata). Zawsze projektuj consumer jako idempotentny — Kafka może dostarczyć wiadomość więcej niż raz!

Gwarancje dostarczania
At-most-once, At-least-once, Exactly-once
Trzy poziomy gwarancji dostarczania wiadomości. Wybór zależy od wymagań biznesowych — każdy wiąże się z kompromisem między pewnością a wydajnością. Exactly-once to najdroższe i najtrudniejsze.
💨 At-most-once (najwyżej raz)

Wiadomość może zostać utracona, nigdy zduplikowana.

Producer: acks=0 lub acks=1

Consumer: commit przed przetworzeniem

Kiedy używać Metryki, logi, analytics gdzie utrata kilku wiadomości jest akceptowalna. Najwyższy throughput.
🔁 At-least-once (przynajmniej raz)

Wiadomość zostanie dostarczona, ale może być duplikat.

Producer: acks=all + retries

Consumer: commit po przetworzeniu (manual)

Duplikaty możliwe gdy: producer retry po sieci, consumer crashuje po przetworzeniu ale przed commitem

Kiedy używać Standard dla większości systemów. Wymagany: idempotentny consumer po stronie aplikacji.
✅ Exactly-once (dokładnie raz)

Wiadomość przetworzona dokładnie raz. Najtrudniejsze.

Producer: enable.idempotence=true + transactional.id

Consumer: isolation.level=read_committed

Kafka Streams: EOS włączone przez processing.guarantee=exactly_once_v2

Kiedy używać Finanse, płatności, deduplikacja przy agregacjach. Duży overhead — używaj świadomie.
Idempotentny Consumer — wzorce implementacji
// At-least-once + Idempotent Consumer = efektywne exactly-once
// bez overhead transakcji Kafki

// ── Wzorzec 1: klucz idempotencji w bazie ─────────────────────
@Service
public class IdempotentOrderProcessor {
  private final ProcessedEventRepo processedEvents;
  private final OrderService orderService;

  @Transactional
  public void process(OrderPlacedEvent event, String eventId) {
    // eventId = topic + partition + offset → unikalny klucz
    if (processedEvents.existsById(eventId)) {
      log.warn("Duplicate event {}, skipping", eventId);
      return;
    }
    orderService.processOrder(event);
    processedEvents.save(new ProcessedEvent(eventId));
  }
}

// ── Wzorzec 2: upsert zamiast insert ──────────────────────────
// SQL UPSERT jest naturalnie idempotentny
// INSERT INTO orders ... ON CONFLICT (order_id) DO UPDATE ...

// ── Wzorzec 3: wersjonowanie / optimistic locking ─────────────
public void updateOrderStatus(String orderId, String status, long version) {
  int updated = orderRepo.updateStatusIfVersion(orderId, status, version);
  if (updated == 0) {
    log.info("Stale event or duplicate, ignoring");
  }
}
// UPDATE orders SET status=? WHERE id=? AND version=?

// ── Eventowy eventId z nagłówków Kafki ───────────────────────
@KafkaListener(topics = "orders")
public void handle(
    @Payload OrderPlacedEvent event,
    @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
    @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
    @Header(KafkaHeaders.OFFSET) long offset) {
  String eventId = topic + "-" + partition + "-" + offset;
  process(event, eventId);
}

KafkaTemplate — wysyłanie wiadomości
// ── KafkaTemplate — podstawowa wysyłka ───────────────────────
@Service
public class OrderEventProducer {
  private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

  // Async — fire and remember (callback)
  public void sendAsync(OrderEvent event) {
    kafkaTemplate.send("orders", event.orderId(), event)
        .whenComplete((result, ex) -> {
          if (ex != null) {
            log.error("Kafka send error: {}", ex.getMessage());
            // fallback: outbox, retry queue, alert
          } else {
            RecordMetadata meta = result.getRecordMetadata();
            log.debug("Sent topic={} partition={} offset={}",
                meta.topic(), meta.partition(), meta.offset());
          }
        });
  }

  // Z nagłówkami (headers)
  public void sendWithHeaders(OrderEvent event) {
    ProducerRecord<String, OrderEvent> record =
        new ProducerRecord<>("orders", event.orderId(), event);
    record.headers()
        .add("eventType", "OrderPlaced".getBytes())
        .add("version", "1".getBytes())
        .add("correlationId", UUID.randomUUID().toString().getBytes());
    kafkaTemplate.send(record);
  }

  // Wysyłka do konkretnej partycji
  public void sendToPartition(OrderEvent event, int partition) {
    kafkaTemplate.send("orders", partition, event.orderId(), event);
  }
}

// ── Transactional KafkaTemplate ──────────────────────────────
@Service
public class TransactionalOrderService {
  private final KafkaTemplate<String, Object> kafkaTemplate;
  private final OrderRepository orderRepo;

  @Transactional("kafkaTransactionManager")
  public void placeOrder(Order order) {
    orderRepo.save(order);
    kafkaTemplate.send("orders", new OrderPlacedEvent(order));
    kafkaTemplate.send("audit", new AuditEvent("order.placed", order));
    // oba topici w jednej transakcji!
  }
}

      @Header(KafkaHeaders.OFFSET) long offset,
      @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
      Acknowledgment ack) {
    try {
      orderService.process(event);
      ack.acknowledge(); // commit offset
    } catch (Exception e) {
      log.error("Processing failed: {}", e.getMessage());
      // nie acknowledge → retry lub DLQ (patrz ErrorHandler)
      throw e;
    }
  }

  // ── Batch listener — przetwarza wiele naraz ────────────────────
  @KafkaListener(topics = "bulk-events", containerFactory = "batchFactory")
  public void handleBatch(
      List<ConsumerRecord<String, BulkEvent>> records,
      Acknowledgment ack) {
    log.info("Processing batch of {} records", records.size());
    try {
      List<BulkEvent> events = records.stream()
          .map(ConsumerRecord::value).toList();
      bulkService.processAll(events);
      ack.acknowledge();
    } catch (Exception e) {
      // obsłuż częściowe błędy w batchu
      throw e;
    }
  }

  // ── Filter messages — tylko VIP orders ───────────────────────
  @KafkaListener(topics = "orders", containerFactory = "filteredFactory")
  public void handleVipOrder(OrderPlacedEvent event) {
    // containerFactory ma ustawiony RecordFilterStrategy
    vipOrderService.process(event);
  }
}

// ── DefaultErrorHandler z exponential backoff i DLQ ──────────
@Configuration
public class KafkaErrorConfig {

  @Bean
  public DefaultErrorHandler errorHandler(
      KafkaTemplate<Object, Object> kafkaTemplate) {

    // Dead Letter Publisher — prześlij do .DLT topic
    DeadLetterPublishingRecoverer recoverer =
        new DeadLetterPublishingRecoverer(kafkaTemplate,
            (rec, ex) -> new TopicPartition(
                rec.topic() + ".DLT", rec.partition()));

    // Exponential backoff: 1s, 2s, 4s, 8s, 16s
    ExponentialBackOffWithMaxRetries backOff =
        new ExponentialBackOffWithMaxRetries(5);
    backOff.setInitialInterval(1000L);
    backOff.setMultiplier(2.0);
    backOff.setMaxInterval(16000L);

    DefaultErrorHandler handler =
        new DefaultErrorHandler(recoverer, backOff);

    // Nie retryuj dla błędów które zawsze failują
    handler.addNotRetryableExceptions(
        ValidationException.class,
        JsonParseException.class
    );
    return handler;
  }

  // ── DLT Consumer — obsługuj martwe wiadomości ─────────────────
  @KafkaListener(topics = "orders.DLT", groupId = "dlt-processor")

Kafka Streams to biblioteka do przetwarzania strumieni danych wbudowana w Kafkę. Działa jako normalna aplikacja Java — bez klastra Spark/Flink. Topology = sieć Source → Processor → Sink. State stores = lokalny RocksDB.
Kafka Streams — podstawowe operacje
@Configuration
public class OrderStreamTopology {

  @Bean
  public KStream<String, OrderEvent> ordersStream(StreamsBuilder builder) {

    KStream<String, OrderEvent> orders =
        builder.stream("orders");

    // ── Filter — tylko duże zamówienia ────────────────────────
    KStream<String, OrderEvent> bigOrders = orders
        .filter((key, order) -> order.getTotal().compareTo(
            new BigDecimal("1000")) > 0);

    // ── Map — transformuj wartość ─────────────────────────────
    KStream<String, OrderNotification> notifications = orders
        .mapValues(order -> new OrderNotification(
            order.getId(),
            order.getCustomerEmail(),
            "Twoje zamówienie: " + order.getTotal()
        ));

    // ── Branch — rozgałęzienie strumienia ─────────────────────
    Map<String, KStream<String, OrderEvent>> branches =
        orders.split(Named.as("branch-"))
            .branch((Named.as("vip"),
                (key, order) -> order.isVip()))
            .branch((Named.as("regular"),
                (key, order) -> !order.isVip()))
            .noDefaultBranch();

    branches.get("branch-vip").to("orders-vip");
    branches.get("branch-regular").to("orders-regular");

    // ── Merge — łącz strumienie ───────────────────────────────
    KStream<String, OrderEvent> domestic = builder.stream("orders-pl");
    KStream<String, OrderEvent> foreign  = builder.stream("orders-intl");
    KStream<String, OrderEvent> all = domestic.merge(foreign);

    // ── To topic — wyślij wynik ───────────────────────────────
    notifications.to("order-notifications");
    return bigOrders;
  }
}

@Bean
public KTable<String, OrderStats> orderStats(StreamsBuilder builder) {

  // ── KTable — zmaterializowany widok (klucz → ostatnia wartość)
  KTable<String, UserProfile> users =
      builder.table("user-profiles",
          Materialized.as("users-store"));

  // ── Stream-Table Join — wzbogać event o dane z tabeli ────────
  KStream<String, EnrichedOrder> enriched =
      builder.stream<String, OrderEvent>("orders")
          .join(users,
              (order, user) -> new EnrichedOrder(order, user),
              Joined.with(Serdes.String(),
                  JsonSerde.of(OrderEvent.class),
                  JsonSerde.of(UserProfile.class)));

  // ── Agregacja z oknem czasowym ────────────────────────────────
  // Zlicz zamówienia per klient w oknie 1h
  KTable<Windowed<String>, Long> ordersPerHour =
      builder.stream<String, OrderEvent>("orders")
          .groupByKey()
          .windowedBy(TimeWindows
              .ofSizeWithNoGrace(Duration.ofHours(1)))
          .count(Materialized.as("orders-per-hour"));

  // ── Sumowanie wartości zamówień per customer ──────────────────
  KTable<String, BigDecimal> totalPerCustomer =
      builder.stream<String, OrderEvent>("orders")
          .groupBy((key, order) -> order.getCustomerId())
          .aggregate(
              BigDecimal::new,  // initializer
              (customerId, order, total) ->
                  total.add(order.getTotal()),  // add
              (customerId, order, total) ->
                  total.subtract(order.getTotal()), // subtract (tombstone)
              Materialized.with(Serdes.String(), BigDecimalSerde.INSTANCE)
          );

  // ── State Store — query ze strony aplikacji ───────────────────
  // ReadOnlyKeyValueStore<String, BigDecimal> store =
  //   streams.store("total-per-customer", QueryableStoreTypes.keyValueStore());
  // BigDecimal total = store.get("customer-123");

  return totalPerCustomer;
}
Typy okien Tumbling: stałe, niepokrywające się (1h, 2h...). Hopping: pokrywające się (1h co 15min). Session: grupuje aktywność z przerwami. Sliding: ciągłe okno o stałej długości.
State Stores Lokalny RocksDB per partycja. Persystentny (przeżywa restart). Dostępny przez Interactive Queries. Changelog topic w Kafce = backup state store.
Stream vs Table KStream: każda wiadomość = nowe zdarzenie. KTable: każda wiadomość = update stanu per klucz (tylko ostatnia wartość). KGlobalTable = replicated table na wszystkich instancjach.
Kiedy Streams, kiedy Consumer Streams: transformacje, joins, agregacje w locie. Consumer: proste przetwarzanie event-by-event, integracja z zewnętrznymi systemami, złożona logika biznesowa.

Wzorce i Pułapki
Dobre praktyki, wzorce projektowe i typowe błędy
Zestawienie wzorców które działają w produkcji oraz pułapek które kosztują wiele godzin debugowania. Każdy wzorzec to lekcja wyniesiona z realnych incydentów produkcyjnych.
✅ Dobre praktyki
Naming topicow
Konwencja: domain.entity.event-type, np. orders.order.placed, payments.payment.failed. Unikaj nazw per-serwis.
Jeden typ eventu per topic
Łatwiejszy schema management i monitoring. Alternatywnie: jeden topic per subdomena z discriminator field.
Klucz wiadomości
Zawsze używaj klucza dla ordering per entity (orderId, customerId). null key = round-robin = brak ordering.
Transactional Outbox
Atomowy zapis do DB i Kafki. DB → outbox table → CDC (Debezium) → Kafka. Brak podwójnego zapisu.
Idempotent consumers
Zawsze zakładaj że wiadomość przyjdzie więcej niż raz. Projektuj logikę jako idempotentną.
DLQ per topic
Każdy topic powinien mieć odpowiadający .DLT topic. Monitoruj DLT — to alarm systemu.
Claim Check Pattern
Duży payload → zapisz w S3/blob storage, wyślij w Kafce tylko referencję (URL + metadata).
Consumer group per service
Każdy mikroserwis = osobna consumer group. Niezależne offsety, replay, skalowanie.
💀 Typowe błędy i pułapki
auto.commit=true
Offset commitowany automatycznie co 5s — utrata wiadomości przy crashu między commit a przetworzeniem. Zawsze false!
Zbyt długie przetwarzanie
max.poll.interval.ms przekroczony → consumer kick z grupy → rebalance → spowolnienie całej grupy.
Zmiana klucza partycji
Zmiana klucza = inne partycje = ordering naruszony! Raz ustalony klucz = niezmienialny dla danego topicu.
Nie monitorujesz DLT
Wiadomości lądują w DLT i nikt nie widzi. System "działa" ale dane giną. Monitoruj lag DLT aktywnie.
Zbyt mało partycji
Nie możesz skalować konsumentów powyżej liczby partycji. Zawsze planuj z 2x zapasem.
Brak Idempotencji
Duplikaty w DB, podwójne płatności, podwójne emaile. At-least-once bez idempotencji = katastrofa.
auto.create.topics=true
Literówka w nazwie topicu tworzy nowy topic. Wyłącz na produkcji!
Brak kompresji
JSON bez kompresji = ogromny narzut. Snappy daje 3-5x mniejszy rozmiar przy niskim CPU.

@KafkaListener(topics = "payment.processed")

}

🎯 Decision Tree — kiedy Kafka?
✅ Wybierz Kafka gdy:

Miliony wiadomości/s, event sourcing, wielu niezależnych konsumentów tego samego strumienia, replay historii danych, Kafka Streams processing, integracja z Flink/Spark, event-driven microservices, audit log.

✅ Wybierz RabbitMQ gdy:

Task queues (worker pattern), złożony routing (exchanges, binding), TTL per wiadomość, priority queues, request-reply pattern, mały/średni wolumen, niska latencja krytyczna.

🛡️ Security — SSL + SASL
SSL/TLS
Szyfrowanie komunikacji broker↔klient i broker↔broker. Konfiguracja: security.protocol=SSL, keystore i truststore.
SASL/PLAIN
Autentykacja username/password. Wymaga SSL w tle! Prosty, ale hasła w konfiguracji.
SASL/SCRAM
Bezpieczniejszy — hasła hashowane. Rekomendowane zamiast PLAIN.
SASL/GSSAPI
Kerberos. Enterprise środowiska z istniejącym Kerberosem.
ACL
Kontrola dostępu per topic per użytkownik. kafka-acls.sh --add --allow-principal. Read/Write/Describe per topic.
Kafka klient
security.protocol=SASL_SSL, sasl.mechanism=SCRAM-SHA-512, sasl.jaas.config z credentials.
Minimum na produkcji SSL + SASL/SCRAM + ACL dla każdego serwisu z tylko potrzebnymi uprawnieniami. Zero PLAINTEXT listenerów dostępnych z zewnątrz.
⚡ KRaft — Kafka bez ZooKeepera

KRaft = Kafka Raft Metadata mode — Kafka 3.3+ GA. ZooKeeper EOL w Kafka 4.0.

Zalety KRaft
Brak ZooKeepera = mniej serwerów do zarządzania. Szybszy startup, szybszy controller failover (ms vs s). Więcej partycji per klaster.
Architektura
Brokers mają nową rolę: controller (Raft quorum) lub broker lub obie. Controller = metadata log.
Migracja
Nowe klastry → KRaft od razu. Istniejące → narzędzie migracji kafka-storage.sh.
Capacity Planning — zasady kciuka

• 1 broker = 30-50 MB/s write throughput (HDD) lub 250 MB/s (SSD)

• Dysk: (throughput × retention × RF) / num_brokers

• Partycje per broker: max 4000 (Kafka 2.x), 200k+ (KRaft)

• RAM: JVM heap 6GB + OS page cache 32GB+ (ważniejszy!)

• CPU: mało potrzebuje, chyba że heavy compression

• Sieć: 10 GbE dla high-throughput klastrów

Troubleshooting — diagnostyka typowych problemów
# ══ Diagnoza Consumer Lag ════════════════════════════════════
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group order-processor --verbose
# Jeśli lag rośnie: skaluj consumers (max do liczby partycji)
# lub zmniejsz max.poll.records + uprzośnij przetwarzanie

# ══ Diagnoza Under-Replicated Partitions ═════════════════════
kafka-topics.sh --bootstrap-server kafka:9092 \
  --describe --under-replicated-partitions
# Możliwe przyczyny: broker down, zbyt wolna replikacja
# Sprawdź: replica.lag.time.max.ms i network bandwidth

# ══ Log Dirs — sprawdź rozmiar na dysku ══════════════════════
kafka-log-dirs.sh --bootstrap-server kafka:9092 \
  --topic-list orders --describe

# ══ Performance Test — ile wiadomości/s możesz wysłać ════════
kafka-producer-perf-test.sh \
  --topic test-perf \
  --num-records 1000000 \
  --record-size 1000 \
  --throughput -1 \
  --producer-props bootstrap.servers=kafka:9092

# ══ Consumer Performance Test ════════════════════════════════
kafka-consumer-perf-test.sh \
  --bootstrap-server kafka:9092 \
  --topic test-perf \
  --messages 1000000

# ══ Dump wiadomości z topicu (debug) ════════════════════════
kafka-console-consumer.sh \
  --bootstrap-server kafka:9092 \
  --topic orders \
  --from-beginning \
  --max-messages 10 \
  --property print.key=true \
  --property print.offset=true

# ══ Preferred Replica Election (równoważ liderów) ═════════════
kafka-leader-election.sh --bootstrap-server kafka:9092 \
  --election-type PREFERRED --all-topic-partitions
🔧 Quick Troubleshooting Checklist

🔴 Consumer nie przetwarza:

→ Sprawdź lag: kafka-consumer-groups.sh --describe

→ Sprawdź logi consumer (rebalance? exception?)

→ Czy max.poll.interval.ms nie jest przekroczony?

→ DLT pełny? Consumer utknął na błędnej wiadomości?

🟡 Wysokie latencje:

→ linger.ms zbyt duże u producenta?

→ Czy broker nie jest przeciążony? (request.handler.idle)

→ Under-replicated partitions?

🟡 Producent nie może połączyć:

→ bootstrap.servers poprawny?

→ Firewall blokuje port 9092?

→ SSL/SASL poprawnie skonfigurowany?

🟢 Zwiększenie throughput:

→ Zwiększ partycje + consumers

→ Włącz compression (snappy)

→ Zwiększ batch.size + linger.ms

→ Batch consumer + paralel processing
