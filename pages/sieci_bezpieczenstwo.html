01 — przeglądarka
CORS — Cross-Origin Resource Sharing

Przeglądarka blokuje żądania JavaScript do innych originów (Same-Origin Policy). CORS pozwala serwerowi powiedzieć "ten inny origin może do mnie pisać".

Origin = protokół + host + port. https://app.com i https://api.app.com to różne originy! CORS chroni użytkownika, nie serwer. Curl i Postman CORS ignorują — tylko przeglądarka go egzekwuje.
🔄 Simple request vs Preflight
Simple Request
GET / POST z prostymi nagłówkami i text/plain lub application/x-www-form-urlencoded. Przeglądarka wysyła bezpośrednio.
1 req
Preflight
PUT/DELETE/PATCH, Content-Type: application/json, własne nagłówki. Przeglądarka NAJPIERW wysyła OPTIONS z pytaniem "czy mogę?"
2 req
── Preflight ────────────────────────────── Przeglądarka OPTIONS /api/users HTTP/1.1 Origin: https://frontend.com Access-Control-Request-Method: PUT Access-Control-Request-Headers: Authorization Serwer odpowiada: Access-Control-Allow-Origin: https://frontend.com Access-Control-Allow-Methods: GET,POST,PUT,DELETE Access-Control-Allow-Headers: Authorization,Content-Type Access-Control-Max-Age: 86400 ── Teraz wysyłane właściwe żądanie ────────
konfiguracja CORS — backend
// Spring Boot
@CrossOrigin(origins = "https://app.com")
@RestController
public class ApiController { ... }

// Lub globalnie:
@Bean
public CorsFilter corsFilter() {
  CorsConfiguration config = new CorsConfiguration();
  config.addAllowedOrigin("https://app.com");
  config.addAllowedMethod("*");
  config.addAllowedHeader("*");
  config.setAllowCredentials(true);
  return new CorsFilter(...);
}

// Nginx — ręczne nagłówki
add_header Access-Control-Allow-Origin
           "https://app.com" always;
add_header Access-Control-Allow-Methods
           "GET,POST,PUT,DELETE,OPTIONS";
add_header Access-Control-Allow-Headers
           "Authorization,Content-Type";
⚠️ NIE używaj Access-Control-Allow-Origin: * z credentials! Jak używasz withCredentials: true (cookies/auth), musisz podać konkretną domenę. Wildcard * z credentials = błąd przeglądarki.
02 — stan sesji
Cookies — zarządzanie stanem

Małe kawałki danych które serwer ustawia w przeglądarce. Wysyłane automatycznie z każdym żądaniem do danej domeny. HTTP jest bezstanowy — cookies dają nam stan.

🍪 Atrybuty cookie
AtrybutCo robi
HttpOnlyJS nie ma dostępu (document.cookie nie działa). Ochrona przed XSS.
SecureWysyłane tylko przez HTTPS. Nigdy przez HTTP.
SameSite=StrictCookie tylko przy nawigacji z tej samej strony. Silna ochrona CSRF.
SameSite=LaxDomyślne w nowoczesnych przeglądarkach. Blokuje cross-site przy POST.
SameSite=NoneWysyłane zawsze (wymaga Secure). Potrzebne dla embedded widgets.
DomainDla jakiej domeny i subdomen. Brak = tylko ta subdomena.
PathDla jakiej ścieżki. / = cała domena.
Max-Age / ExpiresKiedy wygasa. Brak = session cookie (usuwa przy zamknięciu przeglądarki).
ustawianie i czytanie cookies
// Serwer ustawia cookie (HTTP Response)
Set-Cookie: sessionId=abc123;
  HttpOnly; Secure;
  SameSite=Lax;
  Path=/;
  Max-Age=86400

// Java (Spring)
ResponseCookie cookie = ResponseCookie
  .from("sessionId", "abc123")
  .httpOnly(true)
  .secure(true)
  .sameSite("Lax")
  .maxAge(Duration.ofDays(1))
  .path("/")
  .build();
response.addHeader("Set-Cookie",
                   cookie.toString());

// Przeglądarka wysyła automatycznie:
Cookie: sessionId=abc123

// JS - tylko bez HttpOnly!
document.cookie = "theme=dark; path=/";
const cookies = document.cookie;
03 — tożsamość
Uwierzytelnianie — JWT, OAuth, OpenID

Uwierzytelnianie (AuthN) = kim jesteś. Autoryzacja (AuthZ) = co możesz robić. Dwa różne problemy, często mylone.

🎫 JWT — JSON Web Token
Token = 3 części oddzielone kropką: eyJhbGciOiJIUzI1NiJ9 ← Header (base64) . eyJzdWIiOiIxMjMiLCJyb2xlIjoiQURNSU4ifQ ← Payload . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c ← Signature Payload po odkodowaniu: { "sub": "123", ← ID użytkownika "role": "ADMIN", ← role/uprawnienia "iat": 1710000000, ← kiedy wystawiono "exp": 1710086400 ← kiedy wygasa } Serwer weryfikuje podpis swoim kluczem. Nie musi odpytywać bazy danych! → szybko Nie można unieważnić przed exp → wada
użycie JWT
// Wysyłanie tokena
Authorization: Bearer eyJhbG...

// Java (Spring Security)
String token = Jwts.builder()
  .subject("userId123")
  .claim("role", "ADMIN")
  .issuedAt(new Date())
  .expiration(new Date(now + 86400000))
  .signWith(secretKey)
  .compact();

// Weryfikacja
Claims claims = Jwts.parser()
  .verifyWith(secretKey)
  .build()
  .parseSignedClaims(token)
  .getPayload();
🔑 OAuth 2.0 — autoryzacja przez trzecią stronę
── Authorization Code Flow ─────────────── Użytkownik klika "Zaloguj przez Google" │ ▼ Twoja apka redirect → Google /authorize ?client_id=...&redirect_uri=...&scope=email │ ▼ Użytkownik loguje się i zatwierdza Google redirect → Twoja apka ?code=AUTHORIZATION_CODE │ ▼ Twoja apka (backend) → Google /token POST code + client_secret │ ▼ Google zwraca access_token + id_token
📋 Porównanie metod auth
MetodaKiedy używać
Session + CookieTradycyjne web apps. Serwer pamięta sesję. Łatwo unieważnić.
JWT BearerAPI, SPA, mobile. Bezstanowy. Trudno unieważnić przed exp.

04 — infrastruktura
Proxy i Reverse Proxy

Pośrednicy w komunikacji sieciowej. Forward proxy stoi przed klientem, reverse proxy stoi przed serwerem.

➡️ Forward Proxy (po stronie klienta)
Klient → Forward Proxy → Internet Serwer widzi IP proxy, nie klienta. Klient konfiguruje proxy ręcznie. Zastosowania: - Anonimizacja (VPN to też rodzaj proxy) - Filtrowanie treści w firmach - Cache dla całej sieci lokalnej - Ominięcie blokad geograficznych
⬅️ Reverse Proxy (po stronie serwera)
Internet → Reverse Proxy → Serwery Klient widzi tylko proxy, nie backend. Zastosowania: - Load balancing (rozdzielanie ruchu) - SSL termination (proxy kończy TLS, backend dostaje HTTP) - Cache odpowiedzi - Ochrona przed DDoS - Routing per domena/ścieżka Przykłady: Nginx, Caddy, HAProxy, Cloudflare
Nginx — reverse proxy config
# Jedna apka na jednej domenie
server {
  listen 443 ssl;
  server_name apka.com;

  # SSL termination tutaj
  ssl_certificate     /etc/letsencrypt/...;
  ssl_certificate_key /etc/letsencrypt/...;

  location / {
    # Przekaż do Docker kontenera
    proxy_pass http://localhost:3000;

    # Ważne nagłówki!
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For
                     $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # WebSocket support
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
  }

  # /api → inny backend
  location /api/ {
    proxy_pass http://localhost:8080/;
  }
}

# HTTP → HTTPS redirect
server {
  listen 80;
  return 301 https://$host$request_uri;
}
05 — dystrybucja
CDN — Content Delivery Network

Sieć serwerów rozsianych geograficznie (PoP — Points of Presence). Użytkownik dostaje pliki z serwera najbliżej niego, nie z głównego serwera.

BEZ CDN: wszyscy z jednego miejsca Użytkownik (PL) ──────────────────► Serwer (US) ~150ms latencji Z CDN: serwer blisko użytkownika Użytkownik (PL) ──► PoP Warszawa ──► odpowiedź ~5ms latencji! Użytkownik (JP) ──► PoP Tokio ────► odpowiedź Origin serwer jest odpytywany tylko gdy PoP nie ma w cache (cache miss).
Cache-Control nagłówki decydują co CDN cache'uje: Cache-Control: public, max-age=31536000, immutable — dla plików z hashem w nazwie (JS/CSS).
Cache-Control: no-store — dla danych prywatnych (nigdy nie cache'uj).
Cache-Control: public, max-age=3600 — dla publicznych API responses (1 godzina).

06 — skalowalność
Load Balancer

Rozdziela ruch między wiele instancji serwera. Zapewnia wysoką dostępność (jeśli jedna pada — ruch idzie do pozostałych) i skalowalność.

Użytkownicy │││ ▼▼▼ Load Balancer ← jeden punkt wejścia │ │ │ ▼ ▼ ▼ App1 App2 App3 ← wiele instancji │ │ │ ▼ ▼ ▼ Baza danych (shared) Health checks: LB co kilka sekund sprawdza czy instancja żyje. Martwa instancja → wyciągana z puli.

07 — bezpieczeństwo
Bezpieczeństwo sieciowe — OWASP Top 10

Najczęstsze ataki na aplikacje webowe i jak się przed nimi bronić. Jako programista musisz rozumieć te zagrożenia.

⚠️ Najważniejsze ataki
01
SQL Injection
Wstrzykiwanie SQL w dane wejściowe: ' OR '1'='1
✅ Użyj prepared statements / parametrized queries
02
XSS — Cross-Site Scripting
Wstrzykiwanie JavaScript w stronę. Złodziej sesji, keylogger.
✅ Escape outputu, Content-Security-Policy, HttpOnly cookies
03
CSRF — Cross-Site Request Forgery
Złośliwa strona wykonuje akcje w imieniu zalogowanego usera.
✅ SameSite=Lax cookies, CSRF token, sprawdź Origin/Referer
04
Broken Authentication
Słabe hasła, brak rate limiting, nieważne tokeny.
✅ bcrypt, rate limiting, MFA, sprawdzaj exp tokenu
05
MITM — Man in the Middle
Atakujący przechwytuje komunikację między klientem a serwerem.
✅ HTTPS wszędzie, HSTS, certyfikat pinning
SQL Injection — przykład ataku i obrony
// ❌ PODATNE — konkatenacja stringa
String query = "SELECT * FROM users "
  + "WHERE email = '" + email + "'";
// Atak: email = "' OR '1'='1"
// Wynik: SELECT * FROM users
//   WHERE email = '' OR '1'='1'
// → zwraca WSZYSTKICH userów!

// ✅ BEZPIECZNE — prepared statement
PreparedStatement ps = conn.prepareStatement(
  "SELECT * FROM users WHERE email = ?"
);
ps.setString(1, email);
// Parametr jest escapowany automatycznie
// ' OR '1'='1 → traktowane jako tekst

// JPA/Hibernate — bezpieczne domyślnie
userRepo.findByEmail(email); // ok!
Security headers — dobre praktyki
# Nginx — nagłówki bezpieczeństwa

# Wymuszenie HTTPS
Strict-Transport-Security:
  max-age=31536000; includeSubDomains

# Blokuj clickjacking
X-Frame-Options: DENY

# Blokuj MIME sniffing
X-Content-Type-Options: nosniff

# Content Security Policy
Content-Security-Policy:
  default-src 'self';
  script-src 'self' cdn.example.com;
  style-src 'self' 'unsafe-inline';

# Rate limiting w Nginx
limit_req_zone $binary_remote_addr
  zone=api:10m rate=10r/s;

location /api/ {
  limit_req zone=api burst=20;
}
✅ Checklist bezpieczeństwa dla programisty
Hasła i auth
✅ Używaj bcrypt do hashowania haseł (nie MD5, nie SHA1)
✅ Rate limiting na endpointach logowania
✅ Krótki TTL dla JWT access tokenów (15min–1h)
✅ Refresh token w HttpOnly cookie
✅ Weryfikuj exp i iss w JWT
Input / Output
✅ Prepared statements dla wszystkich zapytań SQL
✅ Waliduj i sanityzuj każdy input od usera
✅ Escapuj output w HTML (nie ufaj danym z bazy)
✅ Ogranicz rozmiar żądań (max body size)
✅ Whitelista zamiast blacklisty dla plików upload
Transport i nagłówki
✅ HTTPS wszędzie + HSTS header
✅ Baza danych tylko na 127.0.0.1 (nie 0.0.0.0)
✅ Security headers (CSP, X-Frame-Options)
✅ Ukryj wersje serwerów (Server: nginx bez wersji)
✅ Zasada najmniejszych uprawnień (DB user tylko SELECT/INSERT)
