Architektura
Jak PostgreSQL działa od środka
Senior musi rozumieć architekturę żeby diagnozować problemy i podejmować decyzje projektowe. PostgreSQL to proces-based database server — każde połączenie to osobny proces OS. Shared memory (shared_buffers) to wspólna pamięć dla wszystkich procesów.
🏗️ Kluczowe procesy i komponenty
postmaster
Główny demon. Nasłuchuje na połączenia, fork'uje backend processes. Zarządza shared memory i background workers.
backend process
Jeden per połączenie. Wykonuje zapytania, ma dostęp do shared_buffers. Koszt fork = ~10ms — dlatego używamy connection pooler (PgBouncer).
WAL writer
Write-Ahead Log — każda zmiana trafia do WAL PRZED zapisem na dysk. Gwarancja durability. Podstawa replikacji i point-in-time recovery.
checkpointer
Regularnie flush'uje dirty pages z shared_buffers na dysk. checkpoint_completion_target rozkłada I/O w czasie.
autovacuum
Krytyczny background worker. Usuwa martwe krotki (dead tuples), aktualizuje statystyki, zapobiega wraparound XID. Nigdy go nie wyłączaj.
bgwriter
Proaktywnie zapisuje dirty pages — zmniejsza spike I/O podczas checkpoint. Konfiguruj bgwriter_lru_maxpages.
💾 Ścieżka zapisu danych
INSERT/UPDATE/DELETE
↓
WAL buffer (wal_buffers)
↓ synchronous_commit=on → fsync
WAL segment file (pg_wal/)
↓ równolegle
shared_buffers (dirty pages)
↓ checkpoint
Data files (base/)

Kluczowy insight: Commit potwierdza zapisanie do WAL — nie do heap! Dane są bezpieczne bo WAL = source of truth. Heap update = eventual.

Storage Layout Każda tabela = plik w base/<db_oid>/<table_oid>. Plik podzielony na 8KB pages. Każda page zawiera tuple header + dane. TOAST (The Oversized-Attribute Storage Technique) — wartości >2KB idą do osobnej tabeli TOAST. SELECT pg_relation_filepath('table_name') — znajdź plik tabeli.
Diagnostyka architektury — kluczowe widoki systemowe
-- Aktywne połączenia i ich stany
SELECT pid, usename, application_name, state, wait_event_type, wait_event,
       now() - query_start AS duration, left(query, 80) AS query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC;

-- Rozmiar bazy, tabel i indeksów
SELECT
  schemaname,
  tablename,
  pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total,
  pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) AS table_size,
  pg_size_pretty(pg_indexes_size(schemaname||'.'||tablename)) AS indexes_size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC;

-- Statystyki tabel — hit ratio (powinno być > 99%)
SELECT
  relname,
  heap_blks_read, heap_blks_hit,
  round(heap_blks_hit::numeric / nullif(heap_blks_hit + heap_blks_read, 0) * 100, 2) AS cache_hit_ratio
FROM pg_statio_user_tables
ORDER BY heap_blks_read DESC;

Indeksy
Indeksy — typy, strategie, kiedy pomagają a kiedy szkodzą
Indeks nie jest zawsze dobry. Każdy indeks spowalnia INSERT/UPDATE/DELETE i zajmuje miejsce. Senior wie kiedy tworzyć indeks, jakiego typu użyć i jak weryfikować że jest używany. Reguła: najpierw EXPLAIN ANALYZE, potem decyzja o indeksie.
📚 Typy indeksów — kiedy używać
B-Tree DEFAULT
Domyślny, najczęściej używany. Obsługuje: =, <, >, <=, >=, BETWEEN, LIKE 'prefix%', ORDER BY. Nie obsługuje: LIKE '%suffix', regex, full-text. Dobry dla kolumn o wysokiej selektywności (wiele unikalnych wartości).
Hash
Tylko dla =. Szybszy niż B-Tree dla equality lookups, ale nie obsługuje range queries, ORDER BY, UNIQUE. Od PG 10 w pełni WAL-logged. Rzadko używany.
GiST
Geometria, PostGIS, full-text search, network types (inet/cidr), zakres. Generalized Search Tree — rozszerzalny framework. Obsługuje operatory @>, <&, itp.
GIN
Full-text search, jsonb, arrays. Jeden klucz indeksu → wiele wierszy (odwrócony indeks). Wolniejsze INSERT, szybsze lookup. Idealny dla jsonb @>, tsvector @@, array @>.
BRIN
Block Range INdex. Ogromne tabele z naturalnym porządkiem (logi, time-series, append-only). Bardzo mały rozmiar. Nie nadaje się do tabel z losowymi inserts. Sprawdź correlation w pg_stats.
Partial index
WHERE w definicji. Mniejszy, szybszy. Np. CREATE INDEX ON orders(user_id) WHERE status='pending' — indeksuje tylko podzbiór wierszy. Konieczne żeby planner go użył: zapytanie musi zawierać ten sam WHERE.
Zaawansowane techniki indeksowania
-- Covering index (INCLUDE) — index-only scan bez heap access
-- Planner może odpowiedzieć na zapytanie z samego indeksu
CREATE INDEX idx_orders_user_covering
ON orders(user_id)
INCLUDE (status, created_at, total_amount);

-- Indeks na wyrażeniu — normalizacja w indeksie
CREATE INDEX idx_users_email_lower
ON users(lower(email));
-- Zapytanie MUSI używać tego samego wyrażenia:
SELECT * FROM users WHERE lower(email) = '[email protected]';

-- Partial index — tylko aktywne zamówienia
CREATE INDEX idx_orders_pending
ON orders(created_at)
WHERE status = 'pending';

-- Multicolumn index — kolejność kolumn MA znaczenie
-- (a, b) indeks → działa dla: WHERE a=X, WHERE a=X AND b=Y
-- NIE działa dla: WHERE b=Y (bez a)
CREATE INDEX idx_orders_user_status
ON orders(user_id, status);

-- CONCURRENTLY — tworzenie indeksu bez blokowania tabeli
CREATE INDEX CONCURRENTLY idx_new ON orders(customer_id);
-- Wolniejsze, nie blokuje DML, wymaga unikalnej transakcji

-- GIN dla JSONB
CREATE INDEX idx_metadata_gin ON events USING GIN (metadata);
-- SELECT * FROM events WHERE metadata @> '{"type":"click"}'

-- Znajdź nieużywane indeksy (kandydaci do usunięcia)
SELECT schemaname, tablename, indexname, idx_scan,
       pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelname NOT LIKE '%pkey%'
ORDER BY pg_relation_size(indexrelid) DESC;
Pułapki indeksów NULL wartości — B-Tree indeksuje NULL, ale IS NULL nie używa indeksu (od PG 8.3 używa, ale planner może zrezygnować). Niski cardinality — kolumna boolean z 50/50 rozkładem nie skorzysta na indeksie B-Tree. Casting — WHERE id = '123' gdzie id=int może nie użyć indeksu (implicit cast). Zawsze matchuj typy.
03 / 09
Query Planner
EXPLAIN ANALYZE — czytanie planów wykonania jak senior
Query planner decyduje jak wykonać zapytanie. Opiera się na statystykach (pg_statistics). Senior musi umieć czytać EXPLAIN output, rozumieć koszty, identyfikować problemy i wiedzieć jak na nie wpłynąć — bez hint'ów (PostgreSQL ich nie ma).
📊 Węzły planu — co oznaczają
Seq Scan
Pełny skan tabeli. Nie zawsze zły — dla małych tabel lub gdy zwracasz >10-15% wierszy planner celowo wybiera Seq Scan. Problem gdy koszt jest wysoki a selektywność duża.
Index Scan
Skan indeksu + dostęp do heap dla każdego wiersza. Dobre dla wysoce selektywnych zapytań. Każdy heap access to random I/O — kosztowne przy dużej liczbie wyników.
Index Only Scan
Najszybszy. Dane pobierane z samego indeksu (covering index). Wymaga że kolumny w SELECT/WHERE są w indeksie (INCLUDE). Sprawdź Heap Fetches — powinno być 0.
Bitmap Index Scan
Najpierw zbiera tid'y z indeksu do bitmapy, potem batch read z heap. Kompromis między Seq Scan a Index Scan. Działa dla wielu indeksów jednocześnie (BitmapAnd/BitmapOr).
Hash Join
Build hash table z małej tabeli, probe z dużej. Dobry dla równości JOIN bez indeksu. Wymaga pamięci — jeśli spills to dysk. Sprawdź Batches w output.
Nested Loop
Dla każdego wiersza z zewnętrznej tabeli — lookup w wewnętrznej (przez indeks). Idealny gdy zewnętrzna jest mała. Fatalny przy dużych tabelach bez indeksu.
Merge Join
Obie strony posortowane — merge w jednym przejściu. Efektywny gdy dane posortowane lub Sort jest tani. Dobry dla ORDER BY JOIN.
EXPLAIN ANALYZE — workflow diagnostyki
-- ZAWSZE używaj ANALYZE + BUFFERS + FORMAT TEXT
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, o.total, u.email
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = 'pending'
  AND o.created_at > now() - INTERVAL '7 days';

/*
EXPLAIN output do analizy:
  - rows= (estimated) vs actual rows= — duża różnica = stale statistics
  - cost= pierwszy to startup cost, drugi to total
  - Buffers: shared hit= (cache) vs read= (disk)
  - actual time= czas węzła w ms
*/

-- Kiedy planner się myli — wymuszenie przez konfigurację sesji
SET enable_seqscan = off;   -- wymusz indeks (diagnostic only!)
SET enable_hashjoin = off;  -- wymusz merge/nested loop
SET work_mem = '256MB';      -- więcej RAM dla sort/hash w tej sesji

-- Aktualizuj statystyki gdy planner ma złe szacunki
ANALYZE orders;
ANALYZE orders(status, created_at);  -- tylko wybrane kolumny

-- Extended statistics — korelacje między kolumnami
CREATE STATISTICS stat_orders_status_user
  (dependencies, ndistinct, mcv)
  ON status, user_id FROM orders;
ANALYZE orders;

-- pg_stat_statements — najwolniejsze zapytania
SELECT round(total_exec_time::numeric, 2) AS total_ms,
       calls,
       round(mean_exec_time::numeric, 2) AS mean_ms,
       round(stddev_exec_time::numeric, 2) AS stddev_ms,
       left(query, 100) AS query
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;
Złe szacunki planera — diagnoza Gdy rows=500 actual rows=50000 — planner ma złe statystyki. Rozwiązania: 1) ANALYZE na tabeli. 2) Zwiększ default_statistics_target (domyślnie 100, ustaw 500+ dla ważnych kolumn). 3) Użyj CREATE STATISTICS dla korelacji kolumn. 4) Sprawdź czy autovacuum działa — nie analizuje tabel z <50 zmianami od ostatniego analyze.

MVCC & Transactions
MVCC, izolacja transakcji i anomalie — jak to naprawdę działa
Multi-Version Concurrency Control — PostgreSQL nigdy nie nadpisuje danych in-place. UPDATE = nowa wersja krotki (tuple). Stare wersje zostają jako "dead tuples" aż vacuum je usunie. To jest cena MVCC — bloat i potrzeba vacuum.
🔒 Poziomy izolacji — anomalie i zachowanie
PoziomDirty ReadNon-Repeatable ReadPhantom ReadSerialization Anomaly
READ COMMITTED
Default PG		 Nie Możliwe Możliwe Możliwe
REPEATABLE READ Nie Nie Nie (PG) Możliwe
SERIALIZABLE Nie Nie Nie Nie
Dirty Read
Czytasz dane niezatwierdzonej transakcji. PG nigdy — każda wersja ma XID commitu.
Non-Repeatable
Dwa identyczne SELECT w tej samej transakcji dają różne wyniki. W READ COMMITTED każdy SELECT widzi nowy snapshot.
Phantom Read
Nowe wiersze pasujące do WHERE pojawiają się między zapytaniami. REPEATABLE READ w PG chroni przed tym (snapshot całej transakcji).
Serialization
Write Skew — dwie transakcje czytają te same dane i piszą na podstawie ich wartości. Wymaga SERIALIZABLE lub pessimistic locking.
Transakcje, locking, advisory locks
-- Optymistyczny vs pesymistyczny locking

-- PESYMISTYCZNY: SELECT FOR UPDATE (row-level lock)
BEGIN;
SELECT balance FROM accounts
WHERE id = 42
FOR UPDATE;  -- Lock wiersza - inni czekają
UPDATE accounts SET balance = balance - 100 WHERE id = 42;
COMMIT;

-- FOR UPDATE SKIP LOCKED — job queue pattern (nie blokuj, pomiń)
SELECT * FROM job_queue
WHERE status = 'pending'
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- OPTIMISTYCZNY: version column (CAS pattern)
UPDATE orders
SET status = 'shipped', version = version + 1
WHERE id = 1 AND version = 5;  -- 0 rows = concurrent update!

-- Advisory Locks — application-level distributed locks
SELECT pg_try_advisory_lock(12345);  -- non-blocking, true/false
SELECT pg_advisory_lock(12345);     -- blocking, session-level
SELECT pg_advisory_unlock(12345);
-- Zwalniany automatycznie po zakończeniu sesji/transakcji

-- Savepoints — partial rollback w transakcji
BEGIN;
  INSERT INTO orders(...) VALUES(...);
  SAVEPOINT after_order;
  INSERT INTO payments(...) VALUES(...);
  -- jeśli payment fails:
  ROLLBACK TO SAVEPOINT after_order;
  -- order jest wciąż w transakcji
COMMIT;
Write Skew — klasyczny przykład Dwóch lekarzy jednocześnie sprawdza "czy jest min. 1 lekarz na dyżurze?" (obaj widzą: tak), obaj biorą urlop. Wynik: 0 lekarzy. REPEATABLE READ nie chroni — obaj czytają i piszą różne wiersze. Rozwiązanie: SERIALIZABLE (PG wykryje i rzuci serialization failure) lub explicit lock na wierszu "shift".

Vacuum & Bloat
VACUUM, autovacuum, table bloat — zarządzanie martwymi krotkami
VACUUM to cena MVCC. Bez vacuum: martwe krotki rosną (bloat), visibility map jest nieaktualna (brak index-only scans), XID wraparound grozi utratą danych. Autovacuum jest domyślnie włączony i należy go tuningować, nie wyłączać.
🧹 Jak działa VACUUM i co robi
Dead tuples
UPDATE/DELETE nie usuwa fizycznie — oznacza krotki jako "dead". VACUUM usuwa martwe krotki, ale nie zwraca miejsca do OS — oznacza pages jako "free" do reuse.
VACUUM FULL
Zwraca miejsce do OS. Przepisuje całą tabelę. Wymaga ACCESS EXCLUSIVE LOCK — blokuje wszystkie operacje. Używaj tylko gdy bloat jest krytyczny i możesz sobie pozwolić na downtime (lub pg_repack).
pg_repack
Alternatywa dla VACUUM FULL bez locka. Tworzy nową tabelę w tle, synchronizuje zmiany, swap. Tylko krótki lock na koniec. Produkcyjny standard.
XID Wraparound
Krytyczne! XID to 32-bit licznik (2 mld transakcji). Gdy prawie pełny — PG automatycznie stopuje żeby uniknąć utraty danych. Vacuum musi regularnie freezeować stare XID. Monitoruj age(datfrozenxid).
Visibility Map
Bitmapa — czy strona zawiera tylko "all-visible" tuple. Umożliwia Index-Only Scan (planner wie że nie musi sprawdzać heap). VACUUM aktualizuje visibility map.
Monitoring bloat i autovacuum
-- Monitoruj dead tuples i kiedy był ostatni vacuum
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  round(n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0) * 100, 1) AS dead_pct,
  last_vacuum,
  last_autovacuum,
  last_analyze,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

-- XID Wraparound risk — ALARMUJ gdy > 1.5 mld
SELECT
  datname,
  age(datfrozenxid) AS xid_age,
  pg_size_pretty(pg_database_size(datname)) AS db_size
FROM pg_database
ORDER BY age(datfrozenxid) DESC;

-- Bloat estimate (z pgstattuple lub przybliżenie)
SELECT
  tablename,
  pg_size_pretty(pg_total_relation_size('public.'||tablename)) AS total,
  pg_size_pretty(n_dead_tup * 8192) AS approx_dead_bytes
FROM pg_stat_user_tables
WHERE n_dead_tup > 100000
ORDER BY n_dead_tup DESC;

-- Autovacuum tuning dla tabeli z dużym write volume
ALTER TABLE high_write_table SET (
  autovacuum_vacuum_scale_factor = 0.01,   -- default 0.2 (20%)
  autovacuum_vacuum_threshold = 500,        -- default 50
  autovacuum_analyze_scale_factor = 0.005,
  autovacuum_vacuum_cost_delay = 2          -- ms, default 20 - agresywniejszy
);
Autovacuum nie nadąża — kiedy i co zrobić Symptomy: rosnące n_dead_tup, bloat, slow queries. Przyczyny: zbyt częste UPDATE/DELETE (e.g. session table, queue), autovacuum throttling przez autovacuum_vacuum_cost_delay. Rozwiązania: zmniejsz autovacuum_vacuum_cost_delay per-table, zwiększ autovacuum_max_workers globalnie, rozważ partycjonowanie tabeli (mniejsze partycje → szybszy vacuum).

Partycjonowanie
Table Partitioning — kiedy, jak i jakie strategie
Partycjonowanie rozwiązuje problemy z bardzo dużymi tabelami: wolne queries (pruning), vacuum bloat, drop starych danych (DROP PARTITION = instant, DELETE = wolny). PostgreSQL ma natywny declarative partitioning od PG 10+. Używaj od ~100GB lub gdy vacuum nie nadąża.
🗂️ Strategie partycjonowania
RANGE
Najczęstszy. Czas (created_at miesięcznie/rocznie), sekwencyjne ID. Partition pruning działa dla zapytań z WHERE created_at BETWEEN .... Idealne dla time-series, logów, archiwów.
LIST
Dyskretne wartości — status ('active','inactive'), region ('EU','US','ASIA'). Partition pruning gdy WHERE region = 'EU'. Dobry dla tenant isolation.
HASH
Równomierny rozkład danych. PARTITION BY HASH(tenant_id) z N partycjami. Brak partition pruning dla zakresów — tylko dla =. Używaj gdy nie masz naturalnego klucza partycji.
Partition Pruning — weryfikacja Zawsze sprawdź w EXPLAIN że planner usuwa partycje. Szukaj Partitions removed: N. Warunek: kolumna partycji musi być w WHERE z literałem lub bind parameter — nie z funkcją (np. date_trunc('month', created_at) uniemożliwia pruning na partycji po created_at).
Declarative partitioning — przykład events table
-- Tabela partycjonowana po miesiącach
CREATE TABLE events (
  id          BIGINT GENERATED ALWAYS AS IDENTITY,
  event_type  TEXT NOT NULL,
  user_id     BIGINT NOT NULL,
  payload     JSONB,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);

-- Tworzenie partycji (ręcznie lub automatycznie przez pg_partman)
CREATE TABLE events_2025_01 PARTITION OF events
  FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE events_2025_02 PARTITION OF events
  FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

-- DEFAULT partition — złapie wiersze bez pasującej partycji
CREATE TABLE events_default PARTITION OF events DEFAULT;

-- Indeksy tworzy się na parent — propagują do partycji
CREATE INDEX ON events(user_id, created_at);
CREATE INDEX ON events USING GIN (payload);

-- Usuwanie starych danych — INSTANT! (vs DELETE = pełny scan)
DROP TABLE events_2023_01;  -- lub DETACH PARTITION

-- pg_partman — automatyczne tworzenie i usuwanie partycji
-- SELECT partman.create_parent(
--   'public.events', 'created_at',
--   'native', 'monthly',
--   premake := 3
-- );

-- Lista partycji i ich rozmiary
SELECT
  child.relname,
  pg_size_pretty(pg_relation_size(child.oid)) AS size,
  pg_get_expr(child.relpartbound, child.oid) AS bounds
FROM pg_inherits
JOIN pg_class child ON inhrelid = child.oid
WHERE inhparent = 'events'::regclass
ORDER BY child.relname;

Replikacja
Replikacja strumieniowa, logical replication, HA — architektura produkcyjna
PostgreSQL oferuje kilka mechanizmów replikacji. Senior musi rozumieć różnice, trade-offy i jak skonfigurować HA. Replikacja strumieniowa (WAL-based) to standard produkcyjny. Logical replication to narzędzie dla migracji i CDC.
📡 Typy replikacji — porównanie
TypMechanizmZastosowanieOgraniczenia
Streaming WAL stream, byte-for-byte HA, read replicas, failover Ta sama wersja PG, cała baza
Logical Decoded changes (DML) Migracje, CDC, subskrypcja wybranych tabel Nie replika DDL, wymaga PK lub replica identity
Synchronous Streaming + confirm before commit Zero data loss (RPO=0) Wyższe latency (czeka na standby)
WAL level
wal_level = replica dla streaming. wal_level = logical dla logical replication. Logical = więcej danych w WAL = większy I/O.
Replication slot
Uwaga! Slot zatrzymuje usuwanie WAL segments do czasu potwierdzenia przez replica. Jeśli replica zginie — WAL nakumuluje się bez limitu. Monitoruj pg_replication_slots. Ustaw max_slot_wal_keep_size.
Replication lag
Różnica między primary a standby. Monitoruj pg_stat_replication. Przy synchronous: synchronous_standby_names = 'FIRST 1 (standby1)'.
Konfiguracja replikacji i monitoring
-- postgresql.conf — primary
wal_level = replica
max_wal_senders = 10
wal_keep_size = '1GB'
synchronous_commit = on  -- local / remote_write / remote_apply / on
-- synchronous_standby_names = 'ANY 1 (standby1, standby2)'

-- pg_hba.conf — pozwól replication connections
-- host replication replicator 10.0.0.0/8 scram-sha-256

-- Monitoring replikacji (na primary)
SELECT
  application_name,
  state,
  sync_state,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  (sent_lsn - replay_lsn) AS replication_lag_bytes,
  write_lag, flush_lag, replay_lag
FROM pg_stat_replication;

-- Logical replication — publication / subscription
-- Na source (publisher):
CREATE PUBLICATION my_pub
  FOR TABLE orders, users
  WITH (publish = 'insert, update, delete');

-- Na target (subscriber):
CREATE SUBSCRIPTION my_sub
  CONNECTION 'host=source_host dbname=mydb user=replicator'
  PUBLICATION my_pub;

-- Monitor slotów — KRYTYCZNE
SELECT
  slot_name, plugin, active,
  pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS retained_wal
FROM pg_replication_slots;
Patroni / Patroni + etcd — HA standard Dla produkcyjnego HA używaj Patroni (Python, open source) — automatyczny failover, DCS (etcd/Consul/ZooKeeper) jako distributed lock. Patroni zarządza pg_hba.conf, replikacją i promowaniem standby. Alternatywy: repmgr (prostszy, mniej funkcji), Stolon, managed cloud (RDS, Cloud SQL, Azure Flexible Server).

Performance Tuning
Konfiguracja i tuning produkcyjny — kluczowe parametry
Domyślna konfiguracja PostgreSQL jest konserwatywna (działa na 256MB RAM). Produkcja wymaga tuningu. Reguła: zacznij od pgTune (pgtune.leopard.in.ua), zrozum każdy parametr zanim go zmienisz, monitoruj efekty.
⚙️ Kluczowe parametry konfiguracji
shared_buffers
25% RAM jako punkt startowy. Cache page'ów tabel i indeksów. Postgres też używa OS page cache — nie dawaj >40% RAM.
effective_cache_size
75% RAM. Tylko hint dla planera — ile RAM jest dostępne (shared_buffers + OS cache). Nie alokuje pamięci. Wyższe = planer preferuje indeksy.
work_mem
RAM per sort/hash operation. Ostrożnie — jeden query może mieć wiele operacji, jeden connection = wiele queries. Formuła: (RAM * 0.25) / (max_connections * 2). Ustaw niski globalnie, podnoś per-session.
maintenance_work_mem
RAM dla VACUUM, CREATE INDEX, ALTER TABLE. Ustaw wysoko (1GB+) — używane rzadko, duży impact na szybkość vacuum i budowania indeksów.
wal_buffers
-1 = 1/32 shared_buffers (auto). Rzadko wymaga zmiany. Przy >1000 connections — rozważ 64MB.
random_page_cost
Koszt random I/O. Domyślnie 4.0 (HDD). SSD: ustaw 1.1-1.5. Wpływa na decyzję planer: niższe = bardziej skłonny do Index Scan.
max_connections
Nie ustawiaj wysoko. 100-200 + PgBouncer. Każde połączenie = ~10MB RAM. Wysoki max_connections = mniejszy work_mem per query.
checkpoint_completion
_target
0.9 — rozkłada I/O checkpointu na 90% czasu między checkpointami. Zmniejsza spikes I/O.

Monitoring performance — top queries i zasoby
-- Wolne zapytania aktualnie wykonywane
SELECT pid, now() - pg_stat_activity.query_start AS duration,
       query, state
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > INTERVAL '5 seconds'
  AND state = 'active';

-- Cache hit ratio — powinno być > 99%
SELECT
  sum(blks_hit)::float / (sum(blks_hit) + sum(blks_read)) * 100 AS cache_hit_ratio
FROM pg_stat_database
WHERE datname = current_database();

-- Zabij długo-trwające zapytania
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE query_start < now() - INTERVAL '30 minutes'
  AND state = 'active'
  AND pid != pg_backend_pid();
Reguła nr 1: mierz przed optymalizacją Włącz pg_stat_statements. Zawsze. To jedno rozszerzenie daje więcej wartości niż cokolwiek innego — widać total_time, calls, mean_time, stddev per query pattern. Resetuj po deployach: SELECT pg_stat_statements_reset(). Monitoruj trendy, nie tylko snapshots.

Locking
Lock levels, deadlocks, bezpieczne migracje schema — produkcyjne wzorce
Locking w PostgreSQL to hierarchia. Najgroźniejszy: ACCESS EXCLUSIVE — blokuje wszystko (ALTER TABLE, DROP). Senior musi wiedzieć jak przeprowadzać migracje bez downtime i jak diagnozować deadlocki.

Deadlock
PG automatycznie wykrywa i rollback'uje jedną transakcję (ERROR: deadlock detected). Zapobieganie: zawsze lockuj w tej samej kolejności. Np. zawsze update user przed account, nigdy odwrotnie.
lock_timeout
Ustaw zawsze przy DDL na produkcji! SET lock_timeout = '2s' — jeśli nie dostaniesz locka w 2s, rzuć error zamiast czekać i blokować inne queries w kolejce za tobą.
Bezpieczne migracje schema — zero downtime
-- ❌ NIEBEZPIECZNE: blokuje całą tabelę na czas migracji
ALTER TABLE orders ADD COLUMN metadata JSONB DEFAULT '{}';
-- Problem: NOT NULL DEFAULT = table rewrite na PG < 11
-- PG 11+: ADD COLUMN z DEFAULT bez NOT NULL = metadata only, instant

-- ✅ BEZPIECZNE: ADD COLUMN nullable (instant)
ALTER TABLE orders ADD COLUMN metadata JSONB;
-- backfill w batchach:
UPDATE orders SET metadata = '{}'
WHERE id BETWEEN 1 AND 10000;  -- repeat in batches
ALTER TABLE orders ALTER COLUMN metadata SET DEFAULT '{}';
ALTER TABLE orders ALTER COLUMN metadata SET NOT NULL;

-- ✅ Bezpieczne dodawanie NOT NULL z CHECK CONSTRAINT (PG 12+)
ALTER TABLE orders
  ADD CONSTRAINT orders_meta_not_null
  CHECK (metadata IS NOT NULL) NOT VALID;
-- NOT VALID = nie sprawdza istniejących wierszy, instant lock
ALTER TABLE orders VALIDATE CONSTRAINT orders_meta_not_null;
-- VALIDATE = SHARE UPDATE EXCLUSIVE (nie blokuje DML!)

-- Monitoring locków i kto kogo blokuje
SELECT
  blocked.pid,
  blocked.query AS blocked_query,
  blocking.pid AS blocking_pid,
  blocking.query AS blocking_query
FROM pg_stat_activity AS blocked
JOIN pg_stat_activity AS blocking
  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;
