01 — wydajność
Indeksy i optymalizacja

Indeks to osobna struktura danych (B-Tree) która przyspiesza wyszukiwanie. Bez indeksu baza skanuje każdy wiersz — z indeksem skacze od razu do wyniku.

⚡ Jak działa indeks — B-Tree
Tabela BEZ indeksu
1
Anna
2
Kuba
3
Jan
4
Ola
5
Piotr
6
Marek
Szukasz "Marek"?
Sprawdzam 1,2,3,4,5,6... O(n)
Indeks B-Tree
Anna
→ #1
Jan
→ #3
Kuba
→ #2
Marek
→ #6
Ola
→ #4
Piotr
→ #5
Binary search!
Bezpośrednio do #6. O(log n)
⚠ Indeks ma koszt! Każdy INSERT/UPDATE/DELETE musi aktualizować indeks. Nie dodawaj indeksów wszędzie — tylko na kolumnach które często filtrujesz (WHERE, JOIN, ORDER BY).
rodzaje indeksów
-- Podstawowy indeks
CREATE INDEX idx_email
  ON uzytkownicy(email);

-- Unikalny (jak UNIQUE constraint)
CREATE UNIQUE INDEX idx_email_uq
  ON uzytkownicy(email);

-- Złożony — kolejność MA znaczenie!
-- Dobry dla: WHERE a=? AND b=?
-- Dobry dla: WHERE a=? (sam a)
-- Zły  dla: WHERE b=? (sam b)
CREATE INDEX idx_kraj_miasto
  ON adresy(kraj, miasto);

-- Indeks na wyrażeniu (MySQL 8+)
CREATE INDEX idx_lower_email
  ON uzytkownicy((LOWER(email)));

-- Usuń indeks
DROP INDEX idx_email
  ON uzytkownicy;

-- Pokaż indeksy tabeli
SHOW INDEX FROM uzytkownicy;
01b — diagnoza
EXPLAIN — analiza zapytań

EXPLAIN pokazuje jak baza wykona zapytanie — czy użyje indeksu, ile wierszy sprawdzi, jaką metodę join wybierze. Niezastąpione przy optymalizacji.

EXPLAIN
-- Dodaj EXPLAIN przed zapytaniem
EXPLAIN SELECT *
FROM zamowienia
WHERE uzytkownik_id = 5;

-- Wynik EXPLAIN (co sprawdzać):
-- type:  ALL = skan całej tabeli (źle!)
--        ref = używa indeksu (dobrze)
--        const = jeden wiersz (najlepiej)
-- rows:  ile wierszy sprawdzi
-- key:   który indeks użył (NULL = brak)
-- Extra: Using filesort = sortuje w pamięci

-- EXPLAIN ANALYZE — faktyczny czas
EXPLAIN ANALYZE SELECT *
FROM zamowienia z
JOIN uzytkownicy u
  ON z.uzytkownik_id = u.id
WHERE u.kraj = 'PL';
📊 Co oznaczają kolumny EXPLAIN
KolumnaCo oznacza
typeALL=pełny skan (źle), index=skan indeksu, range=zakres, ref=indeks, const=jeden wiersz (najlepiej)
keyKtóry indeks wybrała baza. NULL = żaden = brak indeksu!
rowsSzacowana liczba wierszy do sprawdzenia. Im mniej tym lepiej.
filtered% wierszy które przejdą filtr WHERE. 100% = żadnego filtrowania.
ExtraUsing index=świetnie, Using filesort=wolne, Using temporary=tabela tymczasowa (bardzo wolne)
02 — spójność danych
Transakcje i ACID

Transakcja to grupa operacji która albo wykona się w całości, albo wcale. Kluczowe przy przelewach bankowych, zamówieniach, wszędzie gdzie dane muszą być spójne.

A
Atomicity
Wszystko albo nic. Albo wszystkie operacje w transakcji się wykonają, albo żadna. Nie ma stanu pośredniego.
C
Consistency
Baza przechodzi z jednego spójnego stanu do drugiego. Wszystkie reguły (constraints) muszą być spełnione.
I
Isolation
Transakcje nie widzą siebie nawzajem w trakcie działania. Jedna transakcja nie wpływa na drugą.
D
Durability
Po COMMIT dane są trwałe. Nawet jeśli serwer padnie zaraz po COMMIT — dane przeżyją.
przelew bankowy — klasyczny przykład
-- Przelew 500 zł od Anny do Jana
START TRANSACTION;

-- Krok 1: Odejmij od Anny
UPDATE konta
SET saldo = saldo - 500
WHERE id = 1;

-- Krok 2: Dodaj Janowi
UPDATE konta
SET saldo = saldo + 500
WHERE id = 2;

-- Sprawdź czy wszystko ok
SELECT saldo FROM konta
WHERE id = 1;

-- Jeśli ok — zatwierdź!
COMMIT;

-- Jeśli coś poszło nie tak
-- ROLLBACK cofa WSZYSTKO
ROLLBACK;
SAVEPOINT — częściowy rollback
START TRANSACTION;

INSERT INTO zamowienia
  (user_id, kwota)
VALUES (1, 200);

-- Zapisz punkt pośredni
SAVEPOINT po_zamowieniu;

INSERT INTO platnosci
  (zamowienie_id, kwota)
VALUES (LAST_INSERT_ID(), 200);

-- Cofnij tylko od savepoint
ROLLBACK TO po_zamowieniu;

-- zamówienie zostało,
-- płatność cofnięta

COMMIT;
Poziomy izolacji transakcji READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE. Wyższy poziom = lepsza izolacja ale wolniej. MySQL domyślnie: REPEATABLE READ.
03 — abstrakcja
Widoki (Views)

Widok to zapisane zapytanie SELECT które zachowuje się jak tabela. Ukrywa złożoność, upraszcza dostęp do danych, pozwala kontrolować co widzi użytkownik.

tworzenie i używanie widoków
-- Stwórz widok
CREATE VIEW v_aktywni_uzytkownicy AS
SELECT
  id,
  imie,
  email,
  COUNT(z.id) AS ile_zamowien
FROM uzytkownicy u
LEFT JOIN zamowienia z
  ON u.id = z.uzytkownik_id
WHERE u.aktywny = 1
GROUP BY u.id;

-- Używasz jak zwykłą tabelę!
SELECT *
FROM v_aktywni_uzytkownicy
WHERE ile_zamowien > 3;

-- Aktualizuj widok
CREATE OR REPLACE VIEW v_aktywni_uzytkownicy
AS ...;

-- Usuń widok
DROP VIEW v_aktywni_uzytkownicy;
💡 Kiedy używać widoków
BezpieczeństwoPokaż userowi tylko widok bez dostępu do surowej tabeli (np. ukryj kolumnę haslo)
UproszczenieSkomplikowany JOIN z 5 tabel → widok → prosta nazwa
KompatybilnośćZmień strukturę tabeli bez zmiany kodu który z niej korzysta
RaportyGotowe widoki do raportowania zamiast pisania za każdym razem
Widok to nie cache! Widok wykonuje zapytanie za każdym razem gdy go odpytujesz. Jeśli potrzebujesz cache'owania — użyj Materialized View (PostgreSQL) lub tabeli pomocniczej.
04 — logika w bazie
Procedury składowane i triggery

Procedury to funkcje napisane w SQL przechowywane w bazie. Triggery to automatyczne akcje uruchamiane przy INSERT/UPDATE/DELETE.

procedura składowana
-- Tworzenie procedury
DELIMITER //

CREATE PROCEDURE dodaj_zamowienie(
  IN  p_user_id  INT,
  IN  p_kwota    DECIMAL(10,2),
  OUT p_order_id INT
)
BEGIN
  -- Walidacja
  IF p_kwota <= 0 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Kwota musi być > 0';
  END IF;

  -- Wstaw zamówienie
  INSERT INTO zamowienia
    (uzytkownik_id, kwota, data)
  VALUES
    (p_user_id, p_kwota, NOW());

  SET p_order_id = LAST_INSERT_ID();
END //

DELIMITER ;

-- Wywołanie procedury
CALL dodaj_zamowienie(5, 299.99, @order_id);
SELECT @order_id;
trigger — automatyczna akcja
-- BEFORE INSERT: waliduj dane
DELIMITER //
CREATE TRIGGER before_user_insert
  BEFORE INSERT ON uzytkownicy
  FOR EACH ROW
BEGIN
  -- Zamień email na lowercase
  SET NEW.email = LOWER(NEW.email);
END //
DELIMITER ;

-- AFTER DELETE: archiwizuj
DELIMITER //
CREATE TRIGGER after_user_delete
  AFTER DELETE ON uzytkownicy
  FOR EACH ROW
BEGIN
  INSERT INTO archiwum_uzytkownikow
    (user_id, email, deleted_at)
  VALUES
    (OLD.id, OLD.email, NOW());
-- OLD = stare wartości (przed zmianą)
-- NEW = nowe wartości (po zmianie)
END //
DELIMITER ;
05 — analityka
Window Functions

Wykonują obliczenia na zestawie wierszy powiązanych z bieżącym wierszem — bez zwijania wyników jak GROUP BY. Każdy wiersz zostaje, a obok pojawia się wynik obliczenia.

Składnia: funkcja() OVER (PARTITION BY ... ORDER BY ...) PARTITION BY = grupuj jak GROUP BY ale nie zwijaj. ORDER BY = kolejność w oknie. Bez PARTITION = jedno okno = wszystkie wiersze.
ROW_NUMBER, RANK, DENSE_RANK
-- ROW_NUMBER: unikalny numer wiersza
SELECT
  imie,
  dzial,
  wynagrodzenie,
  ROW_NUMBER() OVER (
    PARTITION BY dzial
    ORDER BY wynagrodzenie DESC
  ) AS miejsce_w_dziale
FROM pracownicy;
-- Wynik: każdy pracownik + jego
-- miejsce w rankingu swojego działu

-- RANK vs DENSE_RANK
-- wynagrodzenia: 5000, 5000, 3000
-- RANK:       1, 1, 3  (pomija 2)
-- DENSE_RANK: 1, 1, 2  (bez przerwy)

-- TOP 3 zarobki w każdym dziale
SELECT * FROM (
  SELECT *,
    RANK() OVER (
      PARTITION BY dzial
      ORDER BY wynagrodzenie DESC
    ) AS rk
  FROM pracownicy
) t
WHERE rk <= 3;
LAG, LEAD, SUM kumulacyjne
-- LAG: poprzedni wiersz
-- LEAD: następny wiersz
SELECT
  miesiac,
  sprzedaz,
  LAG(sprzedaz) OVER (
    ORDER BY miesiac
  ) AS poprzedni_miesiac,
  sprzedaz - LAG(sprzedaz) OVER (
    ORDER BY miesiac
  ) AS zmiana
FROM sprzedaz_miesieczna;

-- SUM kumulacyjny (running total)
SELECT
  data,
  kwota,
  SUM(kwota) OVER (
    ORDER BY data
  ) AS suma_narastajaco
FROM zamowienia;

-- AVG krocząca (ostatnie 3 miesiące)
AVG(sprzedaz) OVER (
  ORDER BY miesiac
  ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
) AS srednia_3m
06 — czytelność
CTE — Common Table Expressions

CTE to tymczasowe nazwane zapytanie zdefiniowane na początku głównego zapytania klauzulą WITH. Rozbija skomplikowane zapytanie na czytelne kroki. Może być też rekurencyjne.

CTE podstawowe i łańcuchowe
-- Jedno CTE
WITH top_klienci AS (
  SELECT
    uzytkownik_id,
    SUM(kwota) AS suma
  FROM zamowienia
  GROUP BY uzytkownik_id
  HAVING SUM(kwota) > 5000
)
SELECT u.imie, tk.suma
FROM top_klienci tk
JOIN uzytkownicy u
  ON tk.uzytkownik_id = u.id;

-- Wiele CTE po sobie
WITH
zamowienia_2024 AS (
  SELECT * FROM zamowienia
  WHERE YEAR(data) = 2024
),
sumy_per_user AS (
  SELECT uzytkownik_id, SUM(kwota) s
  FROM zamowienia_2024
  GROUP BY uzytkownik_id
)
SELECT * FROM sumy_per_user
ORDER BY s DESC;
CTE rekurencyjne — drzewo kategorii
-- Tabela: kategorie(id, nazwa, parent_id)
-- Chcemy całe drzewo z poziomami

WITH RECURSIVE drzewo AS (

  -- Część bazowa: korzeń (bez rodzica)
  SELECT
    id, nazwa,
    0 AS poziom,
    nazwa AS sciezka
  FROM kategorie
  WHERE parent_id IS NULL

  UNION ALL

  -- Część rekurencyjna: dzieci
  SELECT
    k.id, k.nazwa,
    d.poziom + 1,
    CONCAT(d.sciezka, ' > ', k.nazwa)
  FROM kategorie k
  JOIN drzewo d
    ON k.parent_id = d.id
)
SELECT * FROM drzewo
ORDER BY sciezka;
-- Wynik: Elektronika > Telefony > iPhone
07 — zagnieżdżanie
Subqueries — podzapytania

Zapytanie SQL wewnątrz innego zapytania. Może być w SELECT, FROM, WHERE lub HAVING. Często można zastąpić JOIN-em lub CTE — co jest czytelniejsze zależy od kontekstu.

subquery w WHERE, SELECT, FROM
-- W WHERE: skalarne (jeden wynik)
SELECT * FROM produkty
WHERE cena > (
  SELECT AVG(cena)
  FROM produkty
);  -- droższe od średniej

-- W WHERE z IN
SELECT imie FROM uzytkownicy
WHERE id IN (
  SELECT DISTINCT uzytkownik_id
  FROM zamowienia
  WHERE kwota > 1000
);  -- userzy z zamówieniem > 1000

-- W SELECT: dla każdego wiersza
SELECT
  imie,
  (
    SELECT COUNT(*)
    FROM zamowienia z
    WHERE z.uzytkownik_id = u.id
  ) AS ile_zamowien
FROM uzytkownicy u;
EXISTS, NOT EXISTS, FROM subquery
-- EXISTS: sprawdź czy istnieje
-- Często szybszy niż IN!
SELECT imie FROM uzytkownicy u
WHERE EXISTS (
  SELECT 1
  FROM zamowienia z
  WHERE z.uzytkownik_id = u.id
  AND z.kwota > 500
);

-- NOT EXISTS: brak dopasowania
SELECT imie FROM uzytkownicy u
WHERE NOT EXISTS (
  SELECT 1 FROM zamowienia z
  WHERE z.uzytkownik_id = u.id
); -- userzy bez zamówień

-- W FROM: inline view
SELECT sub.dzial, AVG(sub.wyplata)
FROM (
  SELECT dzial, SUM(wynagrodzenie) wyplata
  FROM pracownicy
  GROUP BY dzial, MONTH(data)
) sub
GROUP BY sub.dzial;
Subquery vs JOIN vs CTE — kiedy co? JOIN jest zazwyczaj najszybszy bo optymalizator ma więcej możliwości. Subquery w WHERE z IN jest wolniejszy przy dużych zbiorach — zamień na JOIN lub EXISTS. CTE używaj gdy chcesz podzielić złożone zapytanie na czytelne kroki — czytelność > mikro-optymalizacja.
