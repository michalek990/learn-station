01 — złożoność obliczeniowa
Notacja Big O

Big O opisuje jak rośnie czas wykonania gdy rozmiar danych n rośnie. Nie mówi ile sekund — mówi jak szybko rośnie. Dla n=1000 różnica między O(n) a O(n²) to 1000 vs 1 000 000 operacji.

// O(n²) — pętla w pętli
for (int i = 0; i < n; i++) {

    }
}

// O(2ⁿ) — naiwna rekurencja

    if (n <= 1) return n;

upraszczanie Big O
// ZASADA 1: stałe mnożniki ignoruj
for (int i = 0; i < n; i++) {   // 3n
    x = a[i];
    y = b[i];
    z = c[i];
}
// O(3n) = O(n)

// ZASADA 2: bierz największy składnik
for (i = 0; i < n; i++) { ... }     // n
for (i = 0; i < n; i++) {
    for (j = 0; j < n; j++) { ... } // n²
}
// O(n + n²) = O(n²)

// ZASADA 3: pętle PO SOBIE, nie zagnieżdżone
for (i = 0; i < n; i++) { ... }  // n
for (j = 0; j < n; j++) { ... }  // n
// O(n + n) = O(2n) = O(n) ← NIE n²!

// RÓŻNE zmienne = RÓŻNE litery!
for (i = 0; i < a.length; i++) {
    for (j = 0; j < b.length; j++) { }
}
// O(a * b) — NIE O(n²) jeśli a ≠ b!
Najważniejsza zasada praktyczna: Szukasz czy kod jest akceptowalny dla produkcji? O(1), O(log n), O(n), O(n log n) — tak. O(n²) — tylko dla małych n (<1000). O(2ⁿ) i O(n!) — nigdy na produkcji bez memoizacji.

03 — jak czytać kod
8 zasad rozpoznawania Big O

Patrzysz na kod i w 10 sekund wiesz jaka złożoność. Zapamiętaj te 8 reguł.

1
Jedna pętla przez n → O(n)
Iterujesz raz przez wszystkie elementy. Prosto.
for (i=0; i<n; i++) → O(n)
2
Dwie pętle zagnieżdżone → O(n²)
Każdy element z każdym. Bubble sort, brute force.
for(i) { for(j) { } } → O(n²)
3
Trzy pętle zagnieżdżone → O(n³)
Każda w każdej. Bardzo rzadko uzasadnione na produkcji.
for(i) { for(j) { for(k) { } } } → O(n³)
4
Dzielisz na pół w każdym kroku → O(log n)
Binary search, BST wyszukiwanie. Za każdym krokiem odrzucasz połowę.
mid = (l+r)/2; → O(log n)
5
Pętla + dzielenie na pół → O(n log n)
Przetwarzasz wszystko (n) ale na każdym poziomie podziału (log n). Merge sort, heap sort.
Arrays.sort() → O(n log n)
6
Dwa rekurencyjne wywołania → O(2ⁿ)
Drzewo wywołań podwaja się z każdym poziomem rekurencji. Naiwna Fibonacci.
return f(n-1) + f(n-2) → O(2ⁿ)
7
Stałe mnożniki ignoruj → 3n = O(n)
O(5n) = O(n). Big O pyta o "jak rośnie", nie "ile dokładnie".
5n + 100 = O(n)
8
Bierz największy składnik → n²+n = O(n²)
Przy dużym n mniejsze składniki nie mają znaczenia. n² dominuje nad n, n nad log n.
n³ + n² + n + 1 = O(n³)
04 — praktyka
Optymalizacja — zanim zaczniesz

Premature optimization is the root of all evil — Donald Knuth. Najpierw profiluj, potem optymalizuj tam gdzie boli.

HashMap — zamień O(n²) na O(n)
// ❌ O(n²) — szukanie w pętli
for (String a : listaA) {
  for (String b : listaB) {   // zagnieżdżona!
    if (a.equals(b)) { ... }
  }
}
// 1000 × 1000 = 1 000 000 porównań

// ✅ O(n) — używaj HashSet/HashMap
Set<String> zbiorB = new HashSet<>(listaB);
for (String a : listaA) {
  if (zbiorB.contains(a)) { ... } // O(1)!
}
// 1000 + 1000 = 2000 operacji
memoizacja — O(2ⁿ) na O(n)
// ❌ O(2ⁿ) — liczy to samo wielokrotnie
int fib(int n) {
  if (n <= 1) return n;
  return fib(n-1) + fib(n-2);
}

// ✅ O(n) — memoizacja
Map<Integer, Long> cache = new HashMap<>();
long fib(int n) {
  if (n <= 1) return n;
  if (cache.containsKey(n)) return cache.get(n);
  long wynik = fib(n-1) + fib(n-2);
  cache.put(n, wynik);
  return wynik;
}
// Każda wartość liczona tylko raz!
wyjdź wcześniej — early return
// ❌ Sprawdza wszystko nawet po znalezieniu
boolean found = false;
for (User u : users) {
  if (u.getAge() > 18) {
    found = true;    // ale pętla idzie dalej!
  }
}

// ✅ break / early return
for (User u : users) {
  if (u.getAge() > 18) return true; // stop!
}
return false;

// Lub Stream z findFirst()
boolean found = users.stream()
  .anyMatch(u -> u.getAge() > 18);

05 — architektura
Zasady SOLID

5 zasad dobrego projektowania obiektowego. Kod który je stosuje jest łatwiejszy do testowania, rozszerzania i zrozumienia.

S
Single Responsibility
Klasa ma jeden powód do zmiany. Nie mieszaj: logika biznesowa ≠ zapis do bazy ≠ wysyłka emaila.
O
Open/Closed
Otwarte na rozszerzenie, zamknięte na modyfikację. Dodajesz nowe zachowanie bez zmiany istniejącego kodu.
L
Liskov Substitution
Podklasa musi działać wszędzie gdzie działa klasa bazowa. Nie psuj kontraktu przy dziedziczeniu.
I
Interface Segregation
Wiele małych interfejsów > jeden duży. Klient nie powinien zależeć od metod których nie używa.
D
Dependency Inversion
Zależność od abstrakcji, nie implementacji. Wstrzykuj zależności (DI) zamiast tworzyć je w środku.
SRP — jeden powód do zmiany
// ❌ Naruszenie SRP — klasa robi za dużo
class UserService {
  saveUser(user)   { /* SQL INSERT */ }
  sendEmail(user)  { /* SMTP send */ }
  generatePDF(user){ /* PDF library */ }
  validateAge(user){ /* business rule */ }
}

// ✅ SRP — każda klasa robi jedno
class UserRepository { save(user)    { ... } }
class EmailService    { send(user)    { ... } }
class PdfGenerator    { generate(user){ ... } }
class UserValidator   { validate(user){ ... } }
DIP — dependency injection
// ❌ Naruszenie DIP — tworzy zależność sam
class OrderService {
  private EmailService email = new EmailService();
  // przywiązany do konkretnej implementacji!
}

// ✅ DIP — wstrzyknięta abstrakcja
interface Notifier {
  void notify(String msg);
}

class OrderService {
  private final Notifier notifier;

  OrderService(Notifier notifier) {
    this.notifier = notifier; // wstrzykiwanie!
  }
}
// Możesz wstrzyknąć Email, SMS, Slack...
06 — filozofia kodowania
DRY · KISS · YAGNI

Trzy zasady które każdy programista powinien mieć wytatuowane na wewnętrznej stronie powiek.

DRY
DRY
Don't Repeat Yourself
Każda logika powinna istnieć w jednym miejscu. Jak widzisz ten sam kod w dwóch miejscach — wyciągnij do funkcji/metody. Jak zmienisz logikę zmieniasz w jednym miejscu, nie szukasz wszystkich kopii.
❌ kopiuj-wklej × 5 miejsc
✅ jedna metoda, 5 wywołań
KISS
KISS
Keep It Simple, Stupid
Najprostsze rozwiązanie które działa jest najlepsze. Nie twórz abstrakcji na zapas. Nie używaj wzorca projektowego jeśli if wystarczy. Kod czytany jest 10× częściej niż pisany — czytający podziękuje.
❌ 3 warstwy abstrakcji dla prostej logiki
✅ jeden if, który każdy rozumie
YAGNI
YAGNI
You Aren't Gonna Need It
Nie koduj funkcji "na przyszłość" której nikt nie zamówił. Kod nieużywany to dług techniczny — trzeba go utrzymywać, testować i rozumieć. Implementuj gdy potrzebujesz, nie gdy "może kiedyś".
❌ "przyda się kiedyś" → kod leży 2 lata
✅ implementujesz gdy naprawdę potrzeba
07 — czytelność
Nazewnictwo — kod który sam się tłumaczy

Dobra nazwa jest lepsza niż komentarz. Komentarz może się zdezaktualizować — nazwa jest zawsze przy kodzie.

złe vs dobre nazwy
// ❌ Co to znaczy?
int d;
int[] lst;
void proc(int x, int y) { ... }
boolean flag = true;
if (u.getT() > 30) { ... }

// ✅ Kod który sam się tłumaczy
int daysSinceCreation;
int[] activeUserIds;
void calculateDiscount(int price, int percentage) { ... }
boolean isEmailVerified = true;
if (user.getAgeInYears() > 30) { ... }

// Booleany — zawsze z is/has/can/should
boolean isActive, hasPermission, canEdit, shouldSend;

// Kolekcje — liczba mnoga
List<User> users; // nie: user, userList, userArr

// Metody — czasownik + rzeczownik
getUserById(), sendConfirmationEmail()
calculateTotalPrice(), validateInput()

08 — design funkcji
Jak pisać dobre funkcje

Funkcja to podstawowy budulec kodu. Kilka zasad które oddzielają dobry kod od złego.

zasady pisania dobrych funkcji
// ❌ Robi za dużo — brak SRP
void processOrder(Order order) {
  // walidacja
  if (order.items.isEmpty()) throw ...;
  if (order.price < 0) throw ...;
  // zapis do bazy
  db.save(order);
  // wysyłka emaila
  emailClient.send(order.user.email, ...);
  // logowanie
  logger.info("Order processed: " + order.id);
  // aktualizacja stock
  inventory.decrease(order.items);
}

// ✅ Każda funkcja robi jedno
void processOrder(Order order) {
  validateOrder(order);          // jasne!
  saveOrder(order);
  notifyCustomer(order);
  updateInventory(order);
}

// ✅ Early return zamiast if zagnieżdżone
void validateOrder(Order order) {
  if (order == null)        throw new NullPointerException();
  if (order.items.isEmpty()) throw new ValidationException(...);
  if (order.price < 0)      throw new ValidationException(...);
  // happy path na końcu, bez zagnieżdżeń
}

unikaj null — Optional
// ❌ Zwracasz null → NPE w przyszłości
User findUser(int id) {
  return db.findById(id); // może null!
}
User u = findUser(5);
u.getEmail(); // NullPointerException!

// ✅ Optional — jawna informacja o braku
Optional<User> findUser(int id) {
  return Optional.ofNullable(db.findById(id));
}
findUser(5)
  .map(User::getEmail)
  .orElse("brak emaila"); // bezpieczne
