// Przykład: auto-rejestracja adnotacji
@Component
public class AuditBeanPostProcessor
        implements BeanPostProcessor {

    @Override
    public Object postProcessAfterInitialization(
            Object bean, String name) {

        // Wywołany po @PostConstruct, przed oddaniem beana
        if (bean.getClass().isAnnotationPresent(Auditable.class)) {
            auditRegistry.register(name, bean);
        }
        return bean; // możesz zwrócić proxy!
    }
}

public class MyBFPP implements BeanFactoryPostProcessor {

    public void postProcessBeanFactory(ConfigurableListableBeanFactory bf) {
        BeanDefinition def = bf.getBeanDefinition("myBean");
        def.setScope(BeanDefinition.SCOPE_PROTOTYPE);
    }
}

🔄 Circular Dependencies

Spring 6 domyślnie zabrania circular dependencies. Jak je eliminować:

1. Refaktoryzacja — najlepsza. Wydziel wspólną logikę do nowego beana (SRP)
2. @Lazy — opóźniona inicjalizacja jednej zależności. Tymczasowe rozwiązanie
3. Setter injection — Spring może rozwiązać przez 3-phase init
4. ApplicationEventPublisher — komunikacja przez eventy zamiast bezpośredniej zależności

Transakcje — głęboka analiza
@Transactional — wszystko co musisz wiedzieć
Transakcje to jeden z najczęstszych obszarów błędów senior developerów. Propagacja, izolacja, rollback rules, self-invocation problem — każdy z nich może zepsuć spójność danych.
📋 Propagacja transakcji
PropagacjaZachowanieKiedy używać
REQUIRED Używa istniejącej lub tworzy nową. Domyślna Operacje wymagające transakcji — standardowy wybór
REQUIRES_NEW Zawsze nowa transakcja, zawiesza poprzednią Audyt, logi — muszą być zapisane nawet przy rollbacku głównej
NESTED Savepoint wewnątrz istniejącej transakcji Partial rollback — cofnij tylko część operacji
SUPPORTS Używa jeśli istnieje, działa bez jeśli nie Operacje read-only, które mogą działać poza transakcją
NOT_SUPPORTED Zawiesza aktywną transakcję, działa poza Operacje które psują transakcje (np. DDL)
MANDATORY Wymaga aktywnej transakcji, rzuca wyjątek jeśli brak Metody które zawsze muszą być wywoływane w transakcji
NEVER Rzuca wyjątek jeśli transakcja istnieje Metody niezgodne z transakcjami
Poziomy izolacji + problemy
// Problem 1: Dirty Read — odczyt niezatwierdzonego
// Problem 2: Non-repeatable Read — 2x odczyt, różny wynik
// Problem 3: Phantom Read — nowe wiersze między odczytami

// READ_UNCOMMITTED — wszystkie problemy możliwe (nie używaj)
@Transactional(isolation = Isolation.READ_UNCOMMITTED)

// READ_COMMITTED — eliminuje Dirty Read (domyślny w Postgres)
@Transactional(isolation = Isolation.READ_COMMITTED)

// REPEATABLE_READ — eliminuje + Non-repeatable Read
@Transactional(isolation = Isolation.REPEATABLE_READ)

// SERIALIZABLE — eliminuje wszystko. Najwolniejszy
@Transactional(isolation = Isolation.SERIALIZABLE)

// Optymistyczne locking — zamiast SERIALIZABLE
@Version
private Long version;  // JPA sprawdzi przy UPDATE
⚠️ Self-Invocation Problem @Transactional nie działa przy wywołaniu wewnątrz tej samej klasy! Spring AOP używa proxy — wywołanie this.method() omija proxy i tym samym transakcję. Rozwiązania: wydziel metodę do innego beana, użyj @Inject samego siebie (anty-pattern!), lub użyj AspectJ compile-time weaving.

Java 21 · Project Loom
Virtual Threads w Spring Boot
Virtual Threads (Project Loom) to rewolucja w modelu współbieżności Javy. Spring Boot 3.2+ obsługuje je out-of-the-box — jeden parametr konfiguracji zastępuje skomplikowany kod reaktywny dla wielu przypadków użycia.
💡 Kiedy Virtual Threads vs WebFlux Virtual Threads (Loom): Idealny gdy masz istniejący kod blokujący i chcesz skalowalności bez przepisywania. Kod wygląda jak tradycyjny synchroniczny — łatwiejszy do debugowania i zrozumienia.
WebFlux/Reactive: Lepszy gdy budujesz od zera i cały stack jest reaktywny. Daje fine-grained control nad backpressure i pipeline'ami danych. Wygrywa przy złożonej kompozycji asynchronicznej.
Włączanie Virtual Threads
# application.yml — jednolinijkowa zmiana!
spring:
  threads:
    virtual:
      enabled: true

// Spring Boot 3.2+ automatycznie konfiguruje:
// - Tomcat: używa VirtualThreads dla requestów
// - @Async: executor oparty na VT
// - @Scheduled: scheduler oparty na VT

// Manualna konfiguracja (jeśli potrzebujesz):
@Bean
public Executor virtualThreadExecutor() {
    return Executors.newVirtualThreadPerTaskExecutor();
}

// Tworzenie ręczne:
Thread.ofVirtual().start(() -> {
    Thread.sleep(1000);  // blokuje VT, nie OS thread!
    processRequest();
});
Pułapki Virtual Threads
// ❌ PUŁAPKA 1: Pinning — synchronized blokuje carrier thread!
synchronized (this) {
    jdbcTemplate.query("...");  // pinned! Zablokuje OS thread
}
// ✅ Zamień na ReentrantLock:
private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try { jdbcTemplate.query("..."); } finally { lock.unlock(); }

// Użyj ScopedValue (Java 21 preview):

Domain-Driven Design
DDD w Spring — praktyczna implementacja
DDD nie jest o adnotacjach — to o organizacji kodu wokół domeny biznesowej. Aggregate, ValueObject, DomainEvent, Repository — każdy z tych wzorców ma konkretne implikacje na strukturę Spring aplikacji.
Aggregate + Domain Events
// Aggregate Root — jedyna klasa z @Entity i publicznymi metodami
@Entity
public class Order extends AbstractAggregateRoot<Order> {

    @Id @GeneratedValue
    private Long id;

    @Embedded
    private Money totalAmount;    // Value Object

    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderLine> lines = new ArrayList<>();

    private OrderStatus status = OrderStatus.PENDING;

    // Logika biznesowa TYLKO tu — nie w Service!
    public void confirm() {
        if (status != OrderStatus.PENDING)
            throw new IllegalStateException("Cannot confirm non-pending order");
        this.status = OrderStatus.CONFIRMED;
        // Rejestruje event — Spring Data publikuje przy save()
        registerEvent(new OrderConfirmedEvent(this));
    }
}

// Value Object — immutable, równość przez wartość
@Embeddable
public record Money(BigDecimal amount, String currency) {
    public Money add(Money other) {
        if (!currency.equals(other.currency))
            throw new IllegalArgumentException("Currency mismatch");
        return new Money(amount.add(other.amount), currency);
    }
}

CQRS · Event Sourcing
CQRS i Event Sourcing w Spring
CQRS (Command Query Responsibility Segregation) rozdziela modele odczytu i zapisu. Event Sourcing przechowuje historię zmian zamiast aktualnego stanu. Razem rozwiązują problemy skalowalności i audytu w złożonych domenach.
CQRS — rozdzielenie odczytu/zapisu
// ─── COMMAND SIDE ────────────────────────────────────────
// Command — intencja zmiany stanu
public record PlaceOrderCommand(String userId, List<Item> items) {}

@Service
public class OrderCommandService {
    @Transactional
    public UUID handle(PlaceOrderCommand cmd) {
        Order order = Order.create(cmd.userId(), cmd.items());
        orderRepo.save(order);  // Write DB (PostgreSQL)
        return order.getId();
    }
}

// ─── QUERY SIDE ──────────────────────────────────────────
// Read Model — zdenormalizowany, zoptymalizowany pod odczyty
@Document  // MongoDB lub ElasticSearch
public class OrderSummaryView {
    private String orderId;
    private String customerName;  // zdenormalizowane!
    private BigDecimal total;
    private String status;
}

@Service
public class OrderQueryService {
    public List<OrderSummaryView> findByCustomer(String userId) {
        return orderViewRepo.findByUserId(userId);  // Read DB

Event Sourcing — uproszczony wzorzec
// Store events, not state
@Entity
public class EventStore {
    @Id private UUID eventId;
    private String aggregateId;
    private String aggregateType;
    private String eventType;
    @Column(columnDefinition = "jsonb")
    private String payload;       // JSON eventu
    private Long sequenceNumber;  // wersja agregatu
    private Instant occurredAt;
}

// Odtwarzanie stanu z eventów:
public Order load(String orderId) {
    List<DomainEvent> events = eventStore
        .findByAggregateId(orderId);   // historyczne eventy
    Order order = new Order();
    events.forEach(order::apply);      // replay!
    return order;
}

// Snapshot — co N eventów zapisz aktualny stan
// żeby nie replayować od początku historii
if (events.size() > 100) snapshotStore.save(order);
✅ Kiedy CQRS

✓ Różne wymagania wydajności odczytu vs zapisu

✓ Złożone raporty wymagające denormalizacji

✓ Różne modele danych dla różnych klientów (mobile vs web)

✗ Prosta CRUD aplikacja — niepotrzebna złożoność

✗ Mały zespół bez doświadczenia w eventech

⚡ Eventual Consistency

Po zapisie na write side, read model jest eventually consistent — aktualizuje się asynchronicznie przez eventy.

Musisz projektować UI który akceptuje opóźnienie (optimistic UI, polling, WebSocket).

Dla operacji wymagających silnej spójności — nie używaj CQRS lub czytaj z write side.

Outbox Pattern — gwarancja dostarczenia
// Problem: save() + kafka.send() — co jeśli Kafka padnie?
// Outbox Pattern: zapisz event do DB w tej samej transakcji

@Entity
public class OutboxEvent {
    @Id private UUID id;
    private String aggregateType;
    private String eventType;
    private String payload;         // JSON
    private Instant createdAt;
    private Boolean published = false;
}

@Transactional
public void placeOrder(OrderRequest req) {
    Order order = orderRepo.save(new Order(req));
    // Atomowo — ten sam commit!
    outboxRepo.save(new OutboxEvent("OrderPlaced",
        toJson(order)));
}

// Relay — osobny wątek/serwis publikuje do Kafka
@Scheduled(fixedDelay = 1000)
public void publishOutbox() {
    outboxRepo.findByPublishedFalse(PageRequest.of(0, 100))
        .forEach(event -> {
            kafkaTemplate.send(event.getEventType(), event.getPayload());
            event.setPublished(true);
        });
}

// Orchestration Saga — centralny koordynator
@Component
public class PlaceOrderSaga {

    public void execute(PlaceOrderCommand cmd) {
        UUID orderId = orderService.create(cmd);
        try {
            inventoryService.reserve(orderId, cmd.items());
            paymentService.charge(orderId, cmd.amount());
            orderService.confirm(orderId);
        } catch (Exception e) {
            // Kompensacja — cofnij co się udało
            compensate(orderId, e);
        }
    }
}

🗄️ Indeksy — strategia

Composite index: kolejność kolumn ma znaczenie. WHERE a=? AND b=? → index(a,b). Query na samo b nie użyje indeksu!

Partial index: WHERE status='ACTIVE' w definicji — mały i szybki

Covering index: INCLUDE dodatkowych kolumn — query tylko z indeksu bez heap scan

Spring Boot Auto-configuration to magia która działa przez @Conditional. Senior musi rozumieć jak pisać własną auto-konfigurację, jak debugować dlaczego bean NIE jest tworzony i jak zarządzać konfiguracją w różnych środowiskach.
Własna Auto-konfiguracja
// 1. Klasa konfiguracyjna z warunkami
@AutoConfiguration
@ConditionalOnClass(RedissonClient.class)
@ConditionalOnMissingBean(RedissonClient.class)
@EnableConfigurationProperties(RedissonProperties.class)
public class RedissonAutoConfiguration {

    @Bean
    @ConditionalOnProperty(name = "redisson.enabled", havingValue = "true")
    public RedissonClient redissonClient(RedissonProperties props) {
        Config config = new Config();
        config.useSingleServer().setAddress(props.getAddress());
        return Redisson.create(config);
    }
}

Strategy Pattern — dynamiczny wybór
// Interfejs strategii
public interface PaymentStrategy {
    String supports();
    PaymentResult pay(PaymentRequest req);
}

// Implementacje jako Spring Beany
@Service class CardPayment implements PaymentStrategy {
    public String supports() { return "CARD"; }
}
@Service class BlikPayment implements PaymentStrategy {
    public String supports() { return "BLIK"; }
}

// Auto-kolekcja wszystkich strategii
@Service
public class PaymentRouter {
    private final Map<String, PaymentStrategy> strategies;

    public PaymentRouter(List<PaymentStrategy> strategies) {
        // Spring wstrzykuje WSZYSTKIE implementacje
        this.strategies = strategies.stream()
            .collect(toMap(PaymentStrategy::supports, identity()));
    }

    public PaymentResult route(PaymentRequest req) {
        return strategies.get(req.getMethod()).pay(req);
    }
}

Pułapki i anty-wzorce
Co często psuje produkcyjne aplikacje
Lista najczęstszych błędów których unikanie odróżnia seniora od juniora. Każdy z nich był prawdziwym incydentem produkcyjnym.
💥 Najgroźniejsze pułapki
Lazy init prod
spring.main.lazy-initialization=true w produkcji → pierwszy request obsługuje inicjalizację → timeouty. Używaj tylko w dev do szybszego startu.
Singleton mutable state
Pole instancji w @Service/@Controller jest dzielone między requestami. Race condition, wycieki danych między sesjami użytkowników.
OpenSessionInView
spring.jpa.open-in-view=true (domyślnie!) trzyma sesję Hibernate przez cały HTTP request → lazy loading w widoku → N+1. Wyłącz!
@Async bez executor
@Async bez własnego ThreadPoolTaskExecutor używa SimpleAsyncTaskExecutor — tworzy nowy wątek per wywołanie, bez limitu. Może zabić serwer.
Tx boundary błąd
@Transactional na prywatnej metodzie lub przy wywołaniu this.method() — proxy nie działa, brak transakcji, brak rollbacku. Cicho gubisz dane.
Cascade ALL wszędzie
CascadeType.ALL + orphanRemoval na relacji @ManyToMany → DELETE rodzica usuwa wszystkie dzieci, nawet te powiązane z innymi obiektami.
fetchType EAGER
@ManyToOne EAGER (domyślnie!) ładuje powiązany obiekt zawsze — nawet gdy go nie potrzebujesz. Użyj LAZY wszędzie i fetch join gdy potrzeba.
@Async — poprawna konfiguracja
// ❌ SimpleAsyncTaskExecutor — unbounded threads
@Async
public void processEmail(Email e) { ... }

// ✅ Bounded pool z queue i rejection policy
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
        exec.setCorePoolSize(4);
        exec.setMaxPoolSize(8);
        exec.setQueueCapacity(100);
        exec.setRejectedExecutionHandler(new CallerRunsPolicy());
        exec.setThreadNamePrefix("async-email-");
        exec.initialize();
        return exec;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, args) ->
            log.error("Async error in {}: {}", method.getName(), ex.getMessage());
    }
}
🚨 Memory Leaks — co sprawdzać Static mutable collections: cache bez eviction, listeners bez deregistracji. ThreadLocal bez remove(): szczególnie w thread poolach — dane z poprzedniego requestu wyciekają. Hibernate session cache: długa transakcja processująca tysiące obiektów → EntityManager rośnie w pamięci → call em.flush(); em.clear(); co N iteracji.

JVM internals · GC Tuning
JVM, GC i optymalizacja pamięci
Senior musi rozumieć model pamięci JVM, różnice między GC algorytmami i jak czytać heap dumps. Bez tej wiedzy nie diagnozujesz OOM, nie rozumiesz latency spikes i nie konfiguracji produkcyjnej JVM.
🧠 Model pamięci JVM
HEAP
├── Young Generation
│ ├── Eden Space     ← nowe obiekty
│ ├── Survivor S0    ← po Minor GC
│ └── Survivor S1    ← po Minor GC
└── Old Generation     ← długożyjące obiekty

NON-HEAP (Metaspace)   ← klasy, metody
Stack                    ← per thread, call frames
Direct Memory         ← NIO, off-heap
Minor GC
Czyści Young Generation. Szybki (<50ms). Częsty.
Major/Full GC
Czyści Old Gen. Wolny, STW (Stop-The-World). Symptom problemu.
⚙️ Wybór GC — kiedy co
G1GC (domyślny)
Dobry balans throughput/latency. Heap 4GB–32GB. Domyślny od Java 9. Większość aplikacji.
ZGC
Ultra-niskie pauzy (<1ms). Heap do terabajtów. Najlepszy dla aplikacji latency-sensitive (API, trading).
Shenandoah
Podobny do ZGC. Lepszy na mniejszych heapach. RedHat/OpenJDK.
SerialGC
Microservices z małą pamięcią (<256MB). Spring Boot w kontenerach.
JVM Flags — produkcja
# Kontener Docker — fundamentalne
-XX:+UseContainerSupport              # czytaj limity cgroups (Java 10+)
-XX:MaxRAMPercentage=75.0             # 75% RAM kontenera na heap
-XX:InitialRAMPercentage=50.0         # zacznij od 50% — szybszy start

# GC — wybierz jedno
-XX:+UseZGC                           # niskie latency
-XX:+UseG1GC                          # dobry balans (domyślny)

# Logi GC — do analizy
-Xlog:gc*:file=/logs/gc.log:time,uptime:filecount=5,filesize=20m

# Heap dump przy OOM (K8s: plik pojawi się na PVC)
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/dumps/heap.hprof

# Exit przy OOM zamiast degradacji
-XX:+ExitOnOutOfMemoryError

# Metaspace — ogranicz jeśli ładujesz dużo klas dynamicznie
-XX:MaxMetaspaceSize=256m

# JFR — low-overhead profiling (zawsze włącz w prod)
-XX:StartFlightRecording=duration=0,filename=/jfr/recording.jfr,maxsize=100m

Senior Checklist
Code Review — o co pytać na każdym PR
Lista pytań które senior powinien zadać przy review każdego nowego kodu Spring — od architektury po bezpieczeństwo i wydajność. Jeśli odpowiedź na większość to "tak" — kod jest produkcyjny.
🏗️ Architektura

□ Czy logika biznesowa jest w domenie, nie w kontrolerze lub repozytorium?

□ Czy serwisy mają jedną odpowiedzialność (SRP)?

□ Czy zależności idą w jednym kierunku (UI → Application → Domain → Infrastructure)?

□ Czy entity nie wychodzą poza warstwę serwisową (używamy DTO)?

□ Czy nowe feature można przetestować bez uruchamiania całego Spring kontekstu?

□ Czy moduły mogą być wymienione bez zmian w reszcie kodu?

⚡ Performance

□ Czy nie ma N+1 (sprawdź z show-sql=true)?

□ Czy używamy projekcji zamiast pełnych entity tam gdzie można?

□ Czy paginacja jest cursor-based dla dużych zbiorów?

□ Czy zapytania w pętlach nie robią dodatkowych zapytań do DB?

□ Czy metody @Transactional są możliwie krótkie?

□ Czy cache ma TTL i eviction policy?

□ Czy @Async ma bounded thread pool?

🔐 Bezpieczeństwo

□ Czy brak sekretów w kodzie i w konfiguracji w repozytorium?

□ Czy validation jest na DTO wejściowym (@Valid)?

□ Czy autoryzacja jest sprawdzana na poziomie metody (@PreAuthorize)?

□ Czy SQL jest paramteryzowany (brak string concatenation)?

□ Czy zewnętrzne ID są walidowane pod kątem dostępu (IDOR)?

□ Czy wrażliwe dane nie trafiają do logów?

🧪 Testy

□ Czy logika biznesowa jest pokryta unit testami (bez Spring)?

□ Czy jest test integracyjny z prawdziwą bazą (Testcontainers)?

□ Czy edge cases są przetestowane (null, puste kolekcje, graniczne wartości)?

□ Czy testy są deterministyczne (nie zależą od kolejności, czasu, losowości)?

□ Czy zewnętrzne serwisy są mockowane (WireMock)?

□ Czy testy mają czytelne nazwy opisujące scenariusz?

📊 Observability

□ Czy są strukturowane logi (JSON) z correlation ID?

□ Czy krytyczne operacje mają metryki (counter, timer)?

□ Czy błędy są logowane z pełnym kontekstem (user id, request id)?

□ Czy health endpoint pokazuje status zależności?

□ Czy tracing jest włączony i działa przez serwisy?

□ Czy są alerty na kluczowe metryki (error rate, latency p99)?

🚀 Produkcja

□ Czy graceful shutdown jest skonfigurowany?

□ Czy timeout na wszystkich zewnętrznych wywołaniach?

□ Czy circuit breaker chroni zewnętrzne zależności?

□ Czy migracje DB są backwards-compatible (nie blokujące)?

□ Czy feature flagi pozwalają rollback bez deploymentu?

□ Czy jest runbook co zrobić gdy serwis padnie?

Każda decyzja architektoniczna to trade-off — senior umie go nazwać i obronić.
